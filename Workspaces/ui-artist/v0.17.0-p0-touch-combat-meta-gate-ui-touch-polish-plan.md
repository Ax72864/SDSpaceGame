## 1. 已读现状与 v0.15 残留

制作人方向要求 v0.17.0 回答“当前纵向切片是否具备进入发布准备的最后一组真实体验证据”，其中触控 / 小屏必须诚实区分真实触控、DevTools / 鼠标替代和未覆盖。PM 看板将 UI 小屏 / 触控 polish 定为 #3 / #8：先定方案，再由真实 P0、自然交火、复盘、Meta 回流证据决定是否修。

v0.15 三视口证据的 UI 残留：

- 小屏叠层：release candidate panel、build panel、status panel、more status、结算 / 复盘路径可能同时占据关键视口区域。
- 深滚动：`更多状态` 展开后 Meta / players 入口在状态面板深处，420x740 和 390x844 需要明确记录滚动可达性。
- 复盘 / 结算按钮中心 hit：强制结算面板与建造 / 状态面板共存时，`开始新局` 中心 hit 被遮挡，不能写作通过。
- build panel 与 release candidate panel 共存：playtest 模式下 `release-candidate-panel` 固定在小屏顶部，`build-panel` 固定在底部，两者都可滚动，但会压缩画布和状态面板可操作空间。
- 真实触控未覆盖：`maxTouchPoints=0`、`pointerCoarse=false`，鼠标 / CDP 只证明布局、DOM 可见性和鼠标语义，不证明真实手指触控、惯性滚动、移动浏览器 UI chrome、软键盘或误触。

## 2. 当前结构观察

当前 UI 已有一些适合低风险 polish 的结构，不需要重构：

- `build-panel` 位于左下，桌面 `max-height: min(68vh, calc(100vh - 112px))`，小屏降为 `max-height: min(34vh, 250px)` 并允许滚动。
- `status-panel` 位于右侧，420px 以下宽度和高度收紧；`moreStatusPanel` 在 761px 以上默认展开，小屏默认折叠。
- `more-status-body` 承载 `combatReviewSummary`、`combatStatusSummary`、选中设施、控制按钮、Meta 入口和 players 入口，是小屏深滚动的主要风险点。
- `runSettlementPanel` 在状态面板内显示，移动端按钮区已使用 sticky，但 `max-height: min(34vh, 300px)` 仍可能导致中心 hit 与其他固定层冲突。
- `combatReviewSummary` 当前在更多状态内，以 2-4 条短句展示战斗复盘；小屏只做字号 / 行高压缩。
- `metaPanel`、`galaxyMapPanel`、`runSettlementPanel` 已进入主流程面板互斥同步，后续不应改成新的面板系统。
- release candidate panel 在 `?playtest=1` 或 `?release-candidate` 下动态插入，固定层级 `z-index: 60`，小屏位于顶部，最大高度 42vh / 38vh，带关闭按钮和滚动。
- release candidate toggle 也是固定层级 `z-index: 60`，小屏右下角出现，可能与底部 build panel 或结算按钮产生视觉 / 命中区竞争。

## 3. 触控假设与真实触控边界

证据分层必须这样写：

- 真实触控通过：来自真实触控设备或真实触控浏览器环境，记录设备、浏览器、viewport、DPR、`maxTouchPoints`、pointer media、操作步骤和人工结论。
- 触控替代证据：DevTools 移动视口、鼠标点击、CDP `Runtime.evaluate`、DOM hit test 或截图，只能证明布局、可见性、滚动容器状态、鼠标点击路径和部分 pointer 事件，不证明真实手指。
- 未覆盖：没有真机、没有真实触控浏览器、没有人工滑动 / 点击路径时，必须明确写“真实触控未覆盖”。

触控边界检查：

- 按钮尺寸：P0 按钮建议有效 hit 区不小于 40px，关键闭环按钮优先 44px；若视觉按钮较小，可通过 padding / min-height 扩大命中区。
- 滚动：验证手指滑动是否落在正确滚动容器，是否被 canvas、固定面板、遮罩、sticky footer 或 `pointer-events` 拦截。
- 误触：检查关闭、刷新、复制、开始新局、留在星系、Meta 入口是否太近，是否容易因滚动结束触发误点。
- 关闭路径：所有覆盖式或固定式面板必须有可见关闭 / 返回 / 继续路径；结算面板如 Esc 不关闭，应记录为设计事实并验证按钮路径。
- 中心 hit：不能只看按钮可见，要检查按钮中心或主命中区是否被其他层覆盖；必要时记录 `elementFromPoint` 与人工点击结果。

## 4. 低风险 polish 准入与禁止项

允许的低风险 polish 只限以下类型，并且必须来自证据路径：

- CSS `max-height` / `min-height`：调整面板高度，避免固定层互相吞掉关键按钮。
- `z-index`：只修复明确可复现的错误遮挡，不建立新的层级体系。
- `pointer-events`：只在装饰层、遮罩或非交互覆盖物阻挡点击时修正；不得让隐藏面板穿透触发游戏操作。
- 滚动区域：补充 `overflow-y: auto`、`overscroll-behavior: contain`、`-webkit-overflow-scrolling: touch` 或 sticky 区域边距。
- 按钮 hit area：提高 `min-height`、padding、gap，优先作用于结算、复盘、Meta、关闭、复制摘要和建造首个按钮。
- 文本折行：使用 `overflow-wrap: anywhere`、合理行高、降低长 pre / 长中文说明的横向溢出。
- 小屏媒体查询：只针对 `760x640`、`420x740`、`390x844` 和邻近断点补齐现有结构。

禁止项：

- 禁止 UI 信息架构重做、全新移动端布局、拆分或重建 HUD、引入新弹窗系统。
- 禁止修改玩法流程、战斗 / 维修 / 护盾 / 伤害 / 敌人 AI / RNG / 结算规则。
- 禁止修改 `SAVE_KEY`、meta schema、storage 迁移、局外成长结构。
- 禁止新增按钮来绕过原路径，禁止用 hook 结果替代真实点击回流。
- 禁止为了取证隐藏 release candidate panel 的核心内容；只能优化共存和关闭路径。

## 5. 三视口目标

`760x640`：

- 目标：桌面窄高视口仍能同时取证 release candidate panel、build panel、status panel 基础状态。
- 风险：`moreStatusPanel` 默认展开，Meta / players 入口落到视口下方；release candidate panel 与状态面板中心 hit 竞争。
- 接受：关键 P0 操作能通过滚动或关闭取证面板继续，不要求所有信息一屏全显。

`420x740`：

- 目标：接近窄屏手机宽度，验证 build panel、release candidate panel、结算 / 复盘、Meta / Galaxy 入口可达。
- 风险：中心 hit、深滚动、sticky 结算按钮、release candidate panel 顶部占位。
- 接受：关键按钮可点击，面板可滚动可关闭；真实触控若未覆盖必须标注。

`390x844`：

- 目标：窄屏高机型，验证更多状态折叠策略、底部 build panel 与顶部取证面板共存、Meta / Galaxy 全屏面板滚动。
- 风险：按钮中心被固定层或 canvas 命中、底部 toggle 与 build panel 竞争、长文本溢出。
- 接受：P0 路径入口可达；复盘 / 结算 / Meta 回流不能只靠 hook 证明。

## 6. 优先级与处理规则

- S0：游戏无法启动、首屏不可见、P0 完全无法进入。UI polish 不能单独消化，应同步 PM / 主程。
- S1：关键路径不可点击或无法关闭，导致 P0、交火复盘、Meta 回流无法完成。必须修复或标记闸门不通过。
- S2：核心信息被遮挡、按钮 hit 明显不足、触控滚动阻塞、结算 / 复盘入口不可点。允许纳入 v0.17 最小 polish。
- S3：非阻塞舒适度、轻微拥挤、需多滚动但路径可达。默认记录为残留，制作人决定是否本轮追。
- Polish：视觉密度、文案折行、按钮间距等不影响 P0 的小修。只有在成本极低且不影响证据路径时才做。

## 7. 推荐最小修复候选

这些不是立即修改项，只是后续 #8 发现同类问题时的低风险候选：

- release candidate panel 与 build panel 共存时，优先检查小屏 `max-height` 是否留出中部操作区；必要时把取证面板高度再压低或要求 PM 关闭后继续 P0。
- 结算面板按钮中心 hit 被遮挡时，优先调整 `runSettlementPanel` 的 `max-height`、sticky actions padding / bottom 安全区和按钮 `min-height`，不要改结算流程。
- `moreStatusPanel` 深滚动导致 Meta 入口难找时，优先记录为证据风险；若纳入修复，只允许提高 summary hit、优化滚动容器高度或保持关键入口折行可读，不新增入口。
- `combatReviewSummary` 被挤在更多状态深处时，优先检查交火后玩家是否能看到复盘摘要；若必须 polish，只允许增加可读性、间距或滚动可达，不创建新复盘弹窗。
- `summaryRestartBtn`、`metaPanelToggle`、`galaxy-map-cancel-btn`、release candidate 关闭 / 复制按钮如低于触控 hit 标准，优先调 `min-height` / padding / gap。
- 如果遮罩或固定层抢点击，先用 `elementFromPoint`、真实点击和截图定位具体层，再做最小 `z-index` / `pointer-events` 修正。

方案验收：

- 已覆盖 v0.15 残留：小屏叠层、深滚动、复盘 / 结算按钮中心 hit、build panel 与 release candidate panel 共存。
- 已明确真实触控边界：鼠标 / CDP / DevTools 不等于真实触控通过。

- 已覆盖 `760x640`、`420x740`、`390x844`。
- UI / 触控检查点不少于 40 条，且可直接给 #8 / #9 使用。

后续实现验收：

- 每个 UI polish 都来自 PM 证据路径的明确现象，而不是凭感觉预改。
- 只修改必要 CSS / hit area / 滚动 / 层级 / 折行，不触碰玩法、数值、存档、Meta schema 或发布流程。

## 10. 回滚标准

任一情况触发回滚或暂停：

- polish 导致游戏启动失败、runtime error、新的 S0 / S1 阻塞或关键按钮不可点。
- polish 使 release candidate panel、复制摘要、runtime errors 或 storage signature 取证能力回潮。
