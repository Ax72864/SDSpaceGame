## 1. 玩家当前困惑

v0.20.0 已经把问题从“只知道还差距离”推进到“团队知道候选路径和失败原因”，但玩家侧仍容易卡在一句话上：看到“还差 X”，却不知道自己是否能通过扩建解决，也不知道下一步该扩到哪、当前资源够不够、还是应该驾驶工站靠近资源后重建采矿站。

当前玩家困惑可以拆成 6 个问题：

1. `还差 X` 是距离不足、采矿站位置错、工站整体太远，还是没有可达路径。
2. 现在能不能靠铺框架进入资源外环，还是最近资源外环在当前结构外不可达。
3. 如果能扩，先点哪几个格子，最后在哪一格建 / 升级采矿站。
4. 如果不能扩，是被金属成本、占用格、未连接核心、缺电、手动关闭，还是资源已采空阻塞。
5. 采矿成功后，收益来自哪座矿站 / 哪个资源点，而不是自然资源增长或其他奖励。
6. 小屏上提示、建造面板、资源外环和地图点击是否同时可读可点。

## 2. 当前 UI 基础

现有结构适合做最小增强，不需要先重构：

- `resourceGuide` 已在右侧状态区用绿色面板展示附近资源、资源外环距离、采矿站状态和放置预览。
- `statusAlerts` 已用低高度警告行展示采矿待命、断电、手动关闭等状态。
- 建造面板设施卡已有角色、用途、成本、电力、状态、推荐 / 警告 tip，并在小屏下滚动压缩。
- 采矿放置预览已能显示 `覆盖良好`、`距资源外环还差 X`、`当前位置采不到资源`。
- 只读诊断 helper 已能产出 `candidates`、`bestCandidate`、`suggestedExpansionPath`、`nearestResource`、`harvestingCount`、`allOutOfRange`、`failure.summary` 等信息。

UI 风险也很明确：把诊断信息全部搬进 HUD 会过载；把候选格做成复杂地图编辑器会扩大 1.0 风险；只保留“还差 X”又无法让玩家行动。

## 3. 信息层级

采矿覆盖 UI 必须按玩家行动顺序组织，而不是按诊断字段堆叠。

| 优先级 | 玩家要回答的问题 | 建议承载位置 | 展示密度 |
| --- | --- | --- | --- |
| P0 | 采矿是否正在产出 | `resourceGuide` 顶部状态 chip + `statusAlerts` 低频提示 | 1 行 |
| P0 | 当前失败原因是什么 | `resourceGuide` 主句 / `statusAlerts` warn | 1 行 |
| P0 | 下一步能不能扩到外环 | `resourceGuide` 建议句 | 1 行 |
| P0 | 如果能扩，先扩几步、目标格在哪 | `resourceGuide` + 可选画布轻高亮 | 1-2 行 |
| P0 | 如果不能扩，为什么不可达 | `resourceGuide` 不可达句 | 1 行 |
| P1 | 具体候选格列表 | 只读诊断面板 / 测试证据 | 默认不常驻 |
| P1 | 收益来自哪座矿站和资源点 | 成功采集时的短行 / chip | 1 行 |
| P2 | 完整路径可视化、逐格箭头、成本模拟 | 后续复杂 UI | 1.0 不优先 |

## 4. UI 方案

### 4.1 候选格提示

目标是让玩家知道“有可用入圈格”而不是阅读 debug 列表。

推荐表达：

```text
覆盖候选：可在 (-4,2) 入圈，先铺 3 步框架。
```

状态规则：

| 状态 | 文案示例 | 视觉 |
| --- | --- | --- |
| 有可达候选格 | `覆盖候选：可在 (-4,2) 入圈，先铺 3 步框架。` | 青绿色 chip / 行首圆点 |
| 候选格已是框架 | `覆盖候选：升级 (-4,2) 为采矿站即可入圈。` | 青绿色 |
| 候选格被占用 | `入圈格被设施占用，需换路径或重新选址。` | 橙色 |
| 无候选格 | `当前结构附近没有入圈格，需移动工站或重新选址。` | 橙红 |

画布层建议只做轻量：

- 最多高亮 1 个 `bestCandidate`，不一次性铺满 12 个候选格。
- 高亮形态为资源色描边 + 细虚线，不遮挡格子点击。
- 没有高亮实现时，文案仍必须能独立解释下一步。

### 4.2 建议扩建路径

建议路径要从“团队诊断字段”翻译成玩家动作。

推荐表达：

```text
下一步：从当前结构向资源外环铺 3 个框架，再在目标格建采矿站。
```

若技术侧能稳定提供逐格路径，小屏和桌面分层显示：

| 视口 | 显示方式 |
| --- | --- |
| 桌面 | `路径：(-2,1) -> (-3,1) -> (-4,2)，目标建采矿站` |
| 小屏 | `路径：先铺 3 格框架 -> 建采矿站` |
| 只读诊断 | 保留完整 `suggestedExpansionPath.frameSteps` 和 `targetGridKey` |

关键限制：

- 不能把理论路径写成“必定可建”，除非实现验证当前资源可支付、位置合法、路径未被占用。
- 如果金属成本不足，优先提示 `路径可达，但当前金属不足，先等核心产金属 / 采集其他资源`。
- 如果路径需要驾驶工站靠近，文案要从“扩建”切换为“移动 / 重新选址”，避免玩家继续点不可达格。

### 4.3 不可达提示

不可达提示要比“还差 X”多给一个决策方向。

| 原因 | 文案示例 | 下一步导向 |
| --- | --- | --- |
| 没有采矿站 | `尚未建造采矿站：先在连接结构上建 1 座。` | 建造采矿站 |
| 全部手动关闭 | `采矿站已关闭：选中采矿站后开启。` | 开关设施 |
| 电力不足 | `采矿站缺电停机：先补发电或关闭低优先级设施。` | 补电 |
| 全部出圈 | `采矿站未覆盖资源外环：需要扩到外环内或移动工站。` | 扩建 / 移动 |
| 有入圈格但路径被阻塞 | `入圈格存在，但扩建路径被占用或不可达。` | 换路径 / 重新选址 |
| 当前结构附近无入圈候选 | `当前结构附近没有入圈格；最近资源还差 X，建议移动工站靠近资源。` | 移动 / 重建 |
| 资源采空 | `最近资源点已采空：前往其他带彩色外环的天体。` | 换资源 |

不可达提示不应每帧刷红。优先放在 `resourceGuide`，只有阻断 P0 或玩家刚尝试失败时才进入 `statusAlerts`。

### 4.4 矿站覆盖状态 chip

在 `resourceGuide` 顶部增加或替换为 1 个采矿状态 chip，帮助玩家先判断“现在到底有没有收益”。

状态 chip 建议：

| 状态 | chip 文案 | 颜色 |
| --- | --- | --- |
| 采集中 | `采矿中 x1` | 绿色 / 呼吸点 |
| 部分采集中 | `部分采矿 x1/3` | 黄绿 |
| 未覆盖 | `未覆盖外环` | 橙色 |
| 缺电 | `采矿缺电` | 橙红 |
| 已关闭 | `采矿关闭` | 灰橙 |
| 未建造 | `未建采矿站` | 灰蓝 |

chip 后跟一条主句即可：

```text
[未覆盖外环] 当前矿站不会产出；可先铺 3 步框架到 (-4,2)。
```

### 4.5 收益归因显示

采矿成功时必须让玩家知道收益来自采矿站，而不是自然增长。

推荐最小表达：

```text
采矿中 x1：采集 矿石外环 · Asteroid A，本轮 +12 矿石。
```

若当前没有“本轮收益”聚合，1.0 可先做更保守的归因：

```text
采矿中 x1：矿站正在采集 矿石外环 · Asteroid A。
```

验收时不能只看 chip 变绿。PM 仍需记录 `harvestingCount > 0`，并用目标资源下降或收益来源字段证明归因成立。UI 只负责把成立的归因读给玩家。

### 4.6 低噪音原则

采矿 UI 的噪音控制比信息量更重要。

- 默认只显示 1 个采矿主状态、1 个最近资源行、1 个下一步建议。
- `statusAlerts` 只显示阻断或玩家刚触发的失败，不重复 `resourceGuide` 的完整说明。
- 候选格最多展示最佳 1 个；完整候选列表保留给只读诊断或测试面板。
- 成功采矿后，失败引导自动降级，不继续显示“还差 X”。
- 普通距离变化不触发 toast；只有建造失败、采矿站全停、路径从可达到不可达时才升级提示。
- 文案避免开发字段：不直接显示 `harvestingCount`、`allOutOfRange`、`suggestedExpansionPath`，除非在只读诊断面板。

### 4.7 小屏注意点

现有小屏下 `build-panel`、`resource-guide`、`status-alerts` 都会压缩字号和面板高度，采矿引导必须更短。

小屏规则：

- chip + 主句优先，候选格坐标和路径细节可折叠或压缩。
- 目标按钮和地图格不能被资源引导遮住；提示面板不得覆盖建造面板的首个可点按钮。
- `resourceGuide` 单次最多 4 行：状态、最近资源、下一步、颜色图例；图例可在已成功采矿后隐藏。
- 建造采矿站时，放置预览行必须可见：`距外环还差 X / 覆盖良好`。
- 真实触控证据必须记录是否能看见资源外环、能否点到候选格、滚动是否误触底部建造按钮。

## 5. 分层实施建议

### A. 仅文案

适用：不改结构、不接入新诊断字段，只优化现有表达。

- 把 `还需靠近 X` 补成 `当前矿站不会产出；需扩到外环内或移动工站`。
- 把 `请进入彩色外环` 补充为 `扩建框架到外环内 / 驾驶工站靠近资源`。
- 成功采矿时强调 `正在采集 资源类型 · 天体名`。

收益：成本最低，能减少误解。限制：仍不能告诉具体候选格和路径。

### B. 状态提示

适用：接入已有 `getMiningStationStatus()` 和放置预览，不展示完整只读诊断。

- 增加矿站覆盖状态 chip。
- 在 `statusAlerts` 中只保留阻断采矿状态。
- 建造面板采矿卡在无矿站或全未覆盖时给一条推荐 / 警告。

收益：1.0 最小闭环主力。限制：不能完整解释路径可达性。

### C. 只读诊断可视化

适用：把 v0.20.0 helper 中的 `bestCandidate` / `suggestedExpansionPath` 以玩家可读方式展示，但不改规则。

- `resourceGuide` 只展示最佳候选：目标格、框架步数、最终动作。
- 画布可选高亮最佳候选格和路径。
- 测试 / PM 证据保留完整候选列表与失败原因。

收益：直接解决“能不能 / 怎么扩”的主要困惑。风险：必须避免把不可支付理论路径包装成可执行路径。

### D. 复杂 UI 重构

适用：后续版本，而非 1.0 优先。

- 建造规划模式、路径 ghost、逐格成本模拟、候选列表面板、资源外环筛选、自动跳转到资源点。
- 全新的移动端建造 / 采矿布局。

收益：长期体验更强。风险：实现和测试面大，可能引入输入、遮挡、状态同步和错误承诺问题，不建议作为 1.0 采矿阻塞的第一解。

## 6. 推荐 1.0 优先 UI 最小方案

1.0 优先采用 `A + B + C 的最小子集`：

1. 保留现有 `resourceGuide` / `statusAlerts` / 建造面板结构，不重构 HUD。
2. 在 `resourceGuide` 顶部展示矿站覆盖状态 chip：采矿中、未覆盖、缺电、已关闭、未建造。
3. 未采矿时展示 1 条失败原因 + 1 条下一步：能扩则说明铺几步框架到目标格；不能扩则说明移动工站 / 重新选址 / 补电 / 开启设施。
4. 仅展示 `bestCandidate` / `suggestedExpansionPath`，不展示全部候选列表。
5. 建造采矿站预览继续展示 `覆盖良好 / 还差 X`，并在不入圈时补一句 `当前位置采不到资源`。
6. 成功采矿时展示收益归因：至少 `采矿中 xN：资源类型 · 天体名`；若收益累计字段可靠，再显示本轮收益。
7. 小屏下压缩为 chip + 下一步短句，图例和详细路径降级。

这套方案不改变规则，却能回答玩家最关键的三个问题：现在有没有产出、为什么没有、下一步能不能扩到资源外环。

## 7. 验收标准

方案验收：

- 已说明玩家看到“还差 X”但不知能否 / 如何扩到资源外环的困惑。
- 已覆盖候选格提示、建议扩建路径、不可达提示、矿站覆盖状态 chip、收益归因显示、低噪音原则和小屏注意点。
- 已区分仅文案、状态提示、只读诊断可视化和复杂 UI 重构。
- 已给出 1.0 UI 最小方案，且没有要求修改采矿规则、数值、存档或发布流程。

后续实现验收：

- 从空档进入，玩家在无外部说明下能理解采矿站是否产出。
- 未产出时，玩家能看到具体失败原因，而不是只看到“还差 X”。
- 若存在可达候选，UI 至少提示目标格和框架步数；若不可达，UI 说明移动工站 / 重新选址 / 补电 / 开启设施等方向。
- 成功采矿时，UI 显示采集资源类型和天体名；PM 证据记录 `harvestingCount > 0` 或等价收益，并证明不是自然增长误判。
- `statusAlerts` 不重复刷屏，成功采矿后失败提示自动降级。
- 桌面真实鼠标路径可读可点；小屏触控或等价触控证据记录资源外环可见性、候选格可点性、建造面板遮挡和滚动误触。
- 无阻断 runtime error、unhandled rejection 或输入锁死。
- 不提交 dirty `42.md`，不创建 tag，不写入 `Releases/`。

## 8. 残留风险

- UI 只能解释路径，不能替代高风险产品决策。如果规则上真实可支付路径不存在，文案不能把失败包装成可完成。
- 若只读诊断未计算成本可支付性，建议路径必须写成“可达候选 / 建议方向”，不能写成“必定可建”。
- 如果收益归因字段不足，UI 可以先显示正在采集的资源和天体，但 1.0 gate 仍需要 PM / 技术证据证明资源变化来源。
- 小屏真实触控仍需人工证据；鼠标、CDP、hook 只能作为替代证据。
