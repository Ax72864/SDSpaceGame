## 0. 文档定位

本文件**只定数值**：

- `META_TALENT_TREE` 12 个节点的字段填表。
- 旧字段映射到新 rank 的对照与不退点策略。
- 点数产出公式与单局期望。
- `effectKey` 命名表与具体 `perRank / value`。
- `START_PROTOCOLS` 4 个协议的初始资源 / 加成 / 限制与文案。
- 平衡边界、上限、风险与 QA 回归清单（≥25 条）。

**不在本文档决策范围**：

- schema 字段名、helper 函数签名、JSON 持久化策略 → ADR §3-§5。
- 面板布局、卡片颜色、文案口吻、动效 → UI 方案。
- 是否做重置 → ADR §7.3（v0.10.0 暂不做）。

数值方案产出后，**ADR 中的 `META_TALENT_TREE = [...]` / `START_PROTOCOLS = [...]` 示例必须替换为本文件下面的填表**。

## 1. 路线分布总览

12 个节点按 ADR §4.1 category 分四条主线 + 解锁/探索类共 4 大类：

| 路线 | 节点数 | id 列表 | 设计意图 |
| --- | --- | --- | --- |
| `resource`（资源型） | 3 | `miningYield`、`metalRefining`、`startingCache` | 滚资源 / 折扣 / 起步资源垫付 |
| `defense`（防御型） | 3 | `hullIntegrity`、`coreFortitude`、`buildDiscount` | 框架耐久 / 核心耐久 / 建造折扣 |
| `weapon`（武器型） | 3 | `weaponCalibration`、`weaponFrame`、`weaponEfficiency` | 武器输出 / 武器框架解锁 / 推力 |
| `exploration`（探索/解锁型） | 3 | `researchInsight`、`efficientCore`、`galaxyForesight` | 科研收益 / 高效核心解锁 / 星系信息提示 |

`tier` 分布：

- tier 1：5 个（每路线至少 1 个低门槛节点，让 1-2 局即可摸到第一次购买）。
- tier 2：4 个（解锁开局协议门槛 + 中段成长，构成"短期目标"层）。
- tier 3：2 个（解锁类强收益，价格陡升）。
- tier 4：1 个（探索型解锁 `galaxyForesight`，作为长期目标）。

总最大投入（满点）：**41 点**（见 §4 成本总表），保证一个有耐心的玩家在 v0.10.0 生命周期内可完整刷满，但不会在 3-5 局内一次性吃完所有收益。

### 4.2 单局点数产出公式

沿用现行代码（不改 `grantObjectiveReward / endRunSettlement`）：

- 单关基础点：`base = 2 + level`；level 0-5。
- 关卡奖励倍率：`rewardMultiplier`（已存在，由 `resolveObjectiveRewardMultiplier` 决定，目前在 1.0-1.5 之间）。
- 单关结算点：`gained = floor((2 + level) * rewardMultiplier)`。
- 终局 bonus：`floor(base * 1.5) - base`，约 +50%。
- 终局自由活动点：`endgameActivityPoints`（由 enemy 击杀触发，单局上限约 5 点）。

**典型单局期望（推荐数值方案前 5 局）**：

| 局情况 | 完成关卡数 | 累计 base | 终局 bonus | 自由活动 | 合计 |
| --- | --- | --- | --- | --- | --- |
| 失败（卡在 level 1） | 2 | 5 | 0 | 0 | **5** |
| 失败（卡在 level 3） | 4 | 14 | 0 | 0 | **14** |
| 失败（卡在 level 5 终局前） | 5 | 20 | 0 | 0 | **20** |
| 通关（全 6 关） | 6 | 27 | 13 | 3 | **43** |
| 通关高倍率（rewardMultiplier 全 1.3） | 6 | 35 | 17 | 5 | **57** |

### 4.3 与制作人约束对齐：单局可购买 1-3 个早期节点

制作人方向要求"单局 15-30 分钟后可购买 1-3 个早期节点，避免雪球"。

- tier 1 节点首次购买成本 1 点。失败 1 局（5-14 点）可购买 5-14 次 tier 1 首级。**但 tier 1 节点首级 cost=1 且只有 4 个**（miningYield / hullIntegrity / weaponCalibration / researchInsight），故"早期可买 1-3 个节点"的体验自然成立：
  - 失败局玩家：完成 1-2 关获得 5-9 点，能买 1-3 个 tier 1 首级。
  - 通关局玩家：43-57 点，能买完 4 个 tier 1 首级 + 2-3 个 tier 1 二级。
- 高额节点（buildDiscount tier 3 = 7 点、galaxyForesight tier 4 = 4 点）单局无法触及，构成跨局长期目标。
- 整套 50 点全满需要约 1-2 次通关 + 多次失败累积，符合"v0.10.0 生命周期内可刷满但不一次吃完"。

### 4.4 雪球抑制

- tier 1 cost 数组 `[1, 1, 2]` 而非 `[1, 1, 1]`：第三级成本翻倍。
- tier 2 cost 数组 `[2, 3]` 或 `[2, 2]`：首级 2 点，明显高于 tier 1。
- tier 3 cost 起 3 点，tier 4 cost 4 点。
- 不允许"单 tier 1 类节点暴富"——玩家想要 4 个 tier 1 满级（12 点）必须放弃 tier 2-4 的关键解锁，存在天然取舍。
- 协议解锁要求 tier 2 节点 rank>=1（`startingCache`、`coreFortitude`、`weaponCalibration rank=2`），形成"先投资再解锁"的引导路径。

## 5. 旧字段映射与不退点策略

### 5.1 映射对照表（与 ADR §3.4 一致并扩展）

| 旧字段 | 新 talent id | rank 计算 | 备注 |
| --- | --- | --- | --- |
| `mining` (int, 旧无上限) | `miningYield` | `clamp(mining, 0, 3)` | 旧每级 12%，新每级 10%；旧最高观测 = 用户实际购买次数 |
| `hull` (int) | `hullIntegrity` | `clamp(hull, 0, 3)` | 同口径 |
| `weapons` (int) | `weaponCalibration` | `clamp(weapons, 0, 3)` | **不附送** `weaponEfficiency`，详见 §5.3 |
| `unlocks.weaponFrame: true` | `weaponFrame` | `rank = 1` | bool → int |
| `unlocks.efficientCore: true` | `efficientCore` | `rank = 1` | bool → int |
| `points` | `points` | 原值保留 | 不动 |

### 5.2 不退点策略（关键）

**原则**：超过 maxRank 的旧 rank 不退还点数，不写回 `points`。

理由：

1. 旧字段每级成本恒为 1 点。如果某玩家有 `mining = 7`，按旧消费实际投入 7 点。新 maxRank = 3 后，迁移后 rank = 3，"看似亏 4 点"——但**v0.10.0 不退点**，保护"已获得能力"的优先级高于"账面公平"。
2. 退点会引入与"未参与超买的玩家"的不公平：例如玩家 A `mining=3`、玩家 B `mining=7`，迁移后 A.rank=3、B.rank=3、若 B 退还 4 点则 B 凭空多 4 点可花。
3. UI 文案必须明确提示（由 UI 方案落地）："v0.10.0 整理了旧成长进度，部分超出的旧投入已折算为当前等级。"

### 5.3 weapons → weaponCalibration 单映射的副作用

旧 `weapons` 同时影响 damage（+10%/+12%）与 thrust（+3%）。

新方案：

- `weaponCalibration` 接管 damage：`+10% / rank`，与旧 +10% / +12% 平均。
- `weaponEfficiency` 独立节点接管 thrust：`+5% / rank`，**但旧字段不映射到 `weaponEfficiency`**。

理由：

1. 拆维度让玩家可单独决定"我要伤害还是要推力"，符合制作人方向 §「路线选择明确」。
2. 旧 `weapons` 玩家不被强制获得新 `weaponEfficiency`，避免双倍收益和"自动塞节点"的体感。
3. 旧 `weapons` rank=3 的玩家迁移后 thrust 加成从 +9% 降到 0%，差值约 9%——在 §10 上限内可被协议 / 后续购买填回。

**ADR 增补提示**：本策略需要在 ADR §3.4 备注一条"weapons 不附送 weaponEfficiency"。

### 5.4 clamp 边界

`clampRank(id, value)` 实现按以下表：

| id | clamp 上限（= maxRank） |
| --- | --- |
| `miningYield` | 3 |
| `hullIntegrity` | 3 |
| `weaponCalibration` | 3 |
| `weaponFrame` | 1 |
| `efficientCore` | 1 |

迁移内 `Math.max(0, Math.min(maxRank, Math.floor(value)))`，对 NaN / undefined / 负数兜底为 0。

## 7. 与 v0.9 galaxyType / resourceMod 的不双倍放大约定

**风险**：v0.9 已有 `GALAXY_TYPES.*.resourceMod`（如 `richBelt.metal: 1.3, ore: 1.5`）通过 `getGalaxyResourceMultiplier` 在 `grantObjectiveReward` 的 assault metal 奖励中生效。如果 v0.10.0 把 meta 加成也按倍率塞入同一公式，玩家会感受到"双倍放大"。

**v0.10.0 约定**：

1. `miningYield` 只乘到**采集设施实时产出**（`getMetaEffect("miningYield")` 在 `game.js:6440` 替换 `miningBonus`），**不进 `grantObjectiveReward` 的 assault metal**。
2. `startingMetal / startingOre / startingGas / startingPlasma` 是开局一次性加值，**不乘 galaxyType.resourceMod**——协议在 `applyMetaStartProtocol` 内**直接 `state.resources.metal += effect`**，绕过 `getGalaxyResourceMultiplier`。
3. `metalRefining / buildDiscount` 只影响**消耗**（`getBuildCost`），不影响**产出**，与 galaxyType.resourceMod 维度正交，无双倍放大。
4. `researchInsight` 在 `grantObjectiveReward` 的 `researchGain` 之后再乘。该位置目前**没有 galaxyType.research 乘子**（galaxyType resourceMod.research 仅用于 v0.5 局内研发流程，与任务奖励 research 是同一资源池）——**潜在风险**：若 `resourceMod.research` 进 `grantObjectiveReward`，则两者会叠乘，需要 ADR 在 #5 实现期复核。本数值方案预设"目前只 `research` 单乘"，若实现发现双乘超过 +50%，把 `researchInsight perRank` 从 0.12 降到 0.08。

**双倍放大兜底**：任何 `effectKey` 与 `galaxyType.resourceMod[*]` 同时作用于同一变量时，必须显式声明乘法顺序，**且总乘子上限 1.8**（即 +80%）。例如富矿星系（metal 1.3）+ miningYield 满 rank（1.3）= 1.69，未超 1.8 √。

## 8. 与 v0.5 局内研发的边界（不被 meta 替代）

ADR §9.2 已规定：

- `researchInsight` 只乘 `state.resources.research` **本局获得量**。
- 不写 `state.research.unlockedMods`、`unlockGlobalResearch`。
- 不进 SAVE_KEY，不持久化研发解锁。

本数值方案数值层面对齐：

- `researchInsight` 满 rank +24%，对应单局 research 收益从 X 提升到 1.24X，需要的研发解锁次数依然由 v0.5 公式决定。
- v0.5 单次研发解锁通常需要 ~30-60 research，+24% 意味着少做 0.2-0.5 次任务即可解锁——可感知但不替代研发系统。
- `efficientCore` 只是解锁蓝图（flag），不直接赠送研发解锁项。

## 9. 平衡边界与上限

### 9.1 软上限（满 rank + 协议最大叠加）

| 维度 | 满 rank 加成 | 协议加成 | 总加成 | 评估 |
| --- | --- | --- | --- | --- |
| 采集产出 | +30%（miningYield） | 0（协议不直接加产出） | +30% | √ 与单 mining 设施增量同档 |
| 武器伤害 | +30%（weaponCalibration） | 0 | +30% | √ 不超过新增武器单元的边际收益 |
| 推力 | +10%（weaponEfficiency） | 0 | +10% | √ 比旧 weapons*0.03 拆出后体感小 |
| 核心 hp | +30%×+20%（hullIntegrity × coreFortitude）= +56% | +8%（defenseStart） | +68% | √ 不让敌人威胁消失 |
| 框架 hp | +30%（hullIntegrity） | +8%（defenseStart） | +40% | √ |
| 科研收益 | +24%（researchInsight） | 0 | +24% | √ |
| 建造金属 | -10%（metalRefining） × -8%（buildDiscount） × -15%（协议设施折扣） | — | ≈ -29% | √ 总折扣不超过 30%，仍保留采集压力 |
| 开局 metal | 0（talent 无） | +25 | +25（130 基线 +19%） | √ |

### 9.2 单局后期消耗约束

- 后期建造单价折扣最高 -29%，但 endgame 关卡资源消耗（assault / guardian）公式不变。
- 武器伤害 +30% 不足以让玩家跳过 v0.6 hostile-station / v0.7 锁敌 / v0.8 encounters 的设计压力。
- 核心 hp +68% 在 endgame guardian 阶段被等比削弱（guardian 伤害与玩家 hp 同步缩放——此设计如 v0.4 已存在；若不存在，需要 ADR 在 #8 审核期复核）。

### 9.3 早期成本曲线

| 局数 | 累计点数（假定每局 12 点） | 推荐购买顺序 | 累计花费 |
| --- | --- | --- | --- |
| 1 局后 | 12 | miningYield×1 (1) + hullIntegrity×1 (1) + weaponCalibration×1 (1) + researchInsight×1 (1) = 4 | 4，余 8 |
| 2 局后 | 24 | + miningYield×2 (1) + hullIntegrity×2 (1) + startingCache×1 (2) → 解锁 miningStart 协议 | 8，余 16 |
| 3 局后 | 36 | + coreFortitude×1 (2) → 解锁 defenseStart 协议 + weaponCalibration×2 (1) → 解锁 weaponStart 协议 | 11，余 25 |
| 4 局后 | 48 | + tier 2 各级 + weaponFrame×1 (3) | ≥ 20 |
| 5 局后 | 60 | + buildDiscount + efficientCore | ≥ 30 |
| 6-7 局后 | ≥ 72 | + galaxyForesight (4)，开始刷满 | 50 |

结论：玩家在 **第 2 局结束**即可解锁第一个协议（miningStart），第 3 局即可同时解锁所有 3 个早期协议。这与制作人方向 §「协议影响下一局前 3 分钟体验」目标对齐。

11. 旧存档 `weapons=3` 迁移后 `weaponCalibration=3`，`weaponEfficiency=0`（不附送，与 §5.3 一致）。

### 11.3 节点购买与效果（8 条）

12. 空存档，`grantPoints(4)` 后买 `miningYield×3`，`getMetaEffect("miningYield") === 1.30`。
13. 不满足 `metalRefining` 前置（miningYield rank < 1）时购买返回 `{ok:false, reason:"prereq_unmet"}`。
14. `buildDiscount` 跨路线 prereq（hullIntegrity rank>=2 + metalRefining rank>=1）任一不满足都拒绝购买。
15. `weaponFrame rank=1` 后 `hasMetaTalent("weaponFrame") === true`，对应 `game.js:5598` 解锁判断生效。
16. `efficientCore rank=1` 后 `game.js:6312` 解锁判断生效。
17. `researchInsight rank=2`，单关 `grantObjectiveReward` 时 `state.resources.research` 增量比 rank=0 时多 24%（在同一 level 与 objectiveType 下）。
18. `metalRefining rank=2` + `buildDiscount rank=2` 同时生效时，基础 metal cost 8 的 mining 设施实际成本 = `floor(8 * 0.95^2 * 0.96^2) = floor(8 * 0.9025 * 0.9216) = floor(6.66) = 6`。
19. 已满 rank 节点 `getMetaPurchaseState` 返回 `{canBuy:false, reason:"max_rank_reached"}`，重复点击不扣点。

### 11.4 协议（6 条）

20. `balanced` 协议恒在 `unlockedProtocols`，空存档可选。
21. 买 `startingCache rank=1` 后 `unlockedProtocols` 包含 `miningStart`。
22. 选 `miningStart` 协议并 reset run 后，`state.resources.metal === 130 + 25 === 155`，`state.resources.ore === 60 + 10 === 70`。
23. 选 `defenseStart` 协议并 reset run 后，核心初始 hp 在 hullIntegrity 加成基础上额外乘 1.08。
24. 选 `weaponStart` 协议并 reset run 后，`state.resources.plasma === 8 + 3 === 11`，武器设施 metal cost 按 0.85 折扣。
25. 协议加成不写入 `state.run.galaxyMap / currentGalaxyType / encounters / objective` 字段（diff 测试）。

### 11.5 边界与不双倍放大（5 条）

26. 富矿星系（`GALAXY_TYPES.richBelt.resourceMod.metal === 1.3`）+ `miningYield rank=3`，采矿产出乘子 = `1.0 * 1.3 * 1.30 = 1.69`，未超 §7 总乘子上限 1.8。
27. `miningStart.startingMetal = 25` **不**乘 `galaxyType.resourceMod.metal`，开局 metal 恒 = 130 + 25 = 155（任何星系一致）。
28. `researchInsight` 不写 `state.research.unlockedMods`，重启浏览器后 v0.5 局内研发解锁状态归零。
29. `weaponEfficiency` 满 rank 推力 +10% 不影响 `state.run.galaxyMap` 跃迁倍率（v0.9 跃迁是独立常量）。
30. 旧 `weapons=5` 迁移后 `weaponCalibration=3`、`weaponEfficiency=0`，玩家可买回 `weaponEfficiency`，**总投入不亏旧花费的 5 点**（旧 5 点 → 新 weaponCalibration 满级仅 4 点 + weaponEfficiency 满级 4 点 = 8 点；§5.2 不退点保证旧花费不被退还但也不影响后续）。

### 13.2 需要 ADR 增补的决策点

向主程提交，由 ADR 在 #5 实现期回写：

1. **D3a**：`startingCache rank=1` 派生解锁 `miningStart`（替换 ADR §4.2 中 `miningStart.unlock = { nodeId: "miningYield", rank: 2 }`）。
2. **D5a**：`mul` op 在 talent 中按 `rank` 次数累乘 `perRankMul`，在 protocol 中按 `value` 单次累乘（扩展 ADR §4.3 表）。
3. **D5b**：`coreFortitude` 与 `hullIntegrity` 都是 `addPercent`，但**跨 effectKey 之间乘法叠加**（核心 hp = `base * (1 + hullIntegrityTotal) * (1 + coreFortitudeTotal)`），非加法。
4. **§5.3 备注**：`weapons → weaponCalibration` 单映射，**不附送 `weaponEfficiency`**。
