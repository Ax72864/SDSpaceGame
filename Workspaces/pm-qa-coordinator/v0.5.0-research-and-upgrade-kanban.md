# v0.5.0 研发与升级 PM 看板

## 本轮目标

让玩家在每局 15-30 分钟内形成可感知、可演化、可平衡的研发路线 build 决策：完成任务获得 research，主动选择设施升级、设施改造或全局加成；同时保留 v0.1.x-v0.4.x 已验收的 `techLevel` 自动难度曲线和既有玩法路径。

## 背景与制作人方向

制作人方向文档：`Workspaces/game-producer-lead-designer/v0.5.0-research-and-upgrade-direction.md`，commit `2b9d640`。基线为 HEAD `2b9d640`（v0.4.0 发布后），当前版本风险级别为项目高风险。

### 方向要点摘要

- 引入局内研发路线 build：任务完成后给 research，玩家用 research 升级 / 改造设施或激活全局加成。
- 建立 `getCellStat(cell, key)` 集中读取层，把散布的 damage / reload / range / repairRate / produceRate 等参数收敛到查询接口。
- 扩展 `TYPES.baseStats / upgrades / modifications`，让 13 种设施具备可调、可回归、可扩展的数据来源。
- 在 cell 上新增 `tier` 与 `mod` 字段；旧 cell 默认 `tier = 0`、`mod = null`，通过 `getCellStat` 应用 tier 倍率与改造效果。
- 扩展任务注册表 `OBJECTIVE_TYPES.researchReward`，任务完成时增加 `state.resources.research`。
- 扩展 `selectedCellPanel` 与全局研发面板，提供单设施升级 / 改造和 `hullMod / weaponMod` 全局节点。
- 保留 `techLevel` 自动升级机制，玩家研发作为额外成长路线，与难度曲线乘法叠加但不替代。

### 体验目标

1. **可感知 build 决策**：玩家完成任务后能在面板看到升级炮塔、改造导弹井、激活船体强化等选项，选择后立刻感知数值或表现变化。
2. **可演化 build 路线**：同一玩家两次 run 可因研发选择不同而呈现差异化 build，形成"再开一局试另一条路线"的动机。
3. **可平衡难度曲线**：保留 v0.1.x-v0.4.x 的 `techLevel` 自动难度曲线；玩家研发是额外路线，不让早期碾压或中后期崩盘。

### 必做范围

- 工程基础：新增 `getCellStat(cell, key)` 查询层，扩展 `TYPES.baseStats / upgrades / modifications`，并保证抽象前后既有 13 种设施行为等价。
- 关键数据结构：为 cell 增加 `tier` 与 `mod`，旧数据默认兼容；明确跨关、重启、旧 meta 的生命周期。
- 研发点积累：为 7 种任务 `mine / explore / battle / survive / guardian / salvage / escort` 增加 `researchReward`，任务完成时写入 `state.resources.research`。
- 升级面板：在 `selectedCellPanel` 增加升级 / 改造入口，显示成本、效果、当前 tier 和当前 mod。
- 改造系统：每个核心设施提供 1-2 个 mutually exclusive 改造，v0.5.0 默认不可逆。
- 全局加成：激活已有死代码 `state.station.hullMod / weaponMod`，分别应用 hull 与 weapon 加成。
- HUD：research 数字保留，并增加增长率 / 秒或等价可读提示。
- 文案与调参：按数值方案落地 research 奖励、升级成本、改造成本、全局加成成本和数值上限。

### 不做范围

- 不做局外天赋树重构。
- 不做联机 / 异步对战。
- 不做战斗深度扩展。
- 不做敌方空间站完整化。
- 不做新设施解锁。
- 不做新建造资源类型。
- 不修改 `SAVE_KEY` / 局外存档结构。
- 不修改 `techLevel` 自动机制。
- 不做 v0.3.0 fragment 系统调整。
- 不做 v0.4.0 任务系统调整，除新增 `researchReward` 字段。
- 不做 v0.4.1 候选项。
- 不做 v0.3.1 候选项。

### 兼容要求

- v0.1.x / v0.2.0 / v0.2.1 / v0.3.0 / v0.4.0 全部回归点必须通过。
- `getCellStat` 抽象后，现有 13 种设施关键行为完全等价。
- 终局守护者与研发系统交互正常，玩家可在 guardian 前研发，但不能因研发碾压 guardian。
- 旧存档 meta 注入后正常读取，覆盖 `{}`、`{points:5}`、无 `unlocks`、无 `research` 字段等形态。
- `state.resources.research` 既有产出 / 消耗路径保持工作，v0.5.0 仅新增任务奖励和玩家选研发两个路径。
- v0.4.0 fragment / wreck / NPC 路径不回潮；wreck fragment 内含 cells 时，`cell.tier` 与 `cell.mod` 必须有默认值兼容。

## 风险定级

- 风险等级：项目高风险。
- 定级原因：触及核心玩法成长循环、关键 cell 数据结构、任务奖励路径、HUD / 面板交互、全局加成死代码激活和大量散布参数迁移。
- 主要风险：`getCellStat` 抽象可能造成 13 种设施行为回潮；`cell.tier / cell.mod` 生命周期不清会污染 fragment / wreck / NPC / nextLevel / resetRun；research 奖励与成本曲线不当会破坏 `techLevel` 难度曲线；改造互斥与全局加成若缺乏上限可能造成数值爆炸；旧 meta 兼容和 `SAVE_KEY` 不应被破坏。
- 高风险流程：制作人方向 → PM 拆分 → 主程技术 ADR → 数值方案 → 核心实现 → 专项审核 → PM 浏览器级 + 主观回归 → 制作人最终决定与发布安排。
- 审核基底约束：专项审核使用 `expert-agent-planner-gpt55`，必须与执行不同基底。

## 缺陷记录格式

- 标题：一句话描述问题。
- 严重程度：S0 阻断 / S1 严重 / S2 一般 / S3 轻微。
- 影响范围：说明影响启动、存档、研发点、升级面板、改造、全局加成、`getCellStat`、设施行为、任务奖励、fragment、wreck、NPC、HUD、WASD、采集、建造、敌人攻击、终局或自由游玩中的哪一部分。
- 复现步骤：按浏览器实机操作顺序列出，包含 seed / level / objective type / build 路线 / station cell / tier / mod / research 数值 / 玩家操作。
- 实际结果：记录玩家实际看到或系统实际发生的情况，必要时附运行态采样。
- 期望结果：记录按制作人方向、ADR 或数值方案应发生的体验。
- 优先级：P0 / P1 / P2，并说明是否阻塞接受。
- 负责人：明确后续处理成员。
- 当前处理：待分配 / 修复中 / 待审核 / 已回归 / 接受为已知问题。
- 回归要求：修复后尽量按原复现步骤复测，并说明失败路径是否关闭。
