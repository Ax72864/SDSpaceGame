## 代码边界判断

当前 `Game/game.js` 的建造规则为：

- `buildAt(x, y, "frame")`：只能在空的已连接邻格新增框架。
- `buildAt(x, y, facility)`：非 frame 设施只能建在已有、未脱落、facility 为 `frame` 的格子上。
- `getPlacementDiagnostics()` / `evaluatePlacementGrid()` 当前用于放置预览，语义是“目标格是否空且连接”，更适合提示框架扩展或设施周边潜在位置；它不能替代“设施实际建造必须点击已有空框架”的成功判定。
- `handleCanvasClick()` 通过 `renderer.screenToWorld()` 和 `worldToGrid()` 将 CDP / 鼠标坐标落到格子，再调用 `buildAt()`。

因此，如果取证脚本在选中 `power/mining/turret/repair` 后继续点击“空邻格”或因坐标/DPR/相机旋转误差落到空邻格，而不是点回新建的 `frame` 格，就会得到 #5 记录中的错误提示。这符合现象，不需要修改建造合法性规则。

## 建议复测脚本口径

下一轮可见浏览器/CDP 复测应按以下顺序记录每一步的格子和建造结果：

1. 选择 `frame`。
2. 点击一个合法空邻格，记录新增 frame 的 `gridKey`。
3. 选择 `power`，点击第 2 步新增的 `gridKey`，确认该格由 `frame` 替换为 `power`。
4. 再选择 `frame`，扩出另一个合法空邻格。
5. 选择 `mining`，点击该新 `frame`，确认 `mining x1`。
6. `turret`、`repair` 同理：先扩 frame，再点已有 frame 替换设施。
7. 每一步记录 `state.selectedBuild`、目标 `gridKey`、`state.station.cells.get(gridKey)?.facility`、资源变化、toast / statusAlerts、runtime errors。

如继续使用 CDP，优先复用 v0.15 的 `mouseWorld + Enter` 方式，或增加“点击前 gridKey 与点击后 facility”断言，避免把“合法格诊断可放置”误写成“实际建造成功”。
