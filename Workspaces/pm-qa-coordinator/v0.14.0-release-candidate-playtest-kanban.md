# v0.14.0-release-candidate-playtest PM 看板

## 本轮目标

版本名：`v0.14.0-release-candidate-playtest`。

本轮目标是把 SSDSpaceGame 从“v0.12/v0.13 功能开发完成但未发布”推进到“发布候选证据可验收”。本版本不继续叠新玩法，不改变平衡，不启动发布流程；只补齐真实浏览器人工体验、小屏触控假设、真实帧耗时粗采样、运行态只读签名和 PM 可复查记录，让制作人能保守判断是否进入发布准备。

## 背景

- 制作人方向文档：`Workspaces/game-producer-lead-designer/v0.14.0-release-candidate-playtest-direction.md`。
- v0.12.0 已完成空间站设计反馈，v0.13.0 已完成战斗威胁可读性；两者均处于开发完成基础状态，但均未正式发布。
- 主要阻塞不是功能列表缺失，而是真实浏览器人工体验、小屏触控、真实帧耗时、完整运行态只读签名等发布证据不足。
- v0.14.0 的 PM 目标是把这些证据缺口拆成可执行任务、可复查验收点和可记录回归结论。
- 没有真实人工体验记录时，不得宣称“发布就绪”；headless 或 VM 结果只能作为自动检查证据。

## 制作人方向摘要

- 玩家路径必须覆盖空存档首局、建造、采矿、资源不足、断电、敌人接近、武器/护盾/维修、战斗后复盘、星系跳转和 Meta 回到新局。
- 浏览器证据必须记录 console/runtime errors、关键 DOM、viewport、小屏或窄屏参数、只读 hooks、frame timing 粗采样和 storage signature。
- release readiness snapshot 或 `runReleaseCandidateChecks` 只能读状态，不写存档，不改变 RNG、战斗、资源、Meta 或星系推进。
- 小屏可读性只允许做低风险打磨，重点是关键容器折行、滚动、遮挡、最大高度、短文案和只读可见性标记。
- 发布候选结论必须区分自动检查、browser/headless、人工体验、未覆盖风险和是否阻塞接受。

## 风险定级

- 风险等级：中低风险。
- 中低风险来源：本版本默认聚焦发布候选取证、只读测试钩子、人工 checklist、browser/headless 记录和小屏可读性小修，不直接改核心玩法规则。
- 默认控制：先做文档化路径和只读取证能力，再做必要小屏可读性补洞；所有记录必须说明证据类型和未覆盖边界。
- 高风险升级触发：若触及核心规则、数值、战斗判定、敌人 AI、RNG、资源公式、维修/护盾/脱落判定、存档结构、`SAVE_KEY`、meta schema、依赖安装、发布产物、tag 或 `Releases/` 归档，必须暂停并升级为高风险，需要制作人重新确认、PM 调整看板、主程专项审核；涉及本机环境危险操作还需要用户确认。

## QA 回归点

### A. 空存档首局

- [ ] A01. 使用空存档或隔离存档进入游戏时，默认入口可见且无阻塞错误。

- [ ] B01. 玩家能找到基础建造入口并看到可建设施。
