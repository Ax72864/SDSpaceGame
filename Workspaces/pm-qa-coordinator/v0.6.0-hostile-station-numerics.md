## §0. 制作人方向回放与硬约束 + ADR 数值对齐

### §0.1 体验目标三条（必须命中）

1. **可感知 BOSS 决战**：玩家在 level 3-5 中段遭遇 hostile-station 时，立刻感受到"这不是普通杂兵"——视觉规模大、攻击多点、生命池厚、被打击有 cells 逐块脱落反馈；击毁后相机震屏 + wreck 爆出 + 任务完成感。
2. **可战术的攻击决策**：玩家可选"先打武器 cell 让它停火"、"先打 power cell 削弱"或"直击核心快速终结"；同 build 在不同战术下表现不同；让 v0.5.0 研发系统 build 决策有"针对 BOSS"的具体使用场景。
3. **可衔接的奖励循环**：击毁后 6-12 个 wreck fragment 与 v0.3.0 拾荒系统强协同；"决战 → 战利品 → 进入下个星系"形成单局节奏高潮。

### §0.2 不做范围（数值层硬约束）

- 不修改 v0.1.x / v0.2.0 / v0.2.1 / v0.3.0 / v0.4.0 / v0.5.0 已发布数值（含 `FRAGMENT_*` / `BRIDGE_*` / wreck 物理 / NPC HP / 奖励倍率 / spawn 曲线 / `techLevel` 公式 / tier 倍率 / mod 倍率 / hullMod / weaponMod 数值）。
- 不修改 `SAVE_KEY` 与局外存档结构；本方案新增字段仅 `state.run.hostileStationSpawnedThisGalaxy` / `state.run.hostileStationAlerted` / `cell.owner = "enemy"` / `enemy.hostileStationMinion = true`，不写入 `state.meta`。
- 不修改 v0.4.0 现有 7 种任务奖励 / 权重 / 触发路径；本方案仅追加 `assault` 第 8 种与其在 level 3-5 的 weight 行扩展。
- 不修改 guardian 数值（HP 1200 / spawnCap 4 / spawn 距离 1100-1300 / 死亡震屏 64/1.1 / 终局触发 endgame）。
- 不修改 v0.5.0 研发系统对玩家 cell 的所有公式；本方案仅依赖 ADR §4.5 给出的 `getCellStat` owner 隔离改动，让敌方 cells 不被玩家 techLevel / hullMod / weaponMod 加成。
- 不引入新设施种类、新建造资源、新弹道类型、新 enemy kind（hostile-station 是唯一新增）。
- 不引入 missile cell / mining cell / processor cell / research cell 给 hostile-station 配置（v0.6.0 hostile-station 只用 core / power / turret / shield / armor / frame 6 种 facility）。
- 不引入"敌方 cells 升级 / 改造"机制（hostile-station cells 固定 tier=0 / mod=null）。
- 不引入"敌方 techLevel"（hostile-station 不进化；level 缩放只通过 HP / 武器 stat / 召唤参数实现）。

### §0.4 与方向文档建议的差异说明

方向文档 §"数值边界建议（给数值策划）" 给出 7 项数值参数；本方案与方向 + ADR 的差异如下：

| 方向建议 | ADR 拍板 | 本方案落地 | 差异理由 |
|---|---|---|---|
| 总 HP 池 1500/2200/3000 | 同方向 | 同 ADR | 完全一致 |
| 单 cell HP 150-300 | 单 cell HP 150-300 | core 160 / power 150 / turret 260-280 / shield 130 / armor 280 / frame 140（level 3 基础值） | turret/armor 在区间上限附近；shield/frame 略低但保证总池 1500 |
| cells 数量 5-8 | 5-8 | **固定 7 cells**（1 core + 1 power + 2 turret + 1 shield + 1 armor + 1 frame） | 7 cells 在区间中位；让玩家有 4 个可拆战术目标（turret×2 / power / shield）+ core 速胜路线 |
| level 权重 25%/35%/30% | **20%/30%/35%**（已拍板） | 同 ADR | ADR 拍板覆盖方向建议；理由：level 5 应是 BOSS 最频繁的位置（玩家 build 最成熟），level 3 最低（玩家 build 尚浅）；与方向"level 4 最高"相反但与"研发完成度递增"匹配 |
| 武器 reload 1.6s | 1.6s | 1.6s（与 pirate / enemy 通用 reload 一致） | 一致 |
| spawn 距离 1100-1300（guardian 同款） | 1000-1300 | 1000-1300（ADR 拍板） | ADR 略放宽下界，让 level 3 出现时玩家更早进入视野 |
| wreck 6-12 / 分布 turret 50/shield 25/armor 15/repair 10 | 同方向 | 同 ADR | 完全一致 |

> **关键决定**：本方案采纳 ADR 拍板的 level 权重 20%/30%/35%（不是方向的 25%/35%/30%），原因如下：
> 1. 玩家在 level 5 已完成 2-3 个核心设施升级与至少 1 个全局加成（按 v0.5.0 numerics §1.4 估算 level 5 累计 47 research），BOSS 决战体验最佳。
> 2. level 3 玩家 research 累计仅 13-15 点（v0.5.0 §1.4），可能尚未完成第一次 tier 升级；过高 weight 会让玩家"被 BOSS 教育"挫败感。
> 3. level 4 居中。
> 4. ADR 已 commit 拍板（`04e88fc`），数值方案不应推翻 ADR。

## §1. hostile-station 总 HP 池 + level 缩放

> 设计意图：让 BOSS 击毁时间在玩家有完整 v0.5.0 研发 build 时 30-45 秒，无研发时 60-90 秒；既高于普通 enemy（pirate ~200 HP / station ~430 HP），又不超 guardian 1200 HP 的"终局封顶"语义（但 hostile-station 因为多 cell 拆解，体感上比 guardian 更"BOSS 化"）。

### §1.1 总 HP 池表（ADR 拍板）

| level | 总 HP 池 | 缩放系数 | 设计理由 |
|---|---|---|---|
| 3 | **1500** | ×1.0（基准） | 玩家 build 浅，BOSS 应"可挑战"；与 v0.5.0 level 3 玩家武器 base damage 7 × 1.05 weaponMod ≈ 7.35 估算，60 秒可输出 ~440 damage，单炮塔难以击破；2-3 炮塔可在 60-90 秒内完成 |
| 4 | **2200** | ×1.47 | 玩家 build 接近成熟（按 v0.5.0 numerics §1.4 累计 30+ research，至少 1 个 tier 2）；HP 池 ×1.47 与玩家武器加成 ×1.2-1.3 匹配 |
| 5 | **3000** | ×2.0 | 玩家 build 成熟（累计 47 research）；HP 池 ×2.0 与玩家 weaponMod 1.05 × tier 2-3 turret 1.4-1.6 ≈ ×1.5-1.7 输出加成匹配；30-45 秒击毁 |

### §1.2 总 HP 池公式

```
HOSTILE_STATION_LEVEL_HP[level] = {
  3: 1500,
  4: 2200,
  5: 3000
}

enemy.maxHp = sum(cell.maxHp for cell in enemy.cells)  # 聚合显示
enemy.hp    = sum(cell.hp for cell in enemy.cells)     # 实时聚合
```

`enemy.maxHp` 与 `enemy.hp` 是聚合值，用于 HUD HP bar 渲染与 `enemy.hp > 0` 过滤；实际伤害逐 cell 累计。

### §1.3 总 HP 含义（cells 占比）

按 ADR §0.3 + 方向 §"数值边界建议"：

- **核心 cells 占比 10-15%**（core HP 含动力）
- **武器 cells 占比 40%**（turret × 2 合计）
- **结构 cells 占比 45-50%**（shield + armor + frame 合计）

本方案精化（见 §2）：

| 类别 | cells | level 3 HP 合计 | 占比 |
|---|---|---|---|
| 核心 | 1 × core + 1 × power | 160 + 150 = 310 | 20.7% |
| 武器 | 2 × turret | 260 + 280 = 540 | 36.0% |
| 结构 | 1 × shield + 1 × armor + 1 × frame | 130 + 280 + 140 = 550 | 36.7% |
| **合计** | **7 cells** | **1400** | 93.3%（差 100 由 §2.3 微调） |

> **§1 备注**：因为方向给出占比建议是 10-15% / 40% / 45-50%，但单 cell HP 150-300 取整后总和 1400，与 1500 差 100；本方案在 §2.3 将 turret 与 armor 上调以填补差距，保证 level 3 总池 = 1500（ADR 拍板）。

## §2. cells 配置精化

> 设计意图：7 cells 固定布局（不随 level 增减 cell 数量），按 level 1.0 / 1.47 / 2.0 系数缩放每个 cell 的 HP；让玩家在 level 3-5 看到的"形状"一致（学习曲线），但难度递增。

### §2.1 cells 7 项布局（ADR §3.2 模板）

| # | cell | 坐标 (x, y) | facility | 名称（HUD 文本） | 设计角色 |
|---|---|---|---|---|---|
| 1 | core | (0, 0) | core | 核心 | 击杀判定中心；HP=0 即死 |
| 2 | power | (0, -1) | power | 动力 | 被破坏后所有武器 reload +50% |
| 3 | turret-A | (1, 0) | turret | 主炮 A | 独立武器 cell |
| 4 | turret-B | (-1, 0) | turret | 主炮 B | 独立武器 cell |
| 5 | shield | (0, 1) | shield | 护盾发生器 | 拦截玩家弹道 |
| 6 | armor | (1, 1) | armor | 装甲 | 高 HP 缓冲，纯 HP |
| 7 | frame | (-1, 1) | frame | 骨架 | 结构补强（与玩家 frame cell 等价） |

> **§2.1 备注**：布局坐标为 3×3 网格 7 占（缺 (-1,-1) / (1,-1)）；视觉上左右基本对称（turret 双侧），上下不对称（power 下 / shield 上）让玩家"看一眼就能识别功能位"。

### §2.2 level 3 各 cell HP（基础值）

| # | cell | facility | HP | 占比 | 备注 |
|---|---|---|---|---|---|
| 1 | core | core | **160** | 10.7% | 核心：故意低于其他 cells，让"直击 core 速胜"可行 |
| 2 | power | power | **150** | 10.0% | 动力：低于 core，让玩家有"先打动力削弱"激励 |
| 3 | turret-A | turret | **260** | 17.3% | 主炮：中高 HP，玩家拆武器需 18-20 秒持续输出 |
| 4 | turret-B | turret | **280** | 18.7% | 主炮：略高于 turret-A，制造"先打哪个"决策成本 |
| 5 | shield | shield | **130** | 8.7% | 护盾发生器：最低 HP，鼓励玩家"先打 shield 让 BOSS 失去保护" |
| 6 | armor | armor | **280** | 18.7% | 装甲：最高 HP，纯 HP 缓冲 |
| 7 | frame | frame | **140** | 9.3% | 骨架：低 HP，与玩家 frame cell baseHp 等价 |
| **合计** | | | **1400** | 93.3% | |

### §2.4 cells 配置常量（实现参考）

按 ADR §3.1 + §4.1 提示，本方案给出 `HOSTILE_STATION_LAYOUT` 与 `HOSTILE_STATION_LEVEL_HP` 配置形式（实现时 T1 可直接复用）：

```js
const HOSTILE_STATION_LEVEL_HP = {
  3: 1500,
  4: 2200,
  5: 3000
};

const HOSTILE_STATION_LAYOUT = {
  cells: [
    { x: 0,  y: 0,  facility: "core",   hpRatio: 160 / 1500 },  // 0.1067
    { x: 0,  y: -1, facility: "power",  hpRatio: 150 / 1500 },  // 0.1000
    { x: 1,  y: 0,  facility: "turret", hpRatio: 300 / 1500 },  // 0.2000
    { x: -1, y: 0,  facility: "turret", hpRatio: 280 / 1500 },  // 0.1867
    { x: 0,  y: 1,  facility: "shield", hpRatio: 130 / 1500 },  // 0.0867
    { x: 1,  y: 1,  facility: "armor",  hpRatio: 320 / 1500 },  // 0.2133
    { x: -1, y: 1,  facility: "frame",  hpRatio: 160 / 1500 }   // 0.1067
  ]
  // 总 hpRatio = 1.0000
};
```

在 `spawnHostileStationNearStation(rng)` 内：

```js
const totalHp = HOSTILE_STATION_LEVEL_HP[state.run.level] || 1500;
for (const slot of HOSTILE_STATION_LAYOUT.cells) {
  const cellHp = Math.round(totalHp * slot.hpRatio);
  const cell = createHostileStationCell(slot.x, slot.y, slot.facility, cellHp);
  cells.set(key(slot.x, slot.y), cell);
}
```

### §2.5 power cell 破坏效果（关键战术）

按 §0 体验目标 "可战术攻击决策"，power cell 被破坏后必须有可感知效果：

| 状态 | 行为 |
|---|---|
| power.hp > 0 | 所有武器 cell `cell.reload = getCellStat(cell, "reload")` 正常 1.6s |
| power.hp <= 0 | 所有武器 cell `cell.reload = getCellStat(cell, "reload") * 1.5` 等效 2.4s reload |

实现时 `updateHostileStationWeapon(enemy, cell, dt)` 内：

```js
let reloadFactor = 1.0;
const powerCell = findCellByFacility(enemy.cells, "power");
if (!powerCell || powerCell.hp <= 0) reloadFactor = 1.5;
cell.reload = getCellStat(cell, "reload") * reloadFactor;
```

> **§2.5 备注**：可选实现方式 B 是 "power 破坏后所有武器停火"（不开火）；本方案选择 reload +50%（×1.5）方式让 BOSS 仍有威胁但显著削弱；选择理由：完全停火过于强力（让玩家"先打 power 必赢"破坏战术多样性）。

### §2.6 shield cell 拦截范围

按 ADR §4.6 `updateHostileStationShield` 路径，shield cell 拦截玩家弹道；本方案给出 shield 数值：

| 字段 | level 3 | level 4 | level 5 | 说明 |
|---|---|---|---|---|
| maxShield | 200 | 280 | 360 | shield cell 的 shield 容量（与 cell.hp 独立；shield 归零后 cell 仍有 hp） |
| regen | 8 | 10 | 12 | shield 每秒回复（已扣除战斗中持续消耗）|
| range | 110 | 130 | 150 | 拦截范围（cell 周围半径，像素） |

> **§2.6 备注**：shield cell 的 `maxShield / shield / regen` 字段沿用 v0.3.0 `createCell` 内 shield 设施初始化（baseMaxShield / maxShield / shield）；getCellStat 路径走 ADR §4.5 owner 隔离，敌方 shield 不享受 hullMod / shieldMod。range 由 getCellStat(cell, "range") 返回，base 110 / 130 / 150 按 level 索引。

## §3. 武器 stat

> 设计意图：hostile-station turret cell 走 player turret 同款 `getCellStat` 路径，但通过 `cell.owner = "enemy"` 隔离玩家 techLevel / hullMod / weaponMod 加成（ADR §4.5 关键隔离）；武器威胁度介于 pirate（damage 6-13） 和 guardian（damage 14） 之间，让玩家有"被 BOSS 压制但不会一击必死"的体验。

### §3.1 turret cell 武器 stat（按 level 缩放）

| 字段 | level 3 | level 4 | level 5 | 取值范围 | 设计理由 |
|---|---|---|---|---|---|
| damage | **12** | **14** | **15** | 10-15 | 与 pirate level 3-5 (8/10/12) 同档但更高；不超过玩家 turret base 14 防止"一击必死"|
| reload | **1.6s** | **1.6s** | **1.6s** | 1.5-1.8 | 与 pirate / enemy 通用 reload 一致；让 BOSS 节奏可预测 |
| range | **480** | **515** | **550** | 480-550 | 介于 pirate 380 与 guardian 600 之间 |
| projectileSpeed | **280** | **300** | **320** | 280-320 | 与 LEVEL_STATS.pirate.projectile 280-320 同档 |
| projectileRadius | 4 | 4 | 4 | 3-5 | 与玩家 turret base 一致 |

### §3.2 武器 stat 公式

```js
// getCellStat(cell, "damage") 走 ADR §4.5 owner 隔离：
function getCellStat(cell, key) {
  // ... 现有 base / tier / mod 计算 ...
  const isPlayerOwned = !cell.owner || cell.owner === "player";
  const techFactor = isPlayerOwned ? getTechLevelFactor(...) : 1;
  const globalFactor = isPlayerOwned ? getGlobalModFactor(...) : 1;
  // hostile-station cell.owner === "enemy"，techFactor = 1, globalFactor = 1
}
```

base stat 由 hostile-station spawn 时按 level 直接写入 `cell.baseMaxHp / baseDamage / baseReload / baseRange / baseProjectileSpeed`；不依赖 TYPES.turret.baseStats（避免 v0.5.0 turret base 改动污染敌方）。

### §3.3 武器 stat 总 DPS 估算

| level | 单 turret DPS | 双 turret 总 DPS | 玩家 frame HP（v0.5.0） | 玩家 5 秒受击容忍 |
|---|---|---|---|---|
| 3 | 12 / 1.6 = 7.5 | 15.0 | core HP 100 + frame 150 = 250（base） | ~16.7 秒至 core 危险 |
| 4 | 14 / 1.6 = 8.75 | 17.5 | ~14.3 秒 |
| 5 | 15 / 1.6 = 9.375 | 18.75 | ~13.3 秒 |

> **§3.3 备注**：玩家 v0.5.0 完整 build（hullMod ×1.05 + tier 2-3 turret + armor cells）frame HP 实际 300-400+，受击容忍可达 30+ 秒；裸船玩家危险窗口 ~13 秒匹配"BOSS 决战 30-90 秒"节奏。

### §3.4 shield cell stat（按 level 缩放）

| 字段 | level 3 | level 4 | level 5 |
|---|---|---|---|
| maxShield | 200 | 280 | 360 |
| regen | 8 | 10 | 12 |
| range | 110 | 130 | 150 |

实现：shield cell 在 `createHostileStationCell` 工厂内额外初始化 `cell.baseMaxShield / maxShield / shield`（仿 v0.3.0 `createCell` shield 分支）；`updateHostileStationShield` 走 ADR §4.6 路径。

### §3.5 武器 cell 死亡即停火

按 ADR §1.3 风险 #4 + §4.2，武器 cell 在 `cell.hp <= 0` 时由 `damageHostileStationCell` 从 `enemy.cells` 删除（`enemy.cells.delete(key)`）；后续 `updateEnemies` 内 `for (const cell of enemy.cells.values())` 自然跳过；该武器停火。**这是直接的战术反馈：玩家先打 turret-A，turret-A hp=0 后该武器停火，BOSS DPS 减半**。

## §4. 召唤手下

> 设计意图：召唤手下增加战场压力但不喧宾夺主；spawnCap 上限保护让 minion 数量稳定可预测；level 缩放让高 level BOSS 召唤更快、上限更高，匹配玩家 build 成熟度。

### §4.1 召唤参数表（ADR 拍板范围内）

| 字段 | level 3 | level 4 | level 5 | 取值范围 | 设计理由 |
|---|---|---|---|---|---|
| spawnInterval | **10.0s** | **8.5s** | **7.0s** | 7-10 | level 缩放：高 level 召唤更频繁 |
| spawnCap | **3** | **4** | **5** | 3-5 | level 缩放：同时存在 minion 上限 |
| minion type | pirate | pirate | pirate | 固定 | 沿用 pirate AI 直接索敌玩家（ADR 拍板）|
| 首次召唤倒计时 | rand(5, 9) | rand(5, 9) | rand(5, 9) | 5-9 | 与现有 station 首召唤一致 |

### §4.2 召唤参数计算

```js
const HOSTILE_STATION_SPAWN_BY_LEVEL = {
  3: { interval: 10.0, cap: 3 },
  4: { interval: 8.5,  cap: 4 },
  5: { interval: 7.0,  cap: 5 }
};

// spawnHostileStationNearStation 内：
const spawnCfg = HOSTILE_STATION_SPAWN_BY_LEVEL[state.run.level] || HOSTILE_STATION_SPAWN_BY_LEVEL[3];
enemy.spawnInterval = spawnCfg.interval;
enemy.spawnCap = spawnCfg.cap;
enemy.spawn = rngFloat(rng, 5, 9);  // 首次召唤倒计时
```

### §4.3 minion 标签 + 行为

按 ADR §2.6 + §4.6：

- minion 是 `state.enemies` 内的 `kind === "pirate"` 实体，沿用 pirate AI（直接索敌玩家）。
- minion 上加标签：`enemy.hostileStationMinion = true`（与 guardian 的 `guardianMinion: true` 等价）。
- minion 等级 = hostile-station 出现的 run.level（`spawnLevel: enemy.spawnLevel`）。
- 击毁 minion 给 metal 奖励（沿用 pirate level 3-5 奖励 20/24/28）；不给 research（不算 enemyKilled 多任务进度，但触发 onEnemyKilled 事件，battle 任务进度可累计）。

### §4.4 60 秒战斗内 minion 数量估算

| level | spawnInterval | spawnCap | 首召唤 | 60s 内最大召唤次数 | 玩家可能面对 minion 数量 |
|---|---|---|---|---|---|
| 3 | 10s | 3 | 7s | 7 + 5 × 10s = 57s 内最多 6 次 | 实际上限 spawnCap 3 |
| 4 | 8.5s | 4 | 7s | 7 + 6 × 8.5s = 58s 内最多 7 次 | 实际上限 spawnCap 4 |
| 5 | 7s | 5 | 7s | 7 + 8 × 7s = 63s 内最多 9 次 | 实际上限 spawnCap 5 |

> **§4.4 备注**：spawnCap 上限保护让"持续战斗"场景下 minion 数量稳定（玩家击毁 1 个后下次 spawn 才补 1 个）；如玩家击毁速度跟得上 spawn 速度，minion 数量始终 ≤ spawnCap；如不跟，minion 持续上限 = spawnCap。

## §5. assault 任务奖励

> 设计意图：assault 是 BOSS 决战任务，奖励应介于 battle（最普通战斗）与 guardian（终局封顶）之间；让玩家完成 assault 后明显感觉到 build 强化的回报。

### §5.1 任务奖励参数表

| 字段 | level 3 | level 4 | level 5 | 取值范围 | 与 v0.4.0 任务对比 |
|---|---|---|---|---|---|
| metal base | **35** | **40** | **45** | 35-45 | 介于 battle ~30 与 guardian ~50 之间 |
| researchReward | **9** | **9** | **10** | 9-10 | 介于 battle 5 与 guardian 12 之间（ADR 拍板）|
| rewardMultiplier | **1.4** | **1.5** | **1.6** | 1.4-1.6 | 介于 battle 1.0 与 guardian 2.0 之间（ADR 拍板）|
| 任务限时（建议）| **8 分钟** | **7 分钟** | **6 分钟** | 6-8 | 避免玩家无限拖延 |

### §5.2 奖励发放公式

```js
// 任务完成时 grantObjectiveReward 路径（沿用 v0.4.0 已有路径）：
metalGained = floor(metalBase × rewardMultiplier × (1 + level × 0.1))
researchGained = floor(researchReward × (1 + level × 0.1))

// 示例：level 4 assault 完成
// metalGained = floor(40 × 1.5 × 1.4) = floor(84) = 84
// researchGained = floor(9 × 1.4) = floor(12.6) = 12
```

> **§5.2 备注**：与 v0.5.0 numerics §1.2 实际发放公式 `floor(base × (1 + level × 0.1))` 完全一致；rewardMultiplier 应用于 metal base（不应用于 research，避免双重加成）。

### §5.3 各 level 实际奖励估算

| level | level 倍率 | metalGained | researchGained |
|---|---|---|---|
| 3 | ×1.3 | floor(35 × 1.4 × 1.3) = 63 | floor(9 × 1.3) = 11 |
| 4 | ×1.4 | floor(40 × 1.5 × 1.4) = 84 | floor(9 × 1.4) = 12 |
| 5 | ×1.5 | floor(45 × 1.6 × 1.5) = 108 | floor(10 × 1.5) = 15 |

### §5.4 任务限时与失败处理

按方向文档与 ADR §11 决策开放项 #2，本方案推荐：

- **设置软限时**：level 3 = 8 分钟 / level 4 = 7 分钟 / level 5 = 6 分钟。
- **限时到达不强制失败**：assault 任务不引入 `isFailed` 字段；超时后仍可击毁 BOSS 完成任务，但 HUD 显示"超时"提示（仿 escort 的"危急"标签）。
- **限时含义**：作为"建议节奏"而非硬性阻塞；玩家可继续战斗，但 PM 可在节点 C 体验测试发现"卡 BOSS 超过 10 分钟"问题时再考虑引入硬性失败。

> **§5.4 备注**：本数值方案推荐 **不引入 isFailed**（与 ADR §11 #2 推荐选项 A 一致）；如制作人决定引入硬限时，可在 v0.6.1 添加。本方案给的限时仅作"PM 监控参考值"。

## §7. wreck 爆出（决战奖励）

> 设计意图：击毁 hostile-station 后立刻爆出 6-12 个 wreck fragment，与 v0.3.0 拾荒系统强协同；玩家拖回 station 桥接获得新设施，让"决战 → 战利品"形成爽快闭环；wreck 设施分布偏战斗设施，让玩家拾荒后 build 直接强化。

### §7.1 wreck 数量（ADR 拍板）

| level | wreck 数量 | 设计理由 |
|---|---|---|
| 3 | **6** | level 3 BOSS HP 1500，wreck 6 个让玩家拾荒 build 增量明显但不过载 |
| 4 | **9** | 中位数；玩家此时已有 station + 拾荒经验 |
| 5 | **12** | 最大值；level 5 决战是版本高潮，奖励量级最高 |

```js
const HOSTILE_STATION_WRECK_COUNT_BY_LEVEL = {
  3: 6,
  4: 9,
  5: 12
};
```

### §7.2 wreck 设施分布（ADR 拍板）

| facility | 权重 | 占比 | 设计理由 |
|---|---|---|---|
| turret | 0.50 | 50% | 偏战斗：让玩家 build 直接强化武器 |
| shield | 0.25 | 25% | 防御加成：玩家可补充防护 |
| armor | 0.15 | 15% | 缓冲：补 frame HP |
| repair | 0.10 | 10% | 维修：少量恢复 |

```js
const ENEMY_WRECK_FACILITY_WEIGHTS = [
  ["turret", 0.50],
  ["shield", 0.25],
  ["armor",  0.15],
  ["repair", 0.10]
];
```

> 与玩家 salvage 用的 `WRECK_FACILITY_WEIGHTS`（turret 0.6 / shield 0.2 / armor 0.15 / repair 0.05）略有差异：敌方爆出 shield 占比更高（25% vs 20%），让玩家拾荒后防御能力提升。

### §7.3 wreck 爆出位置

按 ADR §2.7 + §4.3：

- 中心：`enemy.x, enemy.y`（hostile-station 死亡瞬间位置）。
- 半径：`HOSTILE_STATION_WRECK_SPAWN_RADIUS_MIN = 80` 到 `MAX = 150` 像素。
- 分布：环形分布（每个 wreck 随机 angle 0-2π + 随机 radius 80-150）；spawnEnemyWrecks 内 6 次重试避免相邻 wreck 过近（`WRECK_MIN_SEPARATION × 0.7`）。

### §7.4 wreck 内 cells 数量

按 ADR §4.3 + v0.3.0 `buildWreckShape(rng)`：

- 每个 wreck fragment 含 **2-5 cells**（与 v0.3.0 既有 makeWreckFragment 一致）。
- cell facility 由 `pickEnemyWreckFacility(rng)` 按 ENEMY_WRECK_FACILITY_WEIGHTS 抽取。
- cell HP 由 `createWreckCell` 设为 `hp = maxHp * rngFloat(0.65, 0.9)`（残骸语义）。

### §7.5 fragment.origin 字段

按 ADR §2.7：

- `fragment.origin = "enemy-wreck"`（v0.6.0 新增第 3 态）。
- 与玩家 `origin = "player"` / 玩家 salvage 用 `origin = "wreck"` 区分。
- 桥接 / 拾荒路径视同 wreck（修改 `countWreckFragments / renderFragmentOutline / renderFragments` 判断扩展，但 salvage 任务 `onFragmentDocked` 保持 `origin === "wreck"` 精确匹配，避免 enemy-wreck 误算入 salvage 进度）。

### §7.6 WRECK_FRAGMENT_MAX_COUNT cap 处理

按 ADR §1.3 风险 #7 + §2.7：

- hostile-station 死亡爆出 wreck **不受 `WRECK_FRAGMENT_MAX_COUNT` cap** 限制（决战奖励语义优先）。
- 即使爆出后总 wreck > cap（如同关有 salvage 已 seed 8 wreck + hostile-station 爆 12 wreck = 20 wreck），允许临时高于 cap；下一关 `nextLevel` 内 `state.fragments = []` 时统一清理。

### §7.7 wreck 总价值估算

| level | wreck 数量 | 每 wreck cells 数 | 总 cells 数 | 玩家可桥接 cells（按桥接成功率 60-80%）|
|---|---|---|---|---|
| 3 | 6 | 2-5 | 12-30 | 7-24（绝大多数玩家可获 10+ cells）|
| 4 | 9 | 2-5 | 18-45 | 11-36 |
| 5 | 12 | 2-5 | 24-60 | 14-48 |

> **§7.7 备注**：玩家不一定能拾起全部 wreck（受时间 / 距离 / 拖动操作限制）；典型玩家在 60-90 秒内可桥接约 60-80% 的 wreck，估算 7-48 cells 进 station。

## §8. AI 静止参数

> 设计意图：hostile-station 完全静态悬停（不主动追击），但缓慢自转给视觉活力；不引入移动型敌方空间站的复杂度（v0.6.0 范围外）。

### §8.1 AI 参数表（ADR 拍板）

| 字段 | level 3 | level 4 | level 5 | 取值范围 | 设计理由 |
|---|---|---|---|---|---|
| accel | **4** | **4** | **4** | 3-5 | ADR 拍板：极小加速度；与 enemyUsesStationAi 470 距离悬停模板配合 |
| drag | **0.5** | **0.5** | **0.5** | 0.4-0.6 | ADR 拍板：高阻尼让漂移快速衰减 |
| angularVel | **0.15-0.20** | **0.15-0.20** | **0.15-0.20** | 0.10-0.30 | 缓慢自转（仿 fragment.angularVel）|
| 悬停目标距离 | **470** | **470** | **470** | 固定 | 沿用 enemyUsesStationAi 模板（与 station / guardian 一致）|
| 漂移上限 | **30** | **30** | **30** | ≤ 30 | ADR §2.5 缓解策略：实测如漂移 > 30 px 节点 A 报警 |

### §8.2 angularVel 精化

ADR 给出 angularVel 区间 0.1-0.3 rad/s；本方案精化为 **0.15-0.20 rad/s**（中位偏稳）：

```js
// spawnHostileStationNearStation 内：
enemy.angularVel = rngFloat(rng, 0.15, 0.20) * (rng() < 0.5 ? -1 : 1);
```

设计理由：

- 0.15 rad/s ≈ 8.6°/s（42 秒一圈）
- 0.20 rad/s ≈ 11.5°/s（31 秒一圈）

42 秒一圈让玩家可视感觉到自转但不头晕；31 秒一圈是上限，避免过快。正负随机让 BOSS"个性化"（每次 spawn 转向不同）。

### §8.3 漂移量验证

按 ADR §2.5 测算：

- accel 4 × dt 0.0167 = 0.067 px/s² 每帧加速度
- drag 衰减 `1 - min(0.5 × 0.0167, 0.3) = 0.992` 每帧速度衰减
- 实际峰值速度 ≈ 8 px/s（玩家在 470+ 距离时 BOSS 微推但立刻被 drag 吃掉）
- 1 秒内移动 ≈ 8 px（视觉静止）

如玩家拉远到 1000+ 距离（BOSS 持续推）：

- 持续 10 秒后 BOSS 位置漂移约 30-40 px（仍在容忍范围）
- 节点 A 实测如发现漂移 > 50 px，可调 accel 4 → 2 + drag 0.5 → 0.7

## §9. 体验目标对应的回归校验（共 27 条，超 ≥ 22 下限）

> 设计意图：本节是 PM 节点 A / B / C 全量回归与节点 C 接受测试的可量化校验点；按 §0.1 三类体验目标 + "不回潮"四类组织。

### §9.1 可感知 BOSS 决战（6 条）

1. **spawn 警报**：assault 任务 spawn hostile-station 时，HUD 显示红色警报"敌方空间站接近"，持续 4s 后自动隐藏；`state.run.hostileStationAlerted` 状态防重复触发。
2. **视觉规模**：hostile-station cells 总占地（按 §2.1 7 cells 3×3 网格）≥ 单 enemy（pirate r=20 / station r=42）的 3-5 倍；视觉直径 ≥ 96 像素（CELL × 3）。
3. **攻击多点**：2 个 turret cell 各自独立 reload + 各自 fire，可见两条弹道同时朝玩家发射；视觉上"多点开火"区别于 pirate 单弹道。
4. **生命池厚**：玩家武器 v0.5.0 完整 build（tier 2 turret + weaponMod ×1.05）在 level 3 击破 hostile-station 需 30-45 秒；裸船 60-90 秒。
5. **cells 逐块脱落反馈**：玩家命中某个非 core cell 后该 cell hp 减少；hp ≤ 0 后该 cell 从 `enemy.cells.delete` 移除；视觉上看到 BOSS"少一块"+ 命中粒子 [1, 0.42, 0.32, 0.9]。
6. **击毁庆祝**：core HP=0 触发死亡分支，相机震屏（72 振幅 1.3s）+ 48 个粒子（红黄混合）+ showToast"敌方空间站已摧毁——拾取残骸强化 build"。

### §9.2 可战术攻击决策（5 条）

7. **先打武器**：玩家先击破 turret-A（cell.hp ≤ 0）后，turret-A 立刻停火（该武器 cell 从 enemy.cells 删除）；BOSS 输出 DPS 减半（从 ~17.5 降至 ~8.75 在 level 4）。
8. **先打动力**：玩家击破 power cell 后，所有武器 cell `getCellStat(cell, "reload") × 1.5` 等效从 1.6s 增加到 2.4s reload；BOSS DPS 减少 ~33%。
9. **直击核心**：玩家集中输出 core cell（HP 160 / 235 / 320 按 level），可在 12-20 秒内速胜（如装甲 / shield 不挡 core 投射路径）。
10. **战术差异化**：同 build 下 "拆 turret-A + turret-B 后击 core" 与 "直击 core" 完成时间差 ≥ 30 秒（前者 60-90s / 后者 12-20s + 装甲拦截 = 30-40s）。
11. **研发隔离**：v0.5.0 研发系统（tier / mod / hullMod / weaponMod）对玩家武器有效（玩家 turret damage 7 → 9-12），对 hostile-station cells **无效**（敌方 turret damage 仍为 12/14/15 base，不被 weaponMod ×1.05 影响）；基于 `getCellStat` 内 `!cell.owner || cell.owner === "player"` 判断（ADR §4.5）。

### §9.3 可衔接奖励循环（5 条）

12. **wreck 立刻爆出**：core HP=0 触发 `triggerHostileStationDeath`，0.5 秒内生成 6/9/12 个 wreck fragment（按 level）；origin = "enemy-wreck"。
13. **wreck 可桥接**：玩家拖 enemy-wreck 回 station 桥接（沿用 v0.3.0 `mergeFragmentToStation`），cell 进入 `state.station.cells` 且 owner 字段被清除（默认玩家），享受 techLevel / hullMod。
14. **拾荒强化 build**：玩家桥接 enemy-wreck 后 station 多 turret / shield / armor / repair cells（按 §7.2 50/25/15/10 分布）；总新增 cells 估算 7-48（按 §7.7）。
15. **任务奖励发放**：assault 完成后 `grantObjectiveReward` 一并发放 metal base × rewardMultiplier × level 倍率 + research base × level 倍率（详 §5.2）；level 4 示例 84 metal + 12 research。
16. **下星系 build 更强**：玩家完成 assault + 拾荒后下星系 / 下一 level 时 build 明显增强（武器 cells 数量 +1-3、shield cells +0-2、armor cells +0-2）。

### §9.4 不回潮（11 条）

17. **v0.5.0 研发系统不破坏**：玩家 cell 走 getCellStat 原路径（`isPlayerOwned = !cell.owner || cell.owner === "player"` 默认 true）；turret damage / shield maxShield / armor maxHp 数值与 v0.5.0 完全一致（同 seed 跑 5 个 turret 命中比对）。
18. **v0.4.0 7 种任务保留**：mine / explore / battle / survive / guardian / salvage / escort 全部保留；level 0-2 任务不受影响；level 3-5 任务 weight 按 §6.1 平衡（30 seed 验证 ±10%）。
19. **v0.3.0 fragment 系统不破坏**：player / wreck origin 不变；新增 enemy-wreck origin 与之共存；玩家 fragment 脱落（origin=player）后桥接行为完全等价。
20. **v0.3.0 salvage 任务隔离**：salvage 任务的 `onFragmentDocked` 仅匹配 `origin === "wreck"`（不含 enemy-wreck）；assault + salvage 共存场景下 salvage 进度只统计 salvage 自身 seed 的 wreck。
21. **v0.2.x 三引擎手感不破坏**：玩家船体引擎 accel / drag / turn 数值不变；hostile-station 不影响玩家移动手感。
22. **v0.1.x SAVE_KEY 不变**：旧 meta 兼容；hostileStationSpawnedThisGalaxy / hostileStationAlerted 不写入 meta（仅 state.run）。
23. **guardian 在 level 6 不冲突**：level 6 仍出 guardian（weight 1.0）；不出 assault；guardian 数值 HP 1200 / spawnCap 4 不变。
24. **getCellStat 性能不退化**：敌方 cell.owner = "enemy" 时跳过 techFactor / globalFactor 计算（提前 return 1）；玩家 cell 路径无变化。
25. **WRECK_FRAGMENT_MAX_COUNT cap**：玩家正常 salvage 任务的 wreck 数仍受 cap（约 8）保护；hostile-station 爆出不应用 cap（决战语义）。
26. **state.run 字段重置**：nextLevel / resetRun 内显式重置 hostileStationSpawnedThisGalaxy = false 与 hostileStationAlerted = false（防跨关 / 跨 run 污染）。
27. **collideEnemyWithStation 与 hostile-station**：玩家撞 hostile-station 时玩家 cell 扣血（getCollisionDamage）+ enemy.hp 减 8（按 pirate 同款），但 hostile-station 的 enemy.hp 是聚合值，单次撞击不会触发 BOSS 死亡（cells 仍 hp > 0）。

> **§9 校验条数：27 条**（≥ 22 下限）。

## §10. 平衡兜底

> 设计意图：本节是数值方案的"安全网"——即使具体数值微调失误，硬上限保护体验不爆炸。

### §10.1 总 HP 池硬上限

```js
const HOSTILE_STATION_MAX_TOTAL_HP = 3500;
// spawnHostileStationNearStation 内：
let totalHp = HOSTILE_STATION_LEVEL_HP[state.run.level] || 1500;
totalHp = Math.min(totalHp, HOSTILE_STATION_MAX_TOTAL_HP);
```

即使数值方案微调或人为配置错误，hostile-station 总 HP 不超过 3500（约 guardian 1200 的 3 倍）；防止"无法击毁"卡关。

### §10.2 单 cell HP 硬上限

```js
const HOSTILE_STATION_MAX_CELL_HP = 700;
// createHostileStationCell 内：
cell.hp = Math.min(cell.maxHp, HOSTILE_STATION_MAX_CELL_HP);
```

防止单 cell HP 配置过高（如 armor 误配 1000）让玩家"打一个 cell 打 30 秒"。

### §10.3 武器 damage 硬上限

```js
const HOSTILE_STATION_MAX_WEAPON_DAMAGE = 15;
// getCellStat(cell, "damage") 返回值兜底：
return Math.min(damage, HOSTILE_STATION_MAX_WEAPON_DAMAGE);
```

不超过玩家 turret base 14 × 任何 multiplier，避免"一击必死玩家"；level 5 BOSS turret 15 是上限。

### §10.4 召唤 minion 数量硬上限

```js
const HOSTILE_STATION_MAX_SPAWN_CAP = 5;
// updateEnemies hostile-station 分支内：
const minions = state.enemies.filter((m) => m.hostileStationMinion && m.hp > 0).length;
if (minions >= HOSTILE_STATION_MAX_SPAWN_CAP) return;  // 硬上限
```

防止数值方案微调让 spawnCap 误配 >5；level 5 上限 5 是最高值。

### §10.5 wreck 数量硬上限

```js
const HOSTILE_STATION_MAX_WRECK_COUNT = 12;
// spawnEnemyWrecks 内：
let count = HOSTILE_STATION_WRECK_COUNT_BY_LEVEL[level] || 6;
count = Math.min(count, HOSTILE_STATION_MAX_WRECK_COUNT);
```

即使数值方案放宽，wreck 不超过 12 个，避免性能问题（12 wreck + 玩家 fragments + minion 同帧渲染 cap 已经测试过）。

### §10.6 单星系上限

```js
if (state.run.hostileStationSpawnedThisGalaxy) return null;  // 1 次硬上限
```

确保单星系 hostile-station 出现 1 次；防止 assault 任务被同 galaxy 内重复 spawn。

### §10.7 玩家研发不足时的挑战度

| 玩家状态 | level 3 击毁时间 | level 4 击毁时间 | level 5 击毁时间 |
|---|---|---|---|
| 裸船（无研发） | 60-90 秒 | 90-120 秒 | 120-180 秒（可能超时）|
| 研发完整（tier 2 + 1 mod + 1 全局加成）| 30-45 秒 | 45-60 秒 | 60-90 秒 |
| 研发顶配（tier 3 + 2 mod + 2 全局加成）| 20-30 秒 | 30-45 秒 | 45-60 秒 |

> **§10.7 备注**：裸船在 level 5 击毁时间可能超过 §5.4 软限时 6 分钟，但因为不引入硬 isFailed，玩家不会失败；只是奖励发放延迟。如 PM 节点 C 发现"裸船卡 level 5 BOSS"严重，可在 v0.6.1 引入 isFailed 或调低 level 5 HP 池到 2700。

### §12.2 待制作人 / 主程拍板的开放决策点

以下决策点本方案给出推荐选项，但**最终由制作人 / 主程决定**（实现前需明确）：

1. **minion 是否绕站旋转**（vs 直接索敌玩家）：
   - 本方案推荐：**直接索敌玩家**（与 ADR §2.6 拍板一致）。
   - 节点 B 体验测试若发现"minion 立刻冲玩家显得突兀"，可在 v0.6.1 引入"出生 2-3s 内绕站旋转"过渡 AI。

2. **wreck 是否含 missile 设施**：
   - 本方案推荐：**不含**（v0.6.0 范围控制，hostile-station 配置不含 missile cell，wreck 分布仅 turret/shield/armor/repair）。
   - missile cell 涉及 `state.resources` 检查与玩家 missile 池兼容性，留 v0.7+。

3. **assault 任务限时是否引入 isFailed**：
   - 本方案推荐：**不引入硬 isFailed**（与 ADR §11 #2 一致）；§5.4 软限时仅作监控参考值。
   - 节点 C 体验测试若发现"裸船卡 BOSS 超过 10 分钟"，可在 v0.6.1 引入 isFailed 或下调 level 5 HP 到 2700。

4. **power cell 破坏效果**（reload +50% vs 完全停火）：
   - 本方案推荐：**reload +50%（×1.5）**（§2.5 详述），避免"先打 power 必赢"破坏战术多样性。
   - 节点 B 体验测试若玩家反馈"power 破坏后 BOSS 反而更难感知到削弱"，可考虑改为 ×1.8 或完全停火。

5. **shield cell 拦截范围**（110-150 像素 vs 更大）：
   - 本方案推荐：**110/130/150**（§3.4 + §2.6），让 shield 是"近防"语义；玩家在 shield 外射击不受拦截。
   - 节点 B 测试若发现"shield 拦截太弱"或"太强"，可在 §11.1 ±20% 区间内微调。

6. **hostile-station 击毁是否给 metal 直接奖励**：
   - 本方案推荐：**reward = 0**（与 ADR §11 #1 一致），所有奖励通过 wreck 拾荒 + assault 任务 reward。
   - 节点 C 若发现"击毁后即时反馈不足"，可在 v0.6.1 引入 reward 30 metal。
