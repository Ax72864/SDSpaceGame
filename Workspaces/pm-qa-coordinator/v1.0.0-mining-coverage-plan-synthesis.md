## 三方共同点

1. 都承认 `v0.20.0` 只证明了诊断能力，没有关闭采矿收益闭环；1.0 仍需要真实输入下取得 `harvestingCount > 0` 或等价可归因收益证据。
2. 都同意默认层应尽量不触碰高风险规则：采矿范围 / 公式、资源布局、成本、存档 / Meta schema、移动物理参数都应先保持不变。
3. 都认为玩家卡点不只是“距离不够”，还包括金属缺口不可见、候选路径不可操作、移动 + 重建路径不清楚、小屏可读可点风险。
4. 都要求采矿 UI / 诊断必须把玩家下一步讲清楚：是否产出、为什么不产出、能否扩到外环、需要几步框架或是否应移动工站。
5. 都要求 fallback 涉及数值、资源布局或移动能力时必须走项目高风险流程，由制作人决策后才能进入实现。

## 三方主要分歧

### 默认方案组合

- 数值方案主推 `A`，即先做引导与诊断反馈；把 `B` 首关资源布局保障作为备选。
- 主程 ADR 主推默认层 `A + E + F`，即金属缺口诊断、ghost path、移动 + 重建教学的组合，并明确不动核心规则。
- UI 方案主推 `A + B + C 的最小子集`，这里的 B/C 是 UI 分层：状态提示与只读诊断可视化，不是数值方案的资源布局 `B`。

PM 解释：三方的可合并共识是“默认层做诊断 + 候选路径可视化 + 移动重建教学”。命名上以主程 ADR 的 `A + E + F` 作为本看板默认层，UI 方案中的状态 chip、最佳候选和低噪音原则作为实现时 UI 验收约束。

### fallback 顺序

- 数值方案建议：`A` 失败后优先考虑 `B` 首关资源布局保障，也提到可叠加 `D1` 起始金属补救。
- 主程 ADR 建议：默认层失败后先 `D-d tutorial buff`，再考虑更窄的数值微调，最后才升级 `B` 首关资源布局保障。
- UI 方案不主张直接 fallback 到规则或数值变更，只强调 UI 不能把不可支付路径包装成可完成路径。

PM 裁剪：为降低 1.0 风险，本汇总只保留两个 fallback 候选进入制作人决策：`D-d tutorial buff` 与 `B 首关资源布局保障`。二者都不是默认实施内容；触发后必须重新走高风险流程。

### 触发门槛

- 主程 ADR 中出现过“60 秒内仍无法取得收益”或“真实玩家路径失败率”的技术触发建议。
- 看板与本次 PM 口径采用更保守、更易验证的 gate：真实输入连续 3 次新档，在 120 秒内仍无法取得 `harvestingCount > 0`。

PM 裁剪：#3 阶段先使用“3 次新档 / 120 秒 / 真实输入 / `harvestingCount > 0`”作为 fallback 触发条件；后续 #5/#6 可补充小屏、协议和收益归因细则。

## 默认推荐

PM 默认推荐：先执行低风险默认层 `A + E + F`。

默认层内容：

- `A`：金属缺口诊断。把现有诊断从“还差距离”扩展为玩家可理解的金属 / 矿站 / 路径缺口说明。
- `E`：候选路径或 ghost path。只展示最佳候选路径或目标格，避免候选列表过载；文案必须区分“建议路径”和“当前资源可支付路径”。
- `F`：移动 + 重建教学。把“驾驶工站靠近资源外环，再重建采矿站”作为并列路径讲清楚，但不修改移动物理参数。

默认层禁止内容：

- 不改采矿范围 / 公式。
- 不改资源布局或首关 seed。
- 不改 `frame`、`mining`、`thruster` 等成本。
- 不改起始资源。
- 不改存档 / Meta schema。
- 不改 `PLAYER_PHYSICS`、推进器推力或移动物理参数。
- 不把 CDP / hook 证据替代真实输入体验结论。

## fallback 建议

fallback 触发条件：

- 默认层实现并完成基础自检后，PM 用真实输入连续 3 次新档复测。
- 若每次新档在 120 秒内仍无法取得 `harvestingCount > 0` 或等价可归因收益证据，则默认层不足，进入 fallback 决策。

fallback 候选：

1. `D-d tutorial buff`：首次玩家 / 首局教程补助，作为数值类高风险 fallback。必须明确触发条件、是否只对首次玩家生效、是否复用现有字段、是否影响老玩家和 v0.10 协议价值。
2. `B 首关资源布局保障`：保证首关附近有可达资源，作为资源布局类高风险 fallback。必须明确只作用于首关或新档的边界、seed / 视觉签名影响、旧证据回归方式。

fallback 流程要求：

- fallback 不随默认层自动实施。
- fallback 必须按高风险流程执行：制作人方向 -> PM 拆分 -> 专项方案 -> 执行自检 -> 专项审核 -> PM 汇总 -> 制作人最终决定。
- 若制作人不保留 fallback，则 #4 只允许实施默认层，并把失败风险写入 #5/#6 复测结论。

## 制作人需要决定的问题

1. 是否批准默认层 `A + E + F` 进入 #4 实施与自检。
2. fallback 是否保留：若默认层真实输入失败，是否允许进入 `D-d tutorial buff` 或 `B 首关资源布局保障` 的高风险流程。
3. fallback 优先级：若保留，先评估 `D-d tutorial buff`，还是直接评估 `B 首关资源布局保障`。
4. 哪些采矿覆盖内容列入 1.0 必过 gate：
   - 是否必须真实输入 120 秒内取得 `harvestingCount > 0`。
   - 是否必须证明目标资源 `body.amount` 下降或收益来源可归因。
   - 是否必须覆盖桌面鼠标与小屏触控。
   - 是否必须覆盖空档 + balanced 协议，是否扩展到 `miningStart`。
5. 是否明确否决 1.0 阶段修改采矿范围、frame 成本、整体起始资源、推进器成本或移动物理参数。

## 建议纳入 1.0 必过 gate

PM 建议将以下内容列入 1.0 必过 gate：

- 空档 + balanced 协议下，真实鼠标输入在 120 秒内取得一次 `harvestingCount > 0` 或等价可归因收益。
- 采矿成功时，PM 证据能说明收益来源，不把自然资源增长误判为采矿。
- 玩家能在 UI 中看懂采矿站当前是否产出、失败原因、下一步是扩建 ghost path 还是移动 + 重建。
- 小屏触控或等价触控模式能完成同一采矿主路径，且记录遮挡、误触、资源外环可见性。
- 默认层实现不得改变采矿范围 / 公式、资源布局、成本、存档 / Meta schema 或移动物理参数。
- fallback 若被启用，必须有专项审核结论和制作人再次批准。

## PM 结论

看板 #3 已具备提交制作人决策的条件。PM 建议制作人批准默认层 `A + E + F` 进入 #4 实施与自检，同时保留但不自动启用 fallback：若真实输入连续 3 次新档在 120 秒内仍无法取得 `harvestingCount > 0`，再进入 `D-d tutorial buff` 或 `B 首关资源布局保障` 的高风险流程。

在制作人批准前，#4 不应开始任何代码实现；尤其禁止先改采矿范围、公式、资源布局、成本、存档、Meta schema 或移动物理参数。
