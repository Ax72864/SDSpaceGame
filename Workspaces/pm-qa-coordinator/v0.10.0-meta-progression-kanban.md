版本名：`v0.10.0-meta-progression`

制作人方向 commit：`52ae80d`

## 本轮目标

把当前局外点数从“几个加数值按钮”升级为可规划的持久天赋树与开局策略选择。玩家完成一局后，无论失败还是通关，都能知道本局获得了多少局外成长、现在能买什么、下一局适合尝试哪条路线。

v0.10.0 不追求新增大类战斗内容，而是把 v0.1-v0.9 已有的局内建造、任务、研发、战斗和星系路径接到一个清晰的跨局成长闭环上。

## 背景与制作人方向摘要

- 当前代码已有简化 meta：`SAVE_KEY = "star-ring-station-meta-v1"`、`state.meta.points`、`mining / hull / weapons` 和 `unlocks.weaponFrame / efficientCore`。
- 制作人判断现有局外成长方向成立，但体验仍停留在几个按钮，缺少前置关系、节点等级、路线规划、开局协议、结算引导和存档迁移策略。
- v0.10.0 核心是保留玩家已有点数和旧字段价值，把旧 meta 升级为正式 `META_TALENT_TREE` 注册表、`talents` ranks、`selectedStartProtocol` 和集中效果 helper。
- 首批体验目标：失败也有去处、路线选择明确、开局协议可感知、局内研发和局外天赋边界清楚、不污染 v0.9 galaxyMap。
- 制作人明确不做联机、异步对战、新大类建筑、新 enemy kind、新关卡系统、完整剧情、云存档或付费经济系统。

## 风险定级

- 风险等级：项目高风险。
- 原因 1：涉及 `SAVE_KEY`、旧 meta 字段迁移、玩家点数保留和持久化 schema，任何误判都可能导致点数丢失或旧能力失效。
- 原因 2：新增 `META_TALENT_TREE`、效果 helper、开局协议和购买逻辑，会跨采矿、耐久、武器、科研、结算和 UI 多个系统。
- 原因 3：跨局成长数值一旦过强，会压扁 v0.1-v0.9 的首局难度、研发节奏、星系路径风险和战斗压力。
- 原因 4：Meta UI、协议选择和结算提示都要放进现有单页 UI，信息架构不清会造成玩家读不懂、旧按钮重复或局内 / 局外边界混淆。
- 原因 5：v0.9 `galaxyMap` 是局间路径，v0.10 Meta 是局外长期成长，两者若互相改写状态机会污染已完成系统。
- 高风险流程：制作人方向 -> PM 看板 -> ADR / 数值 / UI 方案 -> 分批实现 -> 专项审核 -> PM 回归 -> 制作人最终决定。

## QA 回归点清单

### 存档与迁移

- 旧 localStorage 中只有 `points / mining / hull / weapons / unlocks` 时，进入游戏后自动补齐 `talents / selectedStartProtocol / migrationVersion`。
- 旧 `points` 数值迁移后不减少、不清零、不被重复结算覆盖。
- 旧 `mining / hull / weapons` 映射到对应新 talent rank，刷新后 rank 保持。
- 旧 `unlocks.weaponFrame / unlocks.efficientCore` 映射到对应解锁节点，刷新后仍可用。
- 旧字段和新 `talents` 同时存在时，效果只按 helper 口径计算一次，不产生双倍收益。
- 空存档首次进入游戏可正常生成默认 meta，默认协议为 `balanced`。
- 损坏或缺字段 meta 不导致游戏启动崩溃，点数保护策略符合 ADR。
- `SAVE_KEY` 策略符合 ADR：保留原 key 或执行明确迁移，浏览器 localStorage 中无幽灵 key 造成的读写混乱。

### 天赋购买与效果 helper

- 节点购买消耗正确点数，点数不足时不可购买且提示明确。
- 前置不足时不可购买，提示缺少的节点和 rank。
- `maxRank` 达到上限后不可继续购买，不重复扣点。
- 多级成本按数值方案计算，刷新后当前等级和剩余点数一致。
- `getMetaTalentRank / hasMetaTalent / getMetaEffect` 或等价 helper 能返回正确值。
- 采矿、耐久、武器、科研、初始资源、解锁项等效果均通过 helper 接入。
- 购买后 UI、存档和 helper 查询结果一致。

### 开局协议

- 默认 `balanced` 协议无需解锁，空存档可正常开局。
- 采矿起步、防御起步、武器起步至少 3 个协议按对应天赋解锁。
- 未解锁协议不可选择，原因提示明确。
- 同一局只能选择一个协议，选择后刷新仍保持。
- 新局开始时协议按 ADR 时机生效，局中不能临时切换改变当前局状态。
- 协议收益符合数值方案，不绕过核心采集、建造、研发、战斗或星系路径。

### 结算与玩家引导

- 失败结算显示本局获得局外点数、当前总点数、可购买推荐或差多少点。
- 通关结算显示本局获得局外点数、当前总点数、可购买推荐或下一局建议。
- 结算推荐基于真实购买状态，不推荐已满级或前置不可能满足的节点。
- 结算后进入 Meta 面板能看到点数变化并可购买符合条件的节点。
- 重启浏览器后结算获得点数、已购节点和已选协议仍持久化。

### 旧系统边界

- v0.5 研发系统仍是本局内临时成长，结算后不被写入局外 meta。
- UI 文案不把 research / 研发称为局外永久天赋，不把 Meta 节点称为局内研究。
- v0.9 `galaxyMap` 的路径、候选、跃迁状态机不被 Meta 购买或协议选择改写。
- v0.9 星系选择仍能正常打开、取消、确认和记录路径。
- v0.1-v0.4 基础采集、建造、任务、敌人波次和奖励主路径无回潮。
- v0.6 hostile-station、v0.7 锁敌 / 齐射、v0.8 随机事件主路径无回潮。

### 浏览器与 localStorage

- Chrome / Edge 至少一个真实浏览器完成空存档、旧存档、购买、刷新、结算、重启主路径。
- localStorage 手动注入旧 meta 后，迁移结果可通过测试钩子或 UI 观察验证。
- 清空 localStorage 后新玩家路径可正常开局、结算和购买。
- 多次刷新不重复迁移、不重复赠点、不重复扣点。
- 控制台无启动错误、购买错误、结算错误和未捕获 promise / runtime error。
