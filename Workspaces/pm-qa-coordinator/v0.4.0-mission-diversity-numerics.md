## §0. 制作人方向回放与硬约束

### §0.1 体验目标三条（必须命中）

1. **可探索**：玩家进入星系时不知道任务类型，需要看 HUD / 引导识别；同一关多次重启不一定是同一任务（任务选择参考 seed）。
2. **可参与**：拾荒任务给"在星系内搜寻老旧残骸 → 靠近桥接回来"明确动作；护送任务给"保护一艘缓慢移动的友方货船"明确动作。
3. **可重玩**：完成一种新任务后，下一局可能遇到完全不同任务，给玩家"再开一局看看下一关是啥"的动机。

### §0.2 不做范围（数值层硬约束）

- 不修改 v0.3.0 `FRAGMENT_*` 物理常量（launch / damp / max speed）与 `BRIDGE_DISTANCE / BRIDGE_ANGLE / FRAGMENT_PROJECTILE_HIT_RADIUS`。
- 不修改 `SAVE_KEY = "star-ring-station-meta-v1"` 与局外存档结构；本数值方案所有新增字段写入 `state.run.*` / 新建 `state.npcs`，不写入 `state.meta`。
- 不修改 level 0 首局 60 秒减压、初始资源 `130/60/35/8/0`、旧 meta 兼容路径。
- 不修改局外点数算法的底层基础公式 `2 + state.run.level`（仅在外层乘以任务类型倍率）。
- 不引入新的设施类型、新武器、新建造资源、新维修类型；wreck fragment 内设施完全复用现有 `turret / shield / armor / repair`。
- 不引入并行多任务（单 `state.run.objective` 结构）。

### §0.3 数值方案对制作人方向的回复

针对方向文档 §"关键技术疑问 / 数值策划"列出的 9 项参数请求，本方案逐条给出推荐值与取值范围（详见 §1-§5）。所有推荐值都落在方向文档建议区间内：

| 方向文档建议 | 本方案落地 | 对应章节 |
|---|---|---|
| 拾荒目标 X（level 0-2 X=2，level 3-5 X=3） | level 1-2 X=2 / level 3-5 X=3（level 0 不出 salvage，减压期） | §2.1 |
| 拾荒残骸数量 N（X+1 到 X+2） | level 1-2 N=3 / level 3-5 N=4 | §2.1 |
| 残骸速度（玩家脱落 30%-50%） | launchSpeed 7-14（玩家 22-36 的 32%-39%） | §2.2 |
| 残骸设施 HP（60%-90%） | 65%-90%（按设施类型微调） | §2.3 |
| 护送 NPC HP / 速度 | HP 200/250/300 阶梯 / 速度 35-45 px/s（玩家 ~100-130 的 30%-40%） | §3.1 §3.2 |
| 护送 spawn 倍率（保持或略降） | level 1-2 0.8× / level 3-4 1.0× / level 5 1.1× | §3.3 |
| 任务选择权重表 | level 0-6 全表（§1） | §1 |
| 拾荒奖励倍率（比 mine / explore 高） | 1.5× | §5 |
| 护送奖励倍率（比 battle 高） | 1.6× | §5 |

## §1. 任务类型权重表

> 设计意图：保持 v0.3.0 节奏曲线（level 0-2 减压 / 3-4 上升 / 5-6 紧张），同时让 seed 化任务选择在每个 level 给玩家"可能遇到 2-3 种任务"的探索感。终局 level 6 锁 guardian，避免随机性破坏终局体验。

### §1.1 各 level 权重表

实现约束：权重表存放在 `OBJECTIVE_TYPES` 注册表外的常量（建议 `OBJECTIVE_LEVEL_WEIGHTS`），由 `createObjective()` 用 `state.run.seed` + level 派生 RNG 抽样；接口细节由 ADR §"任务选择 seed 化"决定。本方案只给参数。

| level | 节奏定位 | 权重表 | 权重和 |
|---|---|---|---|
| 0 | 减压（新玩家友善期） | `{ mine: 0.5, explore: 0.5 }` | 1.0 |
| 1 | 减压尾段 + 引入拾荒 | `{ mine: 0.4, explore: 0.35, salvage: 0.25 }` | 1.0 |
| 2 | 战斗试水 | `{ mine: 0.3, explore: 0.3, battle: 0.2, salvage: 0.2 }` | 1.0 |
| 3 | 压力上升 + 引入护送 | `{ battle: 0.30, survive: 0.20, salvage: 0.25, escort: 0.25 }` | 1.0 |
| 4 | 压力上升续 | `{ battle: 0.30, survive: 0.20, salvage: 0.25, escort: 0.25 }` | 1.0 |
| 5 | 紧张（去 salvage，强战斗 + 护送） | `{ battle: 0.40, survive: 0.30, escort: 0.30 }` | 1.0 |
| 6 | 终局守护者 | `{ guardian: 1.0 }`（不参与抽样） | 1.0 |

设计要点：

- **level 0 不出 salvage / escort**：保持 v0.3.0 首局 60 秒减压不回潮（首次接触 wreck fragment 桥接预览会增加新玩家认知负担）。
- **level 1 引入 salvage**：选择 level 1 作为拾荒首秀，因为玩家在 level 0 已熟悉 mine + explore，可以承担"搜寻并桥接古老残骸"的新动作。
- **level 3 引入 escort**：护送需要更强的战斗经验，放在压力上升期玩家已经具备护卫意识。
- **level 5 去 salvage**：紧张期玩家应该集中精力打硬仗，拾荒的"慢搜寻"节奏与紧张感冲突；保留 escort 是因为护送本身就是高紧张。
- **level 6 锁定 guardian**：终局体验固定，不参与 seed 化抽样；`createObjective()` 中已有 `state.run.level >= ENDGAME_LEVEL` 早返回路径（`Game/game.js:1076`），seed 化只在该路径之后启用。

### §1.2 任务类型 cooldown（避免连续重复）

| 参数 | 推荐值 | 取值范围 | 说明 |
|---|---|---|---|
| `OBJECTIVE_TYPE_COOLDOWN` | 1 level | 0-2 level | 同一类型不连续 2 关出现（即 level N 抽到 type X，则 level N+1 抽样时把 type X 权重置 0 后重新归一化） |
| guardian 豁免 | 不参与 cooldown | 固定 | guardian 仅在 level 6 出现，本身就唯一 |
| 权重表只有 1 种类型时豁免 | 不参与 cooldown | 固定 | level 0 当前权重表只有 mine / explore，若 cooldown 后只剩 1 项，直接选剩下那项 |

接口形式由 ADR 决定（可选实现：`createObjective()` 维护 `state.run.lastObjectiveType`，抽样前过滤；或在 `pickNextForObjective` 内做权重归一化）。本数值方案给参数：`cooldown = 1 level`。

### §1.3 权重表如何与 v0.3.0 节奏共存

- level 0-2 总难度由 spawn 系统（v0.2.0 `levelTimerScale / countMul`）独立控制，不受任务类型影响。
- salvage / escort 不修改 spawn 倍率（除 §3.3 escort 内部微调）。
- 任意 level 在 cooldown 过滤后若所有权重为 0（极端情况，理论不应发生），fallback 到 `mine`（与 level 0 默认一致，确保不出空任务）。

## §2. 拾荒任务（salvage）参数

> 设计意图：让玩家"在星系内搜寻 → 靠近 → 桥接老旧残骸"，复用 v0.3.0 桥接系统，给玩家"赚到了"的奖励感（设施已升级 + 高 HP），但仍需要后续维修。fragment 物理参数明显低于玩家脱落 fragment，让"古老"感觉自然。

### §2.1 每级目标数量与预生成数

| level | 目标 X（需桥接的 wreck fragment 数） | 预生成 N（每星系 wreck fragment 总数） | 余量（N - X） |
|---|---|---|---|
| 1 | 2 | 3 | 1（玩家可挑选） |
| 2 | 2 | 3 | 1 |
| 3 | 3 | 4 | 1 |
| 4 | 3 | 4 | 1 |
| 5 | 3 | 4 | 1（保持 3，避免与紧张感冲突） |

设计要点：

- **余量 = 1**：让玩家有"找最好的"选择空间（例如挑设施数最多 / 位置最近的桥接），但不至于"随便桥接 1 个就完事"。
- **level 5 X=3 而非 4**：紧张期不应让拾荒目标无限拉长。
- **level 0 / 6 不出 salvage**：见 §1.1。

### §2.2 wreck fragment 物理参数

> 实现约束：wreck fragment 共用 v0.3.0 `state.fragments` 数组，但通过 `origin = "wreck"` 字段区分。物理参数在 spawn 瞬间一次性写入；之后由现有 fragment 物理 tick 自然处理（阻尼 / 边界销毁）。`FRAGMENT_*` 共享常量不动，只在 spawn 瞬间用本表的覆盖值。

| 参数 | 推荐值 | 单位 | 取值范围 | 对比玩家脱落 fragment | 说明 |
|---|---|---|---|---|---|
| `WRECK_LAUNCH_SPEED_MIN` | 7 | px/s | 5-10 | 22（32%） | 落入方向文档 30%-50% 区间下沿 |
| `WRECK_LAUNCH_SPEED_MAX` | 14 | px/s | 10-18 | 36（39%） | 同上 |
| `WRECK_LAUNCH_JITTER` | 4 | px/s | 3-6 | 12（33%） | 让多 wreck 之间不同向重叠 |
| `WRECK_ANGULAR_VEL_MIN` | -0.3 | rad/s | -0.45 ~ -0.20 | -0.6（50%） | 50% 玩家脱落角速度 |
| `WRECK_ANGULAR_VEL_MAX` | 0.3 | rad/s | 0.20 ~ 0.45 | 0.6（50%） | 同上 |
| `WRECK_ANGULAR_JITTER` | ±0.08 | rad/s | ±0.05 ~ ±0.12 | ±0.15（53%） | 同上 |
| 阻尼 `linearDamp` | 0.06 | per-frame coef | 固定 | 与玩家脱落一致 | 复用 `FRAGMENT_LINEAR_DAMP` |
| 阻尼 `angularDamp` | 0.35 | per-frame coef | 固定 | 与玩家脱落一致 | 复用 `FRAGMENT_ANGULAR_DAMP` |
| `FRAGMENT_MAX_LINEAR_SPEED` | 180 | px/s | 固定 | 与玩家脱落一致 | 共享上限 |
| `FRAGMENT_MAX_ANGULAR_SPEED` | 2.4 | rad/s | 固定 | 与玩家脱落一致 | 共享上限 |

体感预期：wreck fragment 净速度约 7-14 px/s（约 0.25-0.5 cell/s），3 秒后视觉位移 ~20-40 px（< 1.5 cell），玩家可以从容靠近；角速度 0.3 rad/s ≈ 17°/s，3 秒转 ~50°（远低于桥接角窗 25°），玩家"等转角"等待时间约 1-3 秒。

### §2.3 wreck fragment 内部设施

| 项 | 推荐值 | 取值范围 | 说明 |
|---|---|---|---|
| cells 数量 / fragment | 2-5 cells | 2-6 | 避免过大或过小；过小（1 cell）视觉上像是"碎屑"不像残骸；过大（>6）会占用太多 `FRAGMENT_MAX_CELLS_PER` 预算 |
| 设施类型分布 | turret 60% / shield 20% / armor 15% / repair 5% | 见下 | 保留惊喜（5% 几率得 repair） |
| HP 占 maxHp 比例 | 65%-90% 随机 | 60%-95% | 让玩家觉得"赚到了"（不是满血，但也不是濒死） |
| frameHp 占 70 的比例 | 80%-100% 随机 | 70%-100% | frame 部分相对完整，避免桥接后 frame 立刻塌 |
| reload / fire | 0 | 固定 | 桥接瞬间复位，避免桥接后立即齐射（与 v0.3.0 ADR §2.5 一致） |
| enabled | true | 固定 | 桥接后立即可工作 |
| priority | 与对应设施现有默认值一致 | 固定 | 不引入新优先级机制 |

设施类型分布说明：

| 类型 | 权重 | 设计理由 |
|---|---|---|
| turret | 0.60 | 拾荒主要奖励是火力（玩家最常缺火力） |
| shield | 0.20 | 二级奖励（玩家中后期缺生存能力） |
| armor | 0.15 | 三级奖励（armor 在站体内本就稀缺） |
| repair | 0.05 | 稀有惊喜（v0.3.0 维修无人机在多 fragment 时压力大，repair 设施供给较紧张） |

> 实现注意：单 fragment 内的 cells 设施分布建议按权重独立抽样（不强制每个 fragment 都有 turret）。若某 fragment 抽到全 shield / armor，玩家仍会因为"高 HP 设施回归"获得奖励感。

### §2.4 wreck fragment 位置与数量上限

| 参数 | 推荐值 | 取值范围 | 说明 |
|---|---|---|---|
| 距 station 起点距离窗口 | 200-600 | 150-800 | 太近（< 150）则没有"搜寻"过程；太远（> 800）则玩家可能直接放弃 |
| 避开 planet 半径缓冲 | + 50 px | 30-80 | 不与 planet 视觉重叠 |
| 避开 asteroid 半径缓冲 | + 30 px | 20-50 | 避免 spawn 瞬间被弹道集中 |
| wreck 之间最小距离 | 100 px | 80-150 | 避免视觉聚集，让玩家"搜寻"过程有空间感 |
| wreck fragment 独立 cap | 不计入 `FRAGMENT_MAX_COUNT = 8` | 固定 | wreck 单独计数，独立 cap = N（每 level 最多 4） |
| 实际可见 cap | N（wreck） + 8（玩家脱落） = 12 max | 固定 | 远低于 v0.3.0 `DETACHED_CELLS_TOTAL_HARD_CAP = 120`（4×5 + 8×32 ≈ 276 理论上限，但因玩家脱落和 wreck 同时打满概率极低，实际峰值 < 80 cells） |
| 单 fragment cells 上限 | 仍受 `FRAGMENT_MAX_CELLS_PER = 32` 限制 | 固定 | wreck 设计 2-5 cells，远低于上限 |
| spawn 失败处理 | 重试 5 次，仍冲突则降级 N | 固定 | 例如目标 N=4 但只成功 spawn 3 个，则该 level 实际 N=3，X 同步降到 min(X, N) |

> **关键设计**：wreck fragment 数独立 cap 但共享 `state.fragments` 数组，意味着 v0.3.0 fragment 总数硬上限 `FRAGMENT_MAX_COUNT = 8`（`Game/game.js:1910`）只对玩家脱落计数，wreck 不挤占。实现细节由 ADR 决定（建议在 `state.fragments.filter(f => f.origin !== "wreck").length >= 8` 时才触发降压）。

### §2.5 wreck fragment 在世界中的兼容性

| 兼容项 | 处理 |
|---|---|
| 维修无人机 | **不修 wreck fragment cell**。继承 v0.3.0 §6 决策：fragment cell（无论 origin）不在 `state.station.cells` 中，天然不进入维修候选。 |
| 玩家弹道 | 不命中 wreck（与 v0.3.0 玩家弹道 vs fragment 行为一致）。 |
| 敌方弹道 | 命中 wreck cell，按 v0.3.0 §3.6 规则结算 HP；wreck cell 销毁后 fragment 仍可桥接剩余 cells。 |
| 桥接判定 | 复用 `BRIDGE_DISTANCE = 36 / BRIDGE_ANGLE = 25°`；wreck fragment 与玩家脱落 fragment 在桥接逻辑上完全等价。 |
| 桥接成功后 | wreck 标记移除（合并回 station 后的 cells 不再保留 `origin` 标记）；任务计数 +1（见 §6 校验 #拾荒）。 |
| 跨 level | `nextLevel` 强制清空 `state.fragments`（v0.3.0 既有行为），wreck 在新 level 重新预生成。 |

### §2.6 拾荒任务完成时间窗（兜底）

| 参数 | 推荐值 | 取值范围 | 说明 |
|---|---|---|---|
| 默认时间窗 | 8 分钟（480 s） | 6-10 分钟 | 玩家正常完成约 2-4 分钟；8 分钟兜底宽松，仅防"玩家彻底不参与" |
| 超时触发 | 一次性 toast `残骸已散失，本关任务失败。` | 固定 | 任务失败，但玩家仍可留在本关自由游玩 / 重启 run |
| 超时处理 | 任务 fail（与护送失败选项 B 一致：留在本关自由游玩 / 无法跃迁） | 固定 | 见 §3.5 |
| 超时与 cooldown 关系 | wreck fragment 仍在世界中（不主动销毁），玩家可继续桥接但不计任务进度 | 固定 | wreck 仍受边界销毁 / 总 cap 等 v0.3.0 兜底约束 |

## §3. 护送任务（escort）参数

> 设计意图：让玩家进入星系时同时面对"自己的生存"和"NPC 的生存"，提升参与感；NPC 缓慢移动 + 高 HP 给玩家"可以救得回来"的安全感，但敌方倾向于打 NPC，让玩家必须主动拦截。死亡按选项 B（任务失败 + 留在本关自由游玩 + 无法跃迁），避免"NPC 死=直接 gameOver"的挫败感。

### §3.1 NPC 货船 HP / maxHp

| level | HP / maxHp | 取值范围 | 说明 |
|---|---|---|---|
| 1 | 200 / 200 | 180-220 | 落入方向文档 200-300 区间下沿，新玩家友善 |
| 2 | 200 / 200 | 180-220 | 同上 |
| 3 | 250 / 250 | 230-280 | 上升期玩家应能承担更多保护责任 |
| 4 | 300 / 300 | 280-320 | 接近上限，但任务期间 spawn 倍率 1.0×，敌人压力同步上升 |
| 5 | 300 / 300 | 280-320 | 紧张期 spawn 1.1×，HP 不再继续上调 |

设计要点：

- HP 阶梯与 v0.2.0 难度曲线（level 3-4 上升 / 5 紧张）匹配。
- NPC HP 数量级与中等敌人 station（v0.2.0 数值表中 station 大致 150-250 HP）相当，玩家可以理解"NPC 比一般敌人能扛但不无敌"。
- 不超过 300 是为了让 NPC 不会"超常规扛伤"，玩家必须真正参与保护。

### §3.2 NPC 移动与外观

| 参数 | 推荐值 | 取值范围 | 说明 |
|---|---|---|---|
| `baseSpeed` | 35-45 px/s（随机） | 30-50 | 玩家全速约 100-130 px/s，NPC 速度约 30%-40%；玩家可轻松跟上 |
| 转向速度 | 1.2 rad/s | 1.0-1.5 | 低转向速度，让 NPC 走"平滑长线"，玩家容易预判走向 |
| 半径（碰撞 + 渲染） | 20 px | 18-22 | 与中等敌人接近（v0.2.0 pirate ≈ 16-20） |
| 主色 | 蓝绿色 `[0.35, 0.85, 0.78, 1.0]` | 偏冷调 | 与玩家（橙黄）和敌人（红 / 红紫）明显区分 |
| 友方箭头标记 | 在 NPC 上方 12 px 处绘制朝向跃迁点的三角形 | 固定 | 视觉上立刻识别为"友方目标" |
| 视觉特征额外提示 | 周围加 1 圈虚线浅蓝光环（半径 28 px，闪烁 1 Hz） | 可选 | 加强敌我识别，由 ADR / 主美确认是否实施 |
| 受伤反馈 | 命中粒子用浅蓝色（`[0.5, 0.9, 1.0, 0.9]`） | 固定 | 与敌人受击粒子（红 / 黄）区分 |

### §3.3 起点 / 终点 / 航程

| 参数 | 推荐值 | 取值范围 | 说明 |
|---|---|---|---|
| 起点位置 | 星系内远离跃迁出口的一端（与跃迁出口反向距 station 起点 600-800 px） | 500-1000 | 远离跃迁出口，让 NPC 有"长途旅行"的体感 |
| 终点位置 | 跃迁出口（即现有 `state.run.objective.beacon` 复用思路，或 ADR 决定新字段） | 固定 | NPC 抵达终点 = 任务完成 |
| 抵达判定距离 | 80 px（NPC 中心距跃迁点 ≤ 80 即视为抵达） | 60-100 | 落入方向 60-100 区间 |
| 总航程 | 1500 px（推荐中值） | 1200-1800 | 按 NPC 速度 40 px/s 算 ≈ 37.5 s；含路径偏移、避障、转向，实际 40-60 s |
| 任务持续时间预期 | 40-60 s | 30-90 s | 与玩家"完成一波 spawn"的节奏匹配 |
| 路径策略 | 直线（朝跃迁点），遇到 planet / asteroid 简单绕行（绕行半径 + 60 px） | 固定 | 不引入 A*，由 ADR 决定具体实现 |

### §3.4 任务期间敌人 spawn 倍率

| level | spawn 倍率 | 取值范围 | 说明 |
|---|---|---|---|
| 3 | 1.0× | 0.9-1.1 | 正常 spawn 节奏，与 v0.2.0 level 3 难度一致 |
| 4 | 1.0× | 0.9-1.1 | 同上 |
| 5 | 1.1× | 1.0-1.2 | 紧张期轻微加压 |

> 方向文档建议 level 1-2 0.8×，但 §1.1 权重表已把 escort 锁在 level 3-5，所以本表只覆盖 3-5。如未来权重表下放 escort 到 level 2，再补 0.8× 行（由制作人拍板）。

实现注意：spawn 倍率作用在 `spawnTimer` 的 `levelTimerScale` 之上（v0.3.0 numerics §7.1 减压触发也是同样路径，互不冲突）。具体接口由 ADR 决定。

### §3.5 NPC 死亡处理（选项 B）

| 触发 | 行为 | HUD / 文案 |
|---|---|---|
| NPC HP → 0 | 任务标记 fail；wreck（如果有）和 npc 实体仍保留在世界中或按 ADR 清理；玩家失去跃迁权限（与拾荒超时一致） | 一次性 toast：`护送目标已被摧毁。本关任务失败。`（显示 2.0 s） |
| 任务 fail 后 | 玩家可继续游玩本关（采集 / 击杀），但 `isObjectiveComplete()` 永远 false，跃迁门控不开启 | 跃迁 HUD 显示 `任务已失败，请重启本局` |
| 玩家选择 | 重启 run（按现有 `gameOver` 路径）或继续探索本关采集 / 击杀 | 局外点数 = 已积累的（不补发） |
| HP < 30% 警示 | NPC HUD 切红色 + 闪烁（1.5 Hz） | 持续显示直到 HP ≥ 30% 或 NPC 死亡 |
| HP < 10% 警示 | 一次性 toast `护送目标重伤，立刻支援！` | 仅触发 1 次（用 `state.run.escortLowHpAlerted` 去抖） |
| 跨 level | NPC 在 `nextLevel` 时销毁（不跨关存活） | 固定 |

设计要点：

- **不直接 gameOver**：避免"NPC 一死整局结束"的挫败感；玩家仍可在本关采集 / 击杀，积累局外点数，最后选择重启。
- **跃迁门控**：与"任务未完成"等效，不开启跃迁。
- **HP < 30% 警示**：让玩家明确知道 NPC 危险，主动救援。
- **HP < 10% 警示**：兜底提醒，防止玩家专注战斗忽视 NPC。

## §5. 奖励发放调整

### §5.1 奖励公式

```
gained_base = 2 + state.run.level                    // v0.3.0 现状（Game/game.js:1114）
gained = floor(gained_base * typeMultiplier)          // v0.4.0 新增类型倍率
state.meta.points += gained                           // 写入路径不变
```

### §5.2 任务类型奖励倍率

| 任务类型 | 倍率 | 取值范围 | 设计理由 |
|---|---|---|---|
| mine | 1.0× | 固定 | 基础任务，不调整 |
| explore | 1.0× | 固定 | 基础任务，不调整 |
| battle | 1.0× | 固定 | 基础任务，不调整 |
| survive | 1.0× | 固定 | 基础任务，不调整 |
| salvage | 1.5× | 1.3-1.7 | 反映"主动搜寻 + 桥接成本"；落入方向建议范围 |
| escort | 1.6× | 1.4-1.8 | 反映"全程紧张 + 不可恢复 + 失败回弹高"；略高于 salvage |
| guardian | 1.0×（局内）+ 终局 +50% 加成（已存在） | 固定 | 不叠加；终局结算仍按 `getRunSettlementData()` 的 base × 1.5 一次性加成（`Game/game.js:1129-1132`） |

### §5.3 各 level 实际奖励示例（参考）

| level | base = 2+level | mine/explore/battle/survive | salvage | escort | guardian（终局） |
|---|---|---|---|---|---|
| 0 | 2 | 2 | — | — | — |
| 1 | 3 | 3 | 4（floor(3×1.5)=4） | — | — |
| 2 | 4 | 4 | 6（floor(4×1.5)=6） | — | — |
| 3 | 5 | 5 | 7（floor(5×1.5)=7） | 8（floor(5×1.6)=8） | — |
| 4 | 6 | 6 | 9（floor(6×1.5)=9） | 9（floor(6×1.6)=9） | — |
| 5 | 7 | 7 | — | 11（floor(7×1.6)=11） | — |
| 6 | 8 | — | — | — | 8 + 终局 +50% = base 8 × 1.5 = 12（floor） |

### §5.4 写入路径

- `state.meta.points += gained`（不变）
- `state.run.totalObjectiveRewardBase += gained_base`（**注意**：仍写 base，不写乘倍率后的；终局 settlement 是按 `totalObjectiveRewardBase × 1.5` 计算，倍率不二次叠加；`Game/game.js:1129-1132` 行为不变）
- `saveMeta()` 调用路径不变

> **关键兼容**：`totalObjectiveRewardBase` 写 base 而非 gained，避免终局结算把"已加成的 salvage / escort 奖励"再 ×1.5 二次膨胀。这与方向文档"不叠加"原则一致。

## §6. 体验目标对应的回归校验（≥ 16 条）

> 参考 v0.3.0 numerics §8 风格。PM 在浏览器回归时逐条勾选。覆盖按方向文档"可探索 / 可参与 / 可重玩 / 不回潮"四类目标。

### §6.1 可探索类（5 条）

| # | 校验点 | 量化指标 |
|--:|---|---|
| 1 | 任务选择 seed 化：同 seed + 同 level 任务一致 | 用 seed=`12345` 启动 run，记录 level 1-5 任务序列；重启再用 seed=`12345`，序列完全一致 |
| 2 | 不同 seed 同 level 任务可能不同 | 用 seed=`12345` 和 seed=`67890` 各跑一轮，至少有 2 个 level 任务类型不同 |
| 3 | 终局 level 6 仍固定 guardian | 任意 seed 在 level 6 进入星系时 `state.run.objective.type === "guardian"` |
| 4 | 任务类型 cooldown 生效 | 同一 run 内不出现连续 2 关相同任务类型（guardian 除外） |
| 5 | level 0 不出 salvage / escort | seed 化抽样 100 次（理论或运行时统计），level 0 命中类型仅在 `{mine, explore}` 内 |

### §6.2 可参与类（8 条）

#### 拾荒（5 条）

| # | 校验点 | 量化指标 |
|--:|---|---|
| 6 | 进入 salvage 星系时预生成 wreck fragment 数 = N | level 1-2 N=3 / level 3-5 N=4；`state.fragments.filter(f => f.origin === "wreck").length === N`（spawn 失败降级时按 §2.4 处理） |
| 7 | HUD 显示拾荒目标 `X / 已桥接 Y` | objective HUD 文本含 `已桥接 Y / X 段古老残骸`（或同义文案，由实现确认） |
| 8 | 桥接 wreck 计数 +1 | 桥接 1 段 `origin === "wreck"` 的 fragment 后，`state.run.objective.progress` 增量 = 1 |
| 9 | 桥接玩家脱落 fragment 不计数 | 桥接 `origin !== "wreck"`（玩家脱落）的 fragment 后，progress 不变 |
| 10 | 达到 X 后任务完成 | progress === X 触发 `grantObjectiveReward()`，奖励按 §5.2 1.5× 计算；跃迁门控开启 |

#### 拾荒物理 / 上限（2 条）

| # | 校验点 | 量化指标 |

| 11 | wreck fragment 速度低于玩家脱落 | spawn 瞬间 wreck launchSpeed ≤ 14 px/s；玩家脱落 ≥ 22 px/s（同时存在两者时可对比） |
| 12 | wreck fragment 不计入 `FRAGMENT_MAX_COUNT = 8`（仅对玩家脱落计数） | wreck 4 个 + 玩家脱落 8 个并存时，no toast `残骸过多`，无 wreck 被自动销毁 |

| # | 校验点 | 量化指标 |
|--:|---|---|
| 13 | 护送任务整链路：NPC 生成 → 缓慢移动 → 玩家炮塔不锁定 → 抵达完成 | 进入 escort 星系时 `state.npcs.length` 增量 = 1；NPC 速度 ∈ [30, 50] px/s；玩家自动炮塔目标列表中无 NPC entry；NPC 中心距跃迁点 ≤ 80 时 `grantObjectiveReward()` 触发 |

### §6.3 护送细分（3 条）

| # | 校验点 | 量化指标 |
|--:|---|---|
| 14 | NPC HUD 显示方向 / 距离 / HP | objective HUD 含 NPC 方向箭头、距离（px）、HP/maxHp 三段；HP < 30% 时切红色闪烁 |
| 15 | NPC HP < 10% 触发一次性警示 | HP 首次跌破 10% 时 toast `护送目标重伤，立刻支援！`；HP 后续波动不重复触发 |
| 16 | NPC 死亡按选项 B 处理 | NPC HP → 0 时：toast `护送目标已被摧毁。本关任务失败。`；`isObjectiveComplete()` 持续 false；玩家可继续采集 / 击杀但跃迁门控关闭 |

### §6.4 可重玩类（2 条）

| # | 校验点 | 量化指标 |
|--:|---|---|
| 17 | 奖励倍率正确应用 | level 2 完成 salvage 奖励 = floor(4×1.5) = 6 点；level 3 完成 escort 奖励 = floor(5×1.6) = 8 点 |
| 18 | 终局守护者 +50% 加成保留 | level 6 完成 guardian 后，`getRunSettlementData().total === floor(base * 1.5)`，不受 v0.4.0 任意类型倍率影响 |

### §6.5 不回潮类（4 条）

| # | 校验点 | 量化指标 |
|--:|---|---|
| 19 | v0.3.0 fragment 桥接路径继续可用 | 在 battle / survive level 制造玩家脱落 fragment，桥接成功，行为与 v0.3.0 完全一致（toast `重新接入 N 个结构`） |

## §7. 平衡兜底（参数预留，不强制实现）

> 这两条兜底由 PM 在 v0.4.0 PM 回归时观察是否需要落实。本方案仅做参数预留：如运行时确实出现"拾荒疲劳"或"护送连败"，可按本表数值直接接入，无需另起 ADR。

### §7.1 拾荒疲劳兜底

| 触发条件 | 阈值 | 动作 | HUD / 文案 |
|---|---|---|---|
| 连续 N 关均为 salvage | N = 3 | 下一关抽样时 salvage 权重调到 0.1（其余权重重新归一化） | 无 toast（隐式调整，避免破坏沉浸） |
| 调整持续时间 | 仅作用于下 1 关 | 之后恢复原权重表 | — |
| 实现字段 | `state.run.salvageStreak` | 每关 createObjective 时若 type === "salvage" 则 +1，否则重置为 0 | 不写入 meta |

设计意图：避免 seed 化运气不好时玩家连续 3 关都桥接残骸，破坏节奏多样性。

### §7.2 护送连败兜底

| 触发条件 | 阈值 | 动作 | HUD / 文案 |
|---|---|---|---|
| 连续 N 次 escort 任务 fail | N = 2 | 下一关抽样时 escort 权重置 0（其余归一化） | 无 toast |
| 调整持续时间 | 仅作用于下 1 关 | 之后恢复原权重表 | — |
| 实现字段 | `state.run.escortFailStreak` | escort fail 时 +1，成功完成时重置为 0 | 不写入 meta |

设计意图：避免玩家因连续护送失败产生强烈挫败感。

### §7.3 兜底激活路径

- **默认不激活**：v0.4.0 首次实现时不接入 §7.1 / §7.2，PM 回归时观察 5-10 局玩家体验，由制作人决定是否激活。
- **激活路径**：如果激活，由 PM 在 v0.4.0 PM 回归看板记录，制作人确认后由实现成员按本表参数直接接入（无需另起 ADR / 数值方案微调）。
- **未激活的影响**：方向文档 §"可探索"的"再开一局看看下一关是啥"主要靠 §1.1 权重表 + cooldown 实现，§7 仅是极端情况兜底。

## §9. 自检与决策点

- [x] §2 拾荒参数完整：目标 X / 预生成 N / 物理参数（launch / angular / damp）/ 内部设施分布与 HP / 位置规则 / cap / 时间窗。

- [x] §4 任务完成时间窗完整：salvage 8 分钟 / escort 6 分钟；其他任务不引入时间窗（保留 v0.3.0 现状）。
- [x] §5 奖励发放调整完整：基础公式不变，乘类型倍率（salvage 1.5× / escort 1.6× / 其他 1.0×）；终局 +50% 加成不叠加；`totalObjectiveRewardBase` 仍写 base（兼容终局 settlement）。
- [x] §6 回归校验 22 条 ≥ 16 条；覆盖可探索 5 / 可参与 8 / 护送细分 3 / 可重玩 2 / 不回潮 4。

- [x] §8 阈值放宽 / 收紧规则：PM ±20% 与制作人拍板项分开。
- [x] 数值与方向文档建议范围一致：拾荒目标 2-4 ✓、奖励 1.5× ✓、护送 NPC HP 200-300 ✓、奖励 1.6× ✓。

### §9.2 仍需主程 / 制作人拍板的非阻塞问题

> 这 5 条均不影响数值方案落地；建议在 ADR 完稿 + 实现启动前的节点 A 评审时一次性定。

1. **任务类型 cooldown 接口形式**（§1.2）：由 ADR 决定是 `state.run.lastObjectiveType` 字段还是 `pickNextForObjective` 内部权重归一化。本方案只给参数。
2. **wreck fragment `origin` 字段是否影响 v0.3.0 序列化 / HUD / 维修无人机判定**（方向文档 §"关键技术疑问 #4"）：由 ADR 决定。本方案假设 `origin` 字段对维修无人机透明（不修就是不修），HUD 只按 origin 调整颜色 / 标记。
3. **NPC 移动 AI**（方向文档 §"关键技术疑问 #7"）：本方案给出"直线 + 简单绕行"建议（§3.3），具体算法由 ADR 决定。
4. **NPC 是否在跃迁后留下 wreck**（彩蛋）：本方案默认 `nextLevel` 时 NPC 销毁，不留 wreck。如制作人希望"NPC 抵达后留下一段感谢残骸供玩家桥接"作为彩蛋，由制作人拍板，不在本数值方案微调范围内。
5. **§7 兜底激活时机**：本方案默认不激活，PM 回归后由制作人决定。
