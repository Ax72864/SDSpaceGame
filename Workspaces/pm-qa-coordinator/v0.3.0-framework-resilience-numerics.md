## 1. 制作人方向回放（约束基线）

### 1.1 体验目标三条（必须命中）

“框架脱落 → 玩家重新接入”是一个 **可感知 / 可操作 / 可回收** 的局内循环：

1. **可感知**：站体链路被打断时，脱落部分明显从站体上“飞出来”成为漂浮残骸；HUD 与音画反馈让玩家立刻意识到“丢了一块”。
2. **可操作**：玩家可以驾驶站体靠近残骸，把它视作世界中的目标移动 / 围绕；不强制要求精确对齐到 grid。
3. **可回收**：玩家在合适位置扩展 frame 时，能够把附近的残骸整段重新合并回站体，并恢复其上设施功能（含之前的耐久 / 状态）。

体验风格基线：维持 v0.2.0 节奏紧凑的太空站建造，不引入复杂物理模拟；残骸物理简化到“惯性漂移 + 阻尼”级别；玩家可以选择放弃残骸，但“试图回收”的玩家必须有可读路径与明确反馈。

### 1.2 不做范围（数值层硬约束）

- 不做新设施、新武器、新维修类型、新建造资源。
- 不做完整物理碰撞重做：fragment 不与小行星 / 行星 / 友方弹道 / 站体反弹，仅与桥接判定相关。
- 不做局外天赋树扩展、联机 / 异步对战、新关卡任务类型、发布归档。
- 不动存档：`SAVE_KEY = "star-ring-station-meta-v1"` 不变；新增运行态字段一律放 `state.run` / 新建 `state.fragments`。

### 1.3 制作人对 ADR 的 3 条决策回复（本数值方案必须严格遵守）

1. **fragment 不阻挡敌人**：本版本 fragment 与敌方实体接触 / 路径完全不参与；不为 fragment 设置任何“撞击反弹 / 阻挡 hitbox”数值。
2. **桥接预览 UI 纳入 T4 必做项**：本数值方案必须给出预览触发距离、提示文案、显示位置规则，供 T4 直接落地。
3. **维修无人机不修 fragment cell**：所有维修无人机相关参数仅作用于 station 主 Map 内 cell；fragment cell 在“被桥接合并回站体”之前一律不进入维修候选；合并瞬间立即恢复维修候选资格（hp / frameHp 保留，由现有 `updateRepair` 自然接管）。

## 3. Fragment 物理参数表

> 实现约束：常量集中在 `Game/game.js` 头部，命名采用 `FRAGMENT_*`，与 `PLAYER_PHYSICS` 同级；运行态写入 `state.fragments[i]`，不写入 `state.meta`。

### 3.1 初始线速度（脱落瞬间）

```
fragment.vel.x = station.vel.x + outward.x * launchSpeed + jitter.x
fragment.vel.y = station.vel.y + outward.y * launchSpeed + jitter.y
```

| 参数 | 推荐值 | 单位 | 取值范围 | 说明 |
|---|---|---|---|---|
| `FRAGMENT_LAUNCH_SPEED_MIN` | 22 | px/s | 18-30 | 沿“fragment 几何中心相对站体几何中心”的外向分量；保证脱落后第一秒明显离开站体 |
| `FRAGMENT_LAUNCH_SPEED_MAX` | 36 | px/s | 28-44 | 上限避免飞太快导致玩家追不上 |
| `FRAGMENT_LAUNCH_JITTER` | 12 | px/s | 8-18 | `jitter = { rand(-J, J), rand(-J, J) }`，避免多 fragment 同向飞行视觉重叠 |
| 站体速度继承 | 100% 继承 | — | 固定 | fragment 继承 `state.station.vel`，让“站体在移动中被打掉一块”看起来自然 |

> 量级校准：v0.2.1 站体常态速度约 60-180 px/s；fragment 在站体停时净速度 ~22-36 px/s（约 1.3-2.4 倍 `CELL/s`），1 秒后视觉位移约 1-1.2 cell，足够“飞出来”但 5 秒内仍在站体附近 ~110-180 px 内，玩家可追。

### 3.2 初始角速度（脱落瞬间）

| 参数 | 推荐值 | 单位 | 取值范围 | 说明 |
|---|---|---|---|---|
| `FRAGMENT_ANGULAR_VEL_MIN` | -0.6 | rad/s | -0.9 ~ -0.4 | 负值表示逆时针 |
| `FRAGMENT_ANGULAR_VEL_MAX` | 0.6 | rad/s | 0.4 ~ 0.9 | 正值表示顺时针 |
| `FRAGMENT_ANGULAR_JITTER` | ±0.15 | rad/s | ±0.10 ~ ±0.20 | 在 [-min, max] 抽取后再加一次小抖动 |

> 量级校准：0.6 rad/s ≈ 34°/s；3 秒可转 ~100°。配合 §4 桥接角度阈值 25°，玩家平均等待 ~0.7-1.5 秒就能等到 fragment 转回桥接角窗内，不至于“永远对不上”。

### 3.3 阻尼

```
vel.x *= 1 - min(FRAGMENT_LINEAR_DAMP * dt, FRAGMENT_LINEAR_DAMP)
vel.y *= 1 - min(FRAGMENT_LINEAR_DAMP * dt, FRAGMENT_LINEAR_DAMP)
angularVel *= 1 - min(FRAGMENT_ANGULAR_DAMP * dt, FRAGMENT_ANGULAR_DAMP)
```

| 参数 | 推荐值 | 单位 | 取值范围 | 说明 |
|---|---|---|---|---|
| `FRAGMENT_LINEAR_DAMP` | 0.06 | per-frame coef | 0.04-0.10 | 比旧 `updateDetachedCells` (0.08) 略低，让 fragment 更明显地漂移；远低于站体 `linearDamping=0.27`，避免“一脱落就立刻停下” |
| `FRAGMENT_ANGULAR_DAMP` | 0.35 | per-frame coef | 0.25-0.45 | 让 fragment 旋转明显但不无限自转；约 6-10 秒后角速度衰减到 30% |

### 3.4 速度上限（防越界）

| 参数 | 推荐值 | 单位 | 取值范围 | 说明 |
|---|---|---|---|---|
| `FRAGMENT_MAX_LINEAR_SPEED` | 180 | px/s | 140-220 | 与站体常态速度量级一致；若 fragment 被弹道击中或被多个脱落事件叠加推动，clamp 到此值 |
| `FRAGMENT_MAX_ANGULAR_SPEED` | 2.4 | rad/s | 1.8-3.0 | 约 137°/s；超过则 clamp，避免桥接角窗永远扫过 |

> 应用顺序：先 vel += impulse → 再 damping → 再 clamp。

### 3.5 世界边界处理

| 参数 | 推荐值 | 说明 |
|---|---|---|
| 边界缓冲 | `GALAXY_WORLD_BOUNDS` 外扩 200 px | 即 fragment.pos 满足 `x < minX-200 || x > maxX+200 || y < minY-200 || y > maxY+200` 时销毁 |
| 销毁原因 tag | `"drifted-out"` | 供日志 / 调试 overlay 区分 |
| 玩家提示 | 一次性 toast：“一段残骸已漂出星系。” | 仅在销毁瞬间 push 一条 alert；不重复 |
| 销毁前预警 | 距离边界 < 100 px 时，HUD 残骸条目末尾追加 `(即将漂离)` | 让玩家有 1-3 秒决定是否追 |

### 3.6 弹道命中减 HP 策略（敌方弹道）

> 仅 enemy owner 弹道生效；friendly 弹道直接穿透 fragment，不结算。

| 参数 | 推荐值 | 单位 | 说明 |
|---|---|---|---|
| 命中半径 | `CELL * 0.48 = 14.4` px | px | 与现有 station cell 命中判定一致（ADR §2.8） |
| 单发伤害 | 沿用敌方弹道当前伤害 | — | pirate 弹道 level 0-2 = 10；level 3+ 按 v0.2.0 表 |
| Cell 销毁 | `cell.hp <= 0` 时从 `fragment.cells` 删除，并 `edgeKeys` 重算 | — | 不调 `damageCell`（避免触发 `checkConnectivity`） |
| Fragment 销毁 | `fragment.cells.size === 0` → `fragment.markedDead = true` | — | 下一帧 filter |
| 视觉反馈 | 沿用现有 `spawnParticle` (灰白 4-6 粒，0.4 s) | — | 不新增粒子常量 |
| Fragment 销毁文案 | 一次性 toast：“一段残骸被摧毁。” | — | 与“漂出星系”区分 |

> 敌方弹道 vs fragment cell 在 fragment 内连通性 **不二次检测**：fragment 不会因为中间 cell 被打掉而再分裂；这是性能与可读性的折中，避免“连环 fragment 分裂”。

## 4. 桥接判定阈值

> 触发时机：`buildAt(x, y, "frame")` 成功扩展 frame 后立即调用 `tryBridgeAt(x, y)`（ADR §2.4）。其它建造路径不触发桥接。

### 4.1 距离与角度阈值

| 参数 | 推荐值 | 单位 | 取值范围 | 说明 |
|---|---|---|---|---|
| `BRIDGE_DISTANCE` | `CELL * 1.2 = 36` | px | 30-42（即 1.0×～1.4×`CELL`） | 候选 fragment 边缘 cell 世界坐标到新 frame 世界坐标的最大距离；与制作人方向 + ADR 推荐量级一致 |
| `BRIDGE_ANGLE` | 0.436 rad (= 25°) | rad | 0.349-0.524 (= 20°-30°) | fragment.angle 与 station.angle 的差异上限；超过则不允许桥接 |
| `BRIDGE_PREVIEW_DISTANCE` | `CELL * 1.6 = 48` | px | `BRIDGE_DISTANCE * 1.2-1.5` | 玩家鼠标 hover 可建造 frame 格时，距 fragment 边缘 cell ≤ 此值即显示桥接预览（比真正阈值略宽，让玩家“感觉得到要桥接了”） |
| `BRIDGE_PREVIEW_ANGLE_TOLERANT` | 0.611 rad (= 35°) | rad | 0.524-0.698 (= 30°-40°) | 预览也允许略宽角度，提示玩家“再等一下就能接” |

### 4.2 候选 fragment cell 规则

仅匹配 fragment **边缘 cell**（即 `edgeKeys`：在 fragment 内部 4 邻接至少缺 1 个方向的 cell）。

| 规则 | 内容 |
|---|---|
| 候选源 | `fragment.edgeKeys`（在 split 时一次性算好，merge 或 cell 销毁时重算） |
| 同 fragment 多候选 | 取距离最小的一个 |
| 多 fragment 同时落入阈值 | 按 (cell 总数 desc, 候选距离 asc) 选第一个 fragment；**一次 buildAt 只接入一个 fragment**（与 ADR §2.4 一致） |
| 内部 cell | **不允许**作为候选；只看 edgeKeys |
| grid 冲突 | 整段 grid offset 投影后，任一目标 key 与 `state.station.cells` 冲突 → 全部拒绝（“全部接入或拒绝”） |
| 角度过滤 | `|normalizeAngle(fragment.angle - station.angle)| > BRIDGE_ANGLE` 直接 skip 该 fragment |

### 4.3 桥接成功 / 失败文案

| 场景 | 文案 | UI 通道 | 备注 |
|---|---|---|---|
| 成功桥接 1 段 | `重新接入 N 个结构。` | `showToast`（沿用 ADR §4.3） | N = fragment.cells.size；与现 `reconnectDetached` 文案风格一致 |
| 候选超距离 / 超角度（无候选） | 不弹错误，仅按现有 `setBuildError` 沉默路径（buildAt 已扩展 frame 成功） | 无 | 桥接是“顺手能做就做”，不是建造的必要条件，不应让玩家觉得建造失败 |
| 候选命中但 grid 冲突 | `桥接位置被占用，已扩展框架但残骸未接入。` | `setBuildError`（沿用） | 与现有建造失败风格一致；玩家可移动后再试 |
| 候选 fragment 角度不对（仅角度阈值挡住，距离已落入） | `残骸角度未对齐，请等待残骸自转或换位置。` | `setBuildError`（仅在距离 ≤ BRIDGE_DISTANCE 但角度超阈值时触发，每 1.0 秒去抖一次） | 让玩家知道“是角度问题，不是距离问题” |

### 4.4 桥接预览 UI 规则（T4 必做）

> 制作人方向 #2 决策：桥接预览 UI 纳入 v0.3.0 必做项。

触发条件（全部满足）：

1. 当前建造选择 = `frame`；
2. 鼠标 hover 落在某个**可扩展 frame 的合法 grid 格**（即满足现有 `hasConnectedNeighbor` 与“无现存 cell”）；
3. 该 grid 格世界坐标到任一 fragment.edgeKeys cell 的世界距离 ≤ `BRIDGE_PREVIEW_DISTANCE` (48 px)；
4. 角度差 ≤ `BRIDGE_PREVIEW_ANGLE_TOLERANT` (35°)。

显示内容：

| 元素 | 视觉规格 |
|---|---|
| Hover 格高亮 | 在原 frame 蓝色描边外再加一圈金色（`[0.95, 0.78, 0.32, 0.85]`），宽 2 px，dash `CELL * 0.25` |
| 候选 fragment 整段轮廓 | 用 fragment 本地 grid offset 投影出“将要回归”后的 station-grid 形状；颜色 `[0.95, 0.78, 0.32, 0.55]`，宽 2 px，dash `CELL * 0.30` |
| 候选 fragment 主体 | fragment 自身渲染时 alpha 从默认 0.55 提到 0.70（让玩家明确知道“这一段会接进来”） |
| 文案提示（HUD 下沿 / hover 格下方） | `"按建造键接入 N 个结构"`（N = fragment.cells.size） |
| 角度不达标但距离落入预览 | 文案改为 `"残骸角度未对齐，等待自转"`；hover 高亮颜色降到 alpha 0.4，不显示回归轮廓 |
| grid 冲突 | 投影时若任一目标 key 已被占用，回归轮廓改为红色 `[1.0, 0.35, 0.30, 0.55]`，文案：`"桥接位置被占用"` |

性能预算：预览仅在 hover 状态触发；每帧 ≤ 1 次候选搜索（O(fragments × edgeCells) ≈ 256），无压力。

> 如 T4 实现成本超 PM 预算，按 ADR §11 #2 由 PM 与制作人协商再下沉到 v0.3.1；但本数值方案默认按必做项给参数。

## 5. 最大 fragment / cell 上限与降压策略

### 5.1 数量上限

| 参数 | 推荐值 | 取值范围 | 说明 |
|---|---|---|---|
| `FRAGMENT_MAX_COUNT` | 8 | 6-10 | 同时存在的 fragment 上限；与 ADR §2.6 一致 |
| `FRAGMENT_MAX_CELLS_PER` | 32 | 24-40 | 单 fragment cells 上限；现有站体常见规模 30-60，单次脱落连通分量通常 < 32 |
| `DETACHED_CELLS_TOTAL_SOFT_CAP` | 80 | 64-100 | 所有 fragment cells 之和的软上限；超过触发 §5.3 减压策略 |
| `DETACHED_CELLS_TOTAL_HARD_CAP` | 120 | 100-140 | 硬上限；超过直接按 spawnAt 排序销毁最早 fragment 直到回到 hard cap 内 |

### 5.2 超限降压策略

| 触发 | 动作 | HUD 反馈 |
|---|---|---|
| `state.fragments.length > FRAGMENT_MAX_COUNT` | 按 `spawnAt` 升序销毁多余 fragment（最早脱落的先消失） | 一次性 toast：`残骸过多，最早一段已散失。`（仅在销毁瞬间） |
| 单 fragment `cells.size > FRAGMENT_MAX_CELLS_PER` | 在 split 时按本地 grid 距 fragment 几何中心最远的 cell 截断（保留中心向外 32 个最近 cell；超出部分直接丢弃，不进入 fragment 也不留在 station） | 一次性 toast：`一段残骸过大，外缘结构散失。`（仅 split 触发瞬间） |
| `sum(fragment.cells.size) > DETACHED_CELLS_TOTAL_SOFT_CAP` | HUD 残骸面板顶部加红字 `残骸总量临近上限`；不立即销毁 | 持续显示直到回落 |
| `sum(fragment.cells.size) > DETACHED_CELLS_TOTAL_HARD_CAP` | 按 `spawnAt` 升序销毁 fragment，直到回到 hard cap 内 | 一次性 toast：`残骸过多，部分已散失以保护战场可读性。` |

## 6. 维修无人机对 detached 的策略

> 制作人决策 #3：**维修无人机不修 fragment cell**。本节给出实现细节与 HUD 边界。

### 6.1 判定规则

| 规则 | 内容 |
|---|---|
| Fragment cell 是否进入维修候选 | **不进入**。`updateRepair` 在筛选 `target` 时仅遍历 `state.station.cells`；fragment.cells **不进入该 Map**，天然不被选中 |
| 已派出的 drone 中途 fragment 化 | 现行 `updateRepair` 已有 `target.detached → 返回 repairer` 路径；本版本由于 fragment 拆出后 cell 不再留在 station.cells 中，drone 的 `state.station.cells.get(drone.targetKey)` 自动返回 `undefined`，触发 `drone.life = 0` 自然销毁（ADR §2.8 表“repairDrones”行已确认） |
| Fragment cell 自身 hp / frameHp 在 fragment 内 | 保留脱落瞬间的数值，不被维修无人机回血、不被任何站体内系统主动恢复 |
| Fragment cell 被弹道继续打 | hp 继续减；归零则从 fragment 移除（§3.6） |

### 6.2 合并回 station 后的维修恢复

| 状态 | 行为 |
|---|---|
| 合并瞬间 | `cell.detached = false; cell.drift = null; cell.reload = 0; cell.fire = 0;` `hp / frameHp / facility / enabled / priority / maxHp` 全部保留（ADR §2.5） |
| 维修候选恢复 | 合并瞬间该 cell 立即重新出现在 `state.station.cells` 中；下一次 `updateRepair` 触发时（≤ 1.25 s 内），若 `hp < maxHp || frameHp < 70` 即进入候选 |
| 恢复速率 | 不新增数值；沿用现有 `hp +18 / frameHp +12` 每次到达；玩家会自然感觉到“桥接回来后过几秒就被慢慢修复” |

### 6.3 HUD 提示规则

| 触发 | HUD 文案 |
|---|---|
| 当前存在 fragment 且站体内有 `repair` 设施 | 残骸面板末行追加：`残骸需桥接才可维修`（持续显示，不闪烁） |
| 玩家选中（hover / 点击）某个 fragment cell 试图查看（v0.3.0 仅在 hover 桥接预览时） | 预览文案同 §4.4，不重复提示 |
| 已无 fragment | 残骸面板隐藏整段；上述提示自动消失 |

> 不开“维修无人机有限维修 fragment”分支：与制作人方向 #3 决策严格一致。如未来体验测试反馈“桥接还是残血太苛刻”，由制作人重新拍板再开。

## 7. 平衡兜底

### 7.1 60 秒内大量脱落的减压触发条件

| 条件 | 阈值 | 动作 |
|---|---|---|
| 60 秒滑动窗口内累计新增 fragment cells 数 | ≥ 30 | 触发一次性减压提示 |
| 60 秒滑动窗口内累计 split 次数 | ≥ 4 | 与上一条 OR 关系；任一满足即触发 |
| 减压提示文案 | `站体结构受到严重打击，下一波敌人会推迟。` | 通过 `showToast` |
| 减压效果 | 下一次 `spawnWave` 的 `state.spawnTimer` 乘 1.5（在现有 `levelTimerScale` 之上） | 仅一次性，触发后立即重置滑动窗口计数 |
| 冷却 | 触发后 90 秒内不再触发（即同一压力事件只减压一次，避免玩家“故意丢 cell 换喘息”滥用） | 用 `state.run.lastFragmentReliefAt` 记录时间戳 |

> 实现注意：所有计数字段放 `state.run.fragmentRelief = { window: [], lastReliefAt: 0 }`，不写入 meta。

### 7.2 多 fragment 场景的玩家反馈

| 触发 | HUD / 文案 |
|---|---|
| `state.fragments.length === 1` | 残骸面板显示：`残骸 ×1：方向 / 距离 / 设施数`（最近一段） |
| `state.fragments.length ∈ [2, 4]` | 残骸面板显示：`残骸 ×N：最近一段 方向 / 距离 / 设施数 / 角度差`（仅最近一段详细，其余仅数量） |
| `state.fragments.length ∈ [5, 8]` | 残骸面板顶部加黄字 `残骸较多，建议优先桥接最近一段`；详细信息仍只显示最近一段 |
| 接近 `FRAGMENT_MAX_COUNT` 上限 | 顶部黄字升级为红字 `残骸临近上限，最早一段即将散失` |
| 任一 fragment 与最近站体距离 < `CELL * 4 = 120` 且角度差 ≤ `BRIDGE_ANGLE` | 残骸面板该项末尾追加绿字 `可桥接` | 让玩家知道现在就该出手 |
| 任一 fragment 距离 ≥ `CELL * 12 = 360` | 该项末尾追加灰字 `较远` |

> 所有方向 / 距离 / 角度计算每 0.2 秒采样一次（避免每帧重算），结果存 `state.run.fragmentHudSamples`，由 `buildStatusAlerts` 读取。

### 7.3 战斗惩罚平衡

| 维度 | 设计意图 | 数值锚点 |
|---|---|---|
| 短时间损失大量 cell | 有减压（§7.1），但 fragment 仍需玩家主动桥接才能恢复设施 | 不打折扣，避免“脱落变成爽点” |
| 设施被毁后变 frame，frame 再被毁才会触发 detached | 沿用现有 `damageCell` 链路（§2.1 基线） | 不改 |
| Fragment 上 cell 仍可被敌方弹道继续打 | 让玩家感觉到“不回收的残骸是会丢的” | 沿用敌方弹道伤害（§3.6） |
| Fragment 上设施在脱落瞬间不再射击 / 不再维修 / 不再采矿 / 不再贡献电力 | 由现有 `cell.detached continue` 守卫 + “fragment cell 不在 station.cells”双重保证 | ADR §2.8 |

## 8. 回归校验点（≥ 8 条）

PM 在浏览器回归时逐条勾选，任一不通过即视为该子目标未命中。覆盖类别按 **可感知 / 可操作 / 可回收** 三类目标标注。

| # | 类别 | 校验点 | 量化指标 |
|--:|---|---|---|
| 1 | 可感知 | 制造一次脱落后，fragment 在 1 秒内从站体位置“飞出去”可见 | fragment.pos 与脱落瞬间 station 位置距离 ≥ 22 px（即 `FRAGMENT_LAUNCH_SPEED_MIN * 1.0`）；fragment 不再随 station 移动而平移 |
| 2 | 可感知 | Fragment 视觉与站体明显区分 | 主体 alpha = 0.55、颜色去饱和 0.7；外环虚线颜色 `[0.92, 0.6, 0.32, 0.8]`、dash `CELL * 0.3`（ADR §2.7 参考量，本数值方案确认） |
| 3 | 可感知 | HUD 残骸面板显示最近 fragment 的方向 / 距离 / 设施数量 | 至少 1 个 fragment 存在时，HUD 必有可见的“残骸 ×N”条目；采样频率 0.2 s |
| 4 | 可感知 | Fragment 被摧毁 / 漂出星系时一次性提示 | 销毁瞬间 toast 显示对应文案（“被摧毁” / “漂出星系”）；提示时长 ≥ 1.0 s |
| 5 | 可操作 | 玩家可驾驶站体在 5 秒内靠近 fragment | 在 fragment 几何中心 250 px 外、站体当前位置出发，使用 WASD 任意模式，5 秒内能把 fragment 中心带入 `CELL * 1.6 = 48` px 桥接预览距离内 |
| 6 | 可操作 | 桥接预览 UI 在 hover 合法 frame 格 + 阈值内时正确显示 | hover 格高亮金色、候选 fragment 回归轮廓金色、文案 `"按建造键接入 N 个结构"` 三者同时出现；角度不达标时改为对应文案 |
| 7 | 可操作 | 桥接距离阈值符合预期 | 在 hover 格世界坐标到 fragment 边缘 cell 距离 ≤ 36 px 时触发桥接成功；> 36 px 时不触发；预览阈值 48 px |
| 8 | 可操作 | 桥接失败提示可读 | grid 冲突 → `桥接位置被占用，已扩展框架但残骸未接入。`；仅角度不达标 → `残骸角度未对齐，请等待残骸自转或换位置。` |
| 9 | 可回收 | 桥接成功 1 段后整段 cells 回到 station.cells | `state.station.cells.size` 增量 = fragment.cells.size；fragment 从 `state.fragments` 中消失 |
| 10 | 可回收 | 桥接成功后保留 facility / hp / frameHp / enabled / priority；重置 reload / fire | toast 显示 `重新接入 N 个结构`；下一帧设施按 `updatePower` / `updateTurret` 等自然重启；不出现“桥接后立即齐射” |
| 11 | 可回收 | 桥接成功后维修无人机立刻把该 cell 纳入候选 | 1.25 s 内若该 cell hp < maxHp，可见 drone 飞向它；hp `+18 / +12` 沿用现量 |
| 12 | 可回收 | 多 fragment 场景下，桥接一次只接入一个 fragment | 触发 buildAt 后，`state.fragments.length` 减少恰好 1（除非 hard cap 自动销毁，另算） |
| 13 | 兜底 | Fragment 数量上限生效 | `state.fragments.length` 永远 ≤ 8；超过即按 spawnAt 销毁最早，toast 显示对应文案 |
| 14 | 兜底 | 60 秒大量脱落减压触发 | 60 s 内累计新增 cells ≥ 30 或 split ≥ 4 → 下一次 `state.spawnTimer` 乘 1.5；冷却 90 s 内不再触发 |
| 15 | 不回潮 | v0.2.1 站体手感（三引擎刹停、刹车距离、90 度转向）未受影响 | `PLAYER_PHYSICS` 常量未被本版本读写；fragment 物理与站体物理解耦 |
| 16 | 不回潮 | 旧存档可读取，`SAVE_KEY` 未变 | `loadMeta` 路径不受影响；`state.fragments` 不写入 meta |

> 覆盖统计：可感知 4 条（#1-#4） / 可操作 4 条（#5-#8） / 可回收 4 条（#9-#12） / 兜底 2 条（#13-#14） / 不回潮 2 条（#15-#16），合计 16 条，超过 ≥ 8 条要求。

### 9.2 必须制作人重新拍板（不可由 PM 单独调整）

| 参数 | 默认 | 理由 |
|---|---|---|
| `BRIDGE_DISTANCE` | 36 px (`CELL * 1.2`) | 直接影响“可操作”体验目标；制作人方向给出参考量；改动会改变玩家“能桥接的瞬间”的判定 |
| `BRIDGE_ANGLE` | 25° (0.436 rad) | 同上；影响“角度可读性”体验 |
| `FRAGMENT_MAX_COUNT` | 8 | 触及性能预算 + 战场可读性 + ADR §2.6 上限 |
| `FRAGMENT_MAX_CELLS_PER` | 32 | 同上 |
| `DETACHED_CELLS_TOTAL_HARD_CAP` | 120 | 战场清理上限，影响“是否强制销毁” |
| fragment ↔ 敌方碰撞策略 | 不参与（制作人决策 #1） | 直接由制作人决策回复定 |
| 维修无人机是否修 fragment cell | 不修（制作人决策 #3） | 直接由制作人决策回复定 |
| 桥接成功是否额外回血 | 不回血（沿用 hp / frameHp） | 涉及战斗惩罚 / 韧性平衡 |
| 桥接预览 UI 是否纳入 v0.3.0 | 纳入（制作人决策 #2） | 直接由制作人决策回复定 |
| 边界缓冲值 | 200 px | 影响“漂多远才消失”的体验，与世界尺度耦合 |

### 9.3 跨版本调参原则

- 任何 `FRAGMENT_*` 常量调整都不应影响 `PLAYER_PHYSICS / COLLISION_FEEL / PIRATE_AI`；如发现互相干扰，立即停止调参并升级到主程评审。
- 任何调整都不得写入 `state.meta`；纯运行态。

- [x] 平衡兜底：60 秒大量脱落减压（cells ≥ 30 或 split ≥ 4 → spawnTimer × 1.5，冷却 90 s）；多 fragment HUD 反馈分档（§7）。

- [x] 不动存档：所有新增字段在 `state.fragments / state.run.fragmentRelief / state.run.fragmentHudSamples`，不写 meta。

### 10.2 仍需制作人 / 主程拍板的非阻塞问题

> 这 4 条均不影响实现启动；建议在 T1+T2 完成后的节点 A 评审时顺手定。

1. **fragment 主体 alpha / 闪烁频率 / 外环虚线颜色**（§8 #2）：本方案沿用 ADR §2.7 参考量（alpha 0.55、去饱和 0.7、外环 `[0.92, 0.6, 0.32, 0.8]`、dash `CELL * 0.3`、闪烁 1.5 Hz）。如制作人 / 主美希望更显眼或更隐藏，由制作人拍板。
2. **多 fragment 视觉区分**（ADR §2.3 末段提及）：本方案默认不为不同 fragment 上色区分（依赖空间位置即可识别）。如玩家反馈“多 fragment 时分不清”，可在 v0.3.1 加一套 fragment id → 微色调映射；本版本不开。
3. **桥接预览触发是否要加 0.15 s 去抖**（§4.4）：本方案默认每帧实时计算（性能足够）。如玩家觉得 hover 时预览闪烁，可加 0.15 s 去抖；建议由 PM 在浏览器回归时主观判断。
4. **fragment 距离 < `CELL * 4` 且角度对齐时是否要在站体侧画一条引导线**（§7.2 “可桥接” 绿字提示之外）：本方案默认仅文案提示。如玩家反馈“知道可以桥接但不知道往哪个 frame 格点”，可在 v0.3.1 加引导线；本版本不开。
