## 问题摘要

PM 两轮 CDP 真实输入复测中，按 ghost path 点击时首格 toast 为「框架必须连接到核心或已连接结构」，关键格停留为 `frame`，未建成 `mining` / 后续设施。入口污染已关闭，阻塞点转为**路径顺序可执行性**与**输入可验证性**。

## 实现范围（默认层增强，不改核心规则）

### 路径顺序

- `validateExpansionStepsBuildOrder()`：模拟从已连接结构逐格铺框架，校验顺序是否可执行。
- `normalizeExpansionStepsToBuildOrder()`：正向/反向校验 + 邻接优先级 tie-break，确保 `suggestedExpansionPath.frameSteps` 为「从已连接邻接点 → 候选矿站格」。
- `ghostPath.pathValid` 暴露路径是否通过模拟校验。

### 可执行建造计划

- `computeMiningBuildPlan()` / `__gameTest.playtest.getMiningBuildPlan()`：
  - `steps[]`：`{ build, gridKey, screenPoint, precondition, expectedCost, canAffordStep, selectedBuild, actionable }`
  - 顶层：`currentStep`、`orderedFrameGridKeys`、`affordability`、`canAffordAll`、`selectionDiagnostics`
- 每步含 `clientX/clientY`（`buildGridScreenPointDiagnostics`）与 `mustSelectBuild`。

### UI / ghost 可视化

- `resourceGuide`：「按顺序建造」前 3–5 步 chip；「当前第 N 步必须先建这里」；建造选择不匹配警告。
- playtest URL 下额外显示 PM 点击 client 坐标与应选设施。
- `renderMiningExpansionGhostPath()`：高亮 `currentStep` 框架格；框架未完成时弱化采矿目标虚线，避免误点远端。

## PM 复测建议

1. 全新 profile 打开 `Game/index.html?playtest=1`，点「继续」。
2. 只读采样：`window.__gameTest.playtest.getMiningBuildPlan()`。
3. 确认 `steps[0].build === 'frame'` 且 `steps[0].precondition.connectedNow === true`。
4. 按 `currentStep` 依次：选对应 build 类型 → 用 `screenPoint.clientX/Y` 或肉眼对齐 ghost 高亮格点击。
5. 框架链完成后最后一步 `build === 'mining'`；等待 `harvestingCount > 0`。
