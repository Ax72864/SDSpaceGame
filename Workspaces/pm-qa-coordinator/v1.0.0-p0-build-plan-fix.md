## 问题摘要

- #6 复测中 CDP 按 ghost path 点击时，可能出现首格「框架必须连接到核心或已连接结构」、关键格停留 `frame` 未升级 `mining`、建造选择链不稳定。
- 根因侧重诊断与引导：路径顺序未显式校验、缺少带屏幕坐标的逐步计划、UI 未强调「当前必须先建哪一格 / 选哪种建造类型」、框架铺完后未明确提示切换采矿站。

## 实现范围

### 路径顺序与 ghost path

- `normalizeExpansionStepsToBuildOrder()`：若首格未邻接已连接结构而末格邻接，则反转 `frameSteps`，保证从已连接端向候选矿站格推进。
- `suggestedExpansionPath` / `ghostPath` 增加 `buildOrder: "connected_to_target"`；`totalFrameSteps` 与有序步数对齐。
- `renderMiningExpansionGhostPath()`：仅渲染未建成的框架步；高亮 `buildPlan.currentStep`；框架铺完后高亮采矿目标格。

### 逐步建造计划 helper

- `computeMiningBuildPlan()` + `window.__gameTest.playtest.getMiningBuildPlan(options?)`
- 每步输出：`build`（`frame`/`mining`）、`gridKey`、`screenPoint`（`clientX`/`clientY`/`deviceX`/`deviceY`）、`selectedBuild`、`precondition`、`expectedCost`、`canAffordStep`、`actionable`
- 汇总：`currentStepIndex`、`currentStep`、`selectedBuildRecommended`、`affordability`、`canAffordAll`、`selectionDiagnostics`
- `evaluateMiningUpgradePrecondition()`：只读对齐 `buildAt` 的 frame→mining 前置，不改建造合法性。

### UI / resourceGuide

- `buildMiningExpansionGuidanceHtml()`：展示「按顺序建造」前 3–5 步 chip、当前第 N 步必须先选类型并点击 gridKey、建造选择不匹配提示；修复路径文案重复目标格。

## PM 复测建议（#6）

1. 打开 `Game/index.html?playtest=1`，关闭 release candidate panel，点「继续」。
2. 控制台采样：`const p = __gameTest.playtest.getMiningBuildPlan(); p.steps[0]`，确认 `build==='frame'`、`precondition.connectedNow===true`、`screenPoint.clientX/Y` 有值。
3. 按 `p.currentStep` 逐步：先 `selectedBuild==='frame'` 点 `currentStep.gridKey`；框架完成后 `selectedBuild==='mining'` 点目标格。
4. 对比 resourceGuide「当前第 N 步」与 ghost 高亮格是否一致。
5. 验证 `harvestingCount > 0` 或记录资源不足 / 选择丢失的新阻塞点。

1. 首局最佳候选路径仍可能很长且初始金属不足；helper 会标 `canAffordStep=false`，需 PM 区分「输入误差」与「资源未凑够」。
2. CDP 点击仍依赖 `screenPoint` 与暂停态；未在本任务内跑完整浏览器复测，#6 须 PM 真实输入关闭。
