# v0.2.0 星系节奏 数值方案与参数表

## 制作人方向（已拍板，本文档作为约束基线）

- N = 6：level 0-5 为 6 个普通关卡，level 6 为本局终局星系。
- 终局任务类型 = "击毁本局守护者"：基于现有 `station` 类型参数化强化，**不引入新 AI 行为**。
- 本局结算 +50% 局外点数加成：累计奖励 × 1.5，向下取整。
- 早期减压不回潮：level 0-2 等同或弱于 v0.1.x 减压后水平（即当前 trunk 基线）。

## v0.1.x 基线参考（用于确认"不回潮"）

直接读自 `Game/game.js`，作为 level 0 的天花板：

- `state.spawnTimer` 初始 = 22 秒；后续 `spawnTimer = rand(26, 40) / sqrt(playerCount)`。
- `spawnWave.count = ceil(playerCount * rand(0.6, 2))`。
- `SPAWN_ASTEROID_CHANCE_EARLY = 0.22`，`SPAWN_ASTEROID_CHANCE_LATE = 0.38`，`SPAWN_ASTEROID_RAMP_SEC = 300`。
- 敌人比例（`spawnWave`）：roll < 陨石概率 → asteroid；< 0.82 → pirate；其余 → station（受 `asyncEnabled` 限制）。
- 敌人基础数据（`spawnEnemy`）：
  - asteroid HP=26, r=11, 撞击伤害=18（`COLLISION_FEEL.asteroidCellDamage`）。
  - pirate HP=68, r=20, 撞击伤害=8（`COLLISION_FEEL.pirateCellDamage`）, 弹道伤害=10。
  - station HP=260, r=48, 弹道伤害=16, 增援间隔=7 秒。
- 初始世界：1 恒星 + 2 行星 + 3 小行星，分布在 [-1180..1180, -420..820]，资源量 360-1800 / 9999（恒星）。
- 跃迁补给：`nextLevel` 加 +55 金属/+25 矿/+18 气。
- 任务奖励：`gained = 2 + state.run.level`，存入 `state.meta.points`。

## 1. 程序化星系生成参数

### 1.1 天体数量分布（按 level）

每关固定 1 个恒星（位于世界原点附近，作为星系锚点）。行星与小行星数量按下表生成：

| level | 行星 min | 行星 max | 小行星 min | 小行星 max | 备注 |
|------:|--------:|--------:|----------:|----------:|------|
| 0     | 2       | 2       | 3         | 4         | 等同 v0.1.x 初始 |
| 1     | 2       | 3       | 3         | 5         | 略宽 |
| 2     | 2       | 3       | 4         | 6         | 资源略多 |
| 3     | 3       | 4       | 4         | 6         | 多样化 |
| 4     | 3       | 4       | 5         | 7         | 资源充裕 |
| 5     | 3       | 5       | 5         | 8         | 富有星系 |
| 6     | 2       | 3       | 4         | 6         | 终局，留出战斗空间 |

每关在 min/max 之间用 `rand` 取整。

### 1.2 位置生成规则

世界生成范围：`[-1400, 1400] × [-1100, 1100]`（与现有相机/星点范围匹配；星点保持 `[-2600..2600, -2200..2200]` 用于纵深感）。

- **恒星位置**：`rand(-180, 180) × rand(-180, 180)`，半径 110-150。终局星系（level 6）恒星半径放大到 140-170（更具威胁感）。
- **行星位置**：到恒星中心距离 ∈ [380, 1100]；任意两行星之间最小间距 ≥ `r1 + r2 + 220`。
- **小行星位置**：到恒星中心距离 ∈ [240, 1280]；与其他天体（含恒星、行星、其他小行星）最小间距 ≥ `r1 + r2 + 80`。
- **生成失败回退**：若 80 次随机尝试仍找不到合法位置，则该天体降级（小行星变更小、行星减少 1 颗），避免死循环。
- **空间站初始位置**：`state.station.pos` 与任意天体之间距离 ≥ `body.r + 260`，且与恒星 ≥ 500；优先保留现有 `(-780, -260)` 附近的安全区，若被占用则在世界外环 600-900 半径环上找一个空位。

### 1.3 半径分布

| 类型     | r min | r max | 备注 |
|---------|------:|------:|------|
| star    | 110   | 150   | level 6 提升到 140-170 |
| planet  | 70    | 160   | 不同关卡可拉开差异 |
| asteroid| 22    | 48    | 终局守护者另算（见 §3） |

### 1.4 资源量分布

每个天体 `body.amount`（恒星固定为 9999，与 v0.1.x 一致）：

| 类型     | amount min | amount max | level 影响 |
|---------|-----------:|-----------:|-----------|
| star    | 9999       | 9999       | 不变 |
| planet  | 900        | 1800       | 每 level +5% 容量上限（level 6 不增加，专注战斗） |
| asteroid| 280        | 620        | 每 level +3% 容量上限 |

公式：`amount = round(rand(min, max) * (1 + level * factor))`，`planet factor = 0.05`，`asteroid factor = 0.03`，`level 6` 走 `level = 5` 的容量（避免终局突然爆资源破坏节奏）。

### 1.5 资源倾向类型（4 类星系 + 1 终局）

每关生成时按 level 与 random 决定星系类型，影响每个资源天体的 `body.resource` 抽取权重：

| 星系类型     | ore : metal : gas : plasma | 出现关卡（建议） |
|------------|--------------------------:|----------------|
| 富矿星系     | 50 : 25 : 15 : 10        | level 0、3 优先 |
| 气体星云     | 15 : 15 : 55 : 15        | level 1、4 优先 |
| 等离子风暴   | 10 : 15 : 25 : 50        | level 2、5 优先 |
| 平衡星系     | 30 : 25 : 25 : 20        | 任意 fallback |
| 终末守护星系 | 20 : 30 : 25 : 25        | level 6 固定 |

调度建议（实现简化）：

- level 0：固定"富矿星系"（保证早期玩家先吃到 v0.1.x 同款 ore 体验）。
- level 1：固定"气体星云"。
- level 2：固定"等离子风暴"。
- level 3-5：在前 3 类 + 平衡星系中按 random 抽取，**保证不与上一关重复**。
- level 6：固定"终末守护星系"。

恒星的 `body.resource` 一律为 `plasma`（与 v0.1.x 一致），不参与权重抽取；权重只决定 planet 与 asteroid 的资源类型。

### 1.6 星系视觉调色板（5 套）

每个 level 固定绑定一套调色，确保玩家肉眼可识别：

| 调色名称   | 恒星颜色 [r,g,b,a]         | 背景星点密度 | 星点颜色提示              | 建议星云特效 | 绑定关卡 |
|-----------|---------------------------|-------------|--------------------------|-------------|---------|
| 暖阳金     | [1.00, 0.74, 0.18, 1]     | 360         | [0.65, 0.82, 1.0]        | 无          | level 0 (与 v0.1.x 一致) |
| 翠雾绿     | [0.55, 0.95, 0.65, 1]     | 400         | [0.75, 0.95, 0.85]       | 浅绿星云环  | level 1 |
| 紫电狂涛   | [0.78, 0.40, 1.00, 1]     | 440         | [0.85, 0.70, 1.0]        | 紫色闪电粒子| level 2 |
| 蓝白冷锋   | [0.55, 0.78, 1.00, 1]     | 380         | [0.85, 0.92, 1.0]        | 冷雾环      | level 3-4 任选 |
| 赤红凶兆   | [1.00, 0.30, 0.20, 1]     | 480         | [1.00, 0.65, 0.55]       | 红色危险光环| level 5、6 |

实现指引：

- 背景颜色保留现有 `gl.clearColor(0.012, 0.022, 0.048, 1)` 基底；调色通过 `state.galaxy.palette` 注入恒星与星点。
- 星点数量按 `palette.starCount` 重生成（替换 `initWorld` 中固定的 `360`）。
- 终局星系（level 6）使用"赤红凶兆"，并在恒星外多一圈半径 320、宽 6、alpha 0.18 的危险光环，PM 与制作人能一眼识别。

## 2. 难度爬升曲线

### 2.1 综合压力公式（校准工具）

```
pressureScore(level) = (1 / baseTimer(level))
                     * waveCount(level)
                     * avgEnemyHp(level)
                     * avgEnemyDmg(level)
```

约束：

- `pressureScore(0)` ≤ `pressureScore(v0.1.x baseline)` = 1.00（基准）。
- `pressureScore(1)` ≤ 1.05，`pressureScore(2)` ≤ 1.10。
- `pressureScore(3)` ∈ [1.20, 1.40]，`pressureScore(4)` ∈ [1.30, 1.55]。
- `pressureScore(5)` ∈ [1.40, 1.60]，`pressureScore(6)` 由终局规则决定（见 §3，不走此公式）。

实现成员可在落地后用一个开发态 console 函数打印各 level 的 `pressureScore`，PM 在浏览器回归时对照。

### 2.2 spawnTimer 公式

把 v0.1.x 的 `rand(26, 40)` 改写为按 level 调整的 base，新增 `levelTimerScale`：

```
baseTimer(level) = rand(26, 40) * levelTimerScale(level)
state.spawnTimer = baseTimer(level) / sqrt(playerCount)
```

| level | levelTimerScale | 说明 |
|------:|----------------:|------|
| 0     | 1.00            | 等同 v0.1.x |
| 1     | 0.98            | 几乎不可感知 |
| 2     | 0.96            | 轻微加快 |
| 3     | 0.88            | 玩家能感受到节奏加快 |
| 4     | 0.80            | 显著紧凑 |
| 5     | 0.72            | 频繁压力 |
| 6     | 1.20            | 终局**反而拉长**普通 spawn，主舞台让给守护者 |

`nextLevel()` 中现有 `state.spawnTimer = 5` 保留作为关卡进入缓冲；建议改为 `state.spawnTimer = max(5, 8 - level * 0.5)`，让后期切关也有 ~5 秒喘息但不能完全松懈。

### 2.3 每波敌人数量

```
waveCount(level) = ceil(playerCount * rand(0.6, 2) * countMul(level))
```

| level | countMul | 备注 |
|------:|---------:|------|
| 0     | 1.00     | 不变 |
| 1     | 1.00     | 不变（节奏靠 timer，不增量） |
| 2     | 1.05     | 极轻 |
| 3     | 1.15     | 明显 |
| 4     | 1.25     | 明显 |
| 5     | 1.35     | 强烈 |
| 6     | 0.70     | 终局让位给守护者 |

实现注意：`ceil` 容易让 1 人模式抖动（0.6×1.0 与 0.6×1.05 都进 1），可以在 ceil 之前累加一个 `state.spawnFractional` 累加器，超过 1 再 +1，避免低关卡敌人数量虚增。

### 2.4 敌人类型比例

现有 `spawnWave` 比例计算改为以下结构：

```
roll = Math.random()
asteroidChance = baseAsteroidChance(time) + levelAsteroidBias(level)
pirateChance   = pirateBase(level)
stationChance  = stationBase(level) * (asyncEnabled ? 1 : 0)
// 归一化后抽取
```

| level | asteroidBias | pirateBase | stationBase | 说明 |
|------:|-------------:|-----------:|------------:|------|
| 0     | 0.00         | 0.78       | 0.00        | level 0 完全没有 station，保护新玩家 |
| 1     | 0.00         | 0.78       | 0.04        | 偶发 |
| 2     | 0.00         | 0.78       | 0.08        | 偶发 |
| 3     | +0.02        | 0.74       | 0.14        | station 出现明显 |
| 4     | +0.04        | 0.70       | 0.20        | 多 station |
| 5     | +0.06        | 0.66       | 0.26        | station 高频 |
| 6     | -0.05        | 0.80       | 0.05        | 终局以海盗增援为主 |

实现细节：`asteroidChance` 保留现有 `getAsteroidSpawnChance()` 的 time 渐变曲线（22% → 38%），再叠加 `levelAsteroidBias`；最终 clamp 到 [0.10, 0.50]。`stationChance` 与 `pirateChance` 之和占据剩余概率，按比例归一化。

### 2.5 敌人 HP 与伤害（分段倍率）

不使用线性，使用"分段+对数压缩"，避免后期 HP 爆炸：

```
levelMul(level) = 1 + ln(1 + level) * 0.35
// level 0 → 1.000
// level 1 → 1.243
// level 2 → 1.385
// level 3 → 1.485
// level 4 → 1.563
// level 5 → 1.626
// level 6 → 1.681
```

但 **level 0-2 不允许提升 HP/伤害**（约束：早期减压不回潮）。最终敌人参数：

| 类型     | 字段       | level 0 | level 1 | level 2 | level 3 | level 4 | level 5 | level 6 |
|---------|-----------|--------:|--------:|--------:|--------:|--------:|--------:|--------:|
| asteroid| HP        | 26      | 26      | 26      | 32      | 36      | 40      | 42      |
| asteroid| 撞击伤害  | 18      | 18      | 18      | 20      | 22      | 24      | 24      |
| pirate  | HP        | 68      | 68      | 68      | 88      | 100     | 108     | 112     |
| pirate  | 撞击伤害  | 8       | 8       | 8       | 10      | 11      | 12      | 12      |
| pirate  | 弹道伤害  | 10      | 10      | 10      | 12      | 13      | 14      | 14      |
| station | HP        | 260     | 260     | 260     | 330     | 380     | 410     | 430     |
| station | 弹道伤害  | 16      | 16      | 16      | 18      | 20      | 22      | 22      |
| station | 增援节奏  | 7s      | 7s      | 7s      | 6.5s    | 6s      | 5.5s    | 5.5s    |

终局守护者另算（§3），上表 level 6 指**普通敌人**在终局星系的参数。

实现建议：把 `spawnEnemy` 的 `data` 表改造为函数 `getEnemyData(kind, level)`，返回这张表里对应的字段；现有 `COLLISION_FEEL.asteroidCellDamage` / `pirateCellDamage` 改为查表，避免常量被 level 替换后丢失原值（保留常量作为 level 0 默认，新增 `getCollisionDamage(kind, level)` 包装）。

### 2.6 资源点密度变化

不主动减少资源量（避免后期采集变累赘），改为通过 §1.1 的天体数量增加与 §1.4 的容量微涨，让玩家感受"资源更丰富但敌人更多"。

唯一例外：level 6 资源量按 level 5 不再增长（§1.4），引导玩家"打完守护者就结束"，而不是无限刷资源。

## 3. 本局终局星系参数（level 6）

### 3.1 守护者敌人参数（基于 station 强化）

新增 `kind = "guardian"` 作为 station 的参数化变体（**只是数据扩展，AI 行为完全复用 station 分支，不引入新 AI**）：

| 字段                 | 数值                           | 说明 |
|---------------------|-------------------------------|------|
| HP                  | 1200                          | 约 station(260) × 4.6 |
| r（半径）           | 72                            | 约 station(48) × 1.5 |
| accel               | 16                            | 比 station(20) 略慢，避免追击玩家不掉头 |
| drag                | 0.2                           | 同 station |
| range               | 600                           | 比 station(520) 略远 |
| reward              | 80                            | 击毁后给予 80 金属 + 12 气体（reward × 0.15） |
| spin                | 0.15                          | 缓慢自转 |
| 弹道伤害            | 26                            | 约 station(16) × 1.6 |
| reload              | 1.6（同 station）             | 不加快，避免单 boss 太脏 |
| 增援节奏            | 每 4.5 秒 spawn 一个 pirate    | 比 station(7s) 快，但海盗 HP 用 level 5 数据（108） |
| 增援上限            | 同时存活 ≤ 4 个由守护者刷出的 pirate | 避免雪球；超过则跳过该次 spawn |
| 颜色（rect 主体）   | [0.55, 0.05, 0.10, 1]         | 深红 |
| 颜色（rect 翼）     | [0.85, 0.20, 0.18, 1]         | 警示红 |
| 危险光环            | 半径 r+22，宽 6，[1, 0.25, 0.20, 0.45]，呼吸频率 2.5Hz | 视觉特殊化 |

生成时机与位置：

- `nextLevel()` 推进到 `level === 6` 时，标记 `state.run.endgame = true`，禁用普通 spawnWave 的 station 抽取（已由 §2.4 设为 0.05，可继续允许少量普通 station 作为环境敌人）。
- 在新关卡生成后 6 秒，直接调用 `spawnEnemy("guardian", x, y)`，位置 = 距玩家 1100-1300 半径环上的随机点。
- 守护者只生成 1 个；若被摧毁，标记 `state.run.guardianDefeated = true` 并触发结算面板（§4）。

### 3.2 终局星系视觉特殊化

- 调色板：赤红凶兆（§1.6）。
- 恒星半径：140-170（§1.3）。
- 恒星外加一圈半径 320、宽 6、alpha 0.18 的危险光环。
- 在守护者周围 80 半径范围内永久发射红色粒子（沿用现有 `spawnParticle`，颜色 `[1, 0.25, 0.18, 1]`，每 0.15 秒一次），让玩家从远距离就能定位 boss。
- HUD 上方添加红字提示："终末星系：摧毁守护者结束本局"（实现侧可复用 `currentGoalEl`/`statusAlertsEl`）。

### 3.3 终局任务完成判定

- `state.run.objective` 在 level 6 时由 `createObjective()` 改为返回 `{ type: "guardian", text: "击毁本局守护者", target: 1, progress: 0 }`。
- 在 `updateEnemies` 死亡处理处增加：`if (enemy.kind === "guardian") { state.run.objective.progress = 1; state.run.guardianDefeated = true; }`。
- `updateObjective` 既有 `progress >= target → grantObjectiveReward()` 逻辑可直接复用；`grantObjectiveReward` 内特判：`if (state.run.level === 6) { 触发结算面板逻辑，不要再设置 awaitingObjectiveChoice }`。

### 3.4 终局期间普通敌人 spawn

- **不暂停**，按 §2.4 level 6 数据继续 spawn（频率更慢、station 概率极低、海盗为主）。
- 守护者本身刷出的 pirate 不计入 spawnWave 数量上限。
- 守护者死亡后，普通 spawnWave **停止**（`state.run.guardianDefeated === true` 时跳过 `spawnWave`），但已存在敌人不清场，让玩家自行清理或选择留在终结星系。

## 4. 本局结算面板参数

### 4.1 字段

结算面板（DOM 新增容器，例如 `#runSummaryPanel`，默认隐藏，由守护者死亡触发显示）：

| 字段          | 数据来源                                              | 显示示例 |
|--------------|------------------------------------------------------|---------|
| 完成关卡数    | `state.run.completedObjectives`                       | 7 关（6 普通 + 1 终局） |
| 累计任务奖励  | `state.run.totalObjectiveRewardBase`（本局累加）       | 35 点 |
| 终局奖励倍率  | 固定显示 `×1.5`                                       | ×1.5 |
| 总点数        | `floor(totalObjectiveRewardBase * 1.5)`               | 52 点 |
| 守护者状态    | "已击毁"                                              | （文字+图标） |

实现细节：

- 在 `state.run` 新增字段 `totalObjectiveRewardBase`（每次 `grantObjectiveReward` 累加 `gained`，**注意保留向下兼容：若旧逻辑直接给 meta.points，则改为先累加再 commit**）。
- 终局触发结算时：
  - `bonusPoints = floor(totalObjectiveRewardBase * 1.5) - totalObjectiveRewardBase`（即额外的 50% 部分）。
  - `state.meta.points += bonusPoints`；调用 `saveMeta()`。
  - **注意**：普通关每次完成已经把 `gained` 加进了 `meta.points`，所以这里**只补差额**，而不是再加一次 1.5 倍，避免双倍发放。

```
普通关累计已发：totalObjectiveRewardBase
结算追加：floor(totalObjectiveRewardBase * 1.5) - totalObjectiveRewardBase
总到手 = floor(totalObjectiveRewardBase * 1.5)
```

### 4.2 按钮文案

| 按钮 ID 建议              | 文案                       | 行为 |
|--------------------------|---------------------------|------|
| `summaryRestartBtn`      | "开始新局"                | `location.reload()`，沿用 `gameOver` 同款重启路径，保证 meta 不丢 |
| `summaryStayBtn`         | "留在终结星系自由游玩"     | 关闭结算面板，`state.run.endgameExplore = true`，不再触发任何 spawnWave |

### 4.3 进入新局后

- `location.reload()` 后所有 run 状态由 `state` 初始值重置；`state.meta` 由 `loadMeta()` 从 `localStorage` 恢复 → 局外天赋、点数、解锁全部保留。
- **不需要新增 meta 字段**，所有结算逻辑都在 run 域完成（§6 详述）。

### 4.4 留在终结星系

- 关闭结算面板，不 reload。
- `state.run.endgameExplore = true` 时：
  - 跳过 `spawnWave`（普通敌人不再来）。
  - 不允许再次跃迁（因为没有"下一关"，UI 上的"跃迁下一星系"按钮隐藏或灰显）。
  - 玩家仍可建造、采矿、移动、被现存敌人攻击。
  - 守护者已死，星系标记为"已平定"，可在 HUD 显示"自由探索：再次开始新局请点暂停菜单"（如果有暂停菜单的话；没有则在右上角增加一个常驻"再次开始新局"按钮，沿用 `summaryRestartBtn` 行为）。

## 5. 早期减压边界与回归校验

PM 在浏览器回归时可逐条勾选，任一不通过即视为"减压回潮"：

| # | 校验点 | 量化指标 |
|--:|--------|---------|
| 1 | level 0 进入后 60 秒内 spawnWave 触发次数 | ≤ 1 次（即第一波在 ~22 秒，第二波 ≥ 60 秒） |
| 2 | level 0 前 120 秒内出现的 station 数量 | = 0（受 §2.4 `stationBase = 0` 保护） |
| 3 | level 0 单波敌人数量（1 人模式） | ≤ 2（受 `countMul=1.0` 与 `ceil(rand(0.6, 2))` 上限保护） |
| 4 | level 0 asteroid 撞击伤害 | = 18（不变） |
| 5 | level 0 pirate HP | = 68（不变） |
| 6 | level 0 与 v0.1.x 同 seed 下世界天体数 | level 0 = 6 天体（恒星 1 + 行星 2 + 小行星 3），与 v0.1.x 一致 |
| 7 | level 0-1 资源初始量（金属/矿/气/等离子） | = `130/60/35/8`（不变，由 `state.resources` 决定） |
| 8 | level 0-1 单局核心理论受击次数（pirate 全程触发撞击至少 1 次的可能性下） | ≤ v0.1.x baseline，预期 0-2 次轻微撞击 |
| 9 | level 0-2 spawnTimer 平均值（baseTimer 期望 33） | level 0=33.0，level 1=32.3，level 2=31.7，与基线偏差 ≤ 4% |
| 10 | level 0-2 综合压力分（§2.1 公式） | level 0=1.00，level 1 ≤ 1.05，level 2 ≤ 1.10 |

若 PM 发现某一项超标，应立即标记为阻塞，回退该 level 的对应参数到表中数值。

## 6. 存档兼容

### 6.1 结论：v0.2.0 不新增 meta 字段

- `state.meta` 结构保持：`{ points, mining, hull, weapons, unlocks: { weaponFrame, efficientCore } }`。
- `SAVE_KEY = "star-ring-station-meta-v1"` 不动。
- 所有新增字段（`endgame`、`endgameExplore`、`guardianDefeated`、`totalObjectiveRewardBase`、`galaxy.palette`、`galaxy.type` 等）一律放在 `state.run` 或 `state.galaxy`（新建命名空间），不持久化。
- 旧存档读取：完全兼容（`loadMeta()` 已经用 `{ ...base, ...JSON.parse(...) }`，无字段会自动 fallback）。

### 6.2 若以后需要新增 meta 字段

参考策略（**本版本不执行**，仅供后续版本参考）：

- 新字段在 `loadMeta` 的 `base` 模板中给默认值（0 / false / [] / {}）。
- `SAVE_KEY` 改为 `star-ring-station-meta-v2`，同时在 `loadMeta` 内尝试读取 v1 并迁移：

```
const v1 = JSON.parse(localStorage.getItem("star-ring-station-meta-v1") || "null");
if (v1) { migrated = { ...base, ...v1, newField: defaultValue }; }
```

- 迁移完成后写入 v2，**保留 v1**（防止用户回退游戏版本时丢档）。

### 6.3 SAVE_KEY 不动 → 实现成员必须避免在 saveMeta() 之前往 state.meta 写入任何新字段

例如不能写 `state.meta.endgame = true`；endgame 状态只放在 `state.run`。
