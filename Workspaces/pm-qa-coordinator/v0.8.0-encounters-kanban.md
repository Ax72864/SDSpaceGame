# v0.8.0 局内随机事件系统 PM 看板

## 本轮目标

在 v0.7.0 战斗操控发布后，加入独立于 `OBJECTIVE_TYPES` 的局内随机事件系统，让玩家每关遭遇 1-3 个 trader / distress / ambush / derelict / signal 非主线事件，形成单局记忆点、决策深度和与既有战斗 / NPC / 拾荒系统的协同。

## 背景与制作人方向

制作人方向文档：`Workspaces/game-producer-lead-designer/v0.8.0-encounters-direction.md`，上游指定 commit `a9d9199`。基线为 HEAD `a9d9199`（v0.7.0 发布后），当前版本风险级别为项目中高风险。

### 方向要点摘要

- 新增平行 `ENCOUNTER_TYPES` 注册表，不塞入 `OBJECTIVE_TYPES`，事件作为可拒绝 / 可错过 / 不影响主线的弱语义层。
- 每关按权重抽 1-3 个事件，60-180 秒延迟触发，每种事件单星系冷却，level 6 终末关不刷事件。
- 5 种事件：trader 漂流商船、distress 救援请求、ambush 伪装陷阱、derelict 沉船秘窟、signal 神秘信号。
- 扩展 NPC 系统：`npc.kind` 增加 trader / distress-pilot / pirate-ambush，新增 `npc.role` 和 `npc.encounterId`，隔离 objective NPC 与 encounter NPC。
- 独立 RNG 流：`state.run.seed ^ 0xb5297a4d + level * 0xcc9e2d51`，不能破坏 `__gameTest.findEscortSeed / getObjectiveType`。
- HUD 新增 `buildEncounterHudAlerts()`，`showHostileStationAlert` 增加 priority 队列，fragment 新增 `origin = "derelict" / "signal"` 分色。

### 体验目标

1. **单局记忆点**：玩家每关都会遇到 1-3 个随机遭遇事件，每个有视觉化触发、玩家选择和后果反馈。
2. **决策深度**：5 种事件覆盖交易、援助、真伪判断、自由拾荒和高回报信号，让玩家在主任务外做有意义的取舍。
3. **与已有系统协同**：distress / ambush 复用 v0.7.0 锁敌战斗，trader / distress 复用 v0.4.0 NPC，derelict / signal 复用 v0.3.0 / v0.6.0 fragment 拾荒。

### 必做范围

- 工程基础：`ENCOUNTER_TYPES`、`state.run.encounters`、`state.run.encounterCooldownThisGalaxy`、`notifyEncounters`、`applyEncounterReward`、`seedEncountersForLevel`、独立 RNG、`nextLevel()` 清理与重新滚池。
- 事件实现：trader / distress / ambush / derelict / signal 5 种事件的生成、触发、选择、过期、奖励和完成 / 失败状态。
- NPC 扩展：`npc.kind`、`npc.role`、`npc.encounterId`、`tickNpc()` 分支、`damageNpc()` 分流、`buildNpcHudAlerts()` role 聚合。
- HUD 与视觉反馈：`buildEncounterHudAlerts()`、`showHostileStationAlert` priority 队列、`.alert-encounter-*` CSS、fragment derelict / signal 分色和世界视觉标记。
- 兼容与验证：主程 ADR、数值方案、T1-T5 分批实现、节点 A/B/C 中间审查、专项审核、PM 浏览器级 + 主观回归、制作人最终决定与发布安排。

### 不做范围

- 不破坏 v0.4.0 现有 `OBJECTIVE_TYPES`：事件是完全独立的平行系统，`OBJECTIVE_TYPES` 注册表 / `createObjective` / `notifyObjective` / `grantObjectiveReward` 等不修改。
- 不破坏 v0.5.0 研发系统 / v0.6.0 hostile-station / v0.7.0 锁敌：事件触发战斗时复用现有 enemy 路径，玩家 turret / shield / missile / 锁敌行为不变。
- 不修改 `SAVE_KEY`：`state.run` 仍不存盘，事件状态刷新即重置。
- 不引入新 enemy kind：ambush 使用现有 pirate，distress 围困者也用 pirate。
- 不引入新建造 cell 类型。
- 不做联机 / 异步对战。
- 不做局外天赋树重构。
- 不做新探索 / 地图系统。
- 不做跨关持续事件，全部事件单关结束清空。
- 不做炮塔朝向 / 平面护盾方向性。

### 兼容要求

- v0.1.x / v0.2.0 / v0.2.1 / v0.3.0 / v0.4.0 / v0.5.0 / v0.6.0 / v0.7.0 全部回归点必须通过。
- 现有 5 种 enemy kind + 8 种 `OBJECTIVE_TYPES` 完全等价。
- 现有 `notifyObjective` 调用点不修改，仅在调用之后追加 `notifyEncounters`。
- 现有 `grantObjectiveReward` 不修改，事件独立走 `applyEncounterReward`。
- escort 任务 NPC 行为完全等价，escort 标记 `role: "objective"`，事件 NPC 标记 `role: "encounter"`。
- v0.7.0 锁敌 / F 键齐射 / `[` / `]` salvoSize / HUD 武器行不破坏；事件战斗自动启用。
- `SAVE_KEY` 不变；旧存档兼容，`state.run.encounters` 缺失时默认 `[]`。
- `__gameTest.findEscortSeed / getObjectiveType / resetRun` 等测试钩子完全等价。
- 玩家可完全不参与事件继续游戏，玩法等价 v0.7.0。

## 风险定级

- 风险等级：项目中高风险。
- 原因 1：核心系统扩展，新增与 `OBJECTIVE_TYPES` 平行的 `ENCOUNTER_TYPES`、事件总线、奖励路径和跨关清理。
- 原因 2：5 种新事件同时进入，覆盖交易、救援、伏击、拾荒、信号多条玩法路径，触发与奖励分支多。
- 原因 3：NPC 多 kind 扩展，`friendly-cargo` 既有 escort 语义必须与 trader / distress-pilot / pirate-ambush 隔离。
- 原因 4：HUD 聚合层重构，`buildEncounterHudAlerts` 与 hostile-station、NPC、fragment、武器 HUD 共享视觉空间，priority 队列错误会造成告警堆叠或遮挡。
- 原因 5：RNG 流并行，事件随机必须独立于 objective 随机，不能破坏既有 seed 稳定性和测试钩子。
- 中高风险流程：制作人方向 -> PM 拆分 -> 主程技术 ADR -> 数值方案 -> T1-T5 分批实现 -> 节点 A/B/C 中间审查 -> 专项审核 -> PM 浏览器级 + 主观回归 -> 制作人最终决定与发布安排。
- 审核基底约束：专项审核使用 `expert-agent-planner-gpt55`，必须与执行不同基底。

## 缺陷记录格式

- 标题：一句话描述问题。
- 严重程度：S0 阻断 / S1 严重 / S2 一般 / S3 轻微。
- 影响范围：说明影响启动、存档、`ENCOUNTER_TYPES`、`state.run.encounters`、`notifyEncounters`、`applyEncounterReward`、独立 RNG、trader、distress、ambush、derelict、signal、NPC kind / role、fragment origin、HUD priority、hostile-station、OBJECTIVE_TYPES、研发、锁敌、齐射、WASD、采集、建造、终局或自由游玩中的哪一部分。
- 复现步骤：按浏览器实机操作顺序列出，包含 seed / level / objective type / encounter type / encounter id / spawn 位置 / NPC role / fragment origin / 玩家选择 / HUD 文案 / 任务状态 / 奖励变化。
- 实际结果：记录玩家实际看到或系统实际发生的情况，必要时附运行态采样。
- 期望结果：记录按制作人方向、ADR 或数值方案应发生的体验。
- 优先级：P0 / P1 / P2，并说明是否阻塞接受。
- 负责人：明确后续处理成员。
- 当前处理：待分配 / 修复中 / 待审核 / 已回归 / 接受为已知问题。
- 回归要求：修复后尽量按原复现步骤复测，并说明失败路径是否关闭。
