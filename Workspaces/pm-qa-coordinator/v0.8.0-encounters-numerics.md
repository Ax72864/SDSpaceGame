## §0. 制作人方向回放与硬约束

### §0.1 体验目标三条（按重要程度排序）

1. **单局记忆点**：玩家每关都会遭遇 1-3 个非主线事件，每个事件有视觉化触发 + 玩家选择 + 后果反馈；玩家能在结算时回忆"这局遇到了什么"，形成碎片化的局内叙事。
2. **决策深度**：5 种事件覆盖"友好交易 / 战斗援助 / 真伪判断 / 自由拾荒 / 高风险高回报" 5 种决策模式；每种事件的选项后果明显（不是"一律捡起来就好"），让玩家在战斗和拾荒之外多一层"做选择"的乐趣。
3. **与已有系统协同**：distress / ambush 触发战斗自动启用 v0.7.0 锁敌 + F 键齐射；derelict / signal 复用 v0.3.0 fragment dock；trader / distress 复用 v0.4.0 NPC 系统；事件不抢 OBJECTIVE_TYPES 的注意力，作为平行层叠加。

### §0.2 不做范围（数值层硬约束）

- 不修改 v0.1.x / v0.2.0 / v0.2.1 / v0.3.0 / v0.4.0 / v0.5.0 / v0.6.0 / v0.7.0 已发布数值（含 `FRAGMENT_*` / `BRIDGE_*` / wreck 物理 / NPC HP / 奖励倍率 / spawn 曲线 / `techLevel` 公式 / tier 倍率 / mod 倍率 / hullMod / weaponMod / hostile-station HP 池 / cells 配置 / wreck 分布 / assault 任务奖励 / `priorityTarget` 锁敌参数 / `salvoSize` 4 档配置 / F 键防抖 0.3s）。
- 不修改 `SAVE_KEY` 与局外存档结构；本方案新增字段仅 `state.run.encounters` / `state.run.encounterCooldownThisGalaxy` / `state.hudCenterAlertQueue` / `npc.role` / `npc.encounterId`，不写入 `state.meta`。
- 不修改 `OBJECTIVE_TYPES` 注册表 / `createObjective` / `notifyObjective` / `grantObjectiveReward`；ENCOUNTER 走平行 `ENCOUNTER_TYPES` + `notifyEncounters` + `applyEncounterReward`。
- 不引入新 enemy kind；distress / ambush 海盗复用现有 `spawnEnemy("pirate")` 路径，复用 v0.7.0 锁敌 + F 键齐射。
- 不引入新建造 cell 类型；derelict / signal wreck 复用现有 `WRECK_FACILITY_WEIGHTS` 仅调整权重。
- 不修改 v0.6.0 hostile-station 总 HP 池 / cells 配置 / wreck 数量 / level 权重 / `hostileStationSpawnedThisGalaxy` 行为。
- 玩家可"完全不参与"事件（不靠近 trader / 忽略 distress 信号 / 远离 ambush / 不拾荒 derelict / 不飞 signal）继续游戏，玩法等价 v0.7.0；事件不阻塞跃迁。
- 不做跨关持续事件（如"贸易商队 N 关后归来"），全部事件单关结束清空，避免引入 SAVE_KEY 迁移负担。
- 不破坏 v0.4.0 escort objective NPC 行为（escort npc 标记 `role: "objective"`，事件 NPC 标记 `role: "encounter"`，两条 alert 通道分离）。

## §1. 每关事件数量与权重

> 设计意图：让玩家"每关都有事件 + 但密度不爆炸"——level 越高事件越密集匹配资源压力上升曲线；level 6 终末关 0 事件保护 guardian 决战体验；并发上限 3 防止 HUD 信息过载。

### §1.1 每关事件数量

| level | 事件数量（推荐）| PM 可微调范围（±20%）| 设计理由 |
|---|---|---|---|
| **level 0** | **1** | 不允许微调 | 教学关只触发 1 个事件让玩家熟悉机制；建议优先 trader（最友好） |
| **level 1** | **1-2** | 不允许（上下界）| 引入 distress / derelict / signal，让玩家见识 3-4 种事件类型 |
| **level 2** | **2-3** | 不允许（上下界）| 引入 ambush，玩家开始"真伪判断"决策 |
| **level 3** | **2-3** | 不允许（上下界）| 5 种事件全部可触发，每关有"组合体验" |
| **level 4** | **2-3** | 不允许（上下界）| 与 v0.6.0 hostile-station 共存关；事件 + BOSS 同时存在不冲突 |
| **level 5** | **2-3** | 不允许（上下界）| 跃迁前最后一关，事件密度最高 |
| **level 6** | **0** | ❌ 不允许（制作人拍板）| 终末关 guardian 决战体验保护，无任何事件干扰 |

> **§1.1 备注**：事件数量按 level 递增是为了配合资源压力曲线（v0.5.0 研发系统在 level 3+ 需要大量等离子）；事件数量上下界（如 level 1 "1-2"）由独立 RNG 流 `seedEncountersForLevel` 在 [min, max] 闭区间内均匀采样决定。**事件数量上下界不允许微调**（制作人拍板项），避免 PM 调到 4-5 个事件让 HUD 信息过载。

### §1.2 5 种事件在不同 level 的权重表

> 权重 = 该 level 滚池时该事件被选中的相对概率；权重为 0 表示该 level 不可能出现。

| level | trader | distress | ambush | derelict | signal | 备注 |
|---|---|---|---|---|---|---|
| **level 0** | **100** | 0 | 0 | 0 | 50 | 教学关只有 trader / signal（最友好 + 拾荒入门）|
| **level 1** | **80** | **60** | 0 | **40** | **40** | 引入 distress / derelict，玩家见识"战斗 + 拾荒"|
| **level 2** | **70** | **70** | **50** | **50** | **40** | 引入 ambush，5 种事件首次完整可触发（除非 cooldown 限制）|
| **level 3** | **60** | **70** | **60** | **60** | **40** | distress / ambush 权重升高，战斗事件占比增加 |
| **level 4** | **50** | **70** | **70** | **70** | **40** | ambush 权重持平 distress（玩家此时已有 v0.7.0 锁敌应对）|
| **level 5** | **50** | **70** | **80** | **70** | **50** | ambush 权重最高（跃迁前最后挑战）；signal 权重微升 |
| **level 6** | 0 | 0 | 0 | 0 | 0 | 终末关无事件 |

> **§1.2 备注**：权重表对照方向 §"数值边界建议"第 1 项"trader 各 level 都有 / distress level 1+ / ambush level 2+ / derelict level 1+ / signal level 0+"完全一致。**权重不允许微调**（制作人 / ADR 拍板项），避免 PM 调到"level 1 全是 ambush" 等极端情况破坏决策深度曲线。

### §1.3 总并发上限

| 字段 | 推荐值 | 是否允许微调 | 设计理由 |
|---|---|---|---|
| 同关最大并发事件数 | **3 个** | ❌ 不允许（制作人 / ADR 拍板）| 5 种事件类型 → 每关随机选 1-3 个；3 个并发 HUD 信息密度可控（每事件 1 行 HUD），4 个以上 HUD 行数 ≥ 武器行 + 任务行 + 资源行总和，信息过载 |
| 单星系最大并发（含 cooldown）| **3 个**（与单关一致）| 不允许微调 | 跨关清空 cooldown，所以单星系上限等于单关上限 |

### §1.4 事件数量与权重的滚池伪代码

```js
function seedEncountersForLevel(level, rng) {
  if (level === 6) return [];                              // 终末关无事件
  const [minN, maxN] = ENCOUNTER_LEVEL_COUNT[level];      // §1.1 表
  const n = minN + Math.floor(rng.next() * (maxN - minN + 1));
  const weights = ENCOUNTER_LEVEL_WEIGHTS[level];          // §1.2 表
  const cooldown = state.run.encounterCooldownThisGalaxy;
  const picked = [];
  for (let i = 0; i < n; i++) {
    const type = pickWeightedExcludingCooldown(weights, cooldown, picked);
    if (!type) break;                                       // 所有可选类型耗尽
    picked.push(type);
    if (type !== "trader") cooldown.add(type);              // trader 可 2 次
    else if ((cooldown.get("trader") || 0) >= 1) cooldown.add("trader-2nd");
  }
  return picked.map((type) => createEncounter(type, level, rng));
}
```

> **§1.4 备注**：本伪代码仅给出"按权重 + cooldown 抽样"的数值意图；具体数据结构与 RNG 流接口由主程 ADR 决定。

## §2. 事件触发延迟与过期

> 设计意图：让玩家"先稳住当前情况再应对事件"——按事件复杂度递增延迟（trader 60s 最简单 → signal 180s 最远距离），避免一关开局就堆 5 个 alert；多事件触发时间错开避免同时弹 alert；过期时间按事件类型分类（战斗事件即时 / 拾荒事件永久 / 交易事件短时）。

### §2.1 spawn 后触发延迟参数表

| 事件 | spawn 后触发延迟 | PM 可微调范围（±20%）| 设计理由 |
|---|---|---|---|
| **trader 漂流商船** | **60.0s** | 48 - 72 s | 最早出现，让玩家先体验"友好遭遇"；60s ≈ 玩家完成 1-2 个 fragment dock 的时间 |
| **distress 救援请求** | **90.0s** | 72 - 108 s | 引入战斗事件，玩家此时已有时间装备 turret cell |
| **ambush 伪装陷阱** | **120.0s** | 96 - 144 s | 真伪判断需要玩家"已知 distress 存在"才有迷惑性，故晚于 distress |
| **derelict 沉船秘窟** | **150.0s** | 120 - 180 s | 大规模拾荒，给玩家充足时间应对前面战斗事件 |
| **signal 神秘信号** | **180.0s** | 144 - 216 s | 最远距离，玩家需要"主动飞过去"，故最晚出现让玩家有时间评估值不值 |

### §2.2 事件过期参数表

| 事件 | 过期时间 | PM 可微调范围（±20%）| 设计理由 |
|---|---|---|---|
| **trader** | **30.0s 内可交易** | 24 - 36 s | 友好商船有限时停留，但 30s 足够玩家飞近 + 选择 |
| **distress** | **45.0s 内可援助** | 36 - 54 s | 友方 NPC 在被围困期间可被援助；超时友方 NPC 被海盗击毁 |
| **ambush** | **触发战斗后即时** | 不允许微调 | 玩家进入 300px 触发战斗后没有"过期"概念；战斗清场或玩家逃跑 500px 外 = 事件结束 |
| **derelict** | **关内永久**（直到全部 wreck dock 或跨关清空）| 不允许微调 | 拾荒场景应让玩家"自由决定何时清理"，配合 v0.3.0 fragment dock 节奏 |
| **signal** | **关内永久**（直到 fragment dock 或跨关清空）| 不允许微调 | 玩家要"远征拾取"，永久存在让玩家可在主任务间隙顺路 |

### §2.3 多事件触发时间错开规则

| 字段 | 推荐值 | PM 可微调范围（±20%）| 设计理由 |
|---|---|---|---|
| 相邻事件最小间隔 | **30.0s** | 24 - 36 s | 让玩家有时间"消化"上一事件 alert 再面对下一事件；30s ≈ trader alert 持续 4s + 玩家做决策 + 飞行响应时间 |
| 时间错开算法 | spawn 时按"事件 N 的 spawn time = max(自身延迟, 事件 N-1 spawn time + 30)" | 不允许微调（算法）| 简单可预期；不引入复杂调度优化 |

### §2.4 spawn 时间表的具体推算示例

> 假设 level 3 滚到 3 个事件 `[trader, distress, ambush]`，按 §2.3 错开算法：

| 事件 | 自身延迟 | 上一事件 spawn 时间 | 实际 spawn 时间 | 过期时间 |
|---|---|---|---|---|
| trader | 60s | - | **60s** | 60+30 = 90s |
| distress | 90s | 60s | **max(90, 60+30) = 90s** | 90+45 = 135s |
| ambush | 120s | 90s | **max(120, 90+30) = 120s** | 触发后即时 |

> **§2.4 备注**：本表显示 §2.3 算法在大多数情况下不会强制延迟（因为 §2.1 自身延迟已按 60→90→120→150→180 递增）；仅当 RNG 滚到"derelict + signal"或类似密集组合时才触发"+30s 错开"。

### §2.5 触发延迟与过期的硬约束

```js
function tickEncounter(encounter, dt, now) {
  if (encounter.state === "pending" && now >= encounter.spawnAt) {
    encounter.state = "active";
    encounter.spawnEntities();                              // 生成 NPC / wreck / fragment
    pushHudCenterAlert(encounter.type, "appear");           // §10 alert
  }
  if (encounter.state === "active" && encounter.expireAt !== Infinity && now >= encounter.expireAt) {
    encounter.state = "expired";
    encounter.cleanupEntities();                            // 移除 NPC / wreck
    pushHudCenterAlert(encounter.type, "expire");           // §10 失败 alert
  }
}
```

> 实现时建议在 `updateEncounters(dt)` 内对每个 active encounter 调用 `tickEncounter(encounter, dt, state.time)`。

## §3. trader 漂流商船数值

> 设计意图：trader 是 5 种事件中**唯一无战斗 + 唯一二选一资源转换**事件，是玩家熟悉 ENCOUNTER 机制的入门；2 个选项让玩家学会"评估 50 金属换 1 等离子是否值"。

### §3.1 NPC 数值参数表

| 字段 | 推荐值 | PM 可微调范围（±20%）| 设计理由 |
|---|---|---|---|
| `npc.kind` | `"trader"` | 不允许微调（kind 名称）| 与 ADR §"NPC 系统扩展" 一致 |
| `npc.role` | `"encounter"` | 不允许微调 | 区分 escort objective NPC |
| `npc.hp` | **80** | 64 - 96 | 受攻击会逃跑（500px 外消失），不会真死；HP 仅用于"玩家攻击是否击退"判定 |
| `npc.speed` | **80 px/s** | 64 - 96 px/s | 沿平行玩家速度方向巡航；不追玩家也不逃 |
| `npc.r`（碰撞半径）| **18** | 不允许微调 | 与 friendly-cargo r=18 一致 |
| spawn 距离玩家 | **400-600 px** | 320-480 / 480-720 px | 最近距离的事件 NPC，让 trader 容易"路过被注意"|
| 受攻击触发逃跑距离 | **500 px**（远离玩家）| 不允许微调 | hp <= 0 时不死，飞到 500px 外消失 |

### §3.2 交易选项数值表

| 选项 | 消耗 | 产出 | 设计理由 |
|---|---|---|---|
| **option-1**：金属→等离子 | **50 金属** | **1 等离子** | 等价 v0.5.0 plasma cell 单次产量（50 金属是早期玩家轻松凑齐）|
| **option-2**：金属→科研 | **100 金属** | **3 科研点** | 科研点在 v0.5.0 研发系统单次升级需 ~50 点；100 金属换 3 科研约等于"打 1 个 pirate"科研收益 |
| **option-3**：忽略 | 0 | 0 | 30s 后 trader 飞走，无后果 |

> **§3.2 备注**：交易比例数值与方向 §"数值边界建议"完全一致。**交易比例不允许微调**（制作人 / ADR 拍板项），避免 PM 微调到"50 金属换 2 等离子"等破坏经济平衡。

### §3.3 trader 选项触发方式

| 触发方式 | 数值 | 设计理由 |
|---|---|---|
| 玩家飞到 trader 100 px 内 | 自动打开"交易菜单"HUD | 与 v0.3.0 fragment dock 100px 一致 |
| 玩家按 1 / 2 / 3 数字键选择 | 立即执行交易 | 复用现有 1-9 数字键，但 trader 菜单激活时优先吃 1/2/3 |
| 玩家飞出 100 px | 自动关闭菜单（不取消事件）| 玩家可重新飞回再选 |
| 玩家完全不靠近 | 30s 后 trader 飞走 = option-3 | §2.2 trader 过期 |

### §3.4 trader 资源不足兜底

| 场景 | 行为 |
|---|---|
| 玩家金属 < 50（选 option-1）| 菜单 option-1 灰色禁用 + 显示"金属不足"|
| 玩家金属 < 100（选 option-2）| 菜单 option-2 灰色禁用 + 显示"金属不足"|
| 玩家金属同时不足两项 | option-1 / option-2 都禁用，仅 option-3（忽略）可选 |

## §4. distress 救援请求数值

> 设计意图：distress 是 5 种事件中**第一次引入战斗**的事件，奖励中等（援助风险有限）；友方 NPC HP 60 让玩家"必须及时援助"否则 NPC 死亡导致事件失败。

### §4.1 NPC 数值参数表

| 字段 | 推荐值 | PM 可微调范围（±20%）| 设计理由 |
|---|---|---|---|
| `npc.kind` | `"distress-pilot"` | 不允许微调（kind 名称）| 与 ADR §"NPC 系统扩展" 一致 |
| `npc.role` | `"encounter"` | 不允许微调 | 区分 escort objective NPC |
| `npc.hp` | **60** | 48 - 72 | 略低于 trader 80（distress 是被围困状态，应该比 trader 更脆弱）|
| `npc.speed` | **40 px/s** | 32 - 48 px/s | 跟随玩家但慢于 trader（受伤状态）|
| `npc.r` | **18** | 不允许微调 | 与 friendly-cargo r=18 一致 |
| spawn 距离玩家 | **500-800 px** | 400-640 / 640-960 px | 比 trader 远，玩家需要"主动响应求救"|

### §4.2 海盗数量曲线表

| level | 海盗数量 | 设计理由 |
|---|---|---|
| **level 1** | **4 个** | 引入战斗的"小遭遇"，玩家 turret + 玩家 NPC 协同处理 |
| **level 2** | **4 个** | 与 level 1 持平，玩家熟悉机制 |
| **level 3** | **5 个** | 难度逐渐上升 |
| **level 4** | **5 个** | 玩家此时已有 v0.5.0 研发 + v0.7.0 锁敌应对 |
| **level 5** | **6 个** | 最大海盗数，仍在方向 §"数值边界建议" 4-6 范围内 |

> **§4.2 备注**：海盗数量按 level 递增匹配玩家"battle DPS 上升曲线"；不允许 PM 微调（制作人 / ADR 拍板项），避免破坏 distress / ambush 难度梯度。

### §4.3 海盗 spawn 配置

| 字段 | 推荐值 | 是否允许微调 | 设计理由 |
|---|---|---|---|
| spawn 函数 | `spawnEnemy("pirate")` | 不允许微调 | 复用 v0.6.0 hostile-station 路径，不引入新 enemy kind |
| spawn 距离 distress NPC | **120 - 200 px**（围困状态）| 96 - 240 px | 海盗"围"住 NPC，让玩家看到围困场景 |
| 海盗目标 | **distress NPC 优先 + 玩家次之** | 不允许微调 | distress NPC 不出场玩家死亡风险（海盗优先打 NPC）|
| 海盗 hp | **使用 v0.2.0 / v0.5.0 pirate 基础 hp**（按 level 通过 techLevel 加成）| 不允许微调 | 完全复用现有 enemy 数值路径 |

### §4.4 援助奖励数值表

| 资源 | 推荐值 | PM 可微调范围（±20%）| 设计理由 |
|---|---|---|---|
| **金属** | **+120** | 96 - 144 | 与方向 §"数值边界建议" 一致；约等于 v0.6.0 hostile-station 1 个 turret cell 击破奖励 |
| **科研点** | **+18** | 14 - 22 | 与方向 §"数值边界建议" 一致；约等于打 3 个 pirate 的科研收益 |
| **等离子** | **+5** | 4 - 6 | 与方向 §"数值边界建议" 一致；约等于 5 次 trader 交易 |

### §4.5 援助失败兜底

| 场景 | 行为 | 后果 |
|---|---|---|
| 玩家完全不靠近 | 45s 过期 = 失败 | 友方 NPC 飞走（或被海盗击毁），无奖励，海盗留在原地变成普通敌人 |
| 玩家靠近但 NPC 在援助前死亡 | 事件直接失败 | 无奖励，海盗变成普通敌人，玩家可继续战斗但无 distress 加成 |
| 玩家清完海盗 | 事件成功 | 奖励到账（§4.4），NPC 飞走（或挥手致谢动画后离开）|
| 玩家清部分海盗后逃跑 500px 外 | 不结算，事件仍 active | 直到 45s 过期或 NPC 死亡才结算 |

### §4.6 distress NPC 不受 v0.7.0 priorityTarget 干扰

> distress NPC 是友方 NPC，**玩家右键 priorityTarget 不应锁定 distress NPC**；priorityTarget 仅锁定 enemy（hp / kind / role 字段过滤）。

```js
function onPointerRightClick(worldPos) {
  const target = findNearestEnemyInRadius(worldPos, 48);   // 仅找 enemy，不找 npc
  if (target) {
    state.input.priorityTarget = { enemy: target, setAt: state.time, validUntil: state.time + 12.0 };
  } else {
    state.input.priorityTarget = null;
  }
}
```

> **§4.6 备注**：本约束保护 v0.7.0 锁敌行为不被 distress / ambush NPC 污染；详细实现路径由主程 ADR 决定。

## §5. ambush 伪装陷阱数值

> 设计意图：ambush 是 5 种事件中**唯一真伪判断 + 高风险高回报**事件；伪装阶段显示为 distress（friendly green）诱导玩家靠近，300px 内触发战斗显形为 pirate-ambush（红色）；500px 外可逃跑无奖励。

### §5.1 伪装阶段数值参数表

| 字段 | 推荐值 | PM 可微调范围（±20%）| 设计理由 |
|---|---|---|---|
| 伪装 `npc.kind` | `"distress-pilot"`（伪装时显示）| 不允许微调（kind 名称）| 视觉上与真 distress 完全一致；玩家通过经验 / 距离学会辨别 |
| 显形 `npc.kind` | `"pirate-ambush"`（300 px 内切换）| 不允许微调 | 切换后视觉变红色，玩家开始战斗 |
| `npc.role` | `"encounter"` | 不允许微调 | 区分 escort objective NPC |
| `npc.hp`（伪装）| **60**（与真 distress 一致）| 不允许微调 | 视觉一致性要求"hp 数值也一致"避免穿帮 |
| `npc.hp`（显形）| **80**（pirate-ambush 显形后增强）| 64 - 96 | 略强于真 distress（已变成海盗），让玩家有"被骗了"代价感 |
| `npc.speed` | **40 px/s**（伪装时）/ **120 px/s**（显形后追玩家）| 不允许微调（伪装值）| 显形后变成快速追击的海盗 |
| spawn 距离玩家 | **600-900 px** | 480-720 / 720-1080 px | 比 distress 远，让玩家"主动飞过去"才被骗 |
| 显形触发距离 | **300 px**（玩家进入触发战斗 + 显形）| 240 - 360 px | 短于 trader 交易距离 100 px，让玩家"远观时看不出真假"|
| 逃跑安全距离 | **500 px**（玩家远离不触发战斗）| 400 - 600 px | 长于显形距离 300px，让玩家"飞近又退回"还有机会 |

### §5.2 海盗数量曲线表

| level | 海盗数量 | 设计理由 |
|---|---|---|
| **level 2** | **6 个** | ambush 最低出现 level，海盗数比 distress 同级多 2 个 |
| **level 3** | **6 个** | 与 level 2 持平 |
| **level 4** | **7 个** | 难度上升 |
| **level 5** | **8 个** | 最大海盗数，与方向 §"数值边界建议" 6-8 范围一致 |

> **§5.2 备注**：ambush 海盗数始终比同级 distress 多 1-2 个，匹配"高风险高回报"定位。

### §5.3 海盗 spawn 配置

| 字段 | 推荐值 | 是否允许微调 | 设计理由 |
|---|---|---|---|
| spawn 函数 | `spawnEnemy("pirate")` | 不允许微调 | 复用现有路径 |
| spawn 时机 | **玩家进入 300 px 显形时同步 spawn** | 不允许微调 | 避免玩家看到"远处一群海盗围着 NPC"穿帮 |
| spawn 距离 ambush NPC | **80 - 150 px**（贴近 NPC 出现）| 不允许微调 | 玩家近距离遭遇，没有反应时间 |
| 海盗 hp | 使用 v0.2.0 pirate 基础 hp × **1.1**（略强于普通 pirate）| 不允许微调 | 让玩家有"伏击战比普通战难"的感受 |

### §5.4 战斗奖励数值表

| 资源 | 推荐值 | PM 可微调范围（±20%）| 设计理由 |
|---|---|---|---|
| **金属** | **+200** | 160 - 240 | 与方向 §"数值边界建议" 一致；高于 distress 120（高风险高回报）|
| **科研点** | **+25** | 20 - 30 | 与方向 §"数值边界建议" 一致；约等于 5 个 pirate 击杀科研 |
| **海盗 wreck** | **3 个**（origin="enemy-wreck"，使用 v0.6.0 ENEMY_WRECK_FACILITY_WEIGHTS）| 不允许微调 | 与方向 §"数值边界建议" 一致；3 个 wreck 是 ambush 独有奖励，让玩家"被骗也能赚一笔"|

### §5.5 ambush 状态流转

| 状态 | 触发条件 | NPC 显示 | 海盗存在 | 奖励路径 |
|---|---|---|---|---|
| **伪装阶段** | spawn 后到玩家进入 300px 前 | 绿色 distress-pilot | 无 | - |
| **显形 + 战斗阶段** | 玩家进入 300px | 红色 pirate-ambush | 6-8 个海盗 + ambush NPC | - |
| **战斗成功** | 玩家清掉所有海盗 + ambush NPC | NPC 消失 | 无 | §5.4 奖励到账 |
| **玩家逃跑** | 玩家在显形前飞到 500px 外 | NPC 继续伪装巡航 | 无 | 无奖励，事件保持 active 直到玩家再次靠近 / 跨关清空 |
| **玩家逃跑（显形后）** | 显形后玩家飞到 500px 外 | NPC + 海盗继续追玩家 | 海盗保留 | 海盗变成普通追击敌人，无 ambush 奖励 |

### §5.6 ambush vs distress 视觉差异

> 玩家如何分辨真假？设计上**不引入"显式标记"**（如名字 / icon），让玩家通过以下隐性线索学习：

| 线索 | distress 真 | ambush 假 | 难度 |
|---|---|---|---|
| spawn 距离 | 500-800 px | 600-900 px | 远端略远，玩家不易察觉 |
| 海盗围困状态 | spawn 时已被海盗围困 | spawn 时无海盗（伪装阶段）| **核心线索**——真 distress 远处就能看到红点 |
| 触发延迟 | 90s | 120s | 玩家凭"事件出现时机"难分辨 |
| NPC HP 数值 | 60 | 60（伪装）/ 80（显形）| 玩家显形后才发现 |

> **§5.6 备注**："spawn 时是否有海盗围困"是真假的核心线索，玩家可通过远距观察学习；这是方向 §"决策深度"第 3 项"真伪判断"的具体落地。

## §6. derelict 沉船秘窟数值

> 设计意图：derelict 是 5 种事件中**唯一无战斗 + 唯一大规模拾荒**事件；wreck 数量按 level 递增，权重偏向稀有 cell（plasma / hangar），让玩家"拾荒决策"有显著收益差异。

### §6.1 wreck 数量曲线表

| level | wreck 数量 | 设计理由 |
|---|---|---|
| **level 1** | **4 个** | 引入大规模拾荒，玩家熟悉 fragment dock 机制 |
| **level 2** | **4 个** | 与 level 1 持平 |
| **level 3** | **5 个** | 难度 / 收益逐渐上升 |
| **level 4** | **5 个** | 玩家此时已有较多 cell，可消化 5 wreck |
| **level 5** | **6 个** | 最大 wreck 数，与方向 §"数值边界建议" 4-6 范围一致 |

> **§6.1 备注**：wreck 数量按 level 递增匹配玩家 station 容量上升；不允许 PM 微调（制作人 / ADR 拍板项）。

### §6.2 wreck 物理分布参数表

| 字段 | 推荐值 | PM 可微调范围（±20%）| 设计理由 |
|---|---|---|---|
| 分布形状 | **圆形分布** | 不允许微调（形状）| 视觉上"沉船群"应该是圆形而非线性 |
| 分布半径 | **80 - 120 px** | 64 - 144 px | 4-6 个 wreck 落在 80-120 px 半径圆内，玩家飞近 1 次可清完不必远征 |
| wreck 间最小间距 | **30 px**（避免重叠）| 不允许微调 | 与 v0.3.0 fragment 物理一致 |
| spawn 距离玩家 | **700 - 1100 px** | 560-880 / 880-1320 px | 与方向 §"数值边界建议" 一致；玩家需要主动飞过去拾荒 |
| `fragment.origin` | `"derelict"` | 不允许微调 | 与方向 §"fragment 新 origin" 一致 |
| `fragment` 使用函数 | `makeWreckFragment(pos, rng, { origin: "derelict", weights: DERELICT_WEIGHTS })` | 不允许微调 | 复用现有函数，不引入新接口 |

### §6.3 cell 权重表（DERELICT_WEIGHTS）

> 基线：v0.3.0 `WRECK_FACILITY_WEIGHTS`（假设各 cell 类型权重为 1）。

| cell 类型 | 基线权重 | derelict 倍率 | derelict 最终权重 | 设计理由 |
|---|---|---|---|---|
| **plasma** | 1.0 | **× 2.0** | **2.0** | 稀有 + 玩家需求度最高；derelict 主要诱因 |
| **hangar** | 1.0 | **× 1.5** | **1.5** | 与方向 §"数值边界建议" 一致；高级 cell |
| **fusion** | 1.0 | **× 1.5** | **1.5** | 高级 cell，权重略高 |
| **energy** | 1.0 | **× 1.5** | **1.5** | 高级 cell，权重略高 |
| **metal** | 1.0 | **× 0.7** | **0.7** | 早期 cell，权重略低（玩家此时不缺 metal）|
| 其他（如 turret / missile / shield）| 1.0 | **× 1.0** | **1.0** | 基线，不调整 |

> **§6.3 备注**：权重倍率与方向 §"数值边界建议"完全一致，hangar 在方向中未明确倍率（仅说"偏 plasma / hangar / fusion"），本方案补充为 1.5（与 fusion / energy 同档）。**权重倍率不允许微调**（制作人 / ADR 拍板项），避免 PM 调到"plasma × 5"破坏 v0.3.0 cell 经济平衡。

### §6.4 derelict 持续状态

| 状态 | 触发条件 | 行为 |
|---|---|---|
| **active**（spawn 后）| 玩家进入 1500 px 视野 | wreck 渲染在世界中，HUD 显示 alert |
| **partial dock** | 玩家 dock 部分 wreck | 剩余 wreck 保留，关内永久存在 |
| **all dock** | 玩家 dock 所有 wreck | 事件 success，§10 触发 4s 紫色"沉船清空"alert |
| **跨关清空** | 玩家进入 nextLevel | 未 dock wreck 全部清空（v0.3.0 fragment 行为一致）|

> **§6.4 备注**：derelict 不像 distress / ambush 有"失败"状态——玩家忽略只是"没拿到收益"而非"失败"。这是方向 §"决策深度"中"derelict 部分忽略 → 关内永久 → 跨关清空"的落地。

### §6.5 derelict 收益预估表

> 按权重计算 derelict 单次（level 3，5 个 wreck）的期望资源产出（假设每个 wreck 平均 3 cell）：

| 资源 | 期望产出 | 估算方式 |
|---|---|---|
| 等离子（plasma cell）| **~5 等离子**（权重 2 / 总权重 8 × 15 cell × 1.6 plasma per cell）| 单次拾荒约 5 等离子 |
| 金属（metal cell）| **~80 金属**（权重 0.7 / 总权重 8 × 15 × 60 metal per cell）| 拾荒副产 |
| 科研点（混合）| **~20 科研点** | 每个 fragment dock 触发的研究奖励 |

> **§6.5 备注**：本表为期望值；实际单次拾荒因 RNG 波动可能在 ±30% 内。derelict 总收益对照 v0.7.0 单关基线："不参与时基线 + 0"，"参与 1 次 derelict 时基线 + ~5 等离子 + ~80 金属 + ~20 科研"。

## §7. signal 神秘信号数值

> 设计意图：signal 是 5 种事件中**距离最远 + 单 fragment 高价值**事件；玩家需要"主动远征"才能获得，权重最偏稀有 cell（plasma / fusion），让玩家"评估值不值得远征"。

### §7.1 signal fragment 数值参数表

| 字段 | 推荐值 | PM 可微调范围（±20%）| 设计理由 |
|---|---|---|---|
| fragment 数量 | **1 个** | 不允许微调 | 与方向 §"数值边界建议" 一致；远征事件应该"目标单一"|
| cell 数量（fragment 内）| **6 - 9 个** | 不允许（上下界）| 与方向 §"数值边界建议" 一致；远多于普通 wreck 的 ~3 cell |
| cell 数量算法 | 6 + floor(rng × 4) → 6/7/8/9 均匀 | 不允许微调 | 简单可预期 |
| spawn 距离玩家 | **800 - 1400 px** | 640-1120 / 1120-1680 px | 最远距离，与方向 §"数值边界建议" 一致；玩家飞过去需 ~10s |
| `fragment.origin` | `"signal"` | 不允许微调 | 与方向 §"fragment 新 origin" 一致 |
| `fragment` 使用函数 | `makeWreckFragment(pos, rng, { origin: "signal", weights: SIGNAL_WEIGHTS, cellCount: 6+floor(rng*4) })` | 不允许微调 | 复用现有函数 |

### §7.2 cell 权重表（SIGNAL_WEIGHTS）

| cell 类型 | 基线权重 | signal 倍率 | signal 最终权重 | 设计理由 |
|---|---|---|---|---|
| **plasma** | 1.0 | **× 2.5** | **2.5** | 最稀有，远征首要诱因 |
| **fusion** | 1.0 | **× 2.0** | **2.0** | 高级 cell，远征次要诱因 |
| **energy** | 1.0 | **× 1.5** | **1.5** | 高级 cell |
| **metal** | 1.0 | **× 0.5** | **0.5** | 远征不应该"赚个金属"；权重压低让稀有 cell 占比更高 |
| 其他（turret / missile / shield / hangar 等）| 1.0 | **× 1.0** | **1.0** | 基线，不特别偏好 |

> **§7.2 备注**：权重倍率与方向 §"数值边界建议"一致（"权重偏 plasma / fusion / energy"）。**hangar 在 signal 中权重为 1.0**（与基线一致），仅 derelict 偏好 hangar。**权重倍率不允许微调**。

### §7.3 signal 持续状态

| 状态 | 触发条件 | 行为 |
|---|---|---|
| **active**（spawn 后）| 玩家进入 2000 px 视野（或 HUD 提示）| fragment 渲染在世界中（金色闪烁星标）|
| **dock** | 玩家 dock fragment | 事件 success，§10 触发 3s 金色"信号回收"alert |
| **跨关清空** | 玩家进入 nextLevel | 未 dock fragment 清空 |

### §7.4 signal fragment 视觉差异

| 视觉元素 | 普通 wreck fragment | signal fragment | 设计理由 |
|---|---|---|---|
| 颜色 | 灰色 / 棕色 | **金色 `#ffcc33`** | 远距可见性 + 高价值暗示 |
| 闪烁 | 无 | **闪烁脉冲（周期 0.8s）** | 远距能看到"那有个特殊东西"|
| HUD 标记 | 普通 fragment alert | **金色 alert + 距离指示** | 玩家知道"远处有信号"|
| dock 反馈 | 普通 dock 音效 | （可选）特殊 dock 音效 | v0.8.0 不要求音效；视觉反馈足够 |

### §7.5 signal 收益预估表

> 单 signal 期望资源产出（假设权重 8.5 总权重，7.5 cell 均值）：

| 资源 | 期望产出 | 估算方式 |
|---|---|---|
| 等离子（plasma cell）| **~3.5 等离子**（权重 2.5 / 总权重 8.5 × 7.5 × 1.6）| 单次 signal 约 3-4 等离子 |
| 高级 cell（fusion + energy）| **~2 cell**（权重 3.5 / 总权重 8.5 × 7.5）| 让玩家"远征值得"|
| 科研点 | **~15 科研点** | 7.5 cell 平均触发科研奖励 |
| 金属 | **~30 金属**（权重 0.5 / 总权重 8.5 × 7.5 × 60）| 副产 |

> **§7.5 备注**：signal 单次收益（~3.5 等离子 + ~2 高级 cell）略低于 derelict（~5 等离子 + ~80 金属），但单 fragment 操作成本低于 derelict 拾 4-6 个 wreck。两者形成"密集 vs 单一"的拾荒决策差异。

## §8. 单星系冷却

### §8.1 单星系冷却参数表

| 事件类型 | 单星系最大触发次数 | 是否允许微调 | 设计理由 |
|---|---|---|---|
| **trader** | **2 次** | ❌ 不允许（制作人 / ADR 拍板）| trader 是友好事件，玩家不会反感重复；2 次让玩家"知道还会再来" |
| **distress** | **1 次** | ❌ 不允许 | 重复会让玩家麻木"又是救援" |
| **ambush** | **1 次** | ❌ 不允许 | 重复会让玩家学会"看到 distress 就疑神疑鬼"破坏沉浸 |
| **derelict** | **1 次** | ❌ 不允许 | 大规模拾荒重复破坏稀有感 |
| **signal** | **1 次** | ❌ 不允许 | 远征事件重复让玩家"懒得飞"|

### §8.2 跨关清空规则

| 时机 | 行为 |
|---|---|
| `nextLevel()` 进入新星系 | `state.run.encounterCooldownThisGalaxy.clear()` |
| `nextLevel()` 重新滚池 | `seedEncountersForLevel(state.run.level, encounterRng)` |
| `resetRun()` 重置整局 | cooldown set 自动清空（state.run 重建）|
| `loadSave()` 旧存档兼容 | cooldown set 默认 `new Set()`（state.run 不存盘）|

### §8.3 cooldown set 与 hostileStationSpawnedThisGalaxy 的一致性

| 字段 | 跨关行为 | 写入 SAVE_KEY |
|---|---|---|
| `state.run.hostileStationSpawnedThisGalaxy` | nextLevel 时 `false` | ❌ 否（与 v0.6.0 一致）|
| `state.run.encounterCooldownThisGalaxy` | nextLevel 时 `clear()` | ❌ 否（与 v0.6.0 hostile-station 字段保持一致）|

> **§8.3 备注**：本字段命名与 v0.6.0 `hostileStationSpawnedThisGalaxy` 风格一致（驼峰 + `ThisGalaxy` 后缀），让代码读者一眼看出"这是单星系冷却字段"。

## §9. HUD priority 队列

> 设计意图：5 种事件 + hostile-station + 已有 npc / fragment alert 共用"中央闪烁"通道，必须按 priority 排序避免 DOM 堆叠；同时只显示 1 个，高 priority 抢占低 priority。

### §9.1 priority 数值表

| 事件 / alert 类型 | priority | 是否允许微调 | 设计理由 |
|---|---|---|---|
| **assault hostile-station**（v0.6.0）| **100** | ❌ 不允许（保护 v0.6.0 拍板）| 最高优先级；BOSS 战决战不能被任何事件抢占 |
| **ambush 触发**（玩家进入 300 px 显形）| **80** | ❌ 不允许（制作人拍板）| 战斗高紧急度；玩家必须立刻反应 |
| **distress 友方危险**（NPC HP < 30%）| **70** | ❌ 不允许 | 友方 NPC 即将死亡，玩家必须立刻援助 |
| **trader 出现** | **50** | ❌ 不允许 | 友好事件中等优先级 |
| **derelict 出现** | **40** | ❌ 不允许 | 拾荒事件低紧急度（永久存在）|
| **signal 出现** | **30** | ❌ 不允许 | 拾荒事件最低紧急度（最远距离）|
| **事件结束反馈**（成功 / 失败）| **同事件 priority + 5**（暂时抬升保证不被新事件抢占）| ❌ 不允许 | 让玩家能看完"事件结束反馈"再被新事件 alert 抢 |

> **§9.1 备注**：priority 数值与方向 §"数值边界建议" 完全一致。**priority 数值不允许微调**（制作人 / ADR 拍板项），避免 PM 调到"signal 100"等破坏 alert 优先级语义。

### §9.2 队列数据结构

```js
state.hudCenterAlertQueue = [
  // 每个元素 { text, ms, isSuccess, priority, cssClass, pushedAt }
];
state.hudCenterAlertActive = null;  // 当前正在播放的 alert
```

| 字段 | 类型 | 说明 |
|---|---|---|
| `text` | string | alert 文字（如"伪装陷阱！"）|
| `ms` | number | alert 持续毫秒数（§10 表）|
| `isSuccess` | boolean | 影响视觉色调（绿 / 红 / 灰）|
| `priority` | number | §9.1 表 |
| `cssClass` | string | `.alert-encounter-trader` 等（与方向 §"HUD 集成" 一致）|
| `pushedAt` | number | 入队时间，用于 FIFO 同 priority 时排序 |

### §9.3 抢占规则

```js
function pushHudCenterAlert(alert) {
  // 1. 加入队列
  state.hudCenterAlertQueue.push({ ...alert, pushedAt: state.time });
  // 2. 按 priority 降序 + pushedAt 升序排序
  state.hudCenterAlertQueue.sort((a, b) =>
    b.priority - a.priority || a.pushedAt - b.pushedAt
  );
  // 3. 检查队首是否高于当前播放 alert
  const head = state.hudCenterAlertQueue[0];
  if (!state.hudCenterAlertActive || head.priority > state.hudCenterAlertActive.priority) {
    state.hudCenterAlertActive = head;
    state.hudCenterAlertQueue.shift();
    showHostileStationAlert(head.text, head.ms / 1000, head.isSuccess);  // 复用现有函数
  }
}

function tickHudCenterAlertQueue(dt, now) {
  if (state.hudCenterAlertActive && now >= state.hudCenterAlertActive.expireAt) {
    state.hudCenterAlertActive = null;
    if (state.hudCenterAlertQueue.length > 0) {
      const next = state.hudCenterAlertQueue.shift();
      state.hudCenterAlertActive = next;
      showHostileStationAlert(next.text, next.ms / 1000, next.isSuccess);
    }
  }
}
```

### §9.4 队列上限与溢出策略

| 字段 | 推荐值 | PM 可微调范围（±20%）| 设计理由 |
|---|---|---|---|
| 队列最大长度 | **5 个** | 4 - 6 个 | 大于事件并发上限 3，覆盖"3 事件出现 + 1 事件结束 + 1 hostile-station"高峰场景 |
| 超过上限时 | 丢弃最低 priority 元素 | 不允许微调（策略）| 简单可预期；避免老 alert 阻塞新 alert |
| 同 priority 同时 push | FIFO（按 pushedAt 升序）| 不允许微调 | 公平且可预期 |

## §10. alert 持续时间

> 设计意图：每种事件 alert 持续时间按"紧急度"分配——ambush 最短 3s（玩家必须立刻反应）/ distress 最长 5s（玩家需要时间评估"援助还是忽略"）；事件结束反馈统一短时（4s 成功 / 3s 失败）让玩家"看完反馈继续游戏"。

### §10.1 事件出现 alert 参数表

| 事件 | alert 持续 | 文字 | 颜色 | cssClass | 设计理由 |
|---|---|---|---|---|---|
| **trader 出现** | **4.0s** | "贸易商船出现"| 绿色 `#33cc33`（友好）| `.alert-encounter-trader` | 友好事件中等持续 |
| **distress 出现** | **5.0s** | "求救信号！"| 红色 `#cc3333`（警告）| `.alert-encounter-distress` | 最长持续；玩家需评估时间 |
| **ambush 触发**（300 px 内显形）| **3.0s** | "陷阱！"| 红色 `#ff4444` + 急促闪烁（pulse 0.3s）| `.alert-encounter-ambush` | 最短持续；急促闪烁强化紧迫感 |
| **derelict 发现** | **4.0s** | "沉船秘窟发现"| 紫色 `#8844cc` | `.alert-encounter-derelict` | 中等持续；紫色区别其他 |
| **signal 发现** | **3.0s** | "神秘信号"| 金色 `#ffcc33` | `.alert-encounter-signal` | 短持续；金色 = 远征预期 |

### §10.2 事件结束反馈 alert 参数表

| 触发条件 | alert 持续 | 文字 | 颜色 | 设计理由 |
|---|---|---|---|---|
| trader 成功交易 | **4.0s** | "交易完成 · 资源到账" | 绿色 `#33cc33` | 与出现 alert 同时长，对称感 |
| distress 援助成功 | **4.0s** | "救援成功 · +120 金属"| 绿色 `#33cc33` | 同上 |
| ambush 战斗胜利 | **4.0s** | "陷阱清剿 · +200 金属"| 绿色 `#33cc33` | 同上 |
| derelict 全部 dock | **4.0s** | "沉船清空"| 绿色 `#33cc33` | 同上 |
| signal fragment dock | **4.0s** | "信号回收 · 高级 cell × N" | 绿色 `#33cc33` | 同上 |
| 事件失败 / 过期 | **3.0s** | 按事件类型（如"援助超时"/ "贸易商离去"）| 灰色 `#888888` | 短持续 + 灰色弱化（不打扰玩家继续游戏）|

### §10.3 alert 持续时间汇总表

| alert 名称 | 持续 | PM 可微调范围（±20%）| 设计理由 |
|---|---|---|---|
| trader 出现 | 4.0s | 3.2 - 4.8 s | 中等 |
| distress 出现 | 5.0s | 4.0 - 6.0 s | 最长 |
| ambush 触发 | 3.0s | 2.4 - 3.6 s | 最短 |
| derelict 发现 | 4.0s | 3.2 - 4.8 s | 中等 |
| signal 发现 | 3.0s | 2.4 - 3.6 s | 短 |
| 事件成功反馈（任意类型）| 4.0s | 3.2 - 4.8 s | 中等 |
| 事件失败反馈（任意类型）| 3.0s | 2.4 - 3.6 s | 短 + 灰色弱化 |

> **§10.3 备注**：alert 持续时间 PM 可在 ±20% 微调，但需保证"ambush < trader / derelict < distress"的相对关系不变。颜色 hex 不允许微调（视觉一致性要求）。

### §10.4 ambush 急促闪烁的具体参数

| 字段 | 推荐值 | 说明 |
|---|---|---|
| 闪烁周期 | **0.3s**（pulse 1 → 0.3 → 1）| 比普通 alert 快 3 倍 |
| 闪烁 alpha 范围 | 0.3 - 1.0 | 不完全消失，保证文字始终可读 |
| 闪烁持续 | 3.0s 全程闪烁 | 与 alert 持续时间一致 |
| 闪烁颜色 | `#ff4444`（红色，与 v0.7.0 priorityTarget 锁定框同色系）| 让"紧急 / 危险"视觉语言统一 |

## §11. 体验目标对应的回归校验（共 26 条，超 ≥ 25 下限）

> 设计意图：本节是 PM 节点 A / B / C 全量回归与节点 C 接受测试的可量化校验点；按方向 §"体验目标"三类组织。

### §11.1 单局记忆点（9 条）

1. **每关 spawn 1-3 个事件**：`seedEncountersForLevel(level, rng)` 在 level 0 时返回 1 个、level 1 时返回 1-2 个、level 2-5 时返回 2-3 个、level 6 时返回 0 个；事件数量在 [min, max] 闭区间内由独立 RNG 流均匀采样。
2. **事件按 60-180s 延迟显现**：spawn 后 trader 60s / distress 90s / ambush 120s / derelict 150s / signal 180s 出现；相邻事件按 §2.3 算法错开至少 30s（实际 spawn 时间 = max(自身延迟, 上一事件 spawn + 30s)）。
3. **5 种事件全部可独立触发**：level 3+ 跑 10 局，统计 5 种事件各至少触发 1 次（独立 RNG 流可控）。
4. **事件触发时有 HUD 中央闪烁**：每个事件 spawn 时按 §10.1 触发 alert，文字 / 颜色 / 持续时间符合表格；§10.4 ambush 闪烁周期 0.3s 可视化确认。
5. **事件在世界中有视觉标记**：trader 绿色 NPC / distress 红色 NPC / ambush 伪装阶段绿色显形后红色 / derelict 沉船群 / signal 金色闪烁 fragment 在 viewport 内正确渲染。
6. **玩家可识别每种事件**：trader = 绿色 NPC + 慢速巡航；distress = 红色 NPC + 海盗围困；ambush = 伪装绿色（远处看是 distress）显形后红色 + 海盗贴脸；derelict = 4-6 wreck 圆形分布；signal = 金色闪烁单 fragment 远距。
7. **事件不抢主任务（OBJECTIVE_TYPES）注意力**：v0.4.0 escort / mine / explore / battle / survive / salvage / guardian / assault 8 种主任务在事件触发时行为完全等价；OBJECTIVE 完成 / 失败 / spawn 不受事件影响。
8. **每关结束时 state.run.encounters 完全清空**：`nextLevel()` 执行后 `state.run.encounters.length === 0`，所有事件 NPC / wreck / fragment 从世界中移除。
9. **跨关 encounterCooldownThisGalaxy 清空**：`nextLevel()` 执行后 `state.run.encounterCooldownThisGalaxy.size === 0`，新星系所有事件类型可再次触发。

### §11.2 决策深度（10 条）

10. **trader 选择交易**（option-1 50 金属→1 等离子）：玩家飞到 trader 100 px 内按 1 键，`state.resources.metal -= 50` + `state.resources.plasma += 1`，trader NPC 飞走，§10.2 触发 4s 绿色成功 alert。
11. **trader 选择交易**（option-2 100 金属→3 科研）：玩家按 2 键，`state.resources.metal -= 100` + `state.resources.research += 3`，trader NPC 飞走。
12. **trader 选择忽略 30s 后消失无后果**：玩家完全不靠近 trader，30s 过期后 trader 飞走，`state.resources` 无变化，§10.2 触发 3s 灰色失败 alert"贸易商离去"。
13. **distress 选择援助 → 清海盗 → 奖励到账**：玩家清掉所有 4-6 个海盗，`state.resources.metal += 120` + `state.resources.research += 18` + `state.resources.plasma += 5`，distress NPC 飞走，§10.2 触发 4s 绿色成功 alert。
14. **distress 选择忽略 → 友方 NPC 死亡 → 无奖励**：玩家完全不靠近 distress，45s 过期后 NPC 死亡（HP 60 被海盗打穿），`state.resources` 无变化，§10.2 触发 3s 灰色"援助超时"alert。
15. **ambush 靠近触发 → pirate-ambush 显形 → 战斗**：玩家进入 ambush NPC 300 px 内，NPC 切换 kind 为 `pirate-ambush` 视觉变红色 + HP 80，同步 spawn 6-8 个海盗在 NPC 80-150 px 范围内，§10.1 触发 3s 红色急促闪烁"陷阱！" alert。
16. **ambush 远离不触发战斗 → 事件保持 active**：玩家在显形前飞到 ambush NPC 500 px 外，NPC 保持伪装继续巡航，海盗不 spawn；事件保留在 active 状态直到玩家再次靠近 / 跨关清空。
17. **derelict 拾荒 → wreck dock → fragment 进库**：玩家飞到 derelict 圆形分布内，dock 所有 4-6 个 wreck，每个 wreck 触发 v0.3.0 fragment dock 路径写入 `state.station.cells`；全部 dock 后 §10.2 触发 4s 绿色"沉船清空"alert。
18. **derelict 部分忽略 → 关内永久 → 跨关清空**：玩家只 dock 部分 wreck 后跃迁 nextLevel，未 dock wreck 从世界移除（state.run.encounters 清空 + entity cleanup）；新关重新滚事件池。
19. **signal 飞过去 → 单 fragment dock → 大量稀有 cell**：玩家飞到 signal fragment 位置（800-1400 px 远），dock 单 fragment 写入 6-9 个 cell（plasma / fusion / energy 占多数）；§10.2 触发 4s 绿色"信号回收 · 高级 cell × N"alert。

### §11.3 与已有系统协同（7 条）

20. **distress / ambush 战斗时 v0.7.0 锁敌正常使用**：玩家右键点海盗在 48 px 内写入 `state.input.priorityTarget`，turret / missile / shield 优先打锁定海盗；F 键齐射 + `[/]` salvoSize 调档行为完全等价 v0.7.0。
21. **distress 友方 NPC 不受 v0.7.0 priorityTarget 干扰**：玩家右键点 distress NPC 不写入 priorityTarget（onPointerRightClick 仅查询 enemy 不查询 npc）；priorityTarget 仅锁定 enemy。
22. **derelict / signal fragment dock 走 v0.3.0 路径**：fragment.origin = "derelict" / "signal" 调用同一 `dockFragment(fragment)` 路径，写入 `state.station.cells` + 触发科研奖励 + 触发 v0.6.0 buildFragmentHudAlerts 分色显示；不引入新 dock 路径。
23. **事件 NPC 不破坏 v0.4.0 escort objective 行为**：v0.4.0 escort 任务 NPC 标记 `role: "objective"`，事件 NPC 标记 `role: "encounter"`；`buildNpcHudAlerts()` 按 role 分组聚合，escort objective NPC 的 alert 通道与事件 NPC 的 alert 通道完全分离。
24. **v0.6.0 hostile-station 触发不冲突**：level 3+ 同时存在 hostile-station + 事件时，hostile-station alert priority 100 抢占事件 alert priority 30-80；`hostileStationSpawnedThisGalaxy` 与 `encounterCooldownThisGalaxy` 是两个独立字段不相互影响。
25. **v0.5.0 研发系统 / cell 升级不破坏**：玩家在事件触发期间正常升级 cell（research point 消耗 + cell 等级提升），事件 NPC / wreck / fragment 不影响 research 流；trader option-2 + distress + signal 三个事件产出的 research point 都通过 `state.resources.research += N` 路径写入。
26. **事件不修改 SAVE_KEY / state.meta**：`state.run.encounters` / `state.run.encounterCooldownThisGalaxy` / `state.hudCenterAlertQueue` / `npc.role` / `npc.encounterId` 全部不写入 SAVE_KEY；旧存档读取后这些字段按默认值初始化（encounters = [] / cooldown = new Set() / queue = [] / role 默认 "objective"<sup>*</sup> / encounterId = null），不破坏向后兼容。

> <sup>*</sup> 旧存档兼容 `npc.role` 默认值由主程 ADR 决定（建议默认 "objective" 兼容 v0.4.0 escort NPC 语义）。

> **§11 校验条数：26 条**（≥ 25 下限）。

## §12. 平衡兜底

> 设计意图：本节是数值方案的"安全网"——即使具体数值微调失误，硬保护体验不爆炸；同时确保 v0.4.0 / v0.5.0 / v0.6.0 / v0.7.0 已发布平衡完全不动。

### §12.1 5 种事件资源给予总和 vs OBJECTIVE_TYPES 奖励对比

> 基线：v0.4.0 OBJECTIVE_TYPES 单关完成的"主任务奖励"（按方向 §"不做范围" v0.4.0 完全等价计算）。

| OBJECTIVE 类型 | 单关完成奖励（估算）| 事件比例（事件奖励 / OBJECTIVE 奖励）|
|---|---|---|
| **mine**（采矿任务）| ~150 金属 + ~25 科研 | distress 援助约 80%（120 金属 +18 科研，+5 等离子）|
| **battle**（战斗任务）| ~180 金属 + ~30 科研 | ambush 约 110%（200 金属 +25 科研，+3 wreck）|
| **salvage**（拾荒任务）| ~10 cell + ~30 科研 | derelict 约 50-70%（5 cell 平均 + ~20 科研）|

### §14.2 待制作人 / 主程拍板的开放决策点

以下决策点本方案给出推荐选项，但**最终由制作人 / 主程决定**（实现前需明确）：

1. **ambush 海盗 HP × 1.1 倍率是否引入新字段**：
   - 本方案推荐：**enemy spawn 时 hp 字段直接 ×1.1**（不引入新字段，复用现有 enemy hp 路径）；或者 ADR 可选择"引入 enemy 修饰符字段"。
   - 节点 B 测试若发现 1.1 倍 hp 让 ambush 难度过高，可降到 1.0；本方案放在拍板项是避免 PM 微调到 2.0 等极端值。

2. **trader 菜单 UI 实现方式**：
   - 本方案推荐：**靠近 100 px 自动打开 HUD 菜单**（文字提示 "1=换 1 等离子 / 2=换 3 科研 / 3=忽略"）；不引入弹窗 / 鼠标点击。
   - 节点 B 测试若发现"100 px 自动打开太突兀"，可改为"按 E 键打开菜单"等显式触发；本方案默认走自动触发减少操作步骤。

3. **ambush 真假分辨的难度调优**：
   - 本方案推荐：**仅通过"spawn 时是否有海盗围困"作为唯一线索**（§5.6）；不引入显式 icon / 名字 / 颜色差异。
   - 节点 C 测试若发现"玩家 5 局都被骗"挫败感过强，可考虑在 ambush spawn 后 60s 内 NPC 飞行轨迹略异于真 distress（如"过度迟疑"），或在玩家解锁 v0.5.0 某研发项后获得"侦测能力"标记 ambush；本方案 v0.8.0 不引入此机制。

4. **HUD 队列上限 5 是否合理**：
   - 本方案推荐：**5 个**（覆盖 3 事件出现 + 1 事件结束 + 1 hostile-station 高峰）。
   - 节点 B 测试若发现"高峰场景仍有 alert 被丢弃"，可调到 6-7 个；但 ≥ 8 个会让队列调试成本上升。

5. **事件 + escort objective 同 level 的 NPC 颜色冲突**：
   - 本方案推荐：**escort 用 v0.4.0 灰色 friendly-cargo 颜色 / trader 用绿色 / distress 用红色**，三组 NPC 颜色完全不同。
   - 节点 B 测试若发现"玩家分不清 escort vs distress"（都是友方），可调整 escort 为 cyan 蓝色；本方案 v0.8.0 不修改 v0.4.0 escort 颜色保持兼容。

6. **alert 急促闪烁周期 0.3s 是否生理舒适**：
   - 本方案推荐：**ambush 急促闪烁周期 0.3s + alpha 0.3-1.0**（§10.4）。
   - 节点 B 测试若发现"0.3s 闪烁让玩家头晕"，可降为 0.4-0.5s；但闪烁频率降低会削弱"紧迫感"语义。

7. **signal fragment 金色闪烁的远距可见度**：
   - 本方案推荐：**金色 `#ffcc33` + 闪烁周期 0.8s**（§7.4）；signal 距离最远 1400 px，必须远距可见。
   - 节点 C 测试若发现"1400 px 距离金色闪烁仍不够显眼"，可在 fragment 上方添加箭头指示器或在 HUD radar 上标记 signal 位置。

8. **derelict 拾荒部分忽略后的 HUD 提示**：
   - 本方案推荐：**HUD 显示"沉船秘窟（剩 N 个 wreck）"**实时更新，让玩家知道"还剩多少"。
   - 节点 B 测试若发现"玩家忽略 1-2 个 wreck 后忘了 derelict 存在"，可在 HUD 加"距离 X px"提示。

9. **事件失败 / 过期是否扣科研 / 信誉**：
   - 本方案推荐：**全部不扣**（事件失败仅"无奖励"，不扣资源 / 信誉）。
   - 节点 C 测试若发现"忽略 distress 没有任何代价"让事件深度不足，可在 v0.8.1 引入"distress 失败 -5 信誉"等机制；v0.8.0 保持极简。
