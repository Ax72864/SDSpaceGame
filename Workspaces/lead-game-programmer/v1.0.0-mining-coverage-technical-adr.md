## 1. 问题描述

### 1.1 v0.20 留下的阻塞结论（直接抄录证据）

`v0.20.0-mining-coverage-path-evidence` 阶段虽然把诊断能力建立起来，但**采矿收益依旧未闭环**。`audit-summary.md` 把残留 S2 / S3 列得很清楚：

| 等级 | 内容 |
| --- | --- |
| S2 | 采矿收益阻塞：未取得 `harvestingCount > 0`、稳定 `target` 或 `body.amount` 下降。 |
| S2 | 路径成本阻断：helper `suggestedExpansionPath` 给出的入圈路径在真实铺设中被 `metal` 成本阻断。 |
| S2 | 输入稳定性：原 `mining=-1,1` 路径受 HUD 覆盖、CDP 点击递送、相机 / 站体移动影响，不能稳定复现。 |
| S3 | 提示 / 诊断仍可优化：当前能解释"距离 / 不会产出"，但未直接给出**成本可支付路径**、候选替代路径或输入避让建议。 |

这四条共同构成"采矿收益未闭环"的发布阻塞。在 1.0 玩家承诺里，"采矿收益"是 P0 闭环的中枢节点：没有可归因收益，资源规划、维修取舍、自然交火、复盘和 Meta 回流都没有真实意义。

### 1.2 金属成本阻断的量化根因

把现有代码与初始值放在一起算账，问题就显形：

- 玩家初始金属：`RUN_BASE_STARTING_RESOURCES.metal = 130`，矿石 60。
- 站体默认生成位置（首关 emptyVoid，`findStationSpawn` 偏好分支）：约 `{ x: -780 + rng[-90,90], y: -260 + rng[-90,90] }`，最远情况落到 `star + 600~900 px` 任意角度的回退分支。
- 资源天体 orbit（`tryPlaceOrbitBody`）：行星 380~1100 px、小行星 240~1280 px。
- 最近资源到站体的典型距离：经验上 600~1100 px（v0.20 复测路径 `mining=-1,1` 与最近资源外环差距大致落入该区间）。
- 采矿外环：`range = body.r + 130`；行星 `r ∈ [70,160]` → range 200~290；小行星 `r ∈ [22,48]` → range 152~178。
- 需要扩建的边界距离 ≈ `nearest.distance − range`，典型 350~800 px。
- 每格 frame 占 30 px → 边界距离 / 30 ≈ 12~27 格框架。
- 每格 frame 8 metal → 框架链路成本 96~216 metal。
- 采矿站本身：metal 28、ore 10。
- 合计典型成本：metal **124~244**、ore **10**。

玩家起步金属只有 **130**。在 `balanced` 协议下：

- 顺利时（短距 12 格 + 矿站）≈ 124 metal，刚好临近上限，可能 1 metal 差距即拒绝。
- 中等距离（17 格 + 矿站）≈ 164 metal，**确定不够**。
- 长距离（25+ 格 + 矿站）≈ 228+ metal，**严重不够**。

这正是 v0.20 helper 路径理论可达、玩家实际付不起的根因：helper 只模拟邻接扩建与覆盖关系，不验证 `getBuildCost` 是否可支付。

### 1.3 影响 1.0 发布的具体冲击

| 1.0 Gate | 当前阻塞原因 |
| --- | --- |
| 玩家路径 Gate A05 / C03 / C04 / C07 / D04 | 没有可归因采矿收益，玩家无法形成"建造 -> 收益 -> 再建造"循环，金属 / 矿物变化曲线缺失。 |
| 玩家路径 Gate B05 / B06 | helper 建议的扩建路径在体感上是"沉默失败"——金属看似够建第 1 格，第 12 格才发现不够，玩家无法看清缺口。 |
| 质量 Gate "可理解反馈" | resourceGuide 已经显示"还差 X 距离"，但没显示"还差 Y 金属"。玩家被 UI 暗示沿框架扩建即可，实际中途断粮。 |
| 技术 Gate "存档稳定" | 与本任务无关，需保持不破坏。 |
| 发布 Gate | 没有 P0 闭环之前，发布闸门不能开。 |

### 1.4 本 ADR 解决问题的边界

本 ADR 必须解决的：**找到一条让首局玩家在 1.0 默认协议（balanced）下能产出可归因采矿收益的可执行路径，并明确该路径对核心规则 / 数值 / 布局 / 移动能力的最小侵入度**。

本 ADR 不解决的：

- 不替制作人决定方案；产出候选 + 推荐 + 影响面 + 风险 + 回滚，由制作人决策。
- 不替数值策划定具体数字；涉及成本 / 数量调整时，给出可行**取值范围**与上下限，最终值由 `/systems-numerics-designer` 定稿。
- 不替 UI 设计师定文案与版式；涉及提示 / ghost path 渲染只给出技术接口形态。
- 不解决自然交火、复盘、Meta 回流的后续闭环（依赖本任务完成后再推进）。

## 4. 候选方案逐项评估

为方便对照，每个方案统一按"修改点 / 影响面 / 风险 / 测试方法 / 回滚方式 / 评估结论"展开。

### 4.1 方案 A：只增强诊断与 UI 反馈（only）

#### A.1 修改点

- `buildResourceGuideHtml`（`Game/game.js` ≈ L14940-15013）：把现有"距离 X / 还需靠近 Y"升级为"还需金属 Y、矿石 Z 才能完成扩建+矿站"。
- `getBuildPaletteRecommendation` / `getBuildPaletteStatusText`（≈ L7055-7097）：在 `mining` 设施 tip 中附带"金属链路 ≥ X 才能盖到最近资源"。
- `computeMiningCoverageDiagnostics` 输出新增一组**计算字段**（不破坏现有字段）：`affordability`（candidate frameSteps × frame.cost + mining.cost vs `state.resources`）。
- `getMiningCoverageDiagnostics` test hook 同步暴露新字段。
- 可选：在 `placementPreview` 里给 mining 设施增加"链路成本预估"行。

#### A.2 影响面

- 仅 UI / 诊断层；不动 `world.bodies`、不动 `TYPES`、不动 `RUN_BASE_STARTING_RESOURCES`、不动 `PLAYER_PHYSICS`、不动 `META_TALENT_TREE` / `START_PROTOCOLS`。
- 不影响存档结构、Meta schema。
- 不影响 v0.6 / v0.7 / v0.8 / v0.9 / v0.10 现有状态机。

#### A.3 风险

- 低（属于"必要最小反馈"范畴）。但**单独使用不能解决 §1.2 的金属阻断**——玩家看到"还需 40 metal"也只能干瞪眼。
- HUD 信息密度上升的可读性风险；需要 `/ui-artist` 参与文案与版式取舍。
- 误把"金属链路缺口"和"采矿站本身缺口"混在一行，玩家可能误读为"再采点就够"。

#### A.4 测试方法

- 自动：`__gameTest.playtest.getMiningCoverageDiagnostics()` 断言 `affordability.metalNeeded`、`affordability.oreNeeded` 与 `getBuildCost("frame") × frameSteps + getBuildCost("mining")` 一致。
- 浏览器级：空档进入首局，观察 resource guide 显示金属缺口数。
- 真人体验：玩家是否在 30 秒内能复述"我需要再凑 X 金属"。

#### A.5 回滚方式

- 删除新增 HTML 字符串与 `affordability` 字段；恢复 `buildResourceGuideHtml` 原样。回滚边界清晰，单文件 diff。
- 旧档兼容：完全无影响。

#### A.6 评估结论

- 必选打底（任何主方案都应包含 A 才能让玩家理解）。
- 不能单独闭环。

### 4.2 方案 B：首局资源布局保障

#### B.1 修改点

- `findStationSpawn`（≈ L3142-3154）或 `generateGalaxy`（≈ L3157-3266）：在 `level === 0`（首关）时，强制 stationSpawn 的 `nearest harvestable body` 在 `<= safeBudget` 距离内；不满足则继续放置直到满足。
- 同步在 `tryPlaceOrbitBody` 中加 `minOrbit` 阶段性约束，或在 `generateGalaxy` 末尾追加一次"保障性资源体植入"（在 stationSpawn 附近 280~400 px 半径内额外放置一颗 r ∈ [40, 60]、resource ∈ {"ore","metal"}、`amount = computeBodyAmount("asteroid", 0, rng)` 的小行星）。
- 需要在 `applyGalaxy` 路径同步保证 stationSpawn 与该保障资源不重叠 (`dist > body.r + 260`) 且不和现有 placed bodies 冲突。

#### B.2 影响面

- `state.world.bodies` 结构本身不变（仍是同一字段），但**数量 / 位置发生变化**。
- `state.galaxy.bodies` 持久化字段不变（不进存档），仅运行时影响。
- `generateGalaxyPreview` 与既有"galaxy 视觉签名"自检（v0.9 引入）的输出会变；需同步更新预览基线。
- `getNearestResourceBody` 行为不变，但首关行为更稳定。
- 触动 `generateGalaxy` 的 RNG 序列，导致 v0.8 / v0.9 / v0.10 既有 seed 在首关的 `bodies[]` 不一致；影响所有 PM 之前用 seed 取证的回归基线。
- 不影响 `MINING_RANGE_OFFSET`、`TYPES.*.cost`、`RUN_BASE_STARTING_RESOURCES`、`META_TALENT_TREE`、`START_PROTOCOLS`。
- 不影响存档结构（galaxy 不写盘）。

#### B.3 风险

- **高风险**（"资源布局"是制作人方向显式列出的高风险项之一）。
- RNG 决定性破坏：旧 PM seed 复现路径全部失效；v0.9 视觉签名自检需要重新建立基线。
- 多 galaxyType 一致性：`miningBelt`、`tradeHub` 等型态原本就资源丰富，强制保障可能造成"首关全资源溢出"，破坏 v0.9 引入的星系类型差异化。
- 难度曲线：首关采矿过于轻松后，level 1+ 的"扩建 / 移动"挑战感被压扁。
- 与现有 `findStationSpawn` 的偏好分支 `{ x: -780, y: -260 }` 耦合；偏好失败时回退随机分支后的保障逻辑要避免无穷循环。

#### B.4 测试方法

- 自动：批量跑 `generateGalaxyPreview(0, "emptyVoid", seed)` 在 100 个 seed 下断言 `min(dist(stationSpawn, harvestableBody)) <= safeBudget`。
- 自动：`generateGalaxyPreview(0, X, seed)` 在 X ∈ 所有 galaxyType 下都满足；对 `miningBelt` 等已经资源丰富的型态，不能引入冗余资源体。
- 浏览器级:空档进入首关 + balanced 协议，观察 `__gameTest.playtest.getMiningCoverageDiagnostics()` 的 `bestCandidate.expansionPath.frameSteps`。
- 视觉签名：与现有 v0.9 自检对照，必须主动更新基线并经 PM 接受。

#### B.5 回滚方式

- 单点开关：在 `generateGalaxy` 入口加 `runtimeFlag = true/false`；可一次性关闭"首关保障"逻辑回退到 v0.8 行为。
- 旧档兼容：完全无影响（galaxy 不写盘）。
- 视觉签名基线回滚：在 v0.9 自检文档里同步回滚。

#### B.6 评估结论

- 根治候选，但破坏性高。
- 仅作为 §3 D4 兜底方案使用。
- 实现前必须由制作人 + 数值方案明确"首关与后续关的张力分配"，否则会把 1.0 收益闭环换成 1.0 体验扁平化。

### 4.3 方案 C：调整 `MINING_RANGE_OFFSET`

#### C.1 修改点

- `MINING_RANGE_OFFSET = 130`（L898）改为更大值（例如 180、220、260）。
- `getMiningRange`（L11172）无需改动，因为公式 `body.r + MINING_RANGE_OFFSET` 自动放大。
- 渲染层：`renderer.ring(body, miningRange, ...)`（≈ L14226-14230）会自动跟随，但视觉外环会明显变大，需要 UI 审视是否破坏星系图可读性。

#### C.2 影响面

- **全局影响**：所有星系、所有关卡、所有矿站；玩家从 v0.1 起的所有 PM 取证基线全部失效。
- 与 `TYPES.shield.baseStats.range`、`TYPES.turret.baseStats.range` 等无直接耦合，但视觉感知上"采矿圈"与"武器射程"的比例发生变化。
- 不影响存档 / Meta schema。
- 不影响 `START_PROTOCOLS` / `META_TALENT_TREE`。

#### C.3 风险

- **高风险**（"采矿范围 / 公式"是制作人方向显式列出的高风险项之一）。
- 压扁玩法张力：v0.5+ 设计中"扩建链路靠近资源"是核心循环组成；放大 range 会让玩家"不扩建即采矿"。
- 渲染：外环显著变大，影响星系图可读性，可能与小行星密集区域重叠产生视觉混乱。
- 后期关卡：若 level 0 因 range 增大可达，level 5+ 会变成"啥都可达"，把后期采矿难度归零。
- 与 v0.8 encounters / v0.9 galaxyMap 难度曲线脱钩。

#### C.4 测试方法

- 自动：`computeMiningCoverageDiagnostics()` 在调大 offset 后断言 `harvestingCount > 0`。
- 渲染：视觉对照 v0.20 截图，确认外环是否过度膨胀。
- 多关卡：跑 `generateGalaxyPreview(level, type, seed)` 对照 level 5、ENDGAME_LEVEL 的体感。

#### C.5 回滚方式

- 单值回退：`MINING_RANGE_OFFSET = 130`。
- 旧档无影响。

#### C.6 评估结论

- 见效最快、但 1.0 之后的所有版本都需要承担"采矿门槛已经被砍"的副作用。
- 本 ADR 不推荐作为 1.0 候选；仅在制作人明确判断"采矿挑战不在 1.0 范围"时再启用。

### 4.4 方案 D：调整早期成本 / 起步资源（数值方案）

#### D.1 修改点（候选子集）

- D-a：`RUN_BASE_STARTING_RESOURCES.metal` 由 130 提升到 145~160（提升幅度 15~30 metal）。
- D-b：`TYPES.mining.cost.metal` 由 28 降到 22~25、`TYPES.mining.cost.ore` 由 10 降到 8（仅采矿站自身降本）。
- D-c：`TYPES.frame.cost.metal` 由 8 降到 6~7（影响所有链路成本，会同时压低后期建造门槛）。
- D-d：首局首次开启时，给 balanced 协议追加一次性 "Tutorial Buff"——在 `state.run.startupResourceBonus.metal` 上加 +20，但**仅 level 0 且 metaPoints == 0**生效（首次玩家专属，老玩家不受影响）。

#### D.2 影响面

- D-a：所有局、所有协议、所有难度的初始金属上升 → 协议 `startingCache`（+15 metal）与 `miningStart`（+25 metal）的相对意义被压扁，需要数值方案同步调协议加成。
- D-b：所有局的采矿站建造成本下降 → `miningFacilityDiscount` 协议的相对意义被压扁；后期 `processor` 等链路也会感知到金属节省。
- D-c：所有局的所有 frame 链路成本下降 → 影响整个建造经济，最广泛。
- D-d：仅首次玩家、首局生效 → 影响最小，但需要在 `state.run` 上加一个判断逻辑；可复用现有 `metaPointsGainedThisRun` / `state.meta.points` 字段判断。
- 不影响存档结构、Meta schema、`SAVE_KEY`。
- 不影响 v0.6 / v0.7 / v0.8 / v0.9 状态机；影响 v0.10 协议设计的相对收益。

#### D.3 风险

- **高风险**（"数值"是制作人方向显式列出的高风险项之一）。
- 经济链联动：metal 是所有设施成本的核心资源，调整后会影响整局节奏（建造速度、电力建设、武器建设、维修取舍）。
- 协议失能：v0.10 已发布的协议（`startingCache` / `miningStart`）依赖"初始金属是稀缺资源"作为产品价值锚点；D-a / D-b 直接削弱该锚点。
- "Tutorial Buff"（D-d）的判断条件需要严格隔离，避免老玩家偶尔触发或被刷点击重复触发。
- 数值方案如改幅度过大，可能引入"首关过于轻松、二关断崖"问题。

#### D.4 测试方法

- 计算：构造一张"frame chain 步数 × per-step cost × current resources"的表，由数值方案验证在每个修改值下，玩家走通的最远 frame chain 步数。
- 自动：`getBuildCost("mining") / getBuildCost("frame")` 在调整后的输出与数值方案给的目标一致。
- 自动：`__gameTest.meta.snapshotMeta()` 在 D-d 触发前后断言"首次玩家 +20 metal，老玩家无变化"。
- 浏览器级：空档 + balanced 协议，跑出 `harvestingCount > 0`，记录走通的 frame steps、矿石消耗、剩余金属。
- 旧档兼容：用 `__gameTest.meta.injectOldSave({...v0.9 schema})` 验证 D 修改不影响迁移路径。

#### D.5 回滚方式

- D-a / D-b / D-c：单值回退到原数值。
- D-d：删除"Tutorial Buff"分支即可。
- 旧档兼容：完全无影响。

#### D.6 评估结论

- D-d（首次玩家 Tutorial Buff）侵入度最低，且不破坏 v0.10 协议经济，**作为 v1.0 fallback 推荐**（参见 §3 D3）。
- D-a / D-b 是数值方案最直接的调参，**仅在 D-d 不足时考虑**，且必须由数值方案同步调 v0.10 协议平衡。
- D-c 影响过广，**v1.0 不推荐**。

### 4.5 方案 E：增加采矿路径 helper / ghost path 可视化

#### E.1 修改点

- 在 `placementPreview` / `bridgePreview` 邻近的渲染层（≈ `renderer.ring` 调用点附近），追加 `ghostPath` 渲染：把 `computeMiningCoverageDiagnostics().suggestedExpansionPath.frameSteps` 当作半透明覆盖层渲染成连续矩形或十字网格。
- 同步在 `buildResourceGuideHtml` 给一行"按虚影路径扩建：X 步框架 + 1 座采矿站，总成本 metal Y、ore Z"。
- 不修改 `computeMiningCoverageDiagnostics` 输入参数；仅消费现有输出。
- 可选：把 ghost path 渲染绑定到 `state.selectedBuild === "mining"` 状态，避免视觉污染常规建造。

#### E.2 影响面

- 渲染层 + UI 层；不动核心规则、不动数值、不动 RNG、不动布局、不动 schema。
- 不影响存档结构 / Meta schema。
- 不影响 v0.6 / v0.7 / v0.8 / v0.9 / v0.10 现有状态机。
- 与 v0.9 galaxyMap 渲染层共享 canvas，需要确认渲染顺序不与 `renderer.ring` 冲突。

#### E.3 风险

- 低 - 中（属于"必要最小反馈"扩展）。
- 视觉密度：在小屏 / 触控模式下，ghost path 可能与 placement preview、星系外环混在一起；需 `/ui-artist` 把控分层。
- 性能：每帧绘制 12~27 个矩形对桌面端无压力，但若移动端帧率敏感，需要做"仅 selectedBuild === 'mining' 时绘制"。
- 误导：ghost path 只表达"框架可达"，不表达"金属够不够"；必须配合 A 的金属缺口提示，否则玩家照画 ghost path 仍然中途断粮。

#### E.4 测试方法

- 自动：在 `__gameTest.playtest.getMiningCoverageDiagnostics()` 输出基础上，新增 `__gameTest.render.captureGhostPath()`（如果引入），返回 ghost 渲染的 grid keys，断言与 `suggestedExpansionPath.frameSteps` 一致。
- 浏览器级：截图对照 ghost path 与玩家实际建造结果。
- 真人体验：玩家是否能"照着虚影盖完"。

#### E.5 回滚方式

- 删除 ghost 渲染分支与 resource guide 行；保留 `computeMiningCoverageDiagnostics` 输出本身（v0.20 已有）。
- 旧档无影响。

#### E.6 评估结论

- **推荐**作为 v1.0 默认产品方案的一部分（§3 D3 中的 E）。
- 不能单独闭环——必须配合 A（金属缺口提示）和 F（移动 + 重建）。

### 4.6 方案 F：移动路径证据 + 教学（驾驶工站靠近 + 重建）

#### F.1 修改点

- **不**修改 `PLAYER_PHYSICS`、`updateStationPhysics`、`thrusterNozzle`、`getKeyboardThrustWorld`（移动能力本身）。
- 在 `buildResourceGuideHtml` 或 `buildGuideText` 中追加一段"或：驾驶工站靠近资源外环，再原地重建采矿站"提示；只在 `bestCandidate?.canReachInRange === false`（helper 给的扩建路径不可达 / 成本不够）时显示。
- 可选：在 `state.target` 机制（`updateStationPhysics` 第 12943 行）上挂一个"自动指向最近资源外环"快捷键或按钮；玩家点击后 `state.target` 设为 `nearestBody + 缩进 range × 0.8`。
- 可选：在 HUD 给一个"已进入采矿外环"反馈条（沿用 v0.20 已有的"已进入范围"小提示，但加上"现在可以建采矿站了"动作建议）。
- 可选：在 `computeMiningCoverageDiagnostics` 输出新增 `mobileApproachHint`：返回"从当前 station.pos 移动到资源外环边缘所需的额外距离 + 估算时间"。

#### F.2 影响面

- UI / 教学层 + 可选的 `state.target` 自动指向。
- 不动核心物理参数；不动 RNG；不动布局；不动数值；不动 schema。
- `state.target` 已存在（用于"前往目标自动靠近"），增加一个设定来源不破坏既有语义。
- 不影响存档结构 / Meta schema。

#### F.3 风险

- 低（属于"必要最小反馈"扩展 + 既有移动能力的引导利用）。
- 教学不充分时，玩家可能尝试"移动 + 重建"但仍因金属不够建站；需要配合 A 的金属缺口提示。
- 自动指向若没有"取消 / 中断"入口，可能与玩家手动操作冲突；需复用现有 `state.input.moveKeys` 检测覆盖逻辑。
- 移动期间若敌人接近，可能被战斗打断；需在文案上预留"可暂停 / 取消"。

#### F.4 测试方法

- 自动：模拟 `state.target = nearestBody` 后跑 `updateStationPhysics(dt)` 数百帧，确认 `state.station.pos` 收敛到资源外环内；调用 `computeMiningCoverageDiagnostics` 后 `nearestResource.inRange === true`。
- 浏览器级：空档进入首局，沿 F 提示操作，最终 `harvestingCount > 0`。
- 真人体验：玩家是否能在 60 秒内理解并完成"移动 + 重建"路径。

#### F.5 回滚方式

- 删除 UI 提示与可选自动指向；保留 `state.target` 既有语义。
- 旧档无影响。

#### F.6 评估结论

- **推荐**作为 v1.0 默认产品方案的一部分（§3 D3 中的 F）。
- F 与 E 形成"扩建走 ghost path / 移动走 target"两条平行路径；玩家可以选；二选一都能闭环。
- 不能单独闭环——v0.20 helper 已经提示过"移动 + 重建"但 PM 实际仍未走通，说明现有教学不足，F 必须搭配 A 的反馈。

### 4.7 方案 G：组合方案（推荐基础）

按 §3 D3，G 拆为两层：

#### G.1 G 默认层（v1.0 必做）= A + E + F

- A：金属缺口诊断 + UI 反馈，让玩家看到"还差多少 metal"。
- E：ghost path 可视化扩建路径。
- F：移动 + 重建教学 + 可选自动指向。

#### G.2 G fallback 层（验证后再决定是否启用）

- D-d 优先：首次玩家专属 Tutorial Buff +20 metal，仅 level 0 且 `meta.points == 0` 触发；老玩家与高难关无感知。
- D-d 不够时再升 D-a：`RUN_BASE_STARTING_RESOURCES.metal` 由 130 提到 145；幅度由数值方案给定（**上限 +15**，超出走专项审核）。
- D-a 也不够时再升 D-b：`TYPES.mining.cost.metal` 由 28 降到 24~25；幅度上限 -4 metal，且必须同步调 v0.10 协议。
- B 兜底：仅在以上全部启用后**仍**无法保证空档 + balanced 协议下 `harvestingCount > 0`，才走 B（首局资源布局保障）；B 必须经制作人最终批准。

#### G.3 影响面

- 默认层 G（A + E + F）：UI / 教学 / 渲染层；不动核心规则、数值、布局、移动能力、schema、依赖。
- fallback 层 D-d：仅首次玩家可见；其余玩家与历史关卡无感知。
- fallback 层 D-a / D-b：全局数值变化；需要 v0.10 协议同步调整。
- 兜底层 B：触动资源布局；最高侵入度。

#### G.4 风险

- 默认层 G：低 - 中风险，主要是 UI 设计风险（信息密度、视觉分层）。
- fallback 层 D-d：低 - 中风险，仅在"首次玩家判断"逻辑上需要严密。
- fallback 层 D-a / D-b：高风险（数值），且需同步调协议。
- 兜底层 B：高风险（资源布局）。

#### G.5 测试方法

- 每一层都按 §4.x 的测试方法逐一通过。
- 整链路：空档进入首局 + balanced 协议下，按 ghost path 或移动路径走通，得到 `harvestingCount > 0` + `body.amount` 下降。
- 旧档：`__gameTest.meta.injectOldSave({ ...v0.9 schema })` + reload，断言迁移路径与 v0.10 ADR 行为一致。
- 视觉签名：v0.9 自检在默认层 G 下应无变化；fallback 与兜底层需主动更新基线。

#### G.6 回滚方式

- 按层回滚：默认层 G 独立可回滚；fallback 层独立可回滚；兜底层 B 独立可回滚。
- 旧档：完全无影响。

#### G.7 评估结论

- 本 ADR 的**推荐总体方案**；详细分级见 §3 D3 / D4。

## 5. 推荐方案与分级实施策略

### 5.1 推荐总体方案

**默认层 G（A + E + F），实现优先级 P0**：

- 必做：诊断 / UI 显式金属缺口；ghost path 渲染；移动 + 重建教学。
- 责任：`/lead-game-programmer`（边界把关）+ `/senior-gameplay-engineer`（实现）+ `/ui-artist`（视觉 / 文案）+ `/systems-numerics-designer`（提供 §4.1 / §4.5 / §4.6 中各种数值口径）。
- 验收：空档 + balanced 协议下，玩家在 PM 看板 #5 复测中能稳定走通 ghost path 或移动路径之一，得到 `harvestingCount > 0` + `body.amount` 下降。

**fallback 层 D-d（首次玩家 Tutorial Buff）**：

- 触发条件：默认层 G 实现并通过 PM 看板 #4 自检后，仍有 ≥30% 的真实玩家路径无法在 60 秒内取得 `harvestingCount > 0`。
- 责任：`/systems-numerics-designer`（定 +20 metal 是否合适，给上下限）+ `/senior-systems-engineer`（实现首次玩家判断）。
- 验收：首次玩家在默认层 + D-d 下稳定走通；老玩家无感知；`__gameTest.meta.snapshotMeta()` 输出在两种用户下保持一致 Meta 字段。

**fallback 层 D-a / D-b（数值微调）**：

- 触发条件：D-d 后仍不足；由数值方案与 PM 联合判断。
- 责任：`/systems-numerics-designer` 主责；`/lead-game-programmer` 审核协议影响。
- 验收：v0.10 协议（特别是 `startingCache` / `miningStart`）经数值同步调整后仍有可感知收益。

**兜底层 B（首关资源布局保障）**：

- 触发条件：fallback D-a / D-b 后仍不足。
- 责任：`/lead-game-programmer`（实现）+ `/systems-numerics-designer`（首关与后续关张力分配）+ `/pm-qa-coordinator`（视觉签名基线更新）。
- 验收：100 seed 自动测试首关 stationSpawn 最近资源距离均 ≤ 安全预算；v0.9 视觉签名基线更新且经 PM 接受；制作人最终批准。

**永不启用**：

- 方案 C（改 `MINING_RANGE_OFFSET`）：本 ADR 否决。
- 整体抬升 `RUN_BASE_STARTING_RESOURCES` > 15 metal 或 D-c（改 frame cost）：本 ADR 否决。
- 修改 `PLAYER_PHYSICS` / `updateStationPhysics`：本 ADR 否决。

### 5.2 实施流程对齐

每一层都必须按制作人方向 §"风险流程"执行：

1. `/game-producer-lead-designer` 选定本层范围。
2. `/pm-qa-coordinator` 在 PM 看板 #3 完成方案锁定，拆分到 #4 实现任务。
3. `/systems-numerics-designer` 输出数值表 / 文案口径（涉及数值与文案时）。
4. `/ui-artist` 输出 ghost path 视觉与教学版式（涉及 E / F / A 时）。
5. `/lead-game-programmer` 审核边界：是否触碰 `SAVE_KEY` / `META_SCHEMA_VERSION` / 物理参数 / RNG / 资源布局。
6. `/senior-gameplay-engineer` + `/senior-systems-engineer` 实现 + 自检（PM 看板 #4）。
7. `/pm-qa-coordinator` 复测（PM 看板 #5 + #6）。
8. `/lead-game-programmer` 专项审核（PM 看板 #12）。
9. `/game-producer-lead-designer` 决策是否进入下一层 / 是否进入 1.0 发布闸门（PM 看板 #14）。

## 7. 风险、回滚与回归矩阵

### 7.1 关键风险

| 风险 | 触发条件 | 缓解 | 触发回到 ADR 的条件 |
| --- | --- | --- | --- |
| 默认层 G 仍无法闭环 | 真实玩家在 60 秒内仍无法取得 `harvestingCount > 0` | 升级 fallback D-d | PM #5 复测连续 3 轮失败 |
| fallback D-a / D-b 破坏 v0.10 协议价值 | 调整后 `startingCache` / `miningStart` 玩家感知贬值 | 数值方案同步调协议 effects.value | 数值方案 review 出现"协议失能"判断 |
| 兜底 B 破坏多 galaxyType 一致性 | 在 `miningBelt` / `tradeHub` 等型态触发资源溢出 | 在 B 实现中按 galaxyType 跳过保障逻辑 | `generateGalaxyPreview` 自检报错或 PM 视觉签名异常 |
| 兜底 B 破坏 v0.9 视觉签名 | 旧 seed 在首关产生不一致 bodies | 主动更新 v0.9 视觉签名基线 | 视觉签名自检 fail |
| Ghost path 与现有渲染冲突 | ghost 与 ring / placementPreview 重叠不清 | UI 方案在 P0 阶段完成视觉分层 | 玩家测试出现"看不清现在在哪格" |
| `state.target` 自动指向打断玩家手动操作 | 玩家在 F 触发时同时按键移动 | 在 `updateStationPhysics` 检测 `keyboardThrust` 时优先级高于 `state.target` | 玩家测试出现"被吸走" |
| Tutorial Buff 触发条件失误 | 老玩家因 `meta.points == 0` 被误识别 | 判断条件包含 `meta.unlockedProtocols.length === 1 && schemaVersion === 2 && lifetime stats == 0` 等多重锁 | 老玩家在 PM #5 复测中无意识被 Buff |
| `affordability` 新字段破坏 `storageSignature` | 字段意外被写盘 | 仅 in-memory，不进 `state.meta`、不进 `localStorage` | release candidate checks 报错 |

### 7.2 回归矩阵

实现完成后，PM #12 / #13 / #14 必须覆盖以下回归点：

| 回归点 | 用例 |
| --- | --- |
| v0.19 维修链不回潮 | 空档进入 → 建 frame → 受损 → 建 repair → 观察"下一座资源不足"不会被误读为"刚才失败"。 |
| v0.20 helper 字段不漂移 | `__gameTest.playtest.getMiningCoverageDiagnostics()` 输出包含 `readOnly === true`、`miningStatus`、`reachability`、`miners`、`nearestResource`、`targetResource`、`candidates`、`bestCandidate`、`suggestedExpansionPath`、`failure`、`rules`；如默认层 G 新增 `affordability`，断言其字段类型与数量。 |
| v0.10 Meta 迁移不回潮 | `__gameTest.meta.injectOldSave({ points: 7, mining: 2, hull: 1, weapons: 1, unlocks: { weaponFrame: true } })` + reload，断言点数 / talents / protocols 与 v0.10 ADR §11.1 节点条件一致。 |

| Tutorial Buff 隔离 | 老玩家（已购买过 talents / 已解锁过 protocols）在 fallback D-d 下不被触发；首次玩家被触发；二次进入不被重复触发。 |
| 真实输入 | 空档 + balanced 协议 → 按默认层 G 走通的真实鼠标 / 触控记录；小屏 / 桌面分别记录。 |
| 旧档迁移 | 在 v0.10 schema 旧档基础上跑默认层 G + fallback；点数不丢失、协议状态不丢失。 |

### 7.3 回滚边界

- 任何一层启用后，回滚都必须能在**单次 commit**内完成：要么 revert 该层的 commit；要么删除 `state.run.*` 上的 transient 字段。
- 旧档**永远**不会因为本 ADR 任何方案的回滚而损坏；`SAVE_KEY` / `META_SCHEMA_VERSION` 不变。
- 兜底 B 的回滚需要同步回滚 v0.9 视觉签名基线（PM 维护）；其他层无需基线回滚。

## 10. 与 v0.20 / v0.17 / v0.10 / v0.9 边界对照

### 10.1 与 v0.20 边界对照

- 本 ADR 不再要求"只读 helper"约束（v0.20 是内部证据阶段才约束）。
- 本 ADR 推荐的默认层 G 会**首次写入** `state.run.startupResourceBonus.metal`（fallback D-d 触发时），但仅 transient，不进 `state.meta`、不进 `localStorage`。
- 本 ADR 沿用 v0.20 的 `computeMiningCoverageDiagnostics` 输出 schema；新增字段不破坏既有字段。

### 10.2 与 v0.17 边界对照

- v0.17 强调"先补证，后判断"；本 ADR 已经在 §1 完成 v0.20 证据复用与量化根因分析，符合 v0.17 边界。
- v0.17 列出的"必须升级"清单（战斗 / Meta / 星系 / 存档 / 环境）在本 ADR 中**均不触动**；默认层 G 完全在低风险范围；fallback / 兜底层走数值与资源布局，已在 §3 / §5 / §6 标记为高风险流程。

### 10.3 与 v0.10 Meta ADR 边界对照

- 本 ADR 不修改 `SAVE_KEY = "star-ring-station-meta-v1"`。
- 不修改 `META_SCHEMA_VERSION = 2`。
- 不新增 `state.meta` 字段。
- 不修改 `META_TALENT_TREE[*].id`、`START_PROTOCOLS[*].id`。
- fallback D-a / D-b 若调 `effects.value`，由数值方案同步且不触动 effect op / 顺序。
- Tutorial Buff（D-d）的"首次玩家判断"会读 `state.meta.points` 与 `state.meta.unlockedProtocols`，但**只读不写**。

### 10.4 与 v0.9 galaxyMap 边界对照

- 默认层 G：不动 `generateGalaxy` / `findStationSpawn` / `tryPlaceOrbitBody`，v0.9 视觉签名不变。
- fallback D-a / D-b：同上。
- 兜底 B：触动 `findStationSpawn` / `generateGalaxy`；必须同步更新 v0.9 视觉签名基线并经 PM 接受。
