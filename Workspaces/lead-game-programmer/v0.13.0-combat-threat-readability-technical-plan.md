## 0. 文档定位与边界

本方案只定 **v0.13.0 战斗威胁可读性的技术边界**：5 个新只读 helper 的纯函数语义与只读不变量、4 个 UI 接入点的调用规则、运行时临时统计的 ring buffer 边界、`__gameTest.playtest` 字段扩展、与 v0.12 design health 缓存的协同、HUD 0.12 秒节流下的性能保护、禁止改动清单、T1-T4 任务拆分与节点审查条件。

**不在本方案决策范围**：

- 威胁/武器/维修/复盘文案话术、严重度命名、阈值具体数字、UI 视觉密度 → 文案/阈值表（PM #4）。
- HUD 行内布局、卡片视觉层级、复盘短项视觉密度 → UI 方案（PM #2）。
- 玩家体验取舍（是否能看懂威胁、复盘建议是否实用）→ 制作人 #10 最终决定。
- 浏览器与人工回归证据的具体记录形式 → PM 看板 #9。

本方案只提供 **接入框架**：helper 输入/输出 schema、UI 入口的调用规则、运行时数据生命周期、不变量。UI 方案与文案方案产出后按 §3-§7 边界填入即可，不需要回到本方案再决策。

本方案严格遵守：**只允许新增/修改 `Workspaces/lead-game-programmer/v0.13.0-combat-threat-readability-technical-plan.md`**，不修改 `Game/`、`Docs/`、`.cursor/`、`42.md`、`Releases/`、PM 看板、方向文档、其他工作区文件和历史 ADR；不删除文件；不 stage/commit dirty `42.md`；禁止 `git reset` / `git clean` / `git stash` / checkout 覆盖 / force push。

## 2. 现有代码勘察

### 2.1 敌人与威胁现状

- `state.enemies` 元素结构（`spawnEnemy` 内组装，见 `Game/game.js:10545`）：`{ kind, x, y, vx, vy, angle, reload, spawn, spawnLevel, maxHp, hp, r, accel, drag, range, reward, spin, projectileDamage, reloadTime, spawnInterval, spawnCap, collisionDamage, guardianMinion, hostileStationMinion, ...kind 特有字段 }`。
- 敌种 enum（`enemy.kind`）：`asteroid / pirate / station / guardian / hostile-station`（小行星 / 海盗 / 普通敌站 / 守护者 / 敌方空间站 BOSS）。`enemyUsesStationAi`（`Game/game.js:10204`）已声明 `station / guardian / hostile-station` 三类共享"悬停 470" AI 分支。
- `hostile-station` 由多 cell 组成：`enemy.cells` 是 Map，`enemy.weaponCellKeys` 数组，`enemy.shieldCellKey` 字段；HP 由 `sumCellsHp(enemy.cells)` 计算。
- 距离判定：`dist(state.station.pos, enemy)`（`Game/game.js:utils`）。方向判定建议用 `enemy.x - state.station.pos.x` / `enemy.y - state.station.pos.y` 与 `state.station.angle` 组合，避免在 helper 里引入新坐标变换。
- 已有诊断 helper：`isEnemyPressureLikely()`（`Game/game.js:7520`）：返回是否存在 `enemy.hp > 0 && enemy.dead !== true` 的活体敌人、是否有 combat encounter（`ambush / distress / guardian`），或终末守护者未死。可直接复用为"是否处于战斗"的开关。
- 现有 HUD 体现：`weaponsRowEl`（`Game/game.js:12212-12256`）显示 missile/turret ready 计数与 LOS 阻挡数；`statusAlerts` 中 `endgame` 文案、`hostileStationAlerted` 状态、`fragmentAlerts / hostileStationHudAlerts / encounterAlerts / npcAlerts` 都已聚合。
- **关键发现**：当前没有任何 helper 输出"威胁方向 / 最近敌人类型 / 距离"。这是 §3.2 必须新增的字段。

### 2.3 战斗事件与伤害结算现状

| 写入点 | 触发时机 | 本方案如何"插眼" |
| --- | --- | --- |

| `damageFragmentCell(fragment, cell, damage)`（`Game/game.js:9514`） | 脱落碎片被命中 | 同上，记录 `type: "fragmentHit"` |
| `damageNpc(npc, amount)`（`Game/game.js:8863`） | NPC 被命中（仅作为 collateral，不进入 stationHit 主统计） | 仅记录 `type: "npcHit"` 用于 PM 复核；可选项，不进入复盘必需字段 |
| `enemy.hp -= projectile.damage`（`Game/game.js:10880`） | 我方弹道命中普通敌人 | 在该写入点后追加 `recordCombatEvent({ type: "enemyDamaged", enemy, damage, source: projectile.owner })` |
| `tryDamageHostileStationCell(projectile, enemy)`（`Game/game.js:9655` & `Game/game.js:10872`） | 命中敌方空间站 cell | 在返回 true 路径上追加 `recordCombatEvent({ type: "hostileStationCellHit", enemy, damage })` |
| `state.enemies.filter` 内 `hp > 0` 转 false 分支（`Game/game.js:10478-10501`） | 敌人死亡（pirate/asteroid/station/guardian/hostile-station） | 在分支头部追加 `recordCombatEvent({ type: "enemyKilled", enemy })`；与现有 `notifyObjective / notifyEncounters("enemyKilled", enemy)` **并列**，不替换、不依赖 |
| `cell.fire = 1` in `updateShield`（`Game/game.js:10647`） | 护盾拦截弹体 | 在 `cell.fire = 1` 写入后追加 `recordCombatEvent({ type: "shieldBlocked", cell, projectile })` |
| `cell.fire = 1` in `updateProjectiles` hostile-station shield 分支（`Game/game.js:10866`） | 敌方 BOSS 护盾拦截我方弹体 | 不进入 stationHit 主统计；可选记录 `type: "hostileShieldBlocked"`，默认不开 |
| `fireMissiles()` 内 `cell.reload = getCellStat(...)` 与 `fired += projectileCount`（`Game/game.js:10719-10720`） | 一次齐射完成 | 在 `showToast(fired ? ...)`（`Game/game.js:10722`）**之前** 追加 `recordCombatEvent({ type: "missileSalvoFired", fired, salvoSize, count: toFire.length })` |
| `updateRepair(dt)` 内 `state.repairDrones.push(...)`（`Game/game.js:10979`） | 派出维修无人机 | 追加 `recordCombatEvent({ type: "repairDispatched", repairerKey, targetKey })` |
| `updateRepair(dt)` 内 drone 到达 target 并修复（`Game/game.js:11001+` 后续 step） | 单次修复完成 | 追加 `recordCombatEvent({ type: "repairApplied", repairerKey, targetKey, healed })`（healed 字段需 v0.13 临时统计；不写存档） |
| `gameOver()`（`Game/game.js:11144`） | 核心被摧毁 | 不插眼；战斗窗口由 `isEnemyPressureLikely` 驱动，gameOver 后窗口自然停摆 |

**关键约束**：

1. 所有 "插眼" 必须是 `recordCombatEvent(payload)` 单行调用；该函数体只做 ring buffer push + 窗口聚合（§4），**不返回**任何影响调用方的值，**不抛错**（用 try/catch 包裹）。
2. 所有插眼 **不修改** 调用方上下文中的任何字段（即不能写回 `cell.hp` / `enemy.hp` / `projectile.dead` / `state.run.*` / `state.resources.*`）。
3. `recordCombatEvent` 必须在 §7 禁止清单中明确："禁止用于改变 cell/enemy/projectile/repair 任何字段"。

### 4.2 `recordCombatEvent` 与 `pruneCombatEvents`

```javascript
function recordCombatEvent(event) {
  try {
    if (!event || typeof event !== "object") return;
    const enriched = { ...event, t: state.time };
    combatEventBuffer.push(enriched);
    if (combatEventBuffer.length > COMBAT_EVENT_BUFFER_CAPACITY) {
      combatEventBuffer.splice(0, combatEventBuffer.length - COMBAT_EVENT_BUFFER_CAPACITY);
    }
    updateCombatWindowOnEvent(enriched);
  } catch {
    // 任何异常都吞掉，绝不影响调用方
  }
}

function pruneCombatEvents(now = state.time) {
  const cutoff = now - RECENT_DAMAGE_WINDOW_SEC;
  while (combatEventBuffer.length > 0 && combatEventBuffer[0].t < cutoff) {
    combatEventBuffer.shift();
  }
}
```

**约束**：

1. `recordCombatEvent` **不返回**值，不抛错。
2. `try/catch` 包裹是硬性要求：插眼出错绝不能阻断 `damageCell` / `fireMissiles` 等关键函数。
3. `pruneCombatEvents` 只在 helper 调用入口（`getRecentDamageSummary / getCombatReview / getThreatSummary`）开头调用一次，避免每个事件都遍历。

### 4.3 战斗窗口生命周期

```javascript
function updateCombatWindowOnEvent(event) { /* 仅在伤害事件、射击事件、击杀事件触发时累加 current 字段 */ }
function ensureCombatWindowState(now) {
  const pressure = isEnemyPressureLikely();
  if (pressure) {
    if (!combatWindowState.active) {
      combatWindowState.active = true;
      combatWindowState.startedAt = now;
      combatWindowState.current = createEmptyCombatAggregate(now);
    }
    combatWindowState.lastActiveAt = now;
    combatWindowState.pendingArchive = false;
  } else if (combatWindowState.active) {
    combatWindowState.pendingArchive = true;
    if (combatWindowState.lastActiveAt != null
      && now - combatWindowState.lastActiveAt >= COMBAT_WINDOW_QUIET_SEC) {
      archiveCombatWindow(now);
    }
  }
}
function archiveCombatWindow(now) {
  const current = combatWindowState.current;
  if (current && now - current.startedAt >= COMBAT_WINDOW_MIN_RECORD_SEC) {
    combatWindowState.lastArchived = freezeCombatWindowSnapshot(current);
    combatWindowState.archivedAt = now;
  }
  combatWindowState.active = false;
  combatWindowState.pendingArchive = false;
  combatWindowState.current = null;
  combatWindowState.startedAt = null;
  combatWindowState.lastActiveAt = null;
}
```

**约束**：

1. `ensureCombatWindowState` 仅在 `getThreatSummary / getRecentDamageSummary / getCombatReview / getWeaponEffectiveness` 入口和 `recordCombatEvent` 末尾调用一次；不要求 `update(dt)` 末尾再调用一次。
2. `archiveCombatWindow` 调用条件严格：必须 `state.time - startedAt >= COMBAT_WINDOW_MIN_RECORD_SEC`（默认 2 秒），避免一颗 stray asteroid 弹片产生"战斗复盘"。
3. 归档后 `lastArchived` 通过 `freezeCombatWindowSnapshot`（建议复用 v0.12 `deepFreezeForDiagnostics`）冻结，避免 UI 修改污染。
4. `combatWindowState.current` 不进 `__gameTest.playtest.snapshot()` 主字段（避免 PM 误把"进行中"当成"已结束"）。snapshot 中 `combat.review` 始终来自 `getCombatReview()`。
5. `state.paused === true` 时 `state.time` 不前进，窗口自然冻结；不需要特殊处理 pause。

## 6. 性能与缓存策略（D5）

### 6.1 缓存层级

| helper | 是否缓存 | 缓存 key | 失效条件 |
| --- | --- | --- | --- |
| `getThreatSummary` | 是 | `threatSummaryCache` 模块局部 | `state.time - cache.sampledAt >= COMBAT_THREAT_SAMPLE_INTERVAL` 或 `state.time` 倒退（resetRun 测试场景） |
| `getWeaponEffectiveness` | 是 | `weaponEffectivenessCache` 模块局部 | 同上 |
| `getRecentDamageSummary` | 否 | — | ring buffer reduce 成本低，且玩家修复 / 受损 / 弹道事件刷新频率高，缓存收益小、误差大 |
| `getRepairStatusSummary` | 否 | — | repair drones 数 ≤ 6、damaged cells 数 ≤ 10，reduce O(60) 单帧可承受 |
| `getCombatReview` | 是 | `combatReviewCache` 模块局部 | 同 `getThreatSummary`；归档触发时强制失效 |

**关键约束**：

1. v0.13 缓存与 v0.12 `designHealthSnapshotCache` **并行独立**，不复用任何 key；既避免 v0.13 缓存失效逻辑影响 v0.12 测试钩子，也避免 v0.12 健康摘要刷新意外清空 v0.13 缓存。
2. 缓存对象顶层 + 子层 `Object.freeze`（复用 v0.12 `deepFreezeForDiagnostics`）；PM 在 DevTools 修改返回对象不影响后续调用。
3. 测试钩子（§7）通过 `safeDeepCloneForTest` 返回深拷贝，PM 修改克隆不影响缓存。
4. `state.paused === true` 时 `state.time` 不前进，缓存自然冻结。
5. 归档 `combatWindowState` 时显式清空 `combatReviewCache`，保证下一次 `getCombatReview()` 返回新归档结果。

### 6.2 与 v0.12 的协同

- v0.12 `getStationDesignHealth` 已经在 HUD 0.12 秒周期被多次调用且缓存命中率高；v0.13 helper **不**调用 `getStationDesignHealth`，避免对 v0.12 缓存的命中模式产生意外影响。
- 若 v0.13 的 reasonKey 需要进入 `criticalIssues` 与 v0.12 共享，建议在 `computeStationDesignHealthSnapshot()` **不变**的前提下，由 `buildStatusAlerts` 调用 `getThreatSummary / getRepairStatusSummary` 后**合并** alerts 数组；不要在 `computeStationDesignHealthSnapshot` 内追加新逻辑（防止 v0.12 测试钩子 schema 漂移）。

### 6.3 单帧成本上限

- 单局最坏：`state.enemies.length ≤ 18`、`state.station.cells.size ≤ 80`、`state.repairDrones.length ≤ 6`、`state.fragments.length ≤ 10`、`combatEventBuffer.length ≤ 120`。
- 每周期成本：
  - `getThreatSummary`：O(enemies + activeWeaponCells) ≤ 18 + 12 = 30 操作（不含 inWeaponRange LOS 校验；如启用 LOS，限制为 `nearest` 一次调用 = O(cells) ≤ 80）。
  - `getWeaponEffectiveness`：O(turret+missile cells × LOS) ≤ 6 × 80 = 480 操作（最坏情况；现实场景一般 < 200）。
  - `getRecentDamageSummary`：O(combatEventBuffer × 一次过滤 + damagedCells) ≤ 120 + 10 = 130 操作。
  - `getRepairStatusSummary`：O(repairers × damagedCells + drones) ≤ 6 × 10 + 6 = 66 操作。
  - `getCombatReview`：从 `combatWindowState.lastArchived` 直接复制，O(archivedFields) ≤ 60 操作。
- 总单帧 < 800 操作，纯 JS 对象访问。HUD 0.12 秒 cadence × 5 helper 即 ~6500 操作/秒，远低于 60 FPS 单帧 16ms 预算。
- T4 节点审查要求：`__gameTest.playtest.getCombatReview()` 连续调用 1000 次耗时 < 60ms；`getThreatSummary + getWeaponEffectiveness + getRecentDamageSummary + getRepairStatusSummary` 各 1000 次累计 < 200ms。

### 6.4 LOS 复用建议

- `hasLineOfSight` 每次调用 O(cells)，是热点。建议在 `getWeaponEffectiveness` 内对每个 turret 调用 **一次**；并在 `selectTargetReadOnly` 返回 enemy 后才做 LOS 检查（无目标时跳过）。
- **不**新增"全局 LOS 缓存"，原因：
  1. LOS 受空间站角度、cell 增删、敌人位移影响，缓存失效条件复杂；
  2. 现有 turret 数量 ≤ 6，单帧 < 500 操作；
  3. 多一层缓存增加 v0.13 与 v0.7（LOS 警告环）边界耦合风险。

## 9. 严重性阈值与低噪声提示（D9）

严重性阈值是 helper 返回的 `severity` 字段判定规则。文案/阈值表（PM #4）可微调具体数字，但 **判定函数体必须由本方案 §9 约束**（避免不同 PR 各自定义阈值）。

### 9.1 阈值表

| 维度 | severity = "ok" | severity = "warning" | severity = "critical" |
| --- | --- | --- | --- |
| threat.nearest | `active === false` 或 `distance > 800` | `distance ∈ (300, 800]` 或 `kind === "asteroid"` 且 `distance > 200` | `distance ≤ 300` 或 `kind === "hostile-station"` 且 `inWeaponRange === true` 或 `kind === "guardian"` 且 `state.run.endgame === true` |
| threat.boss | 不存在 hostile-station | hostile-station 距离 > 600 或 `weaponCellAlive >= 2` | hostile-station 在玩家武器射程内 |
| weapon.summary | `weaponReady > 0 && primaryReasonKey === "ready"` | `weaponTotal > 0 && weaponReady === 0`（reloading / LOS / 缺电任一） | `weaponTotal === 0 && isEnemyPressureLikely() === true` |
| shield.coverage | 无 shield 但安全态；或 active 全部 ok | shield 存在但 `recoveringCount > 0` 或 `coverageStatus === "recovering"` | shield 存在但 `active === 0 && isEnemyPressureLikely() === true` |
| recentDamage | `recentEventCount === 0` | `recentEventCount > 0 && noCoreLowHp` | `core.hpRatio < 0.5` 或 `fragmentDetaching > 0` |
| repair.summary | `damagedCellCount === 0` 或 `activeRepairerCount > 0 && uncoveredDamagedCount === 0` | `damagedCellCount > 0 && activeRepairerCount === 0`（但仍有 repairer） | `damagedCellCount >= 3 && (repairerCount === 0 \|\| uncoveredDamagedCount >= 3)` |

### 9.2 与 v0.12 频率控制器复用

- v0.13 reasonKey 加入 `criticalIssues` 后通过 v0.12 `buildLowNoiseDesignIssueAlerts` 的 `designIssueAlertRuntime`（`Game/game.js:1330`）做 8 秒同 key 冷却。
- 若 v0.13 选择不进入 v0.12 `criticalIssues`，必须在 `buildStatusAlerts` 内为 v0.13 reasonKey 复用 `designIssueAlertRuntime.lastRaisedAt` Map（同一 reasonKey 不重复触发），实现细节由 T1/T2 落地时确认；本方案禁止新增独立冷却 Map。

### 9.3 安全态不刷警告

- `isEnemyPressureLikely() === false` 时：
  - `getThreatSummary().active === false`、`nearest === null`、`threats === []`；

### 11.2 版本级回滚

v0.13.0 仅做只读诊断 helper + UI 字符串追加 + 运行时临时统计 + 测试钩子 + 文案 + CSS，**不**改动持久化层、不改动战斗结算、不改动 RNG。回滚成本极低：

1. **代码回滚**：`git revert` v0.13.0 的相关 commits，回到 v0.12 HEAD。
2. **存档兼容性**：v0.13.0 **不**写盘新字段、不修改 meta schema，回滚后 `localStorage[SAVE_KEY]` 与 v0.12 写盘完全一致，玩家无任何感知。
3. **状态兼容性**：诊断 helper、运行时临时统计与测试钩子是纯运行时逻辑，回滚后丢失但不留残余 state；`state.run` 字段集与 v0.12 相同，刷新即等价。
4. **缓存兼容性**：`combatEventBuffer / combatWindowState / threatSummaryCache / weaponEffectivenessCache / combatReviewCache` 是模块局部变量，回滚后自然消失。
5. **PM 取证**：v0.13 期间保存的 `__gameTest.playtest.snapshot().combat` JSON 仍可作为 v0.13 期间的历史证据，但回滚后该字段不存在；测试脚本应做 optional chaining。
6. **DOM 兼容性**：回滚后 `combatReviewSummary` section 在 HTML 中不存在；v0.13 期间任何依赖该选择器的脚本需 fallback。
