## 1. 背景

### 1.1 版本目标摘要

制作人方向把 v0.5.0 定为「研发与升级」：

- 引入局内研发路线 build：玩家完成任务后获得 research，可主动消耗 research 升级 cell tier、选择 cell 改造、激活全局加成。
- 建立 `getCellStat(cell, key)` 集中读取层，把散布在 5+ 个 update 函数体内的硬编码 damage / reload / range / repairRate / produceRate 等参数收敛到查询接口；抽象前后行为完全等价。
- 扩展 `TYPES.baseStats / upgrades / modifications`，让 13 种设施具备可调、可回归、可扩展的数据来源。
- 在 cell 上新增 `tier / mod / upgradePath` 字段；旧 cell 默认 `tier = 0 / mod = null / upgradePath = null`，通过 `getCellStat` 应用倍率。
- 扩展任务注册表 `OBJECTIVE_TYPES.researchReward`，任务完成时增加 `state.resources.research`。
- 扩展 `selectedCellPanel` 与全局研发面板，提供单设施升级 / 改造和 `hullMod / weaponMod` 全局节点。
- 保留 `techLevel` 自动升级机制（v0.1.x-v0.4.x 难度曲线），玩家研发与之乘法叠加但不替代。

### 1.3 关键设计风险（高优先级）

1. **散布硬编码参数全量迁移**：turret / missile / shield / mining / processor / plasma / research / power / thruster / repair 共 10+ 个 update 函数包含硬编码 damage / reload / range / rate / conversion 值。`getCellStat` 抽象必须逐项迁移，**任何一处漏改或公式翻译错误都会导致行为回潮**（damage 差 1% 玩家可能感觉不到，但 tier 倍率失效后整个升级系统就成了"按了没反应"）。
2. **techLevel 既有 4 处散布乘法**：turret reload / thruster thrust / keyboardThrust / power available。`getCellStat` 必须把 techLevel 当作公式中的一项纳入，不能让 update 函数继续直接读 `state.station.techLevel`，否则 tier 与 techLevel 重复叠加或漏叠加。同时 `improveStationHp(1.08)` 直接乘 `cell.maxHp` 的做法是「累积乘法」，与 `getCellStat` 按公式重新计算的「声明式」方式冲突，必须在 §2.6 明确取舍。
3. **cell.tier / cell.mod / cell.upgradePath 生命周期污染**：cell 同时存在于 `state.station.cells`、`state.fragments[].cells`、`state.npcs`（NPC 内不含 cells，但 wreck fragment 内含 cells）。任何一个入口创建 cell 而漏掉默认值，都会让 `getCellStat` 在 `undefined.tier` / `undefined[mod]` 上崩。这是项目高风险点。
4. **state.station.hullMod / weaponMod 死代码激活的语义**：当前初值 `1.0`，未被任何 update 读取。v0.5.0 把它们改为「玩家研发激活后 = 1.05、未激活 = 1.0」时，必须保证「未激活默认 1.0」与现状完全等价（否则 v0.1.x-v0.4.x 行为回潮）。同时由于 `state.station.thrustMod` 已经在使用中（值固定为 1，相当于无效），不要混淆。
5. **TYPES 名称差异（engine / harvester）**：数值方案使用 "engine"（实际 = thruster）和 "harvester"（无对应）。实现按 TYPES 实际名称组织 baseStats / upgrades / modifications；harvester 升级路径暂不实现，留作开放项与数值策划复盘（详见 §11 开放项 #1）。
6. **fragment.cells 中 cell.tier 序列化与桥接回站**：玩家脱落的 fragment 桥接回站后，cell 重新进入 station.cells；其 tier / mod / upgradePath 应保留（玩家不应因脱落导致升级丢失）。wreck fragment 内的 cell 默认 `tier = 0 / mod = null / upgradePath = null`，桥接回来后 `getCellStat` 走 base 公式即可。
7. **2.5× 单 stat clamp 在哪一层应用**：必须在 `getCellStat` **内部最后一步** clamp，不能放到调用方（如 update 函数）。否则 update 函数若忘记 clamp，可能让 damage / reload 突破上限，破坏数值平衡。但 produceRate / mineRate / projectile count / mass / angularDamping 不参与 clamp（数值方案 §6.5）。
8. **research 资源既有路径不能回潮**：当前 `state.resources.research` 由 research 设施产出 + techLevel 自动消耗。v0.5.0 引入「任务奖励 + 玩家升级消耗」两个新路径，必须保证 techLevel 自动消耗 + research 设施产出公式不变。任何一处改动都要由 PM 在节点 A 回归。

## 2. 设计决定逐项（对应方向文档「关键技术疑问」8 项）

### 2.1 `getCellStat(cell, key)` 接口形式

- **候选方案**
  - A. 纯函数 `getCellStat(cell, key)`，按 cell.facility 查 TYPES。
  - B. 在 cell 对象上挂方法 `cell.getStat(key)`。
  - C. 在 TYPES 上挂方法 `TYPES.turret.getStat(cell, key)`。
- **推荐：A**。理由：
  - 纯函数最贴合 v0.3.0 / v0.4.0 既有风格（`cellWorldPosition / thrusterNozzle / nearestEnemy / dist` 都是纯函数）；项目内没有 OO 风格。
  - 不污染 cell 对象的序列化形态（cell 在 fragment 内进出，越纯越好）；如果挂方法会让 cell 不再是纯数据，序列化 / 克隆时容易丢失。
  - 调用点形如 `getCellStat(cell, "damage")`，迁移现有硬编码时只需把 `14` 改成 `getCellStat(cell, "damage")`，最小侵入。
- **签名最终形式**：
  ```js
  function getCellStat(cell, key)
  ```
  - 第三参数（如 station / globalMods）**不需要**：station 是全局唯一的 `state.station`，函数内部直接读 `state.station.hullMod / weaponMod` 即可；不引入参数让调用点更短。
  - 返回值：**number**（不返回 object / function），保持调用点写法 `damage * X` / `if (reload <= 0)` 不变。
  - 未知 facility / 未知 key：返回 `0`（而不是 `undefined` 或 `NaN`），避免数学运算崩溃；同时记一行 `console.warn` 帮助 PM 在浏览器 console 排查。
- **完整 fallback 链**（按顺序，每步覆盖前步）：
  1. `TYPES[cell.facility]?.baseStats?.[key]` → base 默认值（必须存在；缺失返回 0 + warn）。
  2. tier 倍率：根据 `cell.tier`（默认 0）+ `cell.upgradePath`（默认 null，仅核心战斗设施有路径选择）查 `TYPES[cell.facility].upgrades?.[upgradePath ?? "_default"]?.[tier]?.[key]`；返回倍率（如 1.20），与 base 相乘。如果该 upgrades path 不影响 `key`（如 turret damage 路径不影响 range），倍率默认 1.0。
  3. mod 效果：根据 `cell.mod`（默认 null）查 `TYPES[cell.facility].modifications?.[cell.mod]?.[key]`；返回倍率或加值，与上一步结果相乘。
  4. techLevel 因子（仅对相关 stat 启用）：见 §2.6。
  5. globalMod 因子：`maxHp` 受 `state.station.hullMod`（已激活时 1.05，否则 1.0）；`damage` 仅 turret / missile 受 `state.station.weaponMod`（已激活时 1.05，否则 1.0）。
  6. clamp：`Math.min(result, 2.5 × base)`，**仅对 §2.6 列出的 stat 类型生效**；produceRate / mineRate / projectile count / mass / angularDamping 不 clamp。
- **性能约束**：
  - getCellStat 调用频率 ≤ 60 Hz × 战斗 cell 数（典型 ~10-30）× 每帧 1-3 个 stat 查询 = 1800-5400 次 / 秒。
  - 函数内禁止：创建对象 / 数组 / 字符串拼接 / 函数引用；只允许 dict 查表 + 数值运算 + Math.min。
  - 不引入缓存（cache 后变成「tier / mod / techLevel 变化时清缓存」的状态机问题，反而增加 bug 面）。
- **现有 update 函数迁移规则**（T1 范围）：
  - **只改读取层**：把 update 函数体内的硬编码值（如 `14`、`0.60`、`450`）改成 `getCellStat(cell, "damage")` 等；**不改其他逻辑**（不改循环结构、不改判定顺序、不改副作用）。
  - **techLevel 公式不再散布**：例如 turret 原来的 `cell.reload = Math.max(0.28, 0.60 - state.station.techLevel * 0.03);` 改为 `cell.reload = getCellStat(cell, "reload");`，由 `getCellStat` 内部应用 techLevel 公式。
  - **state.meta.weapons / state.meta.mining 等局外加成**：保留在 update 函数外侧（不进 getCellStat），原因是这些是「局外天赋」语义层，与「局内研发」正交；详见 §2.6 公式表。
- **风险**：function 调用开销极小但 ≠ 0；如果实测发现热路径（如 `updateMiningAndResearch` 每帧遍历所有 cell × 多次 getCellStat）造成可观察的 ms 级回归，可在 update 函数顶部把 `getCellStat(cell, "mineRate")` 提取到局部变量后复用，函数本身不变。
- **回滚**：若 `getCellStat` 在节点 A 暴露多处行为差异且修复成本高，可临时把 cell 加 `cell._cachedStats = computeOnce(cell);`，在升级 / 改造 / techLevel 变化时手动 invalidate；但 v0.5.0 默认不开。

### 2.2 `cell.tier` / `cell.mod` / `cell.upgradePath` 序列化

- **跨关 nextLevel 时是否重置**：**重置为 0 / null / null**。理由：
  - 与方向文档 §"关键技术疑问 #2"建议一致（与 fragments / npcs 清空一致，让本局研发是临时收益）。
  - 与数值方案 §2.5 "路径锁定本局有效，跨关 nextLevel 时清空"一致。
  - rogue-like 风格：每关都有 build 决策动机；如果跨关保留 tier，玩家会觉得"研发不重要，反正下一关还能用"，丧失节奏。
- **resetRun 时**：**同步重置**（与 fragments / npcs / enemies 一起在 `resetRun()` 内清理；station.cells 整体重建时 tier / mod / upgradePath 自然为初值）。
- **旧存档（无 cell.tier 字段）兼容**：
  - cell 上不应序列化进 SAVE_KEY（cell 本身就不存盘，meta 只存 `points / mining / hull / weapons / unlocks`）。
  - 但 cell 的 in-memory 创建路径必须保证 tier / mod / upgradePath 默认值；createCell（玩家建造）/ buildFragmentFromCells（玩家脱落）/ makeWreckFragment（拾荒预生成）/ loadMeta 后重建 station 等所有路径都要写默认值。
  - `getCellStat` 内部读 `cell.tier ?? 0` / `cell.mod ?? null` / `cell.upgradePath ?? null`，即便 cell 来自旧版没有这些字段也不崩。
- **state.meta 是否累积总 research 点（局外持久）**：**不累积**。理由：
  - 与方向文档 §"关键技术疑问 #2"建议一致（v0.5.0 研发点是局内即时使用，不带入下一局）。
  - 与数值方案 §0.2 "本数值方案所有新增字段写入 cell.tier / cell.mod / state.station.hullMod / state.station.weaponMod，不写入 state.meta"一致。
  - 保护 rogue-like 节奏：避免玩家把 research 攒到下一局变成"开局就强"。
- **`state.station.hullMod / weaponMod` 跨关与跨局**：
  - **跨关 nextLevel：保留**（与数值方案 §4.4 一致；本 run 永久）。
  - **跨局 resetRun：重置为 1.0**（与数值方案 §4.4 一致；不写入 meta）。
- **wreck fragment / player fragment 的 cell 默认值**：
  - `makeWreckFragment` 创建 cell 时必须显式写 `tier: 0, mod: null, upgradePath: null`。
  - `buildFragmentFromCells`（玩家脱落）由于 cell 来自 `state.station.cells.values()`，已经有 tier 字段，自动带入 fragment；桥接回站时 cell 进入 `state.station.cells`，tier 也自动带回（这是玩家"研发后脱落再桥接回来不丢"的关键路径）。
  - `loadMeta()` 不涉及 cell（meta 只存局外字段）；新 run 启动时 cell 由 `resetRun` 重新创建，默认 tier=0。
- **风险**：如果哪一个 cell 创建入口漏写默认值，`getCellStat(cell, "damage")` 在 `cell.upgrades[undefined][undefined].damage` 上会崩。**必须**在 `createCell` 函数（T1 范围）统一写默认值，所有调用方走这一个入口。
- **回滚**：若节点 A 发现某入口漏改且不易快速修复，可在 `getCellStat` 内部第一行 `cell.tier ??= 0; cell.mod ??= null; cell.upgradePath ??= null;` 兜底；但这是兜底不是设计意图，建议节点 A 修复入口。

### 2.3 research 成本曲线

- **完全采用数值方案 §4.1**：tier 0→1 = 5、tier 1→2 = 10、tier 2→3 = 20（累积 35 升满）、改造一次性 30、全局加成 hullMod / weaponMod 各 50。
- **公式实现选择**：**查表**而非 `Math.pow(2, tier) * 5`。理由：
  - 数值方案 §9.1 允许 PM 在 ±20% 区间内微调（tier 0→1 可调到 4-6、tier 1→2 调到 8-12、tier 2→3 调到 16-24），公式形式 `5 * 2^tier` 不允许这种独立微调。
  - 查表写起来直观，PM 微调时只改一个数字。
- **实现形式**：
  ```js
  const TIER_UPGRADE_COSTS = [5, 10, 20];           // tier 0→1, 1→2, 2→3
  const MODIFICATION_COST = 30;
  const HULL_MOD_COST = 50;
  const WEAPON_MOD_COST = 50;

  function getTierUpgradeCost(currentTier) {
    if (currentTier >= 3) return Infinity;
    return TIER_UPGRADE_COSTS[currentTier];
  }
  ```
- **tier 上限**：**3**（与数值方案 §0.2 一致，v0.5.0 范围内）。tier = 3 时 UI 按钮置灰 + hover "已达最高 tier"。
- **不可升级设施**：core / processor / plasma 由数值方案不在 §2.2 / §2.3 / §2.4 内（如 core 明确不可升级；processor / plasma 没有 §2.2 表项，应理解为单路径升级——见开放项 #2）。如果 PM 与数值策划在节点 A 复盘后认为 processor / plasma 应可升级，本表可补行；否则按当前数值方案视为不可升级。
- **风险**：常量散布在文件多处易遗忘；建议集中在 `Game/game.js` 顶部声明一段 `// === v0.5.0 研发与升级常量 ===` 区块，与 OBJECTIVE_LEVEL_WEIGHTS / WRECK_FRAGMENT_MAX_COUNT 同样位置。

### 2.4 改造（modification）唯一性

- **是否不可逆**：**不可逆**（v0.5.0 范围）。理由：
  - 与方向文档 §"关键技术疑问 #4"建议一致。
  - 与数值方案 §3.6 "改造独立于 tier，但选 A 或 B 后 cell.mod 锁定"一致。
  - 防止玩家"反复横跳"试错（A → B → A → B），破坏 build 决策的"承诺感"。
  - 销毁 cell 重建可作为"曲线救国"（销毁后重建 cell.mod = null，但 cell.tier 也清零，等于重新研发，与"科研重置点"语义接近，不引入新机制）。
- **改造与升级（tier）的关系**：**正交**（与数值方案 §3.6 一致）。
  - tier 改 stat 主属性强度（如 turret damage tier 3 → ×1.60）。
  - mod 改风格（如 turret_heavy → reload ×1.43 + damage ×1.60）。
  - 两者乘法叠加（详见 §2.6 公式）。
  - 玩家可以"先选改造再升级"或"先升级再选改造"，没有顺序约束。
- **改造效果在 getCellStat 中的应用方式**：
  - `TYPES[cell.facility].modifications[cell.mod]` 返回一个对象，包含该 mod 对所有 stat 的倍率（如 `{ damage: 1.60, reload: 1.43 }`）。
  - 未列出的 stat 默认倍率 1.0（即不影响）。
  - 在 fallback 链 step 3 应用：`result = result * (TYPES[cell.facility].modifications?.[cell.mod]?.[key] ?? 1.0)`。
- **v0.5.0 范围内的改造覆盖**：4 个核心设施 = turret / missile / shield / engine（= TYPES.thruster）。其他设施在数值方案 §3.5 明确"不引入"，留 v0.5.1+。
- **改造按钮 UI 互斥**：
  - 未改造时：显示 2 个改造按钮（A / B），都可点击（research 足够时）。
  - 已改造 A 时：显示「已改造：A」 + 两个按钮均置灰；hover 显示"已改造，无法再次改造（销毁后重建可重置）"。
  - 已改造 B 时：同上。
- **风险**：如果 modifications 表中某 mod 误写 `damage: 0`（应写 0.75），cell damage 直接归零，玩家可能困惑。建议在 TYPES.modifications 表上加注释明确"所有 stat 默认倍率 1.0，仅列出 ≠ 1.0 的"。
- **回滚**：如果节点 C 体验测试发现"不可逆太严苛"，可在 v0.5.1 引入"销毁 mod 按钮"（消耗 20 research 取消 mod，不影响 tier）；v0.5.0 不开。

### 2.5 全局加成 `hullMod / weaponMod`

- **是否 binary**：**binary**（与方向 §"关键技术疑问 #5"与数值方案 §4-§5 一致）。理由：
  - 简化 v0.5.0 范围；stackable 需要存储「已激活几次」+ UI 显示「再激活 1 次」+ 倍率累计公式，复杂度上升。
  - v0.5.0 仅 hullMod / weaponMod 2 个全局加成，binary 已能体现"build 决策"价值。
- **存储形式**：
  - `state.station.hullMod` / `state.station.weaponMod` **已经是 number（初值 1.0）**，不是 boolean。
  - **保留 number 类型**，激活后赋值为 `1.05`，未激活保持 `1.0`。
  - 不改为 boolean 的原因：未激活时 `1.0` 与现状完全等价，且可以让 `getCellStat` 直接 `result *= state.station.hullMod`（无需 `if (state.station.hullMod) result *= 1.05`），减少分支。
- **激活路径**：
  - 玩家点击全局研发面板上的 "激活 hullMod / weaponMod" 按钮。
  - 检查 `state.resources.research >= 50`，扣除后 `state.station.hullMod = 1.05`（hullMod 路径）或 `state.station.weaponMod = 1.05`（weaponMod 路径）。
  - showToast("船体强化已激活，全 station HP +5%。") 或 "武器强化已激活，turret / missile damage +5%。"。
- **应用入口**：
  - `getCellStat(cell, "maxHp")` 内部 `result *= state.station.hullMod;`（所有设施都受 hullMod，包括 armor / frame / 战斗设施 / 资源设施，与数值方案 §5.1 一致）。
  - `getCellStat(cell, "damage")` 内部，仅 turret / missile 设施 `result *= state.station.weaponMod;`（用 `WEAPON_MOD_FACILITIES = new Set(["turret", "missile"])` 判断）。
  - 其他 stat（如 range / reload / produceRate）不应用 globalMod。
- **跨关 / 跨局**：
  - nextLevel **保留**（数值方案 §4.4：本 run 永久）。
  - resetRun **重置为 1.0**（数值方案 §4.4：跨 run 重置）。
- **风险**：未激活时 `state.station.hullMod = 1.0`，乘法不变；如果有人误把它初值改成 0，所有 maxHp 归零。`resetRun` 必须显式写 `1.0`（不依赖默认）。
- **回滚**：若数值方案后续把全局加成调整为 stackable，可改为 `state.station.hullModLevel = 0..N`，每级 +0.05；接口不变，仅 getCellStat 内部公式调整为 `1 + 0.05 * level`。

### 2.6 `techLevel` 与玩家研发的叠加方式

- **乘法叠加**（与方向 §"关键技术疑问 #6"与数值方案 §6.1 一致）。
- **完整公式**（写入 `getCellStat`）：
  ```
  final_stat = base
             × tier_multiplier(cell, key)
             × mod_multiplier(cell, key)
             × techLevel_factor(facility, key)
             × globalMod_factor(facility, key)
  if (clampable(key)) final_stat = Math.min(final_stat, 2.5 × base)
  ```
  其中各因子：

  | 因子 | 取值 | 适用范围 |
  | --- | --- | --- |
  | `base` | TYPES[type].baseStats[key]，必须存在 | 所有 facility 所有 key |
  | `tier_multiplier` | TYPES[type].upgrades[upgradePath ?? "_default"]?.[tier]?.[key] ?? 1.0 | cell.tier ≥ 1 时启用，沿 cell.upgradePath 累加（数值方案 §2 表）|
  | `mod_multiplier` | TYPES[type].modifications?.[cell.mod]?.[key] ?? 1.0 | cell.mod 非空时启用 |
  | `techLevel_factor` | 见 techLevel 公式表（保留 v0.1.x-v0.4.x 散布公式）| 仅特定 (facility, key) 对启用 |
  | `globalMod_factor` | maxHp → state.station.hullMod；damage（turret/missile）→ state.station.weaponMod；其他 1.0 | 全局加成激活时生效 |

- **techLevel 公式表**（基于现有散布公式整理，T1 必须保持等价）：

  | facility | key | techLevel 公式 | 现状位置 |
  | --- | --- | --- | --- |
  | turret | reload | `clamp(base × (0.60 - techLevel × 0.03) / 0.60, base × 0.28 / 0.60, base)`（即原 `Math.max(0.28, 0.60 - techLevel * 0.03)` 等价转换为乘法） | `game.js:3559` |
  | thruster | thrust | `1 + techLevel × 0.08` | `game.js:3115-3118` |
  | （keyboardThrust）| — | `1 + techLevel × 0.06`（与 thrustMod 相乘，**不走 getCellStat**，详见下方说明）| `game.js:3134-3136` |
  | power | powerOut | `1 + techLevel × 0.15`（原 `available = 6 + techLevel * 1.5`，单 power cell 增量约 1.5，对应 ×1.15）| `game.js:3052` |
  | (improveStationHp) | maxHp | `1.08^techLevel`（每次自动升级触发 ×1.08；累积乘法）| `game.js:3216-3226` |

- **`improveStationHp` 的处理**：
  - 现状是「每次 techLevel++ 时把所有 cell.maxHp ×1.08，累积乘法持续保留」。这是「就地修改 cell.maxHp」的副作用，与 `getCellStat` 声明式公式冲突。
  - **决定：T1 不动 improveStationHp**。原因：
    - improveStationHp 已经把 maxHp 写入 cell.maxHp（持久值）；`getCellStat(cell, "maxHp")` 应返回 `cell.maxHp` 当前值（即已被 improveStationHp 修改后的值），再叠 tier / mod / hullMod。
    - cell.maxHp 在 `getCellStat` 中作为 base 候选优先：`base = cell.maxHp ?? TYPES[cell.facility].baseStats.maxHp`（即"cell 上覆盖优先"）。
    - 这样 improveStationHp 的累积乘法效果保留（v0.1.x 行为不回潮），同时 `getCellStat` 可叠 tier × hullMod。
    - clamp `2.5 × base` 的 base 是 `TYPES[cell.facility].baseStats.maxHp`（原始 base，**不是** cell.maxHp 当前值）；这样玩家 tier 3 + hullMod + improveStationHp 多次叠加最终值会 clamp 在 `2.5 × 原始 base`，与数值方案 §6 一致。
- **keyboardThrust 的特殊处理**：
  - keyboardThrust（WASD 直推）不绑定到 cell，是 `PLAYER_PHYSICS.keyboardThrust × (1 + techLevel × 0.06) × thrustMod` 的全局倍率。
  - **不走 getCellStat**（因为没有 cell）；保留 `state.station.thrustMod` 的现有用法（值固定 1，相当于无效）。
  - 如果 v0.5.x 后续把 thrustMod 也激活成研发节点，可再走 getCellStat 抽象；v0.5.0 不动。
- **state.meta.weapons / state.meta.mining 局外加成的处理**：
  - 这些是「局外天赋」语义，与「局内研发」正交（meta.weapons 加 turret damage 10% / mining cell rate 12% / thruster thrust 3% 等）。
  - **保留在 update 函数外侧**：getCellStat 不读 state.meta；调用方继续 `damage = getCellStat(cell, "damage") * (1 + state.meta.weapons * 0.1)`。
  - 原因：getCellStat 是"cell 自身能力"查询，meta.weapons 是"玩家整体加成"，两者语义不同。如果混进 getCellStat，clamp 公式会被 meta 加成抬高 2.5×，破坏局内研发上限设计。
- **clamp 适用 stat**（数值方案 §6.5）：
  - **clamp（最大 2.5 × base）**：`damage / reload / range / maxShield / maxHp / regen / thrust`。
  - **不 clamp**：`produceRate / mineRate / repairRate / missile speed / projectile count / mass / angularDamping`。
  - 实现：在 `getCellStat` 内部 `const CLAMPED_STATS = new Set(["damage", "reload", "range", "maxShield", "maxHp", "regen", "thrust"]);` 判断。
- **reload 的 clamp 含义**：reload 是"装填时间"（越小越快），clamp `2.5 × base` 意味着"reload 不能超过 base 的 2.5 倍（即不能慢到 2.5×）"，但 tier 路径是缩短 reload（×0.85 / ×0.70 / ×0.55），与 clamp 方向相反，所以 clamp 对 reload 升级路径无影响；对 reload **改造**（如 turret_heavy reload ×1.43，慢化）会触发 clamp 当且仅当 1.43 × techLevel_reload_factor > 2.5（不可能发生，最大约 1.43 × 1.0 = 1.43）。clamp 仍保留以防未来误配。
- **最坏情况采样**（数值方案 §6.3）：
  - turret damage tier 3（damage 路径）+ weaponMod + techLevel 6：`base × 1.60 × 1.587 × 1.05 = 2.666 × base` → clamp 到 `2.5 × base`。
  - armor maxHp tier 3 + hullMod + techLevel 6：`base × 1.90 × 1.587 × 1.05 = 3.166 × base` → clamp 到 `2.5 × base`。
- **风险**：techLevel_factor 计算分散，T1 必须把所有 techLevel 公式提取到一个集中的 `getTechLevelFactor(facility, key, techLevel)` helper，由 getCellStat 调用；否则 PM 在节点 A 验证 13 种设施时无法快速核对。
- **回滚**：若 techLevel_factor 公式翻译错误导致 v0.1.x-v0.4.x 行为回潮，可在 T1 第一版先让 `getCellStat` 不应用 techLevel_factor（保留 update 函数体内的原 techLevel 散布公式），等 T1 末确认 base 行为完全等价后再做"techLevel 内聚"小重构。

## 4. 关键算法伪代码

### 4.1 getCellStat 完整 fallback 链

```js
const CLAMPED_STATS = new Set(["damage", "reload", "range", "maxShield", "maxHp", "regen", "thrust"]);
const WEAPON_MOD_FACILITIES = new Set(["turret", "missile"]);

function getCellStat(cell, key) {
  if (!cell || !cell.facility) return 0;
  const type = TYPES[cell.facility];
  if (!type) return 0;

  // step 1: base
  let base;
  if (key === "maxHp" && typeof cell.maxHp === "number") {
    base = cell.maxHp;               // cell 上覆盖优先（保留 improveStationHp 累积效果）
  } else {
    base = type.baseStats?.[key];
    if (typeof base !== "number") {
      if (window.DEBUG_GETSTAT) console.warn(`[getCellStat] missing base: ${cell.facility}.${key}`);
      return 0;
    }
  }
  let result = base;

  // step 2: tier multiplier
  const tier = cell.tier ?? 0;
  if (tier > 0 && type.upgrades) {
    const pathKey = cell.upgradePath ?? "_default";
    const tierEntries = type.upgrades[pathKey];
    if (tierEntries && tierEntries[tier]) {
      const mult = tierEntries[tier][key];
      if (typeof mult === "number") result *= mult;
    }
  }

  // step 3: mod multiplier
  if (cell.mod && type.modifications?.[cell.mod]?.effects?.[key]) {
    result *= type.modifications[cell.mod].effects[key];
  }

  // step 4: techLevel factor
  const techMul = getTechLevelFactor(cell.facility, key, state.station.techLevel);
  if (techMul !== 1.0) result *= techMul;

  // step 5: globalMod factor
  if (key === "maxHp") {
    result *= state.station.hullMod;       // 1.0 或 1.05
  } else if (key === "damage" && WEAPON_MOD_FACILITIES.has(cell.facility)) {
    result *= state.station.weaponMod;     // 1.0 或 1.05
  }

  // step 6: clamp
  if (CLAMPED_STATS.has(key)) {
    const baseForClamp = type.baseStats?.[key] ?? base;     // clamp 用原始 base，不用 cell.maxHp 当前值
    result = Math.min(result, 2.5 * baseForClamp);
  }
  return result;
}

function getTechLevelFactor(facility, key, techLevel) {
  if (techLevel <= 0) return 1.0;
  // 由集中表查询，避免散布到 update 函数
  if (facility === "turret" && key === "reload") {
    // 现有公式：Math.max(0.28, 0.60 - techLevel * 0.03)；等价转换为乘法
    const adjusted = Math.max(0.28, 0.60 - techLevel * 0.03);
    return adjusted / 0.60;
  }
  if (facility === "thruster" && key === "thrust") {
    return 1 + techLevel * 0.08;
  }
  if (facility === "power" && key === "powerOut") {
    return 1 + techLevel * 0.15;
  }
  // 其他 (facility, key) 对：techLevel 不直接影响（improveStationHp 已直接修改 cell.maxHp）
  return 1.0;
}
```

### 4.2 upgradeSelected(cell, path) action

```js
function upgradeSelectedCell(cell, path) {
  if (!cell || !cell.facility) return;
  const type = TYPES[cell.facility];
  if (!type || !type.upgrades) {
    showToast("此设施不可升级。");
    return;
  }
  const currentTier = cell.tier ?? 0;
  if (currentTier >= 3) {
    showToast("已达最高 tier。");
    return;
  }
  const cost = getTierUpgradeCost(currentTier);
  if (state.resources.research < cost) {
    showToast(`research 不足，需要 ${cost}，当前 ${Math.floor(state.resources.research)}。`);
    return;
  }
  // 路径锁定
  if (cell.upgradePath && cell.upgradePath !== path && type.upgrades[path]) {
    showToast(`已锁定 ${cell.upgradePath} 路径，无法切换。`);
    return;
  }
  // 单路径设施：path 强制为 "_default"
  const finalPath = type.upgrades["_default"] ? "_default" : path;
  if (!type.upgrades[finalPath]) {
    showToast("无效的升级路径。");
    return;
  }
  // 扣资源 + 升级
  state.resources.research -= cost;
  cell.tier = currentTier + 1;
  if (!cell.upgradePath) cell.upgradePath = finalPath;
  // 如果升级影响 maxHp（如 armor / frame），同步更新 cell.maxHp 以保留 improveStationHp 路径
  const tierEntry = type.upgrades[finalPath][cell.tier];
  if (tierEntry?.maxHp) {
    const newMaxHp = getCellStat(cell, "maxHp");
    const hpRatio = cell.hp / cell.maxHp;
    cell.maxHp = newMaxHp;
    cell.hp = Math.min(newMaxHp, newMaxHp * hpRatio + 8);     // 与 improveStationHp 同款 +8 修复
  }
  showToast(`${type.name} 升级到 tier ${cell.tier}（${finalPath} 路径）。`);
  rerenderSelectedCellPanel();
}

function applyCellModification(cell, modId) {
  if (!cell || !cell.facility) return;
  const type = TYPES[cell.facility];
  if (!type?.modifications?.[modId]) {
    showToast("此设施不可改造。");
    return;
  }
  if (cell.mod) {
    showToast(`已改造：${type.modifications[cell.mod].name}（不可逆）。`);
    return;
  }
  const cost = type.modifications[modId].cost;
  if (state.resources.research < cost) {
    showToast(`research 不足，需要 ${cost}。`);
    return;
  }
  state.resources.research -= cost;
  cell.mod = modId;
  // 改造也可能影响 maxHp，同步 cell.maxHp（虽然 v0.5.0 改造表中无 maxHp 项）
  showToast(`${type.name} 已改造：${type.modifications[modId].name}。`);
  rerenderSelectedCellPanel();
}
```

## 6. 性能预算

| 项 | 估算 | 备注 |
| --- | --- | --- |
| getCellStat 单次 | O(1)：~5 dict 查找 + ~5 乘法 + 1 Math.min | < 0.001 ms |

## 11. 待制作人 / PM 决策的开放项

1. **数值方案的 harvester 在 TYPES 中无对应**：数值方案 §2.2 列出 harvester（产 fuel/gas）的升级，但我们 TYPES 中没有 harvester 设施。建议：
   - 选项 A：跳过 harvester，T3 数据填充时不写 harvester 升级；PM 与数值策划在节点 A 复盘是否补充。
   - 选项 B：把 harvester 映射到 plasma（语义：气体加工 = 采气精炼）；但 plasma 是 gas → plasma **消耗** gas 而非产 gas，语义不准。
   - 选项 C：把 harvester 映射到 mining（语义：采集设施统一）；但 mining 已有自己的 mineRate 升级，会产生路径冲突。
   - **本 ADR 推荐：选项 A**，待 PM 与数值策划在节点 A 复盘。
2. **processor / plasma 是否可升级**：数值方案 §2.2 列出 power / mining / harvester / repair / research，未提及 processor（金属加工）/ plasma（等离子加工）。建议：
   - 选项 A：v0.5.0 范围内 processor / plasma 不可升级（与 core 一致）。
   - 选项 B：补 processor produceRate / plasma produceRate 升级路径（与 power 同款 ×1.25 / 1.50 / 1.75）。
   - **本 ADR 推荐：选项 A**，由 PM 在节点 A 与数值策划确认。
3. **research 设施升级影响 techLevel 自动消耗速度**：当前 techLevel 自动升级条件 `research >= 35 + techLevel * 22`。如果玩家研发了 research 设施 tier 3（产出 +75%），可能让 techLevel 自动升级过快。建议：
   - 选项 A：不动 techLevel 公式；让"玩家研发 research 设施"间接加速 techLevel（设计意图：玩家投资 research 设施获得双重收益）。
   - 选项 B：把 techLevel 升级条件从 `35 + techLevel * 22` 改为 `40 + techLevel * 30`（缓和增速）。
   - **本 ADR 推荐：选项 A**（与方向"不修改 techLevel 自动机制"硬约束一致）；如 PM 节点 C 发现 techLevel 飙升过快，再由制作人决定是否调整。
4. **selectedCellPanel 升级按钮是否需要 hover 显示"升级后值"**：T3 范围是否在升级按钮上显示"damage 14 → 16.8 (+20%)"。建议：
   - 选项 A：仅显示"升级 damage 路径 (5 research)"，玩家点击后自然看到效果。
   - 选项 B：tooltip 显示具体数值预览（"damage 14 → 16.8"）。
   - **本 ADR 推荐：选项 B**（玩家感知更明确，T3 实现时由 gameplay engineer 决定 hover 内容）。
5. **研发树面板"研发树"按钮位置**：DOM 中放在 HUD 哪一处。建议：
   - 选项 A：与现有"导弹齐射"按钮同样位置（HUD 底部）。
   - 选项 B：HUD 顶部 research 数字右侧（更直接关联）。
   - **本 ADR 推荐：选项 B**（语义关联最强；如 T4 实施时发现 DOM 布局冲突，可降级为 A）。
6. **改造按钮 hover 是否显示 DPS 对比**：T3 / T4 范围是否在改造按钮 tooltip 显示"rapid DPS 11.24 / heavy DPS 11.19"（数值方案 §3.1）。建议：
   - 选项 A：仅显示效果说明（"reload +50% / damage -25%"）。
   - 选项 B：显示效果 + DPS 对比。
   - **本 ADR 推荐：选项 A**（DPS 是策划数值，不必直接暴露给玩家；玩家通过实际游玩感知）。
7. **cell.tier 重置范围**：nextLevel 时是否要把 cell.upgradePath 也重置。建议：
   - 选项 A：tier / mod / upgradePath 全重置（与"本局有效"一致）。
   - 选项 B：保留 upgradePath，只重置 tier；玩家下关延续路径偏好。
   - **本 ADR 推荐：选项 A**（与数值方案 §2.5 一致）；如 PM 节点 B 发现"重置路径让玩家挫败"，再由制作人调整。
8. **升级 / 改造日志是否进入 v0.5.0 settlement 面板**：结算时是否展示"本局升级 X 个 cell / 改造 Y 个 cell / 激活 Z 个全局加成"。建议：
   - 选项 A：v0.5.0 不引入（保持结算面板现有内容）。
   - 选项 B：T5 范围内追加 build 路线统计。
   - **本 ADR 推荐：选项 A**（控制 v0.5.0 范围；如 PM 节点 C 体验测试发现"玩家想知道本局做了什么"，再由 v0.5.1 引入）。
