## 2. v0.17 取证与必要修复边界

默认原则：

- 先补证，后判断；没有证据时不能用规则修改绕过。
- 自动 JSON、hook 状态、CDP 或 DevTools 辅助只证明“状态可读/环境可运行”，不证明玩家看懂、点到、走通。
- v0.17 不新增玩法、不调平衡、不重做系统、不改核心规则。
- 必要修复必须来自 T1-T4 证据路径中的明确现象，并由 PM 记录复现、严重度、允许范围和原路径复测。

默认允许的低风险范围：

- UI 可读性 polish：小屏叠层、按钮 hit 区域、滚动、关闭/返回/继续入口、复盘入口、Meta/结算入口。
- 取证可读性 polish：复制摘要、证据文案、非侵入式面板显示、记录模板补充。
- 低风险交互 polish：已存在按钮或面板在真实路径中不可点、不可关闭、被遮挡时的最小修复。

必须升级的范围：

- 战斗：武器开火条件、射程、伤害、命中、弹体、护盾拦截、维修判定、敌人 AI、敌人生成、战斗结算、复盘生成规则。
- Meta：`SAVE_KEY`、`META_SCHEMA_VERSION`、talents schema、points 写入、奖励计算、协议解锁/选择、迁移、局外成长数据结构。
- 星系：星系候选 RNG、星系乘子、跳转判定、节点状态、跨局路径、目标/遭遇权重、星系图状态机。
- 存档：localStorage/sessionStorage key、迁移、持久化结构、清档或伪造进度。
- 环境/发布：依赖安装、系统设置、浏览器全局配置、tag、`Releases/`、构建产物。

升级后处理：

1. 停止把该路径标记为通过。
2. PM 记录缺陷与触发升级的字段或人工现象。
3. 主程给出风险说明，明确是否触及核心规则/存档/数值/发布。
4. 制作人决定是否纳入高风险流程、后续版本，或仅继续补证。
5. 未完成升级审核前，不允许以普通 polish 名义落地修复。

## 3. 自然交火证据读取口径

自然交火通过标准必须来自真实浏览器中的玩家路径或人工观察，hook 只能辅助确认状态。最低证据链应同时覆盖：

- 武器：玩家能看到或理解武器开火、等待目标、断电、射线受限、导弹装填/资源不足等状态。
- 护盾/受击：玩家能看到护盾拦截、受击、防御缺口或防御失效。
- 维修：玩家能看到维修启动、维修缺资源/缺电、无可修目标或仍有损伤。
- 复盘：交火后出现 `combatReview`、复盘入口、复盘内容或等价总结。

只读 hook 的使用方式：

- `getWeaponEffectiveness()`：读取武器可用性、不可开火原因、目标/断电/遮挡/资源等状态，用于佐证“状态确实发生过”。
- `getRecentDamageSummary()`：读取近期受击、来源、区域、严重度，用于佐证护盾/受击或防御缺口。
- `getRepairStatusSummary()`：读取维修能力、派工、缺口、断电或无目标状态，用于佐证维修路径。
- `getCombatReview()`：读取战后复盘摘要，确认是否有 `hasReview`、`shortItems`、持续时间、短板与建议。
- `runReleaseCandidateChecks({ sampleFrames: 30 })`：读取 `snapshotAfter.base.runtimeErrors`、`snapshotAfter.combat.*`、`hookStatus.combat`、`runtimeStateSignature`、`storageSignature`，用于检查取证期间是否出现 runtime error 或异常状态漂移。

不得做的事：

- 不得调用会预置敌人、强制跳关、改存档、改资源或改局外状态的 hook 来宣称自然交火成立。
- 不得把 `hookStatus.combat.available === true`、`combat.review.hasReview === true` 或自动检查 `ok === true` 直接写成“玩家看懂战斗反馈”。
- 不得为了触发交火修改敌人生成、等待时间、武器数值、伤害、护盾、维修或 RNG。
- 不得手写 JSON、手改 storage 或改事件缓存来伪造交火结论。

缺证口径：

- 有 hook 状态、无真实视觉/操作记录：标记为“hook 辅助覆盖”，不能通过自然交火闸门。
- 有敌人/战斗等待问题但未交火：标记为“证据路径问题”，不能改核心战斗节奏绕过。
- 有交火但缺武器、护盾/受击、维修或复盘任一关键反馈：标记为阻塞、非阻塞 polish 或继续补证，由 PM 和制作人决定。

## 4. Meta 点击回流证据边界

Meta 回流必须优先来自真实点击链路：

1. 玩家通过局内结算、Meta 入口、局外入口或等价 UI 进入 Meta。
2. 玩家能看懂点击后的文案、按钮、点数、协议或下一步入口。
3. 玩家真实点击开始新局、返回局内或进入可继续状态。
4. PM 记录点击前后 `storageSignature`、关键 storage 摘要或等价说明。

允许作为辅助的只读证据：

- `snapshotMetaPanelState()`：确认面板 tab、按钮、可购买状态和文案结构。
- `snapshotMeta()`：确认运行态 meta 字段与 storage 摘要一致。
- `previewSettlementHints()`：确认结算提示文案与点数建议。
- `runReleaseCandidateChecks()`：读取 `storage.schemaStable`、`storage.unchanged`、`base.metaStorage.saveKey`、`base.metaStorage.summary.schemaVersion`。

辅助 reset / 注入 / 购买的边界：

- `window.__gameTest.meta.resetMeta()`、`injectOldMeta()`、`grantPoints()`、`purchaseTalent()`、`purchaseViaUi()`、`selectProtocol()` 只能用于隔离环境、回归复现或专项自检。
- 使用这类辅助后，Meta 回流结论必须标为“未覆盖”或“等价辅助”，不能算作真实 UI 回流通过。
- 若为了复现而清理/隔离存档，PM 必须记录起始条件、执行原因、storage 前后差异和不能证明的体验部分。

不得做的事：

- 不得通过控制台购买天赋、赠送点数、选择协议或 reset 后宣称玩家真实回流成功。
- 不得手改 `localStorage`、`SAVE_KEY`、schema、points 或 talents 来制造 Meta 成果。
- 不得把 storage 写入变化本身等同于玩家理解 Meta 或能开始新局。

通过口径：

- 至少一次真实浏览器真实点击链路成立，且 storage 影响被解释为未写、按预期写入、无法判断或异常写入。
- 若只有 hook 或控制台模拟回流，PM #9 不能建议进入发布准备交接。

## 5. `SAVE_KEY` / meta schema / RNG / 战斗判定不变审核

审核目标是证明“v0.17 补证或 polish 没有越界改核心”。主程审核按提交差异、运行态字段和证据包三层执行。

提交差异检查：

- `git diff -- Game/game.js` 应为空；若本轮后续允许 UI polish，必须逐项确认只触及 UI/CSS/低风险交互，且未改以下关键字所在语义：`SAVE_KEY`、`META_SCHEMA_VERSION`、`GALAXY_CHOICE_RNG_XOR`、`GALAXY_CHOICE_RNG_MUL`、`rng`、`damage`、`shield`、`repair`、`reload`、`enemySpawnMod`、`confirmGalaxyJump`、`purchaseMetaTalent`、`saveMeta`。
- `git diff -- Game/` 若出现战斗、Meta、星系、存档、RNG、数值或规则文件变化，直接升级，不按普通 polish 审核。
- 不允许仅凭“改动很小”放行核心字段变化。

运行态字段检查：

- `runReleaseCandidateChecks({ sampleFrames: 30 })` 中 `storage.schemaStable` 必须为 pass，`expectedSaveKey` 与实际 `saveKey` 必须一致，`schemaVersion` 必须等于 `META_SCHEMA_VERSION`。
- `storage.unchanged` 必须按取证目的解释：只读取证期间应稳定；真实 Meta 点击前后允许自然写入，但必须在 PM 记录中说明差异来源。
- `runtimeState.signatureStable` 若为 warn，必须解释是否为 rAF 采样期间自然推进；如果资源、敌人、projectiles、run level、currentGalaxyType、seed、objectiveType 或 priorityTarget 异常跳变，不能作为通过证据。
- `hookStatus.combat/meta/galaxy` 必须无阻塞缺口；缺口不能通过改验收口径覆盖。

RNG 与判定不变检查：

- 星系候选 RNG 常量、星系乘子、目标/遭遇权重、跳转确认状态机不能被 v0.17 polish 修改。
- 战斗判定中的武器 reload、射程、弹体伤害、护盾拦截、维修派工、敌人/炮台命中和敌人生成不能被取证任务修改。
- Meta 奖励、点数、talent rank、协议选择/应用规则不能被取证任务修改。
- 若必须变更上述任一项，先退出 v0.17 普通取证流程，进入项目高风险流程。

## 6. 缺陷等级与修复流程

严重程度：

- S0：游戏无法启动、首局无法进入、页面脚本中断、核心取证入口完全不可用、真实浏览器无法产出任何可复查证据。
- S1：P0 主路径无法完成、关键 UI 不可点击、结算/复盘/Meta/新局回流无法进入、存在阻塞 runtime error。
- S2：证据不可信、小屏核心信息被遮挡、按钮 hit 明显过小、触控滚动阻塞、复盘/Meta 入口可见但难以操作、复制摘要遗漏失败项。
- S3：非阻塞可读性、舒适度或文案问题，不影响 P0 闭环与发布闸门判断。
- Polish：制作人确认的高价值低风险小修，必须来自证据路径，不得夹带新玩法或规则修改。

修复准入：

1. 先有证据：环境、视口、起始条件、操作步骤、实际结果、runtime errors、storage signature、截图/摘要或人工记录。
2. PM 定级：明确 S0/S1/S2/S3/polish、是否阻塞闸门、允许修复范围。
3. 主程复核：确认不触及战斗、Meta、星系、存档、RNG、数值、依赖或发布流程。
4. 制作人决定：S0/S1 默认优先处理；S2/polish 由制作人判断是否纳入 v0.17；S3 默认转后续。
5. 执行最小修复：只改被批准范围。
6. 原路径复测：同一浏览器、同一入口、同一视口、同一起始条件或清晰说明差异。
7. PM #9 汇总：记录修复前后证据、不回潮检查和残留风险。

停止条件：

- 需要改核心规则、数值、战斗判定、RNG、Meta schema、`SAVE_KEY`、存档结构或星系状态机。
- 需要安装依赖、改系统/浏览器全局设置、发布、tag 或归档。
- 修复范围扩大为新功能、重做 UI 架构或新增玩法。
- 修复后出现新的 S0/S1，或无法用原路径复测关闭。

## 7. T1-T4 审核节点

### T1：完整真人 P0 路径补证

审核项：

- 起始条件明确：空存档、隔离存档、现有存档或辅助预置。
- 入口、HUD、建造、采矿、资源不足/断电、敌人威胁、交火、复盘、星系、Meta 回流逐节点标记。
- 每个节点区分真实点击、hook 辅助、DevTools/鼠标替代、未覆盖、失败或阻塞。
- `runtimeErrors` 和 storage signature 有记录。

停止/升级：

- 主路径只能靠 hook 改状态才能完成。
- 出现 S0/S1 阻塞。
- 为了过路径需要改资源、战斗、星系、Meta 或存档。

### T2：自然交火 / 武器护盾维修 / 战后复盘补证

审核项：

- 至少一次真实浏览器自然交火尝试。
- 证据串起武器开火或不可开火原因、护盾/受击或防御缺口、维修状态、战后 `combatReview`。
- hook 仅作为状态佐证，人工记录说明玩家是否看懂反馈。
- 缺任一关键反馈时，记录对发布闸门的影响。

停止/升级：

- 需要改敌人生成、等待时间、AI、伤害、RNG、武器、护盾或维修来触发交火。
- 只有 hook 结果，没有真实交火观察。
- 复盘由手改事件或状态伪造。

### T3：Meta 点击回流补证

审核项：

- 结算、Meta 或局外入口真实可见、可理解、可点击。
- 真实点击后能到达新局或可继续状态。
- 点击前后 storage 影响明确：未写、按预期写入、无法判断或异常写入。
- `SAVE_KEY` 和 schema 稳定，未修改局外成长结构。

停止/升级：

- 只能用 `resetMeta()`、`grantPoints()`、`purchaseViaUi()`、控制台或 storage 注入完成回流。
- 需要改 `SAVE_KEY`、meta schema、奖励、协议、存档迁移或 points 规则。
- storage 差异无法解释为自然 Meta 写入。

### T4：触控 / 小屏叠层 polish 与复测

审核项：

- 真实触控优先；无真机时必须记录 DevTools/鼠标替代边界。
- 小屏 HUD、建造、战斗、复盘、星系、Meta、结算入口可见、可点、可关闭或可滚动。
- 每个 polish 有原始现象、严重度、最小修复范围和原路径复测。
- 修复后不回潮：`runtimeErrors`、`storageSignature`、关键 DOM、combat/meta/galaxy hookStatus 仍可复查。

停止/升级：

- 修复触及战斗、Meta、星系、存档、RNG、数值或发布流程。
- 需要新增移动端系统、重做 UI 架构或扩大为新功能。
- 替代触控证据被包装成真机通过。

## 8. 发布闸门结论口径

可以支持“进入发布准备交接”的最低技术证据：

- 真人 P0 主要路径闭环，辅助使用和未覆盖项已说明。
- 自然交火/复盘至少覆盖武器、护盾/受击或防御缺口、维修、`combatReview`。
- Meta UI 回流有真实点击证据，storage 影响可解释。
- 触控或替代边界诚实；若替代证据发现阻塞级问题，已最小修复并复测。
- 无 S0/S1；S2/polish 均有处理结论和残留风险。
- `SAVE_KEY`、meta schema、RNG、战斗判定、星系状态机和存档结构未变。

只能建议继续补证或小修 polish：

- 缺真人 P0 主路径。
- 缺自然交火、复盘或 Meta 真实点击回流任一关键证据。
- 只有自动 JSON、hook、CDP/headless 或控制台模拟。
- storage 差异无法解释。
- 真实触控未覆盖且替代证据发现阻塞问题尚未关闭。

必须升级或暂停：

- 任何核心规则、战斗判定、Meta schema、星系 RNG、存档结构、依赖、发布产物或系统环境变更需求。
- 取证函数自身污染 storage，或必须伪造状态才能得到通过结论。
- 团队无法区分真实点击、辅助覆盖、未覆盖和失败。
