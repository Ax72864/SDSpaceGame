## 1. 背景

### 1.1 版本目标摘要

制作人方向把 v0.7.0 定为「战斗操控」——把现有"自动追鼠标 → 武器自找最近敌人 → 一次性手动齐射全部 missile"的简单战斗循环，升级为：

- **看得见战斗节奏**：HUD 武器行 + selectedCellPanel 装填进度条 + missile ready 发光环 + turret LOS 遮挡橙色环。
- **能锁定优先目标**：鼠标右键设置 `state.input.priorityTarget`；所有 `nearestEnemy` 调用点经 `selectTarget` 包装后默认尊重玩家锁定。
- **能控制齐射时机**：F 键手动触发 `fireMissiles()`；`[`/`]` 在 1 / 2 / 3 / all 四档之间切换 `state.missile.salvoSize`；仅发射前 N 个就绪的 missile cell（按 cell key 字典序）。

整个版本是「玩家工具升级」而非「敌方/玩家数值改动」，v0.6.0 hostile-station 决战体验只会变好，不会下降；玩家可"完全不用"v0.7.0 新功能（右键不点 / F 不按 / salvoSize 保持 1）继续游戏，行为等价 v0.6.0 + 默认逐发齐射。

### 1.3 关键设计风险（高优先级）

1. **`pointerdown` 行为分支增加**：v0.6.0 之前 `pointerdown` 完全不区分 button；v0.7.0 必须按 `event.button === 2` 分流。如果区分顺序错误（如先建 state.drag 再判 button），右键会先短暂污染 drag 状态再清除，可能让 `pointerup wasClick` 路径误触发 `handleCanvasClick`。**必须**在 `pointerdown` 第一行先判 `event.button`，右键单独路径**不进入** `state.drag = {...}`；同时 `contextmenu` 事件 `event.preventDefault()` 抑制浏览器右键菜单弹出。
2. **`nearestEnemy` 调用点替换为 `selectTarget` 的回归面**：v0.6.0 之前 `nearestEnemy` 被 4 处调用：`updateTurret`（武器索敌）/ `fireMissiles`（每 cell 索敌）/ `shieldDirection`（护盾朝向）/ 无其他玩家路径。如果 `selectTarget` 实现中"有效性检查不严"或"fallback 顺序错误"，会让"无锁定时行为等价 v0.6.0"的硬约束破裂，回潮全部 v0.6.0 战斗体验。**必须**保持 `selectTarget(origin, range)` 在 priorityTarget 为 null / 失效 / 超距时**严格 fallback 到 `nearestEnemy(origin, range)`**；新增检查只是优先级前置，不替代 fallback。
3. **`fireMissiles` 排序逻辑改动**：v0.6.0 之前 `fireMissiles` 遍历 cells.values() 顺序受 Map 插入顺序影响；玩家无法预期。v0.7.0 改为"先 filter ready 再按 cell key 字典序排序"，会让"哪个 cell 先发"对玩家变得**可预期**——但同时**改变了部分 build 在多 missile cell 时的发射顺序**。在 salvoSize = all (999) 时行为等价（全部发射），仅 salvoSize < readyCount 时行为有差异；新行为符合玩家预期"左边的先发"，但**节点 A 必须用 v0.6.0 完整存档跑 5 个 seed**验证 salvoSize = all 时 missile 总伤害与命中分布与 v0.6.0 等价。
4. **资源不足提前 return 的语义保留**：现有 `fireMissiles` 在第一个 cell 资源不足时 `return`，后续 cell 全部不发。salvoSize 模式下"前 N 个 cell 资源不足"时，**必须**保持同款语义（前 N 个排好序后逐个 fire；某个 cell 资源不足时 return；不发"跳过该 cell 用后续 cell 补"）——否则与 v0.6.0 体验差异过大，玩家可能误判资源状态。
5. **F 键与 Space / Enter 键冲突**：现有 Space / Enter → `handleWorldClick`（虚拟光标点击）；F 键当前未占用。F 键 keydown 中**不能调 `event.preventDefault()` 阻止浏览器默认（F 键无默认行为，但出于一致性建议加）**；且与 WASD 移动键并存时，必须保证 F 键路径在 `isMoveKey(key)` 判断之外（F 不是移动键，进 else 分支）。
6. **优先目标 enemy 引用 vs ID 弱引用**：制作人方向已经决定**直接 reference enemy 对象**，由 `selectTarget` 检查 `enemy.hp > 0` 自动失效。**风险**：enemy 被 `state.enemies = state.enemies.filter(...)` 移除后，引用仍持有但 hp 已 0；`selectTarget` 必须每帧做 `hp > 0` 检查。**关键陷阱**：v0.6.0 hostile-station 是聚合 hp（`sum(cell.hp)`），`enemy.hp` 在 cell 破坏时仍 > 0，但玩家直觉上"打了 BOSS 的武器 cell 优先目标应该自动失效"——本 ADR 决定**不**特殊处理 hostile-station：玩家可以一直锁定 hostile-station 整体（与战术体验一致：BOSS 决战的 priorityTarget 就是 BOSS 本身，不是单个 cell）。
7. **HUD 武器行 DOM 与 setHtmlIfChanged 缓存**：updateHud 当前用 `setHtmlIfChanged(element, cacheKey, html)` 防止每帧 DOM 更新。新增"武器"行必须使用新缓存 key（如 `state.hud.weaponsLine`），不能复用 `resources` 或 `status`，否则 HUD 频繁重渲染；同时新 DOM 节点需要在 HTML 中预创建（或 JS 内 createElement 一次），避免 innerHTML 操作的开销。
8. **`renderPriorityTargetLockOverlay` 渲染顺序**：必须在 `renderEnemies` 主循环结束后、HUD canvas 文字之前；如果放错（如放进 renderEnemies 主循环内）会因敌人之间 z-order 错乱（玩家无法判断"哪个敌人才是被锁的"）；如果放在 HUD 之后会被 HUD 文字覆盖。

## 2. 设计决定逐项（对应方向文档「关键技术疑问」8 项）

### 2.1 `priorityTarget` 数据结构

- **决定**：采用**直接 reference enemy 对象**的 `{ enemy, setAt, validUntil }` 结构，state 字段位置 `state.input.priorityTarget`；初始 null。
- **理由**：
  - 弱引用方案（用 `enemyId`）要求 enemy 有稳定 id + 反查表，与现有 `state.enemies` 扁平 array + 无 id 字段的现状不匹配；引入 id 体系需要扩展 spawnEnemy、updateEnemies filter、enemy 序列化等多个路径，复杂度过高。
  - 直接 reference 的失效问题（enemy 死后引用悬挂）由 `selectTarget` 每次访问做 `enemy.hp > 0` 检查解决；hp <= 0 即自动失效（同步清空 `state.input.priorityTarget`，避免下一帧再次访问悬挂引用）。
  - v0.3.0 `fragment` 也用了类似的"直接持有引用 + 状态检查"模式（如 `mining.target.body`），团队已经熟悉。
- **完整有效性检查**（`selectTarget` 内部每次调用）：
  1. `priorityTarget == null` → 走 fallback。
  2. `priorityTarget.enemy.hp <= 0` → 清空 `priorityTarget = null` + 走 fallback。
  3. `state.time - priorityTarget.setAt > PRIORITY_TARGET_DURATION`（默认 12 秒，数值方案精化）→ 清空 + 走 fallback。
  4. `dist(state.station.pos, priorityTarget.enemy) > PRIORITY_TARGET_MAX_DISTANCE`（默认 800 px）→ 清空 + 走 fallback。
  5. `dist(origin, priorityTarget.enemy) > range` → **不清空**（priorityTarget 仍有效，但当前 weapon range 内打不到；让其他 weapon 在自己 range 内继续优先该目标）→ 仅本次走 fallback 即 `nearestEnemy(origin, range)`。
- **`setPriorityTarget(enemy)` / `clearPriorityTarget()` 函数封装**：禁止直接赋值 `state.input.priorityTarget = ...`；统一走两个函数，便于未来扩展（如音效、UI 提示、统计）。
- **`state.input.aimingRightButton: boolean` 字段**：记录右键当前是否按下（pointerdown=true / pointerup=false / pointercancel=false）；v0.7.0 范围**不引入**"按住右键持续瞄准"行为，但保留字段供 v0.7.1+ 扩展（如"按住右键时显示扇形索敌区"）。
- **回滚**：若直接引用引入 BOSS 战回归（如玩家被卡在锁定 hostile-station 的状态），可降级为"priorityTarget 仅对 turret/missile 生效，shieldDirection 保持 nearestEnemy"；2 行 if 即可。

### 2.2 右键事件优先级

- **决定**：
  - `pointerdown` 第一行判断 `event.button === 2`：右键直接走 `handleRightClick(world)` → `setPriorityTarget(world)` 或 `clearPriorityTarget()`；**不进入** `state.drag`。
  - `pointerdown` `event.button === 0` 或默认（兼容触摸 pointerType）：维持原 `state.drag = {...}` 流程，零行为变化。
  - 在 canvas 上单独绑定 `contextmenu` 事件 + `event.preventDefault()`，抑制浏览器右键菜单弹出。
  - `pointerup` / `pointermove` / `pointercancel` 仍按 `state.drag` 路径处理；右键不进入 state.drag 故 pointerup wasClick 不会误触发 `handleCanvasClick`。
  - `state.input.aimingRightButton` 在 `pointerdown(event.button===2)` 设 true、`pointerup(event.button===2)` / `pointercancel` 设 false（仅记录状态，不参与本版本逻辑）。
- **键盘输入与右键独立通道**：
  - WASD / Enter / Space / 1-9 / 0 / - / = → 现有 keydown 路径完全保留。
  - F → 新增 keydown 分支，调用 `fireMissiles()`；F 键防抖（`state.input.fKeyCooldown`，0.3 秒）由数值方案精化。
  - `[` / `]` → 新增 keydown 分支，调用 `adjustSalvoSize(delta)`；无防抖（玩家自己控制连按节奏）。
- **优先级链**（同帧多输入时）：右键独立通道与键盘独立通道**永不冲突**（鼠标事件 vs 键盘事件源不同）；左键 + 拖动旋转 vs 鼠标移动追鼠标由 `updateMouseAim` 内 `state.drag?.moved` 检测自动让路（拖动期间不追鼠标，零回归）。
- **回滚**：若 `contextmenu` 抑制引入浏览器兼容性问题（如某些浏览器禁用 preventDefault），降级为"右键不抑制菜单，但 priorityTarget 仍设置成功"；玩家会看到一次菜单弹出但功能不破坏。
- **关键代码骨架**：
  ```js
  canvas.addEventListener("pointerdown", (event) => {
    // === v0.7.0 右键独立通道 ===
    if (event.button === 2) {
      state.input.aimingRightButton = true;
      const world = renderer.screenToWorld({ x: event.clientX, y: event.clientY });
      handleRightClick(world);
      event.preventDefault();
      return;
    }
    // === 原 state.drag 流程不变 ===
    canvas.setPointerCapture(event.pointerId);
    state.drag = { id: event.pointerId, x: event.clientX, y: event.clientY, lastX: event.clientX, lastY: event.clientY, moved: false };
  });

  canvas.addEventListener("pointerup", (event) => {
    if (event.button === 2) {
      state.input.aimingRightButton = false;
      return;
    }
    // === 原 state.drag 路径完全不变 ===
    if (!state.drag || state.drag.id !== event.pointerId) return;
    const wasClick = !state.drag.moved;
    state.drag = null;
    if (wasClick) handleCanvasClick({ x: event.clientX, y: event.clientY });
  });

  canvas.addEventListener("contextmenu", (event) => {
    event.preventDefault();
  });
  ```

### 2.3 `salvoSize` 默认值与边界

- **决定**：
  - 默认值：`state.missile.salvoSize = 1`（初始 1，逐发模式作为新默认体验）。
  - 边界：1 / 2 / 3 / all（999）四档循环。
  - 调整方式：`[` 减小 / `]` 增大；越界循环（1 ← all → 3 ← 2 ← 1）或 clamp（数值方案精化）。本 ADR 推荐**循环**（玩家从 1 按 `[` 直接跳到 all，体验更顺畅）。
  - 持久化：**仅 run 内**；不写入 SAVE_KEY；`resetRun()` 重置为 1。
- **理由**：
  - 默认 1 让"老玩家"在升级后第一次开局**不会被一次性梭光全部 missile** 惊吓；玩家主动按 `]` 升档到 all 才会进入"v0.6.0 体验"。
  - 仅 run 内的好处：跨 run 不污染、SAVE_KEY 不破坏（不影响存档兼容）、玩家每局有重新选择齐射策略的机会（与"build 选择"一致语义）。
  - 999 作为 "all" 标识比单独 `salvoMode: "single" | "all"` 字段更简单（实现层面用 `Math.min(salvoSize, readyCount)` 自然处理；HUD 显示时检查 `salvoSize >= 999 ? "全部" : salvoSize` 即可）。
- **边界与显示**：
  - HUD 显示：`齐射 1` / `齐射 2` / `齐射 3` / `齐射 全部`；不显示具体数字 999（避免玩家困惑）。
  - `[` 行为：salvoSize === 1 → 切到 all；salvoSize === all → 切到 3；3 → 2；2 → 1。
  - `]` 行为：salvoSize === 1 → 2；2 → 3；3 → all；all → 1。
  - 数值方案如需调整边界（如 1/2/3/5/all 五档），仅需改 `SALVO_SIZE_STEPS = [1, 2, 3, 999]` 常量，循环逻辑不变。
- **回滚**：若节点 B 体验测试发现"默认 1 让导弹不够爽快"，可在 v0.7.1 改默认为 2 或 3；ADR 范围内坚持默认 1。

### 2.4 `fireMissiles` 排序逻辑

- **决定**：
  - 在 `fireMissiles` 内：
    1. `filter`：`cell.facility === "missile" && cell.active && !cell.detached && cell.reload <= 0`，得 `readyCells[]`。
    2. `sort`：按 `cell.key`（即 `"x,y"` 字符串）字典序排序。**确定性**：同 build 的同 cell 集合每次发射顺序固定，玩家可预期"左边的先发"。
    3. `slice`：取前 `Math.min(state.missile.salvoSize, readyCells.length)` 个。
    4. 逐个走原 fire 逻辑（每发独立 `nearestEnemy` 路径作为 fallback，但**经 `selectTarget` 包装**——见 §2.5）。
- **理由**：
  - 字典序排序优于"按 reload 最早完成"：玩家**反复按 F** 时（多次连续齐射），按字典序每次都是固定 cell 顺序；按最早 reload 完成的话每次顺序不同，玩家无法形成肌肉记忆。
  - 字典序对 cell key 格式`"x,y"`（如 `"-1,0"` < `"0,1"` < `"1,0"`）排序结果稳定：x=-1 优先于 x=0 优先于 x=1，符合"左边的先发"直觉。
  - salvoSize = all (999) 时 `slice(0, 999)` 自然返回所有 readyCells，与 v0.6.0 行为等价（只是顺序从 Map 插入顺序变为字典序，**但所有 cell 都发，总伤害不变**）。
- **资源不足语义保留**：
  - 第一个 cell 进入 fire 逻辑后即检查资源；不足则 `showToast` + `return`（与 v0.6.0 一致，不发任何 cell）。**不**改为"跳过该 cell 用后续 cell 补"（避免引入新语义，玩家可能误判资源状态）。
  - 数值方案可决定是否在未来引入"部分发射"行为（如 N=3 但仅够 2 发资源时发 2 发并 toast 警告），但 v0.7.0 范围不做。
- **`fireMissiles` 改造伪代码**：
  ```js
  function fireMissiles() {
    const readyCells = [];
    for (const [cellKey, cell] of state.station.cells) {
      if (cell.facility !== "missile" || !cell.active || cell.reload > 0 || cell.detached) continue;
      readyCells.push({ cellKey, cell });
    }
    if (readyCells.length === 0) {
      showToast("没有可发射的导弹井。");
      return;
    }
    readyCells.sort((a, b) => a.cellKey.localeCompare(b.cellKey));
    const salvo = Math.min(state.missile.salvoSize, readyCells.length);
    let fired = 0;
    for (let i = 0; i < salvo; i++) {
      const { cell } = readyCells[i];
      const origin = cellWorldPosition(cell);
      const enemy = selectTarget(origin, getCellStat(cell, "range"));
      if (!enemy) continue;
      const gasCost = TYPES.missile.baseStats.gasCost;
      const metalCost = TYPES.missile.baseStats.metalCost;
      if (state.resources.gas < gasCost || state.resources.metal < metalCost) {
        showToast("导弹装填资源不足，需要气体和金属。");
        return;
      }
      // ... 原 projectile push 逻辑完全不变 ...
      fired += projectileCount;
    }
    showToast(fired ? `导弹齐射：${fired} 枚` : "导弹齐射：无可锁定目标。");
  }
  ```
- **回滚**：若节点 A 实测"按字典序发射"在某些 build 下显得不自然（如玩家 build 是单边 missile 阵列时所有 missile 都从一侧射出，视觉拥挤），可降级为"按 reload === 0 + 字典序双键排序"（reload === 0 时刻最早的先发，平手时字典序）；但 v0.7.0 范围内保持纯字典序。

### 2.5 `selectTarget(origin, range)` 包装函数

- **决定**：新增 `selectTarget(origin, range)` 函数，所有原 `nearestEnemy(origin, range)` 调用点（4 处：`updateTurret` / `fireMissiles` / `shieldDirection` / 未来扩展点）改为调 `selectTarget`。
- **理由**：
  - 单一入口让"优先目标"在所有武器系统**一致生效**；玩家右键锁定后，turret / missile / shield 都同步朝该目标，体验闭环。
  - `selectTarget` 内部仍保留 `nearestEnemy` 作为唯一 fallback，**不复制索敌逻辑**；性能与 v0.6.0 等价。
  - 命名 `selectTarget` 比 `prioritizedNearestEnemy` 短，且语义清晰（"为此次开火选目标"）。
- **`selectTarget` 完整伪代码**：
  ```js
  function selectTarget(origin, range) {
    const priority = state.input.priorityTarget;
    if (priority && priority.enemy) {
      // 有效性检查
      if (priority.enemy.hp <= 0) {
        clearPriorityTarget();
      } else if (state.time - priority.setAt > PRIORITY_TARGET_DURATION) {
        clearPriorityTarget();
      } else if (dist(state.station.pos, priority.enemy) > PRIORITY_TARGET_MAX_DISTANCE) {
        clearPriorityTarget();
      } else {
        // 当前 weapon range 内打得到则优先返回；range 外仍保留锁定，本次走 fallback
        if (dist(origin, priority.enemy) <= range) {
          return priority.enemy;
        }
      }
    }
    return nearestEnemy(origin, range);
  }
  ```
- **替换点**：
  - `updateTurret` line 4827：`const enemy = nearestEnemy(origin, getCellStat(cell, "range"));` → `const enemy = selectTarget(origin, getCellStat(cell, "range"));`
  - `fireMissiles` line 4878 / 新版：`const enemy = nearestEnemy(cellWorldPosition(cell), getCellStat(cell, "range"));` → `const enemy = selectTarget(cellWorldPosition(cell), getCellStat(cell, "range"));`
  - `shieldDirection` line 5539：`const enemy = nearestEnemy(origin, 420);` → `const enemy = selectTarget(origin, 420);`
  - 任何未来新武器/系统都应通过 `selectTarget` 索敌，**禁止**直接调 `nearestEnemy`（除 selectTarget 内部 fallback）。
- **回归约束**：节点 A 必须验证"priorityTarget == null 时所有武器行为与 v0.6.0 完全等价"（即 selectTarget 严格 fallback 到 nearestEnemy，无任何额外开销/路径偏移）。

### 2.6 优先目标视觉锁定框

- **决定**：在 `renderEnemies` 主循环结束后、HUD canvas 文字之前，独立函数 `renderPriorityTargetLockOverlay(ctx)` 渲染锁定框。
- **样式**：
  - 红色十字准星：线宽 2 / 半臂长 20 px（从 enemy.x/y 出发，4 个方向各延伸 20 px）；颜色 `rgba(255, 50, 50, 0.85)`。
  - 旋转脉冲圆环：半径 `enemy.r * 1.5`（hostile-station enemy.r ≈ 86 时半径 ≈ 130 px）/ 旋转 1 rad/s（每帧 `state.time` 偏移）/ pulse 周期 1.0 秒（`alpha = 0.5 + 0.5 * Math.sin(state.time * 2 * Math.PI)`）；颜色 `rgba(255, 50, 50, alpha)`；线宽 3。
  - 锁定框只在 `priorityTarget` 有效时渲染（`enemy.hp > 0 && 未超时 && 距离 < 800`）；超时立刻消失。

  - 在 enemy 之上、HUD 之下：玩家既能看到锁定框覆盖在 enemy 上（视觉强调"我锁了这个"），又不会被 HUD 文字盖住（避免锁定框被资源条挡住）。
  - 红色十字 + 脉冲圆环组合：十字是"瞄准镜"语义（玩家熟悉），脉冲圆环是"持续锁定状态"语义；两者叠加比单独十字更突出。
  - 不复用 `cellSelect` / `hostile-station 危险圈` 样式：那两种是"系统级提示"（玩家选中 cell / BOSS 出现），priorityTarget 是"玩家主动选择"，应该有独立视觉。
  - 旋转 1 rad/s（每 6.28 秒一圈）+ pulse 1 秒：动感但不刺眼。
- **`renderPriorityTargetLockOverlay` 伪代码**：
  ```js
  function renderPriorityTargetLockOverlay(ctx) {
    const priority = state.input.priorityTarget;
    if (!priority || !priority.enemy || priority.enemy.hp <= 0) return;
    if (state.time - priority.setAt > PRIORITY_TARGET_DURATION) return;
    if (dist(state.station.pos, priority.enemy) > PRIORITY_TARGET_MAX_DISTANCE) return;

    const screen = renderer.worldToScreen(priority.enemy);
    const ringRadius = priority.enemy.r * 1.5 * state.camera.zoom;
    const rotation = state.time * 1.0;  // rad/s
    const pulseAlpha = 0.5 + 0.5 * Math.sin(state.time * Math.PI * 2);  // 1.0 秒周期
    const armLength = 20;

    ctx.save();
    ctx.translate(screen.x, screen.y);
    // 旋转脉冲圆环
    ctx.rotate(rotation);
    ctx.strokeStyle = `rgba(255, 60, 60, ${pulseAlpha})`;
    ctx.lineWidth = 3;
    ctx.beginPath();
    ctx.arc(0, 0, ringRadius, 0, Math.PI * 2);
    ctx.stroke();
    // 十字（不随圆环旋转，回到屏幕坐标）
    ctx.rotate(-rotation);
    ctx.strokeStyle = "rgba(255, 60, 60, 0.85)";
    ctx.lineWidth = 2;
    ctx.beginPath();
    ctx.moveTo(-armLength, 0); ctx.lineTo(armLength, 0);
    ctx.moveTo(0, -armLength); ctx.lineTo(0, armLength);
    ctx.stroke();
    ctx.restore();
  }
  ```
- **渲染顺序集成**：在主 render 流程中，`renderEnemies()` 之后、HUD DOM/canvas 文字之前调用 `renderPriorityTargetLockOverlay(ctx)`。
- **回滚**：若性能或视觉效果不佳，可简化为"仅十字 + 静态半透明圆环"；25 行内可完成简化。

### 2.8 F 键 / `[` `]` 键的可发现性

- **决定**：仅在 HUD 武器行末尾显示固定灰色提示文字 `F=齐射 · [/]=调档 · 右键锁敌`（CSS 颜色 #888 / fontSize 11px / 与武器信息同行）；游戏首次进入**不弹 popup** / **不弹 tour**。
- **理由**：
  - 玩家在熟悉资源条时自然看到武器行末尾的提示；不打断游戏节奏。
  - popup / tour 引入额外 DOM + 状态管理（"是否已显示" / "玩家点击关闭" / SAVE_KEY 标志）成本高，且容易让老玩家反感（每次进游戏被提示窗户挡住）。
  - 玩家自学成本低：F 键一按就有反馈（"导弹齐射：X 枚" toast）；`[`/`]` 一按 HUD 武器行 `齐射 N` 立即更新。
- **数值方案可调**：若节点 C 用户测试发现"玩家完全不知道有 F 键"，可考虑在 v0.7.1 引入"首次按 Space/Enter 后 toast 提示 '试试 F 键齐射？'"；ADR 范围内不做。
- **F 键防抖**：keydown 时检查 `state.input.fKeyCooldown > 0` 则跳过；fire 后设 `state.input.fKeyCooldown = 0.3`（数值方案精化）；每帧 `state.input.fKeyCooldown -= dt`。避免玩家长按 F 导致 keydown 重复触发刷屏。
- **`[`/`]` 无防抖**：调档是离散选择，玩家自己控制连按节奏；HUD 武器行实时反映 salvoSize 变化即可。

### 2.9 LOS 遮挡视觉警告频次

- **决定**：仅在"刚被挡的瞬间"短暂显示 0.5 秒；持续遮挡不闪烁；用 `cell._losBlockedAt` 时间戳节流。
- **触发逻辑**（在 `updateTurret` 内）：
  ```js
  function updateTurret(cell, dt) {
    if (cell.reload > 0) return;
    const origin = cellWorldPosition(cell);
    const enemy = selectTarget(origin, getCellStat(cell, "range"));
    if (!enemy) {
      cell._losBlockedPrev = false;
      return;
    }
    const losOk = hasLineOfSight(origin, enemy);
    if (!losOk) {
      // 上一帧 LOS 通过，本帧失败 → 触发警告
      if (cell._losBlockedPrev !== true) {
        // 5 秒内不重复触发
        if (!cell._losBlockedAt || state.time - cell._losBlockedAt > 5) {
          cell._losBlockedAt = state.time;
        }
      }
      cell._losBlockedPrev = true;
      return;
    }
    cell._losBlockedPrev = false;
    // ... 原 fire 逻辑不变 ...
  }
  ```
- **渲染逻辑**（在 turret cell 渲染路径中）：
  ```js
  function renderTurretLosWarning(cell, screenPos) {
    if (!cell._losBlockedAt) return;
    const elapsed = state.time - cell._losBlockedAt;
    if (elapsed > 0.5) return;
    const alpha = 1.0 - elapsed / 0.5;  // 线性衰减
    ctx.strokeStyle = `rgba(255, 165, 0, ${alpha})`;
    ctx.lineWidth = 2;
    ctx.beginPath();
    ctx.arc(screenPos.x, screenPos.y, CELL * 0.55 * state.camera.zoom, 0, Math.PI * 2);
    ctx.stroke();
  }
  ```
- **理由**：
  - 持续闪烁让玩家视觉疲劳，且无新信息；只在"刚被挡上"瞬间提示，玩家立刻知道"原来这个 turret 此时被挡了"。
  - 5 秒内不重复触发同一 cell：避免快速 station 旋转时同一 cell 反复触发短暂警告，玩家眼花。
  - 0.5 秒持续 + alpha 衰减：足够玩家注意到，不会持续干扰。
- **HUD 武器行的 K 计数**：`K = cells.filter(c => c._losBlockedAt && state.time - c._losBlockedAt < 0.5).length`，与视觉警告同步；玩家可在 HUD 看到"3 个炮塔被遮挡"提示，提示玩家调整 station 朝向。
- **回滚**：若节点 B 发现"0.5 秒太短，玩家看不到"，可调整为 0.8 秒 + 5 秒间隔不变；数值方案精化。

## 4. 关键算法伪代码

### 4.1 `setPriorityTarget` / `clearPriorityTarget` / `handleRightClick`

```js
function setPriorityTarget(enemy) {
  if (!enemy || enemy.hp <= 0) return;
  state.input.priorityTarget = {
    enemy,
    setAt: state.time,
    validUntil: state.time + PRIORITY_TARGET_DURATION
  };
  // 可选：音效 / toast / HUD 提示（数值方案决定）
}

function clearPriorityTarget() {
  state.input.priorityTarget = null;
}

function handleRightClick(world) {
  // 在 PRIORITY_TARGET_LOCK_RADIUS 内找最近敌人
  let nearest = null;
  let nearestDist = PRIORITY_TARGET_LOCK_RADIUS;
  for (const enemy of state.enemies) {
    if (enemy.hp <= 0) continue;
    const d = dist(world, enemy);
    if (d < nearestDist) {
      nearest = enemy;
      nearestDist = d;
    }
  }
  if (nearest) {
    setPriorityTarget(nearest);
  } else {
    clearPriorityTarget();
  }
}
```

### 4.2 `selectTarget(origin, range)`

```js
function selectTarget(origin, range) {
  const priority = state.input.priorityTarget;
  if (priority && priority.enemy) {
    // 有效性检查 + 失效时自动清空
    let invalid = false;
    if (priority.enemy.hp <= 0) invalid = true;
    else if (state.time - priority.setAt > PRIORITY_TARGET_DURATION) invalid = true;
    else if (dist(state.station.pos, priority.enemy) > PRIORITY_TARGET_MAX_DISTANCE) invalid = true;

    if (invalid) {
      clearPriorityTarget();
    } else if (dist(origin, priority.enemy) <= range) {
      // 优先目标在当前 weapon range 内 → 优先返回
      return priority.enemy;
    }
    // priority 仍有效但本 weapon 打不到 → 走 fallback
  }
  return nearestEnemy(origin, range);
}
```

### 4.3 `fireMissiles` 改造

```js
function fireMissiles() {
  const readyCells = [];
  for (const [cellKey, cell] of state.station.cells) {
    if (cell.facility !== "missile") continue;
    if (!cell.active || cell.detached || cell.reload > 0) continue;
    readyCells.push({ cellKey, cell });
  }
  if (readyCells.length === 0) {
    showToast("没有可发射的导弹井。");
    return;
  }
  readyCells.sort((a, b) => a.cellKey.localeCompare(b.cellKey));

  const salvo = Math.min(state.missile.salvoSize, readyCells.length);
  let fired = 0;
  for (let i = 0; i < salvo; i++) {
    const { cell } = readyCells[i];
    const origin = cellWorldPosition(cell);
    const enemy = selectTarget(origin, getCellStat(cell, "range"));
    if (!enemy) continue;

    const gasCost = TYPES.missile.baseStats.gasCost;
    const metalCost = TYPES.missile.baseStats.metalCost;
    if (state.resources.gas < gasCost || state.resources.metal < metalCost) {
      showToast("导弹装填资源不足，需要气体和金属。");
      return;
    }
    state.resources.gas -= gasCost;
    state.resources.metal -= metalCost;

    const projectileCount = Math.max(1, Math.floor(getCellStat(cell, "projectileCount")));
    const projectileSpeed = getCellStat(cell, "projectileSpeed");
    const baseProjectileSpeed = TYPES.missile.baseStats.projectileSpeed;
    const projectileAccel = TYPES.missile.baseStats.projectileAccel * (projectileSpeed / baseProjectileSpeed);
    const launchJitter = TYPES.missile.baseStats.launchJitter;
    const damage = getCellStat(cell, "damage") * (1 + state.meta.weapons * 0.12);

    for (let p = 0; p < projectileCount; p++) {
      state.projectiles.push({
        owner: "missile",
        target: enemy,
        x: origin.x,
        y: origin.y,
        vx: rand(-launchJitter, launchJitter),
        vy: rand(-launchJitter, launchJitter),
        damage,
        life: TYPES.missile.baseStats.life,
        r: TYPES.missile.baseStats.radius,
        maxSpeed: projectileSpeed,
        accel: projectileAccel
      });
    }
    cell.reload = getCellStat(cell, "reload");
    fired += projectileCount;
  }
  showToast(fired ? `导弹齐射：${fired} 枚` : "导弹齐射：无可锁定目标。");
}
```

  ctx.translate(screen.x, screen.y);

## 5. UI 与渲染策略

### 5.1 渲染顺序集成

主 render 流程目前的顺序（概念性）：

```
1. clear canvas
2. renderStars / renderResources / renderFragments
3. renderEnemies         ← hostile-station 内部分支已在
4. renderStation         ← 含 turret / missile / shield cell
5. renderProjectiles / renderParticles
6. HUD overlays (canvas)
```

v0.7.0 新增：

- 在 `renderStation` 内 turret cell 渲染分支末尾调用 `renderTurretLosWarning(cell, screenPos)`（橙色环）。
- 在 `renderStation` 内 missile cell 渲染分支末尾调用 `renderMissileReadyGlow(cell, screenPos)`（cell.reload <= 0 时显示亮色发光环；与 cell.fire 效果区分）。
- 在 `renderProjectiles` / `renderParticles` 之后、HUD canvas 文字之前，调用 `renderPriorityTargetLockOverlay(ctx)`。

### 5.2 性能与缓存

- `renderPriorityTargetLockOverlay` 仅在 priorityTarget 有效时绘制（每帧最多 1 次 ctx.save/restore + 2 个 stroke）；极轻量。
- `renderTurretLosWarning` 仅在 `cell._losBlockedAt && state.time - cell._losBlockedAt < 0.5` 时绘制；同 cell 5 秒内不重复触发，全场 turret 不会同时全部绘制橙色环。
- HUD 武器行用 `setHtmlIfChanged(weaponsLineEl, "weaponsLine", html)` 防止每帧 DOM 更新（与现有 `resources / alerts` 同款节流）。

### 5.3 兼容现有 hostile-station 危险圈

`assault.render` 已在 hostile-station 周围渲染红色危险双环（v0.6.0）；若玩家右键锁定 hostile-station，`renderPriorityTargetLockOverlay` 会在其上叠加红色十字 + 脉冲圆环。两者**视觉协同**（同为红色，玩家不会困惑）；锁定圆环半径 `enemy.r × 1.5 ≈ 130` 比危险圈 `enemy.r + 56 ≈ 142` 略小，视觉层次清晰（"我锁了 BOSS → 危险但已掌控"）。

## 6. 性能预算

| 项 | 估算 | 备注 |
| --- | --- | --- |
| `selectTarget(origin, range)` 单次调用 | O(1) 有效性检查 + 1 次 dist + 1 次 nearestEnemy fallback | 优先路径下 < 0.001 ms；fallback 等价 v0.6.0 |
| `handleRightClick(world)` 一次性 | O(E) E = 敌人数（典型 < 30） | < 0.05 ms |
| `fireMissiles` 改造 | O(M log M) 排序 + 原 fire 逻辑 | M = missile cell 数（典型 1-6）；< 0.05 ms |
| `renderPriorityTargetLockOverlay` 每帧 | 2 个 ctx.beginPath + stroke | < 0.05 ms |
| `renderTurretLosWarning` 每帧 | 每个被挡 turret 1 次 stroke | typical < 0.05 ms |
| `updateHud` 武器行计算 | O(C) C = cells 数（典型 < 50）的 4 次 filter | < 0.1 ms（每帧）|
| `updateTurret` 内 LOS 节流 | 1 次 hasLineOfSight + 状态对比 | < 0.05 ms / turret |
| F 键防抖 | 1 次浮点比较 + 1 次 dt 衰减 | 可忽略 |
| **合计新增（priorityTarget 激活 + 武器行刷新）** | < 0.5 ms / 帧 | 远低于 v0.5.0 `getCellStat` 总开销 |

- **关键决定**：HUD 武器行**每帧重算**（4 次 filter ≈ 200 次比较）；不引入"按 cell.reload 变化时才重算"缓存（复杂度过高，节省可忽略）。
- **如节点 B 实测发现 HUD 占用 > 0.5 ms**：可改为每 100ms 重算一次（仍流畅，玩家无感差异）。

## 11. 待制作人 / PM 决策的开放项

1. **`adjustSalvoSize` 循环 vs clamp**：
   - 选项 A：循环（1 ↔ all ↔ 3 ↔ 2 ↔ 1）。
   - 选项 B：clamp（在 1 按 `[` 不变；在 all 按 `]` 不变）。
   - **本 ADR 推荐：选项 A**（玩家从 1 按 `[` 直接跳 all 体验顺畅；clamp 让玩家在 1 时按 `[` 无反馈，可能误以为按键失效）；如节点 B 体验发现"循环让玩家不小心切到 all 太多发"，可改 clamp。

2. **HUD 武器行小屏响应式**：
   - 选项 A：单行水平（默认）。
   - 选项 B：屏宽 < 800 px 自动换两行（导弹井 + 炮塔 分行）。
   - **本 ADR 推荐：选项 A**（v0.7.0 范围控制；移动端兼容留 v0.8+）；如节点 B 实测移动端拥挤，再考虑选项 B。

3. **selectedCellPanel reload 进度条颜色**：
   - 选项 A：0-25% 暗红 / 25-75% 黄 / 75-100% 亮绿（数值方案默认）。
   - 选项 B：纯进度色（如 hsl(120 × pct, 60%, 50%)）。
   - **本 ADR 推荐：选项 A**（语义清晰：红 = 危险/未就绪、黄 = 进度、绿 = 就绪）；数值方案可微调阈值。

4. **F 键失败 toast**：
   - 选项 A：salvoSize > readyCount 时仅发 readyCount 个 + toast `导弹齐射：X 枚`（不报错）。
   - 选项 B：salvoSize > readyCount 时 toast `齐射档位 N，但仅 X 个就绪`（提示玩家档位与现实不符）。
   - **本 ADR 推荐：选项 A**（少打扰）；节点 B 视玩家反馈再决定。

5. **priorityTarget 超时主动提示**：
   - 选项 A：超时静默清空（玩家自己注意锁定框消失）。
   - 选项 B：超时 toast `优先目标已失效`。
   - **本 ADR 推荐：选项 A**（避免打扰）；如节点 C 玩家"不知道为什么 turret 又自动找最近"，可加 toast。

6. **右键确认 / 二次右键取消**：
   - 选项 A：右键在 PRIORITY_TARGET_LOCK_RADIUS 内有敌则锁定，否则清空（默认）。
   - 选项 B：在已锁敌人上再次右键取消锁定；右键空白不清空（保持当前锁定）。
   - **本 ADR 推荐：选项 A**（玩家学习成本最低，与 RTS / MOBA 右键习惯一致）；选项 B 引入"右键状态机"复杂度。

7. **missile ready 发光环样式**：
   - 选项 A：cell.reload <= 0 时持续显示亮色发光环（脉冲）。
   - 选项 B：仅在 reload 刚归 0 的瞬间显示 0.5 秒发光环（与 LOS 警告同款节流）。
   - **本 ADR 推荐：选项 A**（持续可见让玩家清晰知道"哪些 missile 可发"；与 LOS 警告"瞬时状态"语义不同）。

8. **dev 调试入口**：
   - 选项 A：T1 临时 `window.DEBUG_LOCK_NEAREST = () => setPriorityTarget(nearestEnemy(state.station.pos, 9999))`；T6 移除。
   - 选项 B：不引入（与 v0.6.0 一致策略）。
   - **本 ADR 推荐：选项 A**（节点 A PM 可快速验证）；T6 必须移除。
