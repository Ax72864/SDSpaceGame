## 1. 背景

### 1.1 版本目标摘要

制作人方向把 v0.3.0 的核心体验定为「框架脱落 → 玩家重新接入」可感知、可操作、可回收的循环：

- 脱落部分明显从站体「飞出来」，HUD 同时提示玩家「丢了一块」。
- 玩家可以驾驶站体靠近残骸，按候选位置扩展 frame 时整段桥接回站体。
- 不引入复杂物理模拟，残骸物理简化为「惯性漂移 + 阻尼」。
- 不强制玩家回收；放弃残骸是合法选择，但「试图回收」要有可读路径与明确反馈。

### 1.3 关键设计风险（高优先级）

1. **坐标系交叉**：fragment 离开 station grid 后，需要独立的 pos / angle / vel / angularVel，再合并回 station 时要做 grid snap。
2. **连通性 BFS 复用**：`checkConnectivity` 现在产生「灰色 cell」；新版要在同一个出口把灰色 cell 转抽离 fragment。
3. **桥接边界条件**：候选 cell 的世界距离阈值、角度阈值、grid 冲突需要明确拒绝路径，否则容易出现「整段消失 / 整段重叠」。
4. **回归**：v0.1.x / v0.2.0 / v0.2.1 验收路径里大量遍历 `state.station.cells`，新版要保证 fragment cell 不进入这些循环，且 detached cell 不再留在 `state.station.cells`。

## 2. 设计决定逐项（对应制作人方向 §「待评估关键技术决定」8 项）

### 2.1 fragment 数据结构与生命周期

- **候选方案**
  - A. 复用现有 `Map<"x,y", Cell>` 的 detached 标记，仅在渲染 / HUD 层补「飞出去」表现。
  - B. 新增 `state.fragments[]`，每个 fragment 持有 cells 子集 + 自己的 pos/vel/angle/angularVel。
  - C. 把每一片 cell 转成完全独立的「散件」单元，扔掉 grid 结构，仅靠世界坐标维护。
- **推荐：B**。复用 grid 结构最小代价支持「整段桥接回站」；A 无法做到真正「飞出去」，C 让重连失去 grid 锚点。
- **理由**：保持原 cell 字段不变，便于复用渲染与运行时数据；fragment 用统一壳子承载 station 外的「连通子图」。
- **风险**：fragment.cells 的 key 必须固定为「脱落瞬间的相对 grid 坐标」；如果中途让它「自由变形」，重连时 grid snap 会失败。约定 fragment 一经创建，cell 内部 `x,y` 不再改变。
- **回滚**：保留 `cell.detached + cell.drift` 字段不删除（兼容字段），如本版本 fragment 流程出现重大缺陷，可禁用 fragment 抽离、回退到现行「灰色 cell」表现，仅丢失新版桥接体验。

### 2.2 station ↔ fragment 坐标系切换

- **脱落瞬间**：从 `checkConnectivity` 的「未访问 cell 集合」按 4 邻接再 BFS 拆出多个连通分量，每个连通分量生成一个 fragment；fragment 初始 `angle = station.angle`、`pos = 该分量所有 cell 世界中心的几何均值`、`vel = station.vel + 平均向外推力`、`angularVel = 平均切向速度`（细节见 §4.2）。
- **重连瞬间**：玩家扩展 frame 的目标 grid 是 `(gx, gy)` (站体本地)。从 fragment 中找出最近候选 cell `c`，把 fragment 整体做 grid snap：
  - 把 fragment.angle 吸附到 station.angle（差异在 §2.4 阈值内）；
  - 求 fragment 整体 grid offset = `(gx - c.x, gy - c.y)` (以站体本地 grid 为基准)；
  - 对 fragment 所有 cell，新 station-grid key = `(cell.x + dx, cell.y + dy)`；
  - 检查每个 key 是否与现有 `state.station.cells` 冲突。
- **冲突策略（推荐）**：「全部接入或拒绝」。任何一个目标 key 已有 cell → 直接拒绝并提示「桥接位置被占用」。不做部分接入，避免新一轮孤岛产生连环 detached。
- **风险**：玩家在 fragment 重叠角度上修建会卡住；ADR 建议 HUD 同步标出「桥接预览框」（在 numerics 决定阈值后再实现）。
- **回滚**：如果冲突拒绝过于严苛，可在后续版本扩展「自动 grid offset 微调」（在不超过 ±1 grid 的范围内寻找无冲突 offset）。

### 2.3 物理简化策略

- **fragment ↔ station 碰撞**：不参与。fragment 仅做漂移 + 阻尼，不与站体反弹。理由：物理碰撞重做超出本版本范围，且容易导致玩家无法靠近 fragment。
- **fragment ↔ 敌方弹道**：参与。fragment cell 可被敌方弹道命中减 HP；hp 归零的 cell 从 fragment 中删除；fragment cells 为空 → fragment 整体销毁。理由：让「残骸继续在战斗中被吃掉」是制作人方向中明确希望的反馈。
- **fragment ↔ 友方弹道 / 维修无人机**：不参与。friendly 弹道直接穿过，维修无人机继续显式跳过 fragment cell（数值策划可考虑后续允许有限维修；默认不开）。
- **fragment ↔ 敌方实体接触**：不参与。fragment 不阻挡敌人路径、不被敌人撞击产生反弹。理由：避免桥接判定时被推走，也避免 fragment 变成「移动盾牌」破坏现有 collide 平衡。
- **fragment ↔ 小行星 / 行星**：不参与。与制作人方向「不与小行星 / 行星实体碰撞」一致。
- **fragment 间碰撞**：不参与。多个 fragment 可重叠，玩家视觉上能区分（不同 fragment 给不同微小色调或外环。具体由 numerics）。
- **风险**：fragment 不阻挡敌方碰撞会让玩家觉得「敌人穿模」，但与「最小物理」目标一致；如果体验测试中明显出戏，再升级为「fragment 阻挡敌人但不参与桥接」。

### 2.4 桥接判定算法

- **触发时机**：仅在 `buildAt(x, y, "frame")` 成功扩展 frame 后调用 `tryBridgeAt(x, y)`。其它建造 / 升级路径不触发桥接，避免和现有 reconnectDetached 行为冲突。
- **候选距离阈值**：制作人方向给定 `~CELL * 1.2`；ADR 推荐为可调常量 `BRIDGE_DISTANCE = CELL * 1.2`，便于 numerics 调参。
- **角度阈值**：fragment.angle 与 station.angle 差异 ≤ `BRIDGE_ANGLE`（推荐 25° ≈ 0.44 rad；超出则不允许桥接，玩家需要把 fragment「转回来」——但因为 fragment 没有玩家控制，实际效果是「等到 fragment 自转到合适角度」）。numerics 可决定是否允许 90° 整数倍对齐（先不开）。
- **是否允许多 fragment 同时桥接**：v0.3.0 不允许。`buildAt` 一次只接入一个 fragment（按 fragment cell 总数最大、距离最近的排序选第一个）。理由：多 fragment 同时桥接会让 grid snap 冲突检测变得指数复杂，且玩家难以预测。
- **是否允许桥接到 fragment 内部 cell**：不允许，仅匹配 fragment「边缘 cell」（即 fragment 内 4 邻接缺失至少 1 个方向的 cell）。理由：内部 cell 物理上没有空位与 station 接触；只考虑边缘 cell 能大幅减少候选集。
- **候选选择逻辑**：
  1. 在所有 fragment 中，过滤出与 `(x, y)` 世界距离 ≤ `BRIDGE_DISTANCE` 的边缘 cell；
  2. 按 fragment 分组，每个 fragment 取一个最近的候选 cell；
  3. 在剩余 fragment 中，按 (cell 总数 desc, 候选距离 asc) 选第一个 fragment；
  4. 用该 fragment + 候选 cell 计算 grid offset，做 §2.2 描述的 merge。
- **风险**：玩家可能希望「精确接到某一个 fragment cell」，但不会有 UI 选择；通过候选 cell 距离 asc 排序，最近 cell 通常符合玩家直觉。
- **回滚**：v0.3.0 上线后若反馈不准，可在 v0.3.1 增加「按鼠标 hover 高亮预览候选 fragment」。

### 2.5 重连后状态恢复

- **保留字段**（merge 完成后照搬，不重置）：
  - `facility`（设施类型，决定接入后是否仍是 turret / missile / mining 等）；
  - `hp` `frameHp` `maxHp`（保留耐久，避免「桥接还能回血」滥用）；
  - `enabled` `priority`（玩家手动禁用状态保留）；
  - `detached = false`（一定要清零）；
  - `drift = null`（一定要清零）；
- **归零字段**（避免运行时累积异常）：
  - `reload`（重新进入站体后从 0 开始计算冷却，避免桥接瞬间齐射）；
  - `fire`（视觉脉冲清零，下一帧自然重算）；
  - `active`（下一帧 `updatePower` 自然重算，不必手动设值）；
- **风险**：保留 hp / frameHp 与「无人机不维修 fragment」呼应；如果玩家觉得「桥接回来还是残血」太苛刻，可由 numerics 给一次性「桥接维修」buff（不在 v0.3.0 默认范围）。
- **回滚**：保留 / 归零策略集中在 `mergeFragmentToStation` 一个函数里，调整成本低。

### 2.6 性能预算

- **fragments 上限**：≤ **8**（制作人建议）。超过上限触发「平衡兜底」，按生成时间排序销毁最早的 fragment 并 toast 提示（与方向 §「平衡与可读性兜底」一致）。
- **单 fragment cells 上限**：≤ **32**。理由：现有站体常见规模 30~60 cell，单次脱落连通分量通常 < 32；上限只用于保护，触发上限直接销毁 fragment 多余 cell（落 hp = 0 自然销毁）。
- **每帧 budget**：
  - `updateFragments`: O(fragments × cellsPerFragment) ≈ 8 × 32 = 256，无问题；
  - `tryBridgeAt`: 仅在 buildAt 成功扩展 frame 时触发（玩家驱动），单次 O(fragments × edgeCells) ≤ 256；
  - `checkConnectivity + splitDetachedIntoFragments`: O(总 cells) BFS，触发频率约「每次 frame 删除」，单次 < 200 节点；
  - 渲染：station ≤ 60 cells + fragments ≤ 8 × 32 = 256 cells，总 < 320 cell rect，对当前 Canvas2D pipeline 无压力。
- **风险**：极端「连环 detached」场景下 fragment 数会冲到上限；用兜底销毁保证不爆。

### 2.7 渲染管线

- **复用还是新增**：**抽出公共函数 `renderCellAt(cell, originPos, angle, opts)`**，由 `renderStation` 与新增 `renderFragments` 共同调用；不要在 `renderStation` 内塞 fragment 渲染。
- **新增 renderFragments**：遍历 `state.fragments`，每个 fragment 用自身 `pos/angle`，调用同一 `renderCellAt`；opts 控制视觉差异。
- **视觉差异建议（参考量，数值由 numerics 拍板）**：
  - cell 主体 alpha = 0.55；颜色去饱和 ×0.7；
  - 外环虚线（fragment 整体）：颜色 [0.92, 0.6, 0.32, 0.8]，dash ≈ `CELL * 0.3`；
  - 闪烁：每秒 ~1.5 Hz 的低幅 alpha 抖动，告诉玩家「这是漂浮中的残骸」；
  - 不再渲染单 cell 血条（fragment cell 的 hp 已是「过去时」，避免 HUD 噪声）。
- **桥接预览**：当 `buildAt` 准备扩展 frame 的鼠标 hover 落在 `BRIDGE_DISTANCE` 内 fragment 候选边缘时，在 hover 格画一圈高亮 + 在候选 fragment 上画整段「将要回归」轮廓。建议作为子任务 (gameplay)，可在 v0.3.0 主版本 cut 掉，留作 v0.3.1 增强。
- **多 fragment 旋转**：天然支持。每个 fragment 各自维护 angle / angularVel。

## 3. 数据结构草案

### 3.1 新增 state 字段

```js
state.fragments = [];
```

放在 `state` 顶层（与 `state.station` 同级），独立维护、不写入 SAVE_KEY。

### 3.2 Fragment 形态

```js
{
  id: number,
  pos: { x: number, y: number },
  vel: { x: number, y: number },
  angle: number,
  angularVel: number,
  cells: Map<"x,y", Cell>,
  spawnAt: number,
  bounds: { minX: number, minY: number, maxX: number, maxY: number },
  edgeKeys: Set<"x,y">
}
```

- `id`：自增 id，方便日志 / 调试 overlay 区分 fragment；
- `pos / vel / angle / angularVel`：fragment 自身世界态；
- `cells`：原 cell 完整结构搬迁（含 `x,y` 字段，仍为「脱落瞬间的本地 grid 坐标」）；
- `spawnAt`：`state.time` 作时间戳；用于「超龄 / 超上限」时按最早销毁；
- `bounds`：本地 grid 包围盒，加速 tryBridge 候选过滤；
- `edgeKeys`：本地 grid 的边缘 cell key 集合（4 邻接至少缺 1）；merge 完成或 cell 被打掉时增量维护；如代价高也可每次 split 时一次性算好（fragment 不再增长，cell 减少时重算 edgeKeys 即可）。

### 3.3 Cell 兼容字段保留

`createCell` 新增字段不变；保留 `detached` 与 `drift` 字段（默认 `false / null`），仅作兼容标记，新版不再在 `state.station.cells` 中长期保留 `detached=true` 的 cell。

## 4. 关键算法伪代码

### 4.1 `splitDetachedIntoFragments(unreachableKeys)`

```js
// 由 checkConnectivity 调用，参数为 BFS 找出的所有「无法连通 core」的 station.cells key 集合
function splitDetachedIntoFragments(unreachableKeys) {
  if (unreachableKeys.size === 0) return;
  // 1. 在 unreachableKeys 内做 4 邻接 BFS，拆出多个连通分量
  const visited = new Set();
  const components = [];
  for (const startKey of unreachableKeys) {
    if (visited.has(startKey)) continue;
    const comp = [];
    const queue = [startKey];
    while (queue.length) {
      const k = queue.shift();
      if (visited.has(k) || !unreachableKeys.has(k)) continue;
      visited.add(k);
      comp.push(k);
      const cell = state.station.cells.get(k);
      for (const n of neighbors(cell.x, cell.y)) {
        const nk = key(n.x, n.y);
        if (unreachableKeys.has(nk) && !visited.has(nk)) queue.push(nk);
      }
    }
    components.push(comp);
  }
  // 2. 每个连通分量 -> 一个 fragment
  for (const comp of components) {
    const fragment = createFragmentFromKeys(comp);
    state.fragments.push(fragment);
  }
  // 3. fragments 上限兜底
  enforceFragmentBudget();
  if (components.length) {
    pushFragmentAlert(`${components.length} 段结构脱落`);
  }
}

function createFragmentFromKeys(compKeys) {
  const cells = new Map();
  const centers = [];
  // 在抽离前先记录每个 cell 的世界位置 + 速度估算
  for (const k of compKeys) {
    const cell = state.station.cells.get(k);
    const worldP = cellWorldPosition(cell); // 利用现有函数读站体坐标系
    centers.push(worldP);
    cell.detached = false;  // fragment 内部统一视作「连接」(由 fragment 自身管)
    cell.drift = null;
    cells.set(key(cell.x, cell.y), cell);
    state.station.cells.delete(k);
  }
  const pos = mean(centers);
  const vel = {
    x: state.station.vel.x + rand(-12, 12),
    y: state.station.vel.y + rand(-12, 12)
  };
  // 估算 angularVel：取 fragment 几何中心到各 cell 的切向 + 一点 random
  const angularVel = rand(-0.6, 0.6);
  const bounds = computeBounds(cells);
  const edgeKeys = computeEdgeKeys(cells);
  return {
    id: ++FRAGMENT_ID_SEQ,
    pos,
    vel,
    angle: state.station.angle,
    angularVel,
    cells,
    spawnAt: state.time,
    bounds,
    edgeKeys
  };
}
```

### 4.2 `updateFragments(dt)`

```js
function updateFragments(dt) {
  for (const fragment of state.fragments) {
    fragment.pos.x += fragment.vel.x * dt;
    fragment.pos.y += fragment.vel.y * dt;
    fragment.angle += fragment.angularVel * dt;
    fragment.vel.x *= 1 - Math.min(FRAGMENT_LINEAR_DAMP * dt, FRAGMENT_LINEAR_DAMP);
    fragment.vel.y *= 1 - Math.min(FRAGMENT_LINEAR_DAMP * dt, FRAGMENT_LINEAR_DAMP);
    fragment.angularVel *= 1 - Math.min(FRAGMENT_ANGULAR_DAMP * dt, FRAGMENT_ANGULAR_DAMP);
    // 边界限位：超过 GALAXY_WORLD_BOUNDS 外扩 200 即销毁
    if (isOutOfWorld(fragment.pos)) {
      destroyFragment(fragment, "drifted-out");
    }
  }
  state.fragments = state.fragments.filter((f) => f.cells.size > 0 && !f.markedDead);
}
```

参考量（待 numerics 拍板）：`FRAGMENT_LINEAR_DAMP = 0.06`、`FRAGMENT_ANGULAR_DAMP = 0.35`。

### 4.3 `tryBridgeAt(x, y)`

```js
// 由 buildAt 在「frame 扩展成功」后立即调用
function tryBridgeAt(x, y) {
  if (!state.fragments.length) return;
  // station 本地 grid (x, y) 对应的世界点（fragment 与 station 都在同一世界坐标）
  const worldP = cellWorldPosition({ x, y, detached: false }); // 复用站体坐标变换
  let best = null;
  for (const fragment of state.fragments) {
    // 角度过滤
    const angleDiff = Math.abs(normalizeAngle(fragment.angle - state.station.angle));
    if (angleDiff > BRIDGE_ANGLE) continue;
    // 选 fragment 内最近的边缘 cell
    let nearest = null;
    let nearestDist = BRIDGE_DISTANCE;
    for (const ek of fragment.edgeKeys) {
      const cell = fragment.cells.get(ek);
      const cp = fragmentCellWorldPosition(fragment, cell);
      const d = dist(cp, worldP);
      if (d < nearestDist) {
        nearest = cell;
        nearestDist = d;
      }
    }
    if (!nearest) continue;
    const candidate = {
      fragment,
      anchor: nearest,
      distance: nearestDist,
      cellCount: fragment.cells.size
    };
    if (!best || candidate.cellCount > best.cellCount
        || (candidate.cellCount === best.cellCount && candidate.distance < best.distance)) {
      best = candidate;
    }
  }
  if (!best) return;
  const merged = mergeFragmentToStation(best.fragment, best.anchor, x, y);
  if (merged) {
    showToast(`重新接入 ${best.fragment.cells.size} 个结构。`);
  } else {
    setBuildError("桥接位置被占用，已扩展框架但残骸未接入。");
  }
}
```

### 4.4 `mergeFragmentToStation(fragment, anchorCell, gridX, gridY)`

```js
function mergeFragmentToStation(fragment, anchorCell, gridX, gridY) {
  const dx = gridX - anchorCell.x;
  const dy = gridY - anchorCell.y;
  // 1. 冲突预检：先复制 keys，不修改 station.cells
  const projected = [];
  for (const cell of fragment.cells.values()) {
    const newKey = key(cell.x + dx, cell.y + dy);
    if (state.station.cells.has(newKey)) return false; // 全部接入或拒绝
    projected.push({ cell, newKey, nx: cell.x + dx, ny: cell.y + dy });
  }
  // 2. 写入站体
  for (const item of projected) {
    item.cell.x = item.nx;
    item.cell.y = item.ny;
    item.cell.detached = false;
    item.cell.drift = null;
    item.cell.reload = 0;
    item.cell.fire = 0;
    // hp / frameHp / facility / enabled / priority 保留
    state.station.cells.set(item.newKey, item.cell);
  }
  // 3. 标记 fragment 销毁
  fragment.cells.clear();
  fragment.markedDead = true;
  // 4. 触发连通性复检（防御性，理论上 anchor 已连，整段一定连）
  checkConnectivity();
  return true;
}
```

## 5. 渲染策略

- 抽 `renderCellAt(cell, transform, opts)`：
  - `transform = { pos, angle }`；
  - `opts = { alpha, colorScale, drawHpBar, drawDashedRing }`；
  - 内部完成 rect / 描边 / facility 标志（core 圆点、thruster 火焰、shield 圆环、selected 圈）；
- `renderStation`：保持原签名，循环每个 cell 时把 `transform = { pos: state.station.pos, angle: state.station.angle }` 传入；
- `renderFragments`：遍历 `state.fragments`，先在 fragment 包围盒外画虚线轮廓；再对每个 cell 用 fragment 自己的 transform 渲染，`alpha 0.55、colorScale 0.7、drawHpBar=false`；
- 渲染顺序：fragment 在站体之下（避免遮挡站体战斗反馈），fragment 之上画外环虚线。

## 10. 死代码清理建议

- **`shouldFire` 第二条 ternary 死分支**（`Game/game.js:1839`）：
  - 代码：`const shouldFire = targetVector ? dot(pushWorld, targetVector) > 0.12 : keyboardThrust ? length(station.vel) > 3 : false;`
  - 由于上文 `targetVector = keyboardThrust ? keyboardThrust : (state.target ? normalize(...) : null);`，当 `keyboardThrust` truthy 时 `targetVector === keyboardThrust` truthy，必走第一条分支；当 `keyboardThrust` 为 null 时 `targetVector` 也只能是 state.target 推导值或 null，进入 ternary 第二条时 `keyboardThrust` 一定为 null/false，仍然只能走 `: false`。因此 `keyboardThrust ? length(station.vel) > 3 : false` 永不可达，可直接简化为 `targetVector ? dot(pushWorld, targetVector) > 0.12 : false`。
  - **建议**：作为 T6 顺手清理，行数极小，不增加回归面。

- **`updateShield`（`Game/game.js:2320`）**：
  - 制作人方向中说「实现但未在主循环被调用（疑似死代码）」**与代码现状不符**。代码确认 `updateShield(cell)` 在 `Game/game.js:2206` 由站体设施主循环按 facility=="shield" 调用。
  - **建议**：**不清理**。仅在 ADR 中纠正这一事实，由 PM 把这条更新同步给制作人方向文档（不直接修改方向文档）。
