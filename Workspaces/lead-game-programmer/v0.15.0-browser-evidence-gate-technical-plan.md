## 2. 浏览器取证路径排序

### P0：真实浏览器人工 DevTools

首选路径。适用于 PM、制作人或执行成员直接在 Edge/Chrome 中打开游戏，人工完成 DevTools Console 调用、P0 路径和三视口记录。

入口：

```powershell
Start-Process "msedge.exe" "file:///D:/work/project/agentgame/Game/index.html?playtest=1"
```

若 `msedge.exe` 不在 PATH，改用本机已存在的完整路径，例如：

```powershell
Start-Process "${env:ProgramFiles(x86)}\Microsoft\Edge\Application\msedge.exe" "file:///D:/work/project/agentgame/Game/index.html?playtest=1"
```

Chrome 等价入口：

```powershell
Start-Process "chrome.exe" "file:///D:/work/project/agentgame/Game/index.html?playtest=1"
```

验证方式：

1. 打开 DevTools。
2. 确认 Console 无初始化阻塞 error。
3. 在 Console 执行 `runReleaseCandidateChecks({ sampleFrames: 30 })` 取 JSON。
4. 人工记录 P0 路径和三视口。

优点：最接近真实玩家浏览器，能同时观察 Console、DOM、视口、手感和可读性。

限制：人工复制 JSON 和截图容易漏字段，必须按 PM 模板记录。

### P1：现有 Chrome/Edge 远程调试 / CDP

仅当本机已存在 Edge/Chrome，且不需要安装任何工具时使用。CDP 只作为取证辅助，不替代人工体验。

允许临时启动浏览器命令：

```powershell
$GameUrl = "file:///D:/work/project/agentgame/Game/index.html?playtest=1"
$Profile = Join-Path $env:TEMP "ssds-v015-cdp-profile"
$Edge = "${env:ProgramFiles(x86)}\Microsoft\Edge\Application\msedge.exe"
$Proc = Start-Process -FilePath $Edge -ArgumentList @(
  "--remote-debugging-port=9222",
  "--user-data-dir=$Profile",
  "--new-window",
  $GameUrl
) -PassThru
```

超时策略：

- 启动后最多等待 10 秒确认 `http://127.0.0.1:9222/json/version` 可访问。
- 单次取证窗口建议不超过 10 分钟。
- 若 10 秒内端口不可访问，记录为 CDP 不可用，回到 P0 人工 DevTools 或 P2。

清理策略：

```powershell
Stop-Process -Id $Proc.Id -ErrorAction SilentlyContinue
Remove-Item -Recurse -Force $Profile -ErrorAction SilentlyContinue
```

CDP 可用性检查：

```powershell
Invoke-RestMethod "http://127.0.0.1:9222/json/version"
Invoke-RestMethod "http://127.0.0.1:9222/json/list"
```

CDP 取证要求：

- 可以使用已有 CDP 客户端、浏览器内 DevTools Protocol 工具或 PowerShell/.NET 临时 WebSocket 脚本。
- 不允许为 CDP 安装新 npm/pip 包。
- CDP 只记录浏览器级 JSON、console、DOM 或截图证据；P0 人工路径仍需人工结论。
- 若 CDP 能设置 viewport，可使用 `Emulation.setDeviceMetricsOverride` 覆盖 `760x640`、`420x740`、`390x844` 并记录参数。

### P2：file URL + `?playtest=1`

这是 P0/P1 的默认页面入口，不依赖本地静态服务：

```text
file:///D:/work/project/agentgame/Game/index.html?playtest=1
```

使用要求：

- 不启动本地服务。
- 不修改浏览器全局设置。
- 不写入游戏规则、数值、存档 schema 或测试 hook。
- 若浏览器因 file URL 安全策略导致 clipboard、localStorage 或 module 相关能力异常，记录具体字段和失败现象。

### P3：失败时 Node 静态取证 fallback

仅在真实浏览器 JSON 或 CDP 都无法取得时使用，用于定位文件、静态 hook 名称和可读字段，不作为浏览器通过证据。

允许范围：

- 使用仓库已有文件做静态扫描。
- 使用系统已存在的 Node 读取文本、正则定位函数名和导出入口。
- 不安装 jsdom、不模拟 DOM、不把 Node 结果写成 browser pass。

fallback 结论口径：

- 可以支持“hook 存在性静态确认”。
- 不能支持“真实浏览器运行态 JSON 已通过”。
- 不能支持“三视口已覆盖”。
- 不能支持“P0 人工路径已通过”。
- 若只能到 P3，PM #9 只能建议继续补证。

### 3.3 证据文件格式建议

PM 工作区建议保留两类文件：

- 摘要 Markdown：`Workspaces/pm-qa-coordinator/v0.15.0-browser-evidence-gate-evidence.md`

Markdown 摘要建议格式：

```markdown
### RC JSON 证据 <timestamp>

- 证据类型：真实浏览器 / CDP / file URL / Node fallback。
- 浏览器：Edge/Chrome 版本或可识别 UA。
- 页面入口：file URL + 查询参数。
- 视口：宽 x 高；设置方式。
- 调用方式：DevTools Console / releaseCandidate panel / CDP Runtime.evaluate。
- JSON 文件：相对路径。
- `ok`：
- `durationMs`：
- `storageSignature`：before/after 是否一致；save/corruptRaw hash。
- `runtimeStateSignature`：before/after 是否一致；若不一致，是否为自然帧推进。

- 结论：通过 / 失败 / 未覆盖 / 阻塞 / 非阻塞风险。
```

## 4. 字段验证口径

### 4.1 `storageSignature`

检查位置：

- `result.snapshotBefore.storageSignature`
- `result.snapshotAfter.storageSignature`
- `result.checks[]` 中 `id === "storage.unchanged"`
- `result.residualRisks[]` 中是否存在 `kind === "storage_mutation"`

通过条件：

- `snapshotBefore.storageSignature.save.hash32 === snapshotAfter.storageSignature.save.hash32`
- `snapshotBefore.storageSignature.corruptRaw.hash32 === snapshotAfter.storageSignature.corruptRaw.hash32`
- `checks.storage.unchanged` 为 `pass`
- 无 `storage_mutation`

失败口径：

- 任一 signature 变化，证据可信度阻塞。
- 立即停止把本次 JSON 当作通过证据。
- 记录浏览器、入口、操作、取证时间和前后 hash，转 T4 缺陷复核。

### 4.2 `runtimeStateSignature`

检查位置：

- `result.snapshotBefore.runtimeStateSignature`
- `result.snapshotAfter.runtimeStateSignature`
- `result.checks[]` 中 `id === "runtimeState.signatureStable"`
- `result.residualRisks[]` 中是否存在 `kind === "runtime_state_drift"`

通过条件：

- 关键运行态签名一致，或仅存在自然帧推进可解释变化。

注意：

- `runReleaseCandidateChecks` 会等待 rAF 采样，游戏时间可能自然推进。
- `runtime_state_drift` 默认是 warn，不直接等价于阻塞；但如果资源、敌人、projectiles、run 关键字段异常跳变，必须记录为可信度风险。

### 4.3 `frameTiming`

检查位置：

- `result.snapshotAfter.frameTiming`
- `result.checks[]` 中 `frameTiming.available`、`frameTiming.longFrame`

通过条件：

- `available === true`
- `sampleCount` 接近或等于 30
- `mode` 为 `rAF` 或可解释的浏览器采样模式
- 记录 `avgMs`、`maxMs`、`longFrameCount`、`longFrameThresholdMs`

失败/未覆盖口径：

- `available === false`：本次缺少真实帧耗时证据。
- `longFrameCount > 0`：不直接阻塞，但必须结合人工体验和视口状态判断。
- Node fallback 的 frame timing 不可替代浏览器 rAF。

### 4.4 `runtimeErrors`

检查位置：

- `result.snapshotAfter.base.runtimeErrors.count`
- `result.snapshotAfter.base.runtimeErrors.latest`
- DevTools Console 手工观察

通过条件：

- 自动 buffer 中无阻塞 runtime error。
- DevTools Console 无未捕获异常、Promise rejection 或脚本中断。

注意：

- 自动 buffer 不等价于完整 DevTools Console。
- PM 摘要必须单独写 Console 面板观察结果。

### 4.5 key DOM

检查位置：

- `result.snapshotAfter.base.keyDom`
- `result.snapshotAfter.hud.keyDom`
- `result.checks[]` 中 `id === "keyDom.present"`

关键 DOM 包括：

- HUD：`hud`、`runInfo`、`resources`、`pauseBtn`
- 引导：`currentGoal`、`nextStep`、`statusAlerts`、`resourceGuide`
- 建造/状态：`selectedCellPanel`、`selectedCellInfo`、`selectedCellDiagnostics`
- 战斗/复盘：`combatStatusSummary`、`combatReviewSummary`
- 星系/目标：`galaxyMapPanel`、`objectiveJumpBtn`、`objectiveJumpDeferredBtn`
- Meta/结算：`metaPanelToggle`、`runSettlementPanel`、`runSettlementStats`、`summaryStayBtn`

失败口径：

- 缺失关键 DOM 或关键按钮不可见时，按 S1/S2 判断是否阻塞 P0。
- 不能通过改验收规则绕过缺失 DOM；必须记录实际 UI 状态。

### 4.6 `hookStatus`

检查位置：

- `result.snapshotAfter.hookStatus`
- `checks[]` 中 `design.hooks`、`combat.hooks`、`build.hooks`、`meta.hooks`、`galaxy.hooks`

通过条件：

- design/combat/build/meta/galaxy 均 `available === true`
- `missingHooks`、`missingFields` 为空或仅有可解释非阻塞项

失败口径：

- hook 不可用导致 JSON 无法证明关键路径时，按证据路径缺陷处理。
- hookStatus 失败不允许靠手改 JSON 或 PM 口头说明覆盖。

### 4.7 evidence layers

检查位置：

- `result.snapshotAfter.evidence.layers.automatic`
- `result.snapshotAfter.evidence.layers.browser`
- `result.snapshotAfter.evidence.layers.manual`
- `result.snapshotAfter.evidence.coverage`
- `result.residualRisks` 中 `browser_uncovered`、`manual_uncovered`

通过解释：

- automatic：自动检查层。
- browser：真实浏览器环境、viewport、headlessHint 等浏览器级证据。
- manual：P0 人工路径，默认不能由自动检查标记通过。

闸门口径：

- automatic 全 pass 仍不能替代 manual。
- 有真实浏览器 JSON 但没有 P0 人工路径，PM #9 只能建议继续补证。
- 有 P0 体验但没有真实浏览器 JSON，也只能建议继续补证。

## 5. 三视口证据采集

目标视口：

- `760x640`
- `420x740`
- `390x844`

### 5.1 人工调整窗口

适合 P0 人工 DevTools 路径。

步骤：

1. 打开 `file:///D:/work/project/agentgame/Game/index.html?playtest=1`。
2. 手工调整窗口接近目标视口。
3. 在 Console 执行：

```javascript
window.innerWidth + "x" + window.innerHeight
```

1. 记录实际 viewport。若无法精确达到目标，写“等价近似”并保留实际尺寸。
2. 在每个视口下至少记录 HUD、建造面板、战斗提示、复盘、星系图、Meta 入口的可见性、滚动、可点性。

限制：

- Windows 外框尺寸不等于 viewport，必须以 `window.innerWidth/innerHeight` 或 JSON 中 `base.viewport` 为准。

### 5.2 DevTools Device Toolbar

推荐用于精确 viewport。

步骤：

1. 打开 DevTools。
2. 切换 Device Toolbar。
3. 选择 Responsive。
4. 分别输入 `760 x 640`、`420 x 740`、`390 x 844`。
5. 每个视口执行：

```javascript
const result = await window.__gameTest.playtest.runReleaseCandidateChecks({ sampleFrames: 30 });
copy(JSON.stringify(result, null, 2));
result.snapshotAfter.base.viewport;
```

记录：

- `base.viewport.width`
- `base.viewport.height`
- `devicePixelRatio`
- `narrowViewport`
- HUD/面板是否遮挡
- 关键按钮是否可点
- 面板是否可关闭
- 复盘/星系/Meta 是否可滚动

### 5.3 CDP emulation

仅当 P1 CDP 已可用时使用。建议 CDP 命令：

- `Emulation.setDeviceMetricsOverride`
- `Runtime.evaluate`
- `Page.captureScreenshot`
- `Emulation.clearDeviceMetricsOverride`

每个 viewport 执行同一段 Runtime.evaluate：

```javascript
JSON.stringify(await window.__gameTest.playtest.runReleaseCandidateChecks({ sampleFrames: 30 }))
```

限制：

- CDP emulation 不等于真实触控设备。
- 若没有真实触控设备，本轮必须写“真实触控未覆盖，保留残留风险”。

### 5.4 不可用时的记录

如果任一视口无法设置：

- 不得用默认窗口结果替代该视口。
- 在 PM 证据中标记“未覆盖”。
- 写明失败方式：无法调整窗口、DevTools 不可用、CDP 不可用、截图失败、JSON 调用失败等。

## 6. 缺陷修复边界

如果在 T1-T3 发现 S0/S1/S2 缺陷，流程如下：

1. 先记录证据：浏览器、入口、视口、操作步骤、实际结果、JSON 字段、Console error、截图或复制摘要。
2. 停止把该路径记为通过。
3. PM 标记严重程度和是否阻塞。
4. 制作人判断是否纳入 v0.15.0 小修。
5. 另派执行任务做最小修复。
6. 修复后必须用原浏览器、原入口、原视口、原操作路径复测。
7. 复测 JSON 和人工记录进入 PM 证据包。

严重程度口径：

- S0：游戏无法启动、核心路径无法进入、`runReleaseCandidateChecks` 完全不可用、真实浏览器 JSON 无法产出且无可行回退。
- S1：P0 路径无法完成、关键 UI 不可点击、关键状态错误、阻塞 runtime error。
- S2：证据不可信、关键 DOM 缺失、小屏核心信息被遮挡、只读字段明显错误、复制摘要丢失关键失败项。

禁止借取证改规则：

- 不改 `SAVE_KEY`。
- 不改 meta schema。
- 不改核心数值、伤害、敌人 AI、RNG、资源公式、维修/护盾/脱落判定。
- 不为通过证据伪造战斗结果、Meta 进度或存档。
- 不把手感 polish 混入 S0/S1/S2 修复。

## 7. T1-T4 节点审查

### T1：真实浏览器运行态 JSON

对应 PM #4。

审查项：

- 使用 P0 或 P1 路径取得真实浏览器 JSON。
- JSON 来自 `runReleaseCandidateChecks({ sampleFrames: 30 })`。
- 记录浏览器、入口、视口、调用方式。
- `storageSignature` 前后一致。
- `frameTiming` 有 30 帧左右真实浏览器采样，或明确未覆盖。
- `runtimeErrors` 和 DevTools Console 已分别记录。
- `hookStatus` 无阻塞缺口。
- `evidence.layers` 明确 automatic/browser/manual coverage。

停止条件：

- 无法取得真实浏览器 JSON。
- `storage_mutation`。
- `keyDom.present` blocking fail。
- `hookStatus` blocking fail。
- 需要安装依赖或改系统设置。

回滚/处理：

- T1 本身不应修改代码；失败时记录为继续补证或转 T4。
- 临时浏览器进程按 §2 P1 清理。

### T2：P0 人工路径

对应 PM #5。

审查项：

- 空存档首局可进入。
- 建造/采矿链路可完成。
- 资源不足/断电反馈可理解。
- 敌人接近提示可见。
- 武器/护盾/维修状态可读。
- 战斗后复盘可读。
- 星系跳转入口和结果可理解。
- Meta 能回到新局且状态不混乱。
- 每个节点标记通过、失败、未覆盖、阻塞或非阻塞风险。

停止条件：

- 没有真实浏览器人工记录。
- 任一 S0/S1 阻塞。
- 人工路径只能靠 DevTools 改状态才能通过。

回滚/处理：

- 不回滚游戏状态作为证据修复方式。
- 记录起始条件和可能的存档污染；必要时换临时 profile 或清理浏览器测试数据，但必须在 PM 记录中说明。

### T3：三视口与复制摘要

对应 PM #6。

审查项：

- `760x640`、`420x740`、`390x844` 均有记录或明确未覆盖。
- 每个视口记录 viewport 设置方式和实际 JSON 中的 `base.viewport`。
- HUD、建造、战斗提示、复盘、星系、Meta 入口均被检查。
- 复制摘要或 JSON 保存路径可复查。
- 真实触控未覆盖时写入残留风险。

停止条件：

- 视口无法设置且未记录原因。
- 某视口核心按钮不可点击或关键面板不可关闭。
- 复制摘要遗漏失败项、runtime error 或未覆盖项。

回滚/处理：

- T3 证据采集不修改代码。
- 若需 UI 修复，转 T4，修复后按同一视口复测。

### T4：阻塞缺陷小修

对应 PM #7。

审查项：

- 缺陷来自 T1-T3 证据，不是新增愿望。
- 修复范围最小。
- 未触及禁止清单。
- 原失败路径已复测关闭。
- `runReleaseCandidateChecks({ sampleFrames: 30 })` 复测仍能证明 storage、runtime、frameTiming、keyDom、hookStatus、evidence layers。

停止条件：

- 修复需要安装依赖、改系统设置、改浏览器全局配置。
- 修复需要改核心规则、数值、RNG、存档 schema 或发布产物。
- 修复范围扩大为 polish 或新玩法。
- 修复后出现新的 S0/S1。

回滚/处理：

- 由执行任务按其修改文件回滚或补丁修复；本技术方案不直接定义代码回滚命令。
- 禁止 `reset/checkout/clean/stash` 覆盖用户或其他成员工作。
- 若无法安全回滚，停下交给 PM 和制作人判断。

## 8. 发布闸门结论口径

可以建议进入发布准备交接的最低证据：

- 至少一份真实浏览器 JSON。
- 至少一次真实浏览器 P0 人工路径记录。
- 三视口均有证据或明确未覆盖且不构成阻塞。
- 无 S0/S1。
- S2 要么修复并复测，要么由制作人明确接受为非阻塞残留风险。
- PM 证据包区分真实浏览器、CDP/headless、Node fallback、人工体验和未覆盖项。

只能建议继续补证的情况：

- 没有真实浏览器 JSON。
- 没有 P0 人工路径记录。
- 三视口无记录且无失败说明。
- 只能拿到 Node fallback。
- CDP/headless 结果被误当作真实人工体验。

必须停止并升级的情况：

- 需要安装依赖或浏览器驱动。
- 需要修改系统设置、权限、服务或浏览器全局配置。
- 需要发布、打 tag 或归档 `Releases/`。
- 需要改核心规则、数值、RNG、存档 schema、`SAVE_KEY` 或 meta schema。
- 证据显示取证函数自身污染 storage。
