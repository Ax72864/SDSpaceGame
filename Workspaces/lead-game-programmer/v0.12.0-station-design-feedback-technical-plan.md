## 0. 文档定位与边界

本方案只定 **v0.12.0 空间站设计反馈的技术边界**：诊断 helper 的纯函数语义与只读不变量、建造面板/选择/放置/健康摘要四个 UI 接入点的 helper 调用规则、HUD 0.12 秒节流下的缓存策略、`__gameTest.playtest` 子命名空间字段扩展、禁止改动清单、T1-T4 任务拆分与节点审查条件。

**不在本方案决策范围**：

- 卡片字段排版、角色标签视觉、健康摘要短项视觉密度 → UI 方案（PM #2）。
- 设施角色映射表、诊断阈值具体数字、短诊断文案话术 → 数值/文案方案（PM #4）。
- 体验取舍判断（玩家能否说出空间站短板）→ 制作人 #10 最终决定。
- 浏览器回归证据的记录形式 → PM 看板 #9。

本方案只提供 **接入框架**：诊断 helper 的输入/输出 schema、UI 入口的调用规则、性能与缓存边界、不变量。UI 方案与数值方案产出后按 §3-§7 边界填入即可，不需要回到本方案再决策。

本方案严格遵守：**只允许新增/修改 `Workspaces/lead-game-programmer/v0.12.0-station-design-feedback-technical-plan.md`**，不修改 `Game/`、PM 看板、`42.md`、`Releases/`、`Docs/`、`.cursor/`、其他工作区文件和历史 ADR；不删除文件；不 stage/commit dirty `42.md`。

### 3.2 `getBuildPaletteDiagnostics()` schema

**目的**：给 `createBuildUi` / `updateBuildButtons` 提供每张卡的角色、成本、可负担、已建数量与短推荐。

**输入**：无（读 `state.resources / state.station / state.meta / state.run / TYPES / FACILITY_ORDER`）。

**输出**（示意；具体字段名可由 T1 落地时与 UI 方案对齐，但**不能**新增写操作分支）：

```javascript
{
  generatedAt: state.time,
  facilities: [
    {
      id: "mining",                        // 等于 TYPES key
      name: "采矿站",                       // = TYPES[id].name
      role: "mining",                       // §3.2.1 角色映射
      roleLabel: "采矿",                    // 由 UI/文案方案在数值表中定义；T1 fallback 用 role 原值
      cost: { metal: 28, ore: 10 },         // = getBuildCost("mining")
      affordable: true,                     // 全部资源 >= cost
      missingResources: [                   // 资源差值（空数组表示不缺）
        { name: "metal", need: 28, have: 17, gap: 11 }
      ],
      powerImpact: {                        // 不调用电力结算；查 TYPES 静态字段
        powerOut: 0,                        // = TYPES[id].powerOut || TYPES[id].baseStats.powerOut || 0
        powerUse: 3,                        // = TYPES[id].powerUse || 0
        netDelta: -3                        // powerOut - powerUse
      },
      count: 2,                             // 当前已建该类型（不含 detached）
      countLimitHint: null,                 // 默认 null；未来若有上限可填，当前不引入限制
      recommendation: {                     // 由数值/文案表提供 fallback；本 helper 仅给信号
        priority: "neutral",                // "high" / "medium" / "neutral" / null
        reasonKey: "core_capability_missing" // 见 §3.2.2 reasonKey 白名单
      },
      flags: {
        firstOfKind: false,                 // count === 0
        unaffordableButCloseToAffordable: false  // missingResources.gap 总和 < 10 时为 true
      }
    },
    /* ...frame / power / thruster / processor / plasma / research / turret / missile / shield / armor / repair... */
  ],
  summary: {
    affordableCount: 5,
    totalCount: 11,
    powerHeadroom: 12,                      // state.power.available - state.power.used
    overallPowerTight: false                // 与 buildStatusAlerts 行 10321 同判定
  }
}
```

**约束**：

1. **只读**：函数体内禁止出现 `state.X = ...` / `localStorage.setItem` / `state.station.cells.set` / `cell.active = ...` 等写操作。Lint checklist 见 §8.5。
2. `cost` 必须直接 `getBuildCost(id)`，不重新实现折扣逻辑；否则与实际扣费分叉。
3. `missingResources` 顺序与 `Object.entries(cost)` 一致，便于 UI 显示稳定。
4. `count` 必须排除 `cell.detached`，与 `getMiningStationStatus().miners` 计数口径一致。
5. `recommendation.reasonKey` 是 enum 字符串，**不**返回完整文案；文案查表交给数值/文案方案。
6. 不在 helper 内调用 `setBuildError / showToast / setHtmlIfChanged`，只返回数据。

#### 3.2.1 角色映射（建议默认值）

T1 落地时定义一个常量表（在 `FACILITY_ORDER` 附近）：

```javascript
const FACILITY_ROLE = {
  frame: "support",
  power: "power",
  thruster: "thrust",
  mining: "mining",
  processor: "production",
  plasma: "production",
  research: "research",
  turret: "weapon",
  missile: "weapon",
  shield: "defense",
  armor: "defense",
  repair: "repair"
};
```

数值/文案表（PM #4）若调整 role 命名，本方案不阻塞；但 **role 字面量集合必须保持有限且文档化**，否则健康摘要分组失效。

#### 3.2.2 `reasonKey` 白名单

T1 落地时只允许返回如下 enum（未列入的 reasonKey 必须先增补本节再实现）：

- `null`：无推荐
- `"core_capability_missing"`：当前空间站无该角色任何设施
- `"capability_limited"`：已有该角色但 count < 2 或全部 inactive
- `"power_pressure"`：当前 powerHeadroom < 该 facility 的 powerUse
- `"resource_pressure"`：missingResources 非空但 unaffordableButCloseToAffordable
- `"redundant_high_count"`：count >= 4 且该角色已有 activeMiners / activeTurrets >= 2（仅展示，不阻止建造）

### 3.3 `getSelectedCellDiagnostics(cell)` schema

**目的**：给 `updateSelectedCellPanel` 在现有 stateNote 之外提供结构化的诊断分区。

**输入**：cell 对象（可能为 null，应返回 `null` 而不是抛错）。

**输出**：

```javascript
{
  cellKey: "12,7",                          // cell.key 或 key(x,y)，便于 PM 取证
  facility: "mining",                       // cell.facility
  facilityName: "采矿站",                    // TYPES[cell.facility].name 或 "核心"
  role: "mining",                           // FACILITY_ROLE
  status: {
    enabled: true,                          // cell.enabled
    active: true,                           // cell.active
    detached: false,                        // cell.detached
    powerStarved: false,                    // enabled && !active && powerUse > 0
    nozzleBlocked: false                    // facility==="thruster" && isThrusterNozzleBlocked
  },
  connection: {
    connected: true,                        // !cell.detached
    fragmentSize: 18,                       // 所在 fragment 的 cell 数（用 cell.fragmentId 或现有 fragment 引用）
    isCoreFragment: true                    // 该 fragment 是否包含核心
  },
  range: {                                  // 仅对 mining / turret / missile / shield / repair 返回；其余 null
    kind: "mining",                         // "mining" / "weapon" / "shield" / "repair" / null
    nominalRange: 130,                      // mining 复用 MINING_RANGE_OFFSET；weapon 用 baseStats.range
    currentTarget: { bodyName: "矿带 A", resource: "ore" } | null,
    nearestHint: {                          // mining 用 getNearestResourceBody；其他 facility 暂为 null
      bodyName: "矿带 A",
      distance: 215,
      range: 130,
      gap: 85,
      inRange: false
    } | null
  },
  // T2 落地时如果 turret / missile 类没有现成 LOS helper，此字段必须明确 returns null 而不是猜
  weapon: {
    losAvailable: null,                     // 仅当存在可信 LOS 计算时填 boolean；本版本默认 null
    rangeText: "约 450"                     // 仅展示用，源于 baseStats.range
  } | null,
  repair: {                                 // facility==="repair" 时填；本方案不引入新覆盖判定
    coverageStatus: "unknown",              // "unknown" / "active" / "inactive"，由现有 enabled/active 推导
    note: null                              // 数值/文案表填入；helper 不写死
  } | null,
  reasonKeys: ["power_starved"]             // 复用 §3.2.2 与新增分支：power_starved / nozzle_blocked / detached / out_of_range / target_depleted / disabled_manual
}
```

**约束**：

1. cell == null → 返回 null，调用方按 null 隐藏分区。
2. 不修改 cell 任何字段；不调用 `updateBuildButtons / updateHud`。
3. range / weapon / repair 三个子字段只对应 facility 类返回，**严禁**在 helper 内引入新的范围计算（如新 LOS 算法）。如果当前代码没有 LOS / repair coverage 判定，本字段就返回 `null` / `"unknown"`。
4. reasonKeys 集合允许扩展，但与 §3.2.2 共享白名单语义：未列入需先增补文档。

#### 3.4.1 采样缓存（D6）

```javascript
const DESIGN_HEALTH_SAMPLE_INTERVAL = 0.18;             // 秒；HUD 周期 0.12 的 1.5 倍
let designHealthCache = null;

function getStationDesignHealth() {
  const now = state.time;
  if (designHealthCache && now - designHealthCache.sampledAt < DESIGN_HEALTH_SAMPLE_INTERVAL) {
    return designHealthCache.value;
  }

```

**约束**：

1. 缓存只在 `getStationDesignHealth` / `getResourceReachability` 这两个最重的 helper 上启用；`getBuildPaletteDiagnostics / getSelectedCellDiagnostics / getPlacementDiagnostics` 不缓存（输入会因 selectedCell / hover 频繁变化，缓存反而增加错误率）。

3. `state.paused` 时 `state.time` 不前进，缓存自然冻结，与玩家暂停语义一致。
4. **缓存对象禁止被外部突变**：T1/T4 节点审查必须验证 `getStationDesignHealth() === getStationDesignHealth()`（同一周期内引用相同），且 PM 修改返回对象时不影响后续调用（必要时 `Object.freeze` 顶层 + 子层）。

### 3.5 `getPlacementDiagnostics(facility, gridX, gridY)` schema

**目的**：给"准备建造但尚未点放置"状态提供位置相关诊断。当前游戏在 `state.selectedBuild != null` 时通过 `state.buildHint` / `bridgePreview` 等更新提示；本 helper 把这些零散判定聚合为一个数据对象。

**输出**：

## 5. 严重性阈值与低噪声提示（D9）

严重性阈值是 helper 返回的 `severity` 字段判定规则。数值/文案表（PM #4）可微调具体数字，但 **判定函数体必须由本方案 §5 约束**（避免不同 PR 各自定义阈值）。

| 维度 | severity = "ok" | severity = "warning" | severity = "critical" |
| --- | --- | --- | --- |
| power | `tight === false && starvedCount === 0` | `tight === true` 或 `starvedCount > 0 && starvedCount < 3` | `starvedCount >= 3` 或 `available <= 0` |
| mining | `harvestingCount > 0` | `minerCount > 0 && harvestingCount === 0`（待命 / 远离） | `minerCount === 0 && state.run.level > 0`（开局阶段不算 critical） |
| thrust | `thrusterCount > 0 && activeCount > 0` | `blockedCount > 0` 或 `activeCount === 0 && thrusterCount > 0` | `thrusterCount === 0`（开局可放宽到 warning，由数值表决定） |
| weapon | `activeWeaponCount > 0` | `weaponCount > 0 && activeWeaponCount === 0` | `weaponCount === 0` **且** （`state.run.endgame` 或 `nearestEnemyDistance < 800`；敌袭 / 终末才升 critical） |
| defense | `shieldCount + armorCount > 0` | `shieldCount + armorCount > 0 && activeShieldCount === 0` | `shieldCount + armorCount === 0` **且** `state.run.endgame` 或敌人来袭中 |
| repair | `repairCount > 0` | `repairCount > 0 && activeRepairCount === 0` | 默认不升 critical（repair 缺失不影响立即生死） |

低噪声提示触发：

- `criticalIssues` 数组 push 来源 = 每个维度 severity = "critical"。
- 低噪声 alerts 频率上限：同一 reasonKey 在 8 秒内只触发一次 `statusAlerts.danger` 级别。频率控制器实现允许放在 `buildStatusAlerts` 内（用 `state.run` 命名空间下的 `lastDangerEmitMap`，不进入持久化）。
- 不引入新的 alert level（good / warn / danger 不变）。

**严禁**：

1. **不**在 severity 计算中调用 RNG。
2. **不**在 severity 计算中读取 `state.meta.points / unlockedProtocols / selectedProtocol`（除了 `getMetaEffect("...")` 已经被 `getBuildCost` 包装的部分，本 helper 不需要再次访问）。
3. **不**让 severity 反向影响 `state.station.cells` 任何字段。
