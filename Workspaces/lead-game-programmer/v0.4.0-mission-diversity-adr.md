## 1. 背景

### 1.1 版本目标摘要

制作人方向把 v0.4.0 定为「任务多样化」：

- 让 rogue 局内每个关卡有真正不同的玩法体验，不再被「清剿敌人 + 采矿」单一循环主导。
- 引入与 v0.3.0 fragment 系统强协同的**拾荒任务**（搜寻并桥接古老残骸）。
- 引入需要保护友方 NPC 的**护送任务**（首次出现「友方实体」）。
- 把现有任务系统重构为**可扩展的注册表 + 事件分发框架**，为后续任务类型扩展铺路。
- 任务选择**参考 seed**，同 seed + 同 level 任务一致，不同 seed 任务可不同；终局 level 6 仍固定 guardian。

### 1.3 关键设计风险（高优先级）

1. **散落进度触点合并**：现有 4 处直接修改 `objective.progress` 的代码必须改走统一事件入口（`notifyObjective`），否则注册表无法承接新类型的进度。一旦漏改任何一处，旧 5 种任务行为会回潮（漏计 / 重计）。
2. **HUD 与文案硬编码 type 链**：`updateHud()` / `pickNextForObjective()` / `buildGuideText()` 三处都把 5 种 type 写死成 switch / 三元链。注册表必须把这些链改成查表，否则新增任务无法显示 detail。
3. **任务选择 seed 化对节奏的影响**：现行 `level % choices.length` 看似随机，实际是每个 level 永远固定一种任务（level 0 = mine、level 1 = explore、level 2 = battle、level 3 = survive、level 4 = mine ...），改成 seed 化后必须保留「level 节奏曲线」（0-2 减压 / 3-4 上升 / 5-6 紧张），否则可能在 level 1 蹦出 escort，体验崩盘。
4. **friendly NPC 引入对现有 enemy / projectile / turret pipeline 的污染**：当前 `enemyAi / spawnWave / collideEnemyWithStation / state.enemies.filter` 全部默认 "any item in state.enemies 都是敌对"；若用扩展 `enemy.kind = "friendly"` 方案，要在 ≥ 6 处加 `if (kind === "friendly") return;` 兜底，散布风险大。
5. **拾荒 wreck fragment 与 v0.3.0 fragment 系统耦合**：`origin = "wreck"` 字段必须不进入 `detachComponents` 旁路、不污染玩家脱落 fragment 的 HUD / 上限 / 命中判定；任务计数仅统计 wreck 桥接，玩家脱落桥接不计入。
6. **任务失败兜底**：现有所有 5 种任务都没有"失败"概念，只有"未完成"。引入 escort 后必须新增"失败但本关可继续游玩 / 跃迁禁用"分支，影响 `isObjectiveComplete()` 与 HUD 状态。

## 2. 设计决定逐项（对应方向文档 §「关键技术疑问」8 项）

### 2.1 `OBJECTIVE_TYPES` 注册表结构

- **候选方案**
  - A. 注册表仅声明 `make / getDetail / getHint`，进度仍由调用方直接 `objective.progress += ...`（最小改动）。
  - B. 注册表声明完整生命周期：`make / tick / onXxx 系列事件 / isComplete / getDetail / getHint / render / reward`，进度走事件分发。
  - C. 注册表 + 多接口继承（OO 风格类），每个类型继承基类。
- **推荐：B**。理由：
  - 5 种现有任务的散落触点必须收口，否则注册表只是「换皮硬编码」。
  - 拾荒 / 护送都需要监听新事件（`fragmentDocked / npcArrived / npcDestroyed`），注册表必须支持事件回调。
  - 不上 C（OO），因为 v0.3.0 ADR 已确立「不引入额外架构开销」的取舍；纯对象字面量足够。
- **必选方法**（每个 type 必须实现）：
  - `make(rng, level, galaxy) -> objective` —— 生成任务对象（含 type / text / target / progress / 类型特有字段）。
  - `getDetail(objective, displayProgress) -> string` —— HUD `#objective` 的 progressDetail 行（取代 `updateHud()` 当前三元链）。
  - `getHint(objective) -> string` —— `pickNextForObjective` 用的引导文案。
- **可选方法**（按需实现，未实现走默认 no-op）：
  - `tick(objective, dt)` —— 每帧调用（survive 计时、escort NPC AI tick 走这里）。
  - `onMined(objective, amount)` —— 采集事件（mine）。
  - `onEnemyKilled(objective, enemy)` —— 敌人击杀（battle）。
  - `onGuardianKilled(objective, enemy)` —— 守护者击杀（guardian）。
  - `onFragmentDocked(objective, fragment)` —— fragment 桥接成功（salvage 仅响应 `origin === "wreck"`，玩家脱落 fragment 默认 no-op）。
  - `onNpcArrived(objective, npc)` —— 友方 NPC 抵达跃迁点（escort）。
  - `onNpcDestroyed(objective, npc)` —— 友方 NPC 死亡（escort）。
  - `isComplete(objective)` —— **默认**走 `objective.progress >= objective.target`；guardian 已天然满足（progress = 1 / target = 1），无需重写；escort 也用默认（progress = 1 / target = 1，NPC arrived 时 progress = 1）。
  - `render(objective, renderer)` —— 任务专属世界渲染（explore beacon、guardian 包围环、escort NPC 路径线、salvage wreck 高亮）。
- **奖励倍率是否进注册表**：**进**。在注册表加 `rewardMultiplier`（默认 1.0），`grantObjectiveReward()` 改为 `gained = Math.round((2 + state.run.level) * (type.rewardMultiplier ?? 1))`。salvage / escort 由数值方案给 > 1（如 1.2 / 1.4），mine / explore / battle / survive 保持 1.0；guardian 由 endRunSettlement 走终局加成路径，不受影响。
- **任务失败判定是否进注册表**：**进**。新增可选 `isFailed(objective)`，默认 `false`。escort 在 `onNpcDestroyed` 后置 `objective.failed = true`，`isFailed` 返回该字段。`isObjectiveComplete()` 不变（仍只判完成），但 HUD 与跃迁门控参考 `isObjectiveFailed()`（详见 §2.7）。
- **风险**：注册表条目变成 11 个方法，部分类型只用 2-3 个，看上去稀疏。可接受，因为这是「为扩展铺路」的版本。
- **回滚**：若注册表后续被证明过于重，可逐步拆为「核心注册表 + 事件分发」+ 「expansion table」，但 v0.4.0 不做。

### 2.2 `notifyObjective(event, payload)` 事件分发

- **事件命名约定**：小驼峰，固定一组 string 常量（建议 `Game/game.js` 顶部声明 `const OBJECTIVE_EVENTS = { MINED: "mined", ENEMY_KILLED: "enemyKilled", GUARDIAN_KILLED: "guardianKilled", FRAGMENT_DOCKED: "fragmentDocked", NPC_ARRIVED: "npcArrived", NPC_DESTROYED: "npcDestroyed" }`）。
- **是否支持多类型监听同一事件**：**不支持**。理由：
  - v0.4.0 仍保持单 `state.run.objective` 结构，单 objective 至多对应一种 type，不会出现「同事件 → 多类型同时响应」。
  - 即便后续考虑「击杀加分」之类的副作用，也应放在事件源（如 `state.enemies.filter` 旁路）直接做，不走任务系统。
  - 不引入广播复杂度，避免一处事件触发多处副作用导致回归追查困难。
- **是否需要异步事件**（如 NPC 状态延迟通知）：**不需要**。所有事件都在主循环同步路径调用，payload 由调用方填好。
- **是否要把"5 种现有类型重构"放在引入新类型之前**：**是**。强制顺序：
  - T1（注册表 + 事件分发框架 + 5 种现有类型迁移） 必须独立提交并由 PM 做最小行为等价回归（节点 A），再启动 T3 / T4。
  - 理由：5 种现有类型的回归是「项目高风险」核心，必须在没有新类型干扰的情况下证明等价；否则节点 C 全量回归无法判断回潮原因是注册表问题还是新类型问题。
- **实现框架**（伪代码见 §4.2）：
  ```js
  function notifyObjective(event, payload) {
    const obj = state.run.objective;
    if (!obj) return;
    const def = OBJECTIVE_TYPES[obj.type];
    if (!def) return;
    if (isObjectiveComplete() || isObjectiveFailed()) return;
    const handlerKey = OBJECTIVE_EVENT_HANDLERS[event];
    const handler = def[handlerKey];
    if (typeof handler !== "function") return;
    handler(obj, payload);
    if (def.isComplete ? def.isComplete(obj) : obj.progress >= obj.target) {
      grantObjectiveReward();
    }
  }
  ```
- **风险**：notifyObjective 在事件源每次都查表，但 OBJECTIVE_TYPES 是常量 dict，查询 O(1)，可忽略。
- **回滚**：若发现某事件源调用频率过高（实际不会），可在事件源加 cheap guard（如 `if (obj.type !== "mine") return;`）短路 notifyObjective。

### 2.3 任务选择 seed 化

- **候选方案**
  - A. 直接把 `Math.random()` 改成 `mulberry32(state.run.seed + level)`，权重均匀。
  - B. 每个 level 有「任务类型权重表」，用 seed 派生 RNG 按权重抽样。
  - C. 完全随机，不保留 level 节奏（最大随机性，破坏 v0.2.0 难度曲线）。
- **推荐：B**。理由：
  - 保留 v0.3.0 / v0.2.0 的 level 节奏曲线（0-2 减压 / 3-4 上升 / 5-6 紧张）是兼容性硬要求。
  - 权重表能让数值方案精确控制「level 0 不可能出现 escort」等约束。
  - level 6 终局直接早回路走 guardian，根本不进入权重抽样。
- **权重表接口**：
  ```js
  const OBJECTIVE_LEVEL_WEIGHTS = [
    { mine: 0.6, explore: 0.4 },                                // level 0：减压
    { mine: 0.4, explore: 0.35, salvage: 0.25 },                // level 1：减压 + 少量 salvage
    { mine: 0.3, explore: 0.25, salvage: 0.25, battle: 0.2 },   // level 2：减压收尾
    { battle: 0.35, salvage: 0.25, survive: 0.2, escort: 0.2 }, // level 3：上升
    { battle: 0.3, escort: 0.3, salvage: 0.2, survive: 0.2 },   // level 4：上升
    { battle: 0.3, escort: 0.35, survive: 0.25, salvage: 0.1 }, // level 5：紧张
    // level 6 强制 guardian，不走权重表
  ];
  ```
  - 具体数值由数值方案拍板；ADR 仅给参考量级（明确「level 0 必无 escort / salvage 占比 ≤ 0.3 / level 6 不进入权重表」）。
  - 权重表必须是常量（不写入 SAVE_KEY），位于 `OBJECTIVE_TYPES` 注册表附近。
- **seed 派生 RNG**：
  ```js
  function getObjectiveRng(seed, level) {
    return mulberry32((seed ^ 0x9e3779b9) + level * 0x85ebca6b);
  }
  ```
  - 与 `${seed}:${level}` 字符串派生隔离（galaxy 生成自己一份），避免互相影响。
  - 如果 v0.3.0 已有 mulberry32 / xorshift 工具就复用，没有就在 createObjective 上方加一个 12 行的 mulberry32 函数。
- **跨 level 任务类型连续重复处理**：**不主动避免**。理由：
  - 用 seed 抽样的本意就是「同 seed 重启可复现」，强制去重会破坏可复现性。
  - 真正出现连续 escort 概率 ≤ 0.3 × 0.3 = 0.09，体验不至于过载。
  - 如果数值方案后续验证需要，可在权重表上加「上一关类型权重 × 0.5」的可选 decay，但 v0.4.0 默认不开。
- **终局 level 6 兜底**：保持现有 `if (state.run.level >= ENDGAME_LEVEL)` 早回路，强制塞 guardian，**不进入权重表**。这条早回路是「seed 化不打乱终局」的硬保证，PM 回归必须复测。
- **回归方式**：
  - PM 在节点 B 用同 seed + 同 level 多次进入，确认任务类型一致。
  - 用不同 seed 至少 3 个，确认 level 1-5 能出现至少 2 种不同任务（不要求覆盖所有）。
- **风险**：权重表数据若由数值方案延迟交付，T2 可以先用「均匀权重」占位（即每个 level 所有类型权重均为 1），等数值方案到位再 swap 数据；接口 / 算法不变。
- **回滚**：若 seed 化引起强烈回潮（例如某 seed 在 level 0 抽到了 salvage 但拾荒任务有 bug），可临时把 `OBJECTIVE_LEVEL_WEIGHTS[0]` 强制收窄为 `{mine: 1}`，不动框架。

### 2.4 拾荒 fragment 预生成时机与 origin 字段

- **生成时机选项**
  - A. `applyGalaxy → createObjective` 之间，由外部 helper 注入。
  - B. `createObjective` 内部，由 `OBJECTIVE_TYPES.salvage.make(rng, level, galaxy)` 调用 `seedSalvageFragments(galaxy, count)`。
- **推荐：B**。理由：
  - 让"生成残骸"成为 salvage 任务的天然副作用，外部不需要知道任务类型决定要不要生成。
  - 其他任务类型的 make 不调用 seedSalvageFragments，自然不预生成 wreck fragment。
  - 任务失败 / 重置时若需要清理 wreck，逻辑也集中在注册表里。
- **`origin` 字段加在哪里**：在 fragment 对象上加 `origin: "player" | "wreck"`（默认 "player"，玩家脱落分支 `detachComponents` 内部填 "player"）。Cell 上不加（cell 不需要区分来源；区分用 fragment.origin）。
- **`origin` 字段对 v0.3.0 影响逐一确认**：
  - **序列化**：`state.fragments` 本来就不写入 SAVE_KEY（v0.3.0 ADR §1.2 明确），origin 字段同样不写入。✅
  - **HUD 区分**：`buildFragmentHudAlerts` / 最近 fragment HUD 需读 `fragment.origin` 决定显示色与文案（`origin === "wreck"` → 古老残骸 X 段；玩家脱落 → 我方残骸 Y 段）。少量改动，T3 范围内。
  - **维修无人机**：当前显式只遍历 `state.station.cells`，不会修 fragment cell；无论 origin 都不修。✅ 兼容
  - **桥接判定**：`tryBridgeAt` 不需要看 origin，行为对 wreck / player 一致（玩家可正常桥接 wreck 取回设施）。✅
  - **弹道命中**：现有 enemy 弹道 → fragment 命中分支不区分 origin。wreck 也可被敌方打掉（"残骸继续在战斗中被吃掉"是体验目标延续）。✅
  - **fragment 上限**：见 §2.8 单独决定。
- **`origin === "wreck"` 与 `state.fragments` 上限关系**：见 §2.8。
- **拾荒残骸位置策略**：
  - 在 `state.galaxy.bounds` 内随机散布，避开 station 起点 `< CELL * 12` 距离窗口（避免开局立刻桥接）。
  - 与 planet / asteroid 体积**允许重叠**（fragment vs planet/asteroid 本身就不碰撞，按 v0.3.0 ADR §2.3）。
  - 与已有 wreck fragment 之间保持 `> CELL * 6` 最小间距（避免桥接判定时选错目标）。
  - 每个 wreck fragment 由 `seedSalvageFragments` 生成时给定 `vel` < `FRAGMENT_LAUNCH_SPEED_MIN * 0.4`（明显比玩家脱落慢，便于追上）。
  - 详细数量 / 速度 / 间距由数值方案拍板。
- **wreck fragment 内的 cells 构造**：复用现有 facility 集合（turret / shield / armor / repair / mining / power），不引入新设施类型。每个 wreck fragment 选 1-3 个 cell，facility 由数值方案权重表给。位置在 fragment 本地 grid 中可形成简单形状（如 1×3 直线或 L 形）。
- **风险**：wreck fragment 与玩家脱落 fragment 共用上限会导致开局玩家脱落多了反而吃掉 wreck（不应发生），见 §2.8 单独 cap 方案。
- **回滚**：若 wreck 生成位置在某些 seed 下与 planet 撞重叠引起视觉问题（fragment 在 planet 下层渲染），可加一次性 reroll（最多 5 次找无重叠位置）。

### 2.5 友方 NPC 数据结构

- **候选方案**
  - A. 新建 `state.npcs[]` 数组 + `Npc` 类型；与 `state.enemies` 完全隔离。
  - B. 扩展 `state.enemies` + `enemy.kind = "friendly"`，全部 enemy pipeline 加 friendly 兜底。
- **推荐：A**。理由：
  - `enemyAi`（`Game/game.js:2715-2810`）、`collideEnemyWithStation`（`3018-3030`）、`updateTurret` 自动锁敌、`state.enemies.filter`（死亡清算 → 计 battle 进度 / 给资源）等 ≥ 6 处都默认「item ∈ state.enemies 就是敌对」。
  - 选项 B 需要在每处加 `if (enemy.kind === "friendly") return / continue;`，散布修改高，回潮风险散布；任何一处漏改都会让 NPC 被自己人计 battle 进度、给资源、被炮塔打。
  - 选项 A 把 NPC 抽成独立数组，所有 NPC 相关逻辑集中在 `updateNpcs / renderNpcs / collideNpcWithEnemy` 三个新函数中；现有 enemy pipeline 不动一行，回归面最小。
  - 代码量多 ~30 行（一个独立 Npc 类 + 三个函数），但全部是新增、零侵入。
- **Npc 数据结构**：
  ```js
  {
    id: number,
    pos: { x, y },
    vel: { x, y },
    angle: number,           // 渲染朝向（朝向目标点）
    target: { x, y },        // 移动目标（跃迁出口）
    hp: number,
    maxHp: number,
    radius: number,          // 碰撞半径
    speed: number,           // 速度上限
    color: [r, g, b, a],     // 友方蓝绿色，与红 enemy 区分
    state: "moving" | "arrived" | "destroyed",
    spawnAt: number,
    role: "freighter"        // 预留字段，未来若扩展 escort 类型可分 "freighter / scout / refugee"
  }
  ```
- **state 顶层位置**：`state.npcs = []`，与 `state.enemies / state.fragments` 同级，不写入 SAVE_KEY。
- **跨关卡清理**：`nextLevel()` 中追加 `state.npcs = [];`（与 fragments / enemies 一致）。
- **风险**：玩家炮塔自动锁敌（`updateTurret`）只遍历 `state.enemies`，天然不锁 NPC；玩家手动鼠标点击若选中 NPC，可走「state.target」路径（仅作为航行目标），不触发开火。
- **回滚**：若选项 A 在体验测试中暴露问题（如 NPC 与敌人渲染层级混乱），可降级为「`state.npcs[i]` 上加 `friendly: true` 标记后并入 enemies」，但 v0.4.0 默认不做。

### 2.6 NPC 移动 AI 与伤害

- **移动方式**：**直线移动 + 简单避障**。理由：
  - 星系内只有 planet / 大行星少量静态障碍，避障复杂度低。
  - A* 在动态障碍（asteroid / enemy）下也只能给参考路径，且增加 ≥ 80 行复杂度，不值得。
- **AI 伪代码**：
  ```js
  function tickNpc(npc, dt) {
    if (npc.state !== "moving") return;
    const toTarget = sub(npc.target, npc.pos);
    let dir = normalize(toTarget);
    for (const body of state.world.bodies) {
      if (body.type !== "planet" && body.type !== "star") continue;
      const toBody = sub(body, npc.pos);
      const distBody = length(toBody);
      const avoidR = body.r + NPC_AVOID_MARGIN;
      if (distBody > avoidR) continue;
      const tangent = perpendicular(normalize(toBody), npc.target);
      dir = normalize(add(dir, scale(tangent, NPC_AVOID_WEIGHT)));
    }
    npc.vel.x = dir.x * npc.speed;
    npc.vel.y = dir.y * npc.speed;
    npc.pos.x += npc.vel.x * dt;
    npc.pos.y += npc.vel.y * dt;
    npc.angle = Math.atan2(dir.y, dir.x);
    if (dist(npc.pos, npc.target) < NPC_ARRIVE_RADIUS) {
      npc.state = "arrived";
      notifyObjective("npcArrived", npc);
    }
  }
  ```
- **速度建议**：`npc.speed` 由数值方案给，建议 v0.2.1 玩家全速的 30%-40%（让玩家能跟上）。NPC_ARRIVE_RADIUS 建议 `CELL * 2.5`。
- **受伤来源**：
  - 敌方 projectile：在 `updateProjectiles` 弹道行进末尾追加 NPC 命中检查（球 vs npc 中心距离 < `npc.radius`），仅 enemy owner 触发；命中后 NPC 减 hp，弹道销毁。
  - 撞击：`collideNpcWithEnemy(npc)` 在 `updateNpcs` 末尾调用；与 enemy 距离 < `npc.radius + enemy.r` 时，NPC 减 hp（NPC_COLLISION_DAMAGE，由数值方案给），enemy 不受影响（避免 NPC 变盾牌）。
  - asteroid 伤害：与 enemy 撞击共用 `collideNpcWithEnemy`（asteroid 也是 enemy.kind）。
  - 玩家弹道：**不命中**（敌我识别，玩家弹道 owner === "player" 时跳过 NPC 命中检查）。
  - 玩家撞击：**不伤害**（碰撞检查不包含 player ↔ NPC，玩家可穿过 NPC）。
- **NPC 死亡处理**：hp ≤ 0 → `npc.state = "destroyed"` → `notifyObjective("npcDestroyed", npc)` → escort 任务记 `objective.failed = true`。NPC 不立刻从 `state.npcs` 移除，保留 destroyed 状态 1 秒供视觉特效（破碎 + toast），随后清理。
- **NPC 抵达跃迁点**：跃迁点定义为「跃迁出口」world 坐标，建议放在 galaxy 中心相对玩家起点的对侧（`-station.startPos`），数值方案决定具体距离。npc.target 在 spawn 时固定写入；NPC 抵达后 `notifyObjective("npcArrived", npc)`，escort 任务 `progress = 1`，完成。
- **风险**：直线避障极端情况下可能在 planet 切线方向「左右摇摆」（dir 来回切换），加 `NPC_AVOID_WEIGHT * dt` 平滑或加 hysteresis 即可，T4 实现时验证。
- **回滚**：若直线避障在某些 galaxy 布局下卡死（NPC 在两 planet 之间死循环），降级为「忽略避障、直接穿过 planet」（视觉上 planet 不挡 NPC），影响小于体验崩盘。

### 2.7 NPC 死亡处理 / 任务失败兜底

- **候选方案**
  - A. 任务失败 → `gameOver()`（重启本局，保留点数）。
  - B. 任务失败 → 留在本关自由游玩，跃迁禁用，玩家可继续探索 / 重启。
  - C. 任务失败 → 重置进度 + spawn 新 NPC（学习成本高、节奏拖延）。
- **推荐：B**。理由：
  - 与现有「自由游玩」机制对齐（终局守护者击毁后玩家可留在终结星系自由游玩，结构相同）。
  - A 过于严苛，新手一次 NPC 死亡就 GG，会很挫败。
  - C 在 escort 上下文中等于「再失败一次」，体验拖沓。
- **失败状态字段**：在 `state.run.objective` 上加 `failed: bool`（默认 false，仅 escort onNpcDestroyed 时置 true）。
- **`isObjectiveComplete()` 是否要兼顾 failed**：**不改**。理由：
  - 完成态走 `objectiveCompleteAt > 0` 路径，仍只代表「正常完成」。
  - 新增 `isObjectiveFailed()` 单独函数，由 HUD / 跃迁门控分别引用。
- **跃迁门控**：`jumpToNextGalaxy()` 改为 `if (!isObjectiveComplete()) return;` + 同样不允许失败状态跃迁（即 escort 失败本关无法跃迁），玩家必须重启或留在当前星系自由游玩。
- **HUD 状态**：escort 失败时 `#objective` 显示「护送失败 · 本关无法跃迁」红色提示，statusAlerts 追加 escort 失败条目，`progressDetail` 改为「友方货船已损毁」。
- **奖励**：escort 失败不给奖励（`grantObjectiveReward` 内 `isObjectiveComplete()` 守护已天然兜底）。
- **结算面板**：escort 失败的关算 0 个 completedObjectives，进入下一局重启时正常重置。
- **风险**：若玩家在 escort 失败后选择留在本关「赚资源」，需要确保 spawnWave 等敌人逻辑仍可继续（与终局自由游玩一致）。
- **回滚**：若 B 在体验测试中暴露「失败感不够」，可加可选数值「失败时扣局外点数（最多 1-2 点）」，但 v0.4.0 默认不做。

### 2.8 性能预算

| 项 | 估算 | 备注 |
| --- | --- | --- |
| 注册表查找 | O(1) | OBJECTIVE_TYPES dict 常量 |
| notifyObjective 单次 | O(1) | 取注册表 + 调 handler |
| notifyObjective 调用频率 | mined: 60 Hz、enemyKilled: 数 Hz、fragmentDocked: 玩家驱动、npcArrived/Destroyed: 1 次 / 关 | mined 频率最高但 handler 仅 `obj.progress += amount`，无问题 |
| `updateNpcs` 单帧 | 1 × NPC（每关 1 艘）× 4-8 vector op | 可忽略 |
| `state.npcs` 数量上限 | 1 / 关（escort 任务时） | 严格限制，不允许扩到多艘（v0.4.0 范围） |
| `collideNpcWithEnemy` 单帧 | 1 × 20 (enemies) ≈ 20 距离判定 | 远低于 enemy ↔ station 命中 |
| 弹道命中 NPC | enemy 弹道 ≤ 60 × 1 NPC ≈ 60 距离判定 / 帧 | 可忽略 |
| `seedSalvageFragments` 单次 | 一关一次性生成 3-5 fragment × 2-3 cells | < 30 cells，远低于 FRAGMENT_MAX_CELLS_PER |
| wreck fragment 是否计入 `FRAGMENT_MAX_COUNT = 8` | **不计入** | 单独 cap `WRECK_FRAGMENT_MAX_COUNT = 6`（数值方案拍板），独立 `state.fragments.filter(f => f.origin === "wreck")` 限制；避免任务一上来就因为玩家脱落多吃掉 wreck 失败 |
| 渲染 npc | 1 × 5-10 个 renderer.circle / triangle 调用 | 可忽略 |
| 总预算 | v0.4.0 新增 < 50 line/frame 计算量 | 与 v0.3.0 fragment 系统同量级 |

- **关键决定**：wreck fragment **独立 cap**，不与玩家脱落 fragment 共用 `FRAGMENT_MAX_COUNT = 8`。`enforceFragmentBudget` 改为分类计数（player / wreck 各自维护）。
- **风险**：分类 cap 让 `state.fragments.length` 不再单一上限，T3 实现需明确：
  - `state.fragments.length <= FRAGMENT_MAX_COUNT + WRECK_FRAGMENT_MAX_COUNT = 8 + 6 = 14`。
  - 渲染 / 物理 / 边界清理仍统一遍历 `state.fragments`，cap 只在生成入口生效。
- **回滚**：若分类 cap 让代码复杂度上升超出预期，临时让 wreck 不计入 cap（即 wreck fragment 无上限，仅由 `seedSalvageFragments` 一次性生成 N 个，没有持续 spawn 不会失控）。

## 4. 关键算法伪代码

### 4.1 任务选择 seed 化

```js
function createObjective() {
  ensureRunRuntimeState();
  if (state.run.level >= ENDGAME_LEVEL) {
    state.run.endgame = true;
    state.run.guardianSpawned = state.run.guardianDefeated;
    if (!state.run.guardianDefeated) {
      state.run.guardianSpawnDelay = Math.max(state.run.guardianSpawnDelay, 6);
    }
    state.run.objective = OBJECTIVE_TYPES.guardian.make();
    if (state.run.guardianDefeated) state.run.objective.progress = 1;
    state.run.objectiveCompleteAt = state.run.guardianDefeated ? Math.max(state.time, 1e-6) : 0;
    resetObjectiveChoiceState();
    return;
  }
  const rng = getObjectiveRng(state.run.seed, state.run.level);
  const weights = OBJECTIVE_LEVEL_WEIGHTS[state.run.level] || OBJECTIVE_LEVEL_WEIGHTS[0];
  const type = pickWeighted(rng, weights);
  const galaxy = state.galaxy;
  state.run.objective = OBJECTIVE_TYPES[type].make(rng, state.run.level, galaxy);
  state.run.objectiveCompleteAt = 0;
  state.run.endgame = false;
  state.run.guardianSpawned = false;
  state.run.guardianDefeated = false;
  state.run.guardianSpawnDelay = 0;
  state.run.endgameExplore = false;
  state.run.settlementShown = false;
  resetObjectiveChoiceState();
}

function pickWeighted(rng, weights) {
  let total = 0;
  for (const w of Object.values(weights)) total += w;
  let r = rng.next() * total;
  for (const [type, w] of Object.entries(weights)) {
    if ((r -= w) <= 0) return type;
  }
  return Object.keys(weights)[0];
}

function getObjectiveRng(seed, level) {
  return makeMulberry32((seed ^ 0x9e3779b9) + level * 0x85ebca6b);
}
```

### 4.2 notifyObjective 实现框架

```js
const OBJECTIVE_EVENT_HANDLERS = {
  mined: "onMined",
  enemyKilled: "onEnemyKilled",
  guardianKilled: "onGuardianKilled",
  fragmentDocked: "onFragmentDocked",
  npcArrived: "onNpcArrived",
  npcDestroyed: "onNpcDestroyed",
};

function notifyObjective(event, payload) {
  const obj = state.run.objective;
  if (!obj) return;
  if (isObjectiveComplete() || isObjectiveFailed()) return;
  const def = OBJECTIVE_TYPES[obj.type];
  if (!def) return;
  const handlerKey = OBJECTIVE_EVENT_HANDLERS[event];
  const handler = handlerKey && def[handlerKey];
  if (typeof handler !== "function") return;
  handler(obj, payload);
  if (def.isComplete ? def.isComplete(obj) : obj.progress >= obj.target) {
    grantObjectiveReward();
  }
}

function tickObjective(dt) {
  const obj = state.run.objective;
  if (!obj || isObjectiveComplete() || isObjectiveFailed()) return;
  const def = OBJECTIVE_TYPES[obj.type];
  if (def?.tick) def.tick(obj, dt);

    grantObjectiveReward();
  }
}

function isObjectiveFailed() {
  return !!state.run.objective?.failed;
}
```

### 4.4 seedSalvageFragments（拾荒预生成）

```js
function seedSalvageFragments(galaxy, rng, level, count) {
  const bounds = galaxy.bounds || GALAXY_WORLD_BOUNDS;
  const minDistFromStart = CELL * 12;
  const minDistFromOther = CELL * 6;
  const placed = [];
  const ids = [];
  for (let i = 0; i < count; i++) {
    let pos = null;
    for (let tries = 0; tries < 12; tries++) {
      const candidate = {
        x: rng.range(bounds.minX + 80, bounds.maxX - 80),
        y: rng.range(bounds.minY + 80, bounds.maxY - 80),
      };
      if (dist(candidate, state.station.startPos || state.station.pos) < minDistFromStart) continue;
      if (placed.some((p) => dist(candidate, p) < minDistFromOther)) continue;
      pos = candidate;
      break;
    }
    if (!pos) continue;
    placed.push(pos);
    const fragment = makeWreckFragment(pos, rng, level);
    fragment.origin = "wreck";
    state.fragments.push(fragment);
    ids.push(fragment.id);
  }
  return ids;
}

function makeWreckFragment(pos, rng, level) {
  const cells = new Map();
  const shape = pickWreckShape(rng);                       // 1×3 / L / cross 等小形状
  for (const { x, y, facility } of shape) {
    const cell = createCell(x, y, facility, getWreckHp(rng, level, facility));
    cells.set(key(x, y), cell);
  }
  return {
    id: ++FRAGMENT_ID_SEQ,
    pos,
    vel: { x: rng.range(-WRECK_DRIFT_SPEED_MAX, WRECK_DRIFT_SPEED_MAX),
           y: rng.range(-WRECK_DRIFT_SPEED_MAX, WRECK_DRIFT_SPEED_MAX) },
    angle: rng.range(-Math.PI, Math.PI),
    angularVel: rng.range(-0.3, 0.3),
    cells,
    spawnAt: state.time,
    bounds: computeBounds(cells),
    edgeKeys: computeEdgeKeys(cells),
    origin: "wreck",
  };
}
```

```js
    role: "freighter",
    hudLastSampleAt: 0,
  };
  state.npcs.push(npc);
  return npc;
}

function updateNpcs(dt) {
  for (const npc of state.npcs) {
    if (npc.state === "destroyed") {
      if (state.time - npc.destroyedAt > 1) npc.markedDead = true;
      continue;
    }
    if (npc.state === "moving") tickNpc(npc, dt);
    collideNpcWithEnemies(npc);
  }
  state.npcs = state.npcs.filter((n) => !n.markedDead);
}

function collideNpcWithEnemies(npc) {
  if (npc.state !== "moving") return;
  for (const enemy of state.enemies) {
    if (enemy.hp <= 0) continue;
    if (dist(npc.pos, enemy) < npc.radius + enemy.r) {
      damageNpc(npc, NPC_COLLISION_DAMAGE);
    }
  }
}

function damageNpc(npc, dmg) {
  if (npc.state !== "moving") return;
  npc.hp -= dmg;
  if (npc.hp <= 0) {
    npc.hp = 0;
    npc.state = "destroyed";
    npc.destroyedAt = state.time;
    notifyObjective("npcDestroyed", npc);
    showToast("友方货船被击毁，护送任务失败。");
  }
}
```

## 5. HUD 与渲染策略

### 5.1 HUD `#objective`

- progressDetail 走 `OBJECTIVE_TYPES[type].getDetail()`，硬编码三元链全部移除。
- 失败态 (failed) 新增显示分支（红色 bad 样式，与现有 `.alert-danger` 同色系），文案「任务失败 · 本关无法跃迁」。
- escort 任务在 statusAlerts 追加一条 NPC HP 警告（≤ 30% 时 level="warn"，文案「友方货船 HP 低」）。
- salvage 任务在 statusAlerts 不追加（HUD 已有"古老残骸 X / Y"提示足够）。

### 5.2 HUD fragment 区分

- 在 `buildFragmentHudAlerts` 中区分 `origin`：
  - `origin === "player"` → 文案「我方残骸 X 段」。
  - `origin === "wreck"` → 文案「古老残骸 X 段（任务 Y / Z）」。
- 最近 fragment 方向 HUD 同步显示 `origin` tag（如「最近：古老残骸 · 320m / 285°」）。

### 5.3 NPC HUD

- 新增 `buildNpcHud()` 在 `#objective` 右下追加一行（escort 任务期间显示）：
  - 方向（角度，相对玩家朝向）+ 距离（米 / world unit）+ HP 百分比 + 状态（moving / arrived / destroyed）。
  - 节流：每 `NPC_HUD_SAMPLE_INTERVAL = 0.2s` 更新一次，避免每帧重算。

### 5.4 世界渲染

- 新增 `renderNpcs(renderer)`，由 `render()` 主循环调用：
  - 在 station / fragment 之上渲染（玩家需清楚看到 NPC 位置）。
  - 渲染形状：六边形 / 菱形（与 enemy 圆形、station 方形明显区分）。
  - 蓝绿色 `[0.4, 0.95, 0.7, 1]` 主体 + 白色描边。
  - destroyed 状态 1 秒内画破碎特效（圆形扩散 + 颜色淡化）。
- wreck fragment 渲染复用现有 `renderFragments`，仅多加一档：
  - `origin === "wreck"` 时 outline 色改为冷色 `[0.6, 0.85, 0.95, 0.8]`（与玩家脱落的橙色 `[0.92, 0.6, 0.32, 0.8]` 区分）。
- 任务专属世界渲染（explore beacon ring、guardian 红环、escort 路径线 + 跃迁点 ring、salvage 不需要）通过 `OBJECTIVE_TYPES[type].render(obj, renderer)` 调用，原 `renderTargetAndObjective` 中的硬编码 switch 移除。

## 11. 待制作人 / PM 决策的开放项

1. **是否允许 wreck 内部含 power 设施**：本 ADR 推荐复用现有 facility 集合（含 power）。如果数值方案认为玩家「桥接拾来发电站」过强，可在 wreck shape 表中排除 power，留作数值方案决定。
2. **escort 任务是否允许玩家手动驾驶 NPC**：本 ADR 推荐**不允许**，NPC 完全由 AI 控制。若数值方案后续认为玩家应能「指挥 NPC 等待」，需要额外加 UI / 输入路径，留作 v0.4.1 评估。
3. **NPC 渲染层级**：本 ADR 推荐渲染在 station / fragment 之上、enemy 之下。若节点 C 体验测试发现遮挡，可调换层级，T4 实现时由 PM 在体验闭环中决定。
4. **escort 任务期间是否降低敌人 spawn 倍率**：本 ADR 不动 spawnWave；由数值方案决定（建议保持或略降）。
5. **任务失败界面是否要单独面板**：本 ADR 推荐复用 `#objective` HUD + statusAlerts，不新建面板。若节点 C 体验测试认为失败感不清晰，可在 v0.4.1 加专属面板。
6. **wreck 桥接成功后维修无人机是否优先维修 wreck cell**：本 ADR 默认不区分（维修无人机按现有规则）。若数值方案认为「拾来的设施应该先维修」，可在 T3 加优先级标记，但不在 v0.4.0 范围。
7. **`OBJECTIVE_LEVEL_WEIGHTS` 是否允许 mod 配置**：本 ADR 推荐写死在代码中，不暴露 mod 接口。后续若有 mod 需求，可加 `loadObjectiveWeightsFromConfig()`，但 v0.4.0 不做。
8. **是否在 v0.4.0 加 "上一关任务类型 decay" 权重**：本 ADR 推荐不开（默认允许连续重复）。如节点 B 体验测试发现连续 escort 太频繁，再开 decay。
