## 0. 文档定位与边界

本方案只定 **v0.14.0 发布候选取证的技术边界**：发布候选只读快照 `getReleaseReadinessSnapshot()` / 发布候选检查 `runReleaseCandidateChecks()` 的字段 schema、存储签名前后比对策略、frame timing 粗采样窗口、可选隐藏 UI 入口的最小形态、与 v0.11-v0.13 已有 helper 的复用规则、性能与只读不变量、禁止改动清单、T1-T4 任务拆分与节点审查条件、回滚方式。

**不在本方案决策范围**：

- 人工 P0-1~P0-8 路径具体步骤、记录字段、缺陷模板、PM 验收结论模板 → PM 人工 checklist 方案（PM 看板 #3，已落地）。
- 小屏断点具体像素值、HUD 折行视觉密度、按钮可点击尺寸 → UI 方案（看板 #6 T3 落地时由 `/ui-artist` 决定）。
- 浏览器 / headless / Node-VM 选型与具体命令行 → PM 看板 #7 T4 由 `/pm-qa-coordinator` 与 `/senior-systems-engineer` 选择已有工具。
- 发布判断本身、tag / 归档执行 → 制作人 #9 与 PM/IT #10。

本方案只提供 **接入框架**：函数 schema、调用前后不变量、生命周期、与现有 snapshot 的关系、误报 / 漏报失败时的降级路径。UI 方案、PM 回归脚本与人工 checklist 产出后按 §3-§9 边界填入即可，不需要回到本方案再决策。

本方案严格遵守：**只允许新增/修改 `Workspaces/lead-game-programmer/v0.14.0-release-candidate-playtest-technical-plan.md`**，不修改 `Game/`、`Docs/`、`.cursor/`、`42.md`、`Releases/`、PM 看板、方向文档、PM 人工 checklist 方案、其他工作区文件和历史 ADR；不删除文件；不 stage/commit dirty `42.md`；禁止 `git reset` / `git clean` / `git stash` / checkout 覆盖 / force push。

## 1. 决策摘要

| 编号 | 决策 | 主要理由 |
| --- | --- | --- |
| D1 | 新增 2 个**只读纯函数**入口：`getReleaseReadinessSnapshot()` 返回结构化运行态快照，`runReleaseCandidateChecks(options)` 在快照之上做分层评估并返回 `{ ok, layers, checks, residualRisks, capturedAt, durationMs }`；命名空间挂到 `window.__gameTest.playtest.releaseCandidate.*`，与现有 `__gameTest.playtest.snapshot` 等并列，**不**新增顶层 `__gameTest` 字段以避免冲撞 v0.11-v0.13 接入面 | 与 v0.11 / v0.13 增量同模式；`releaseCandidate` 子命名空间集中表达"本版本特有发布候选取证"，PM 在 DevTools 里一行可拿到结果；不破坏现有 snapshot 字段语义，避免回潮 v0.11 readiness |
| D2 | `getReleaseReadinessSnapshot()` **必须**先调用现有 `window.__gameTest.playtest.snapshot()` 取得 base，然后在 base 之上扩展四个新字段：`storageSignature` / `frameTiming` / `environment` / `evidence`；现有 base 字段（`panels / viewport / keyDom / metaStorage / resource / design / objective / guide / selfChecks / runtimeErrors / objectiveChoice / combat`）按原结构透传，**不**重命名、**不**精简 | base snapshot 已是 v0.11 readiness 与 v0.13 战斗可读性的稳定回归证据来源，重新打包会冲掉 PM 看板 N06 / N07 / N08 不回潮验收；release readiness 是 base 的**超集**，不是替代 |
| D3 | `storageSignature` 字段同时覆盖 `localStorage[SAVE_KEY]` 与 `sessionStorage[META_CORRUPT_RAW_KEY]`；签名由 `{ supported, hasValue, length, head32, tail32, hash32 }` 组成，**不**走 `crypto.subtle`（避免 async 与依赖）。`hash32` 用 32 位 djb2 / xor 等简单 stable hash 实现，纯 JS，不引入新模块 | 与 `captureStorageSnapshot`（`Game/game.js:15781`）思路一致；不能依赖完整 value 字符串，因为 PM 可能需要将签名复制到 PR / 看板 / 截图记录，长字符串不可读；djb2 + length + head/tail 已足够覆盖"是否被修改"判定；任何不一致都直接阻塞 |
| D4 | 在 `runReleaseCandidateChecks()` 的**第一行**捕获 `storageSignature` 作为 `before`，在最后一行捕获作为 `after`；若不一致，整个返回的 `ok` 强制为 `false`，并在 `residualRisks` 中追加 `{ kind: "storage_mutation", before, after, severity: "blocking" }`；该不变量是 PM 看板 M01-M06 的代码级证据 | M03 / M06 的可复查证据；若 release candidate checks 自身就污染了存档，则后续所有 PM 结论失效；这是本方案的强制底线 |
| D5 | `frameTiming` 采样走 **请求-响应模式**：调用方在 `runReleaseCandidateChecks({ sampleFrames: N })` 时，函数内部用 `requestAnimationFrame` 跑 `clamp(N, 6, 120)` 帧，记录每帧 `performance.now()` 差值；默认 `N = 30`（≈ 0.5 秒）；采样不写 `state`、不调用 `update / render`、**只读** `performance.now()`；采样结束后返回 `{ sampleCount, avgMs, maxMs, longFrameCount, longFrameThresholdMs, mode }`，`mode = "rAF"` / `"unavailable"`；`longFrameThresholdMs` 默认 33（≤ 30 fps 即长帧）；**不**修改 `state.lastTime` / `state.hud.timer` | 与 `loop()` 节奏并行而非耦合，避免改造 v0.11 readiness 的 `loop()`；只读不变量保证连续两次调用 readiness 仍能保持 storage signature 与 RNG 不变；rAF 不可用时（如 jsdom / VM）降级为 `mode: "unavailable"` 而不是抛错 |
| D6 | `environment` 字段表达浏览器/运行环境**提示**：`userAgent / language / platform / viewport / devicePixelRatio / online / prefersReducedMotion / prefersColorScheme / hasRequestAnimationFrame / hasPerformanceNow / hasLocalStorage / hasSessionStorage / hasStructuredClone / isHeadlessHint / capturedAt`；`isHeadlessHint` 是 best-effort 启发式（`navigator.webdriver`、`!hasRequestAnimationFrame`、UA 命中 `HeadlessChrome / jsdom / node.js / Puppeteer / Playwright` 等），**仅记录**给 PM 作为参考，**不**用于决定 `ok`；headless / VM 与真实浏览器结果由 PM 在记录侧区分（看板 O01-O06、P01-P06） | 自动检查不能替代人工体验；`isHeadlessHint` 只服务"PM 录入证据时少漏标"，不为程序自动判断；不引入 UA sniffing 战争 |
| D7 | `evidence` 字段是 **快照层级语义标注**：`{ layers: { automatic: [...], browser: [...], manual: [...] }, coverage: { automaticPassed, browserCovered, manualCovered, unknownCount } }`；每条 `evidence.layers.*` 元素结构 `{ id, label, layer, source, summary, severity, blocking }`；`layer` 取值 `"automatic"` / `"browser"` / `"manual"`；自动检查的 `getReleaseReadinessSnapshot` 只填 `automatic / browser` 两层的**自动可读项**，并将所有 `manual` 项的 `summary` 显式标 `"未由自动检查覆盖，需人工 / 真实浏览器记录"` | 与制作人方向第 §"验收路径" 节、PM 看板"自动 / browser / human / 未覆盖"四象限严格一致；自动调用方拿到结构后，能直接告诉 PM "automatic 通过、browser 未覆盖、manual 全部未覆盖"，避免把 headless 通过冒充人工通过 |
| D8 | `runReleaseCandidateChecks()` 返回 `checks` 数组结构 `{ id, layer, status, severity, details, capturedAt }`；`status` 取 `"pass"` / `"fail"` / `"warn"` / `"unknown"`；`unknown` 表示"自动检查未覆盖该项，需要 PM 在外部补证"；自动 `checks` 集合**仅**包含可只读自动判定的 13-18 条 id（见 §3.4 列表），覆盖 `runtimeErrors`、关键 DOM 存在 / 可见、storage signature 一致、frame timing 长帧阈值、selfChecks 子项、design / combat / galaxy / meta hook 字段完整性；**不**包含玩家是否能"看懂"、是否"小屏体验通过"、是否"真实交火通过"等主观项 | 自动 fail 必须能在浏览器 DevTools 内复现；主观项只能由 PM 人工 checklist 标记；与 PM 看板 #9 "自动检查、browser/headless、人工体验、未覆盖风险" 四类证据严格对应 |

| D10 | 性能保护：`getReleaseReadinessSnapshot()` **不**进入 `loop()` 0.12 秒 cadence；调用方负责频率上限。函数内部沿用 v0.11 / v0.12 缓存：design / combat 子树通过现有 `getStationDesignHealth` / `getThreatSummary` 等的内部缓存机制；release candidate 自身**不**再加二级缓存。`frameTiming` 采样窗口最长 2 秒（120 帧 × 16.67ms 上限）；超过后强制截断并标记 `truncated: true` | 单次 readiness 调用不会改变 HUD cadence；frameTiming 上限保证 PM 哪怕传 `sampleFrames: 99999` 也不至于卡住浏览器主线程；与 v0.11 §6 性能边界一致 |

**v0.14 缺口**（必须新增字段，**不**得移除任何已有字段）：

- `storageSignature` 子树：`{ before, after, consistent }` 形态，并对 `SAVE_KEY` / `META_CORRUPT_RAW_KEY` 各算签名；当前 `metaStorage.hasSave` / `summary.hasCorruptRaw` 只能表达"有没有"，无法表达"前后一致"。
- `frameTiming` 子树：当前完全没有；snapshot 不能直接做窗口采样（snapshot 是瞬时读取），必须由 `runReleaseCandidateChecks` 主动跑一窗口后填入或单独 helper 提供。
- `environment` 子树：当前 `viewport` 只覆盖宽高 / DPR / narrowViewport，缺 UA / platform / online / hasFoo / isHeadlessHint。
- `galaxy` 子树：当前 `objective.currentGalaxyType` 单字段，`getGalaxyHudSnapshot`（`Game/game.js:15193`）已存在但**没有**接入 snapshot；需要补 `{ currentGalaxyType, panelOpen, pendingChoices[], galaxyChoicesShown, currentGalaxyInfoHtmlPresent, pathDotsHtmlPresent, galaxyMapPathNodes[] }`。
- `build` 子树：当前 snapshot 没有 build palette 直接镜像；`getBuildPaletteDiagnostics`（`Game/game.js:7123`）已存在；需要补 `{ palette[], selectedCellDiagnostics?, placementDiagnostics? }`，但只放摘要字段（卡片数、可负担数、电力压力）避免 HTML 注入。
- `meta` 子树扩展：当前 `metaStorage` 走 `summarizeMetaStorageForPlaytestSnapshot`，**没有**面板状态；需要补 `{ panelOpen, metaPanelToggleVisible, lastTab }` 等纯读字段（不打开 meta panel）。
- `evidence` 与 `checks` 层级语义标注：当前完全没有；§3.5-3.6 定义。

### 3.3 `StorageSignatureBundle`

```text
{
  saveKey: "star-ring-station-meta-v1",
  corruptRawKey: "star-ring-station-meta-corrupt-raw",
  capturedAt: <epoch ms>,
  save: <StorageEntrySignature>,
  corruptRaw: <StorageEntrySignature>
}

StorageEntrySignature = {
  supported: boolean,        // localStorage / sessionStorage 是否可访问
  hasValue: boolean,         // 当前 key 是否存在
  length: <int>|null,        // 值字节长度，null 表示不存在
  head32: <string>|null,     // 前 32 个字符；不存在或不足时返回完整值或 null
  tail32: <string>|null,     // 末 32 个字符；不存在时 null
  hash32: <"0x"+8 hex>|null  // djb2 32-bit hash；不存在时 null
}
```

实现约束：

- `hash32` 用纯 JS：`h = 5381; for c of value: h = ((h*33) ^ c.charCodeAt(0)) >>> 0`；返回 `"0x" + h.toString(16).padStart(8, "0")`；该实现稳定、无依赖、足够区分小修改。
- `head32` / `tail32` 用 `String.prototype.slice`；非 ASCII 直接保留。
- 失败（`storage.getItem` 抛 `SecurityError`）时 `supported: false, hasValue: false, length: null, head32: null, tail32: null, hash32: null`，**不**抛出。
- 不调用 `setItem` / `removeItem`。

签名等价判定（`runChecks` 用）：

```text
isEqual(a, b) = a.supported === b.supported
             && a.hasValue === b.hasValue
             && a.length === b.length
             && a.hash32 === b.hash32
             // head32 / tail32 是辅助证据，不进入等价判定，但记录在 result 里
```

任一字段不等价即 `storage_mutation` blocking。

### 3.4 `FrameTimingResult` 与采样窗口

```text
{
  available: boolean,                  // false 表示 rAF / performance.now 不可用
  reason: <string>|null,               // available 为 false 时填原因
  mode: "rAF" | "performance" | "unavailable",
  requestedFrames: <int>,              // 调用者传入的 sampleFrames
  effectiveFrames: <int>,              // clamp 后实际采样帧数
  durationMs: <number>,                // 实际窗口耗时
  sampleCount: <int>,                  // 与 effectiveFrames 一致；冗余字段方便 PM 解读
  avgMs: <number>|null,
  maxMs: <number>|null,
  longFrameCount: <int>,
  longFrameThresholdMs: <number>,      // 默认 33
  truncated: boolean,                  // requestedFrames > 上限时 true
  context: {
    paused: <boolean>,
    capturedAt: <epoch ms>,
    sampleStartedAt: <epoch ms>,
    sampleEndedAt: <epoch ms>
  }
}
```

实现约束：

- `effectiveFrames = clamp(requestedFrames|0, 6, 120)`，默认 30。
- 采样循环：在 `runChecks` 内开始时记录 `t0 = performance.now()`；递归注册 `requestAnimationFrame((now) => { frames.push(now - prev); prev = now; if (frames.length < effectiveFrames) raf(...) else resolve() })`；返回 Promise。
- `avgMs = sum / length`；`maxMs = Math.max(...frames)`；`longFrameCount = frames.filter(f => f >= longFrameThresholdMs).length`。
- 不支持 rAF 时（jsdom / VM）：`available: false, mode: "unavailable", reason: "requestAnimationFrame not available"`；上层 `runChecks` 对应 check 走 `unknown`。
- 不支持 `performance.now` 时退化为 `Date.now()`；mode 标 `"performance"`；精度记录在 evidence 内。
- 采样过程**不**调用 `update / render / updateHud`；浏览器会在 rAF 间隔继续运行 `loop`，与 frameTiming 相互独立但互相影响（PM 视角看到的就是真实运行帧耗时）。
- 采样不写 `state.lastTime` / `state.hud.timer`；不持有任何 state 锁。

调用方时序：

```text
runChecks(options):
  bindPlaytestErrorListeners()           // 幂等
  signatureBefore = collectStorageSignature()
  baseSnapshot = playtest.snapshot()
  frameTiming = await sampleFrameTiming(options?.sampleFrames ?? 30)
  environment = buildEnvironmentHint()
  galaxy = collectGalaxyHook()
  build = collectBuildHook()
  meta = collectMetaHook()
  hud = collectHudHook()
  evidence = buildEvidenceLayers({ baseSnapshot, frameTiming, ... })
  signatureAfter = collectStorageSignature()
  checks = evaluateChecks({ baseSnapshot, signatureBefore, signatureAfter, frameTiming, ... })
  return safeDeepCloneForTest({ version, capturedAt, baseSnapshot, storageSignature: { before, after, consistent }, frameTiming, environment, galaxy, build, meta, hud, evidence, checks, residualRisks, durationMs })
```

`sampleFrameTiming` 必须**最先**异步完成的步骤之后再 collect signatureAfter，保证签名前后涵盖**最长**的可能写入窗口。

### 3.8 `residualRisks` 自动填项

`runReleaseCandidateChecks` 自动追加以下残留风险（PM 在外部记录里补 manual / browser 未覆盖项）：

- `storage_mutation` — signature before/after 不等价；severity `blocking`；`details: { before, after, changedKey }`
- `rAF_unavailable` — frameTiming `available === false`；severity `warn`；提示 PM 此次取证未拿到帧耗时
- `narrow_viewport_uncovered` — base `viewport.narrowViewport === true` 且 frameTiming 未在该窗口下重复采样；severity `info`
- `field_unavailable` — 任一 hook 子树返回 `{ available: false }`；severity `warn`；列出缺失 hook
- `schema_mismatch` — `base.metaStorage.summary.schemaVersion !== META_SCHEMA_VERSION` 或 `saveKey !== SAVE_KEY`；severity `blocking`
- `frame_long_warning` — `longFrameCount > 0` 但 < 20%；severity `info`，让 PM 知道但不阻塞

`browser_uncovered` / `manual_uncovered` 默认自动填入"全部 P0-1 ~ P0-8 未自动覆盖"，PM 在外部 evidence 中转化为 `pass / fail` 后由 PM 看板手动汇总；自动检查**不**写"已通过"。

## 6. 性能与回潮边界

### 6.1 频率上限

- `getReleaseReadinessSnapshot` 单次调用 worst case：30-80 cells + 4-12 bodies + 全 `state.enemies / state.projectiles / state.repairDrones` 遍历 + 字符串拼接，预算 **< 5ms**（参考 v0.12 §6 已落地的 0.18 秒采样缓存，单次调用与一次 HUD cadence 同量级）
- `runReleaseCandidateChecks({ sampleFrames: 30 })` 总耗时 ≈ **0.5 - 0.6 秒**（30 帧 × 16.67ms = 500ms 主线程窗口），其中函数本体执行 < 10ms
- PM 在 DevTools 中**不应**循环调用 `runChecks`；UI 面板"运行检查"按钮在运行中**禁用**防止重入

### 6.2 与 HUD cadence 关系

- `loop()` 每帧 `update / render`；HUD `updateHud` 0.12 秒 cadence 与 readiness 调用**完全独立**
- readiness 内部调用的 helper（`getStationDesignHealth` 等）有 v0.12 内部 0.18 秒采样缓存；连续两次 `runChecks` 在 0.2 秒内调用时，二级缓存命中，readiness 间接节流
- readiness 不修改 `state.hud.timer`；调用前后 HUD 渲染节奏不变

### 6.3 内存占用

- 单次 readiness 返回深拷贝预算 ≈ **10-30KB JSON 字符化**（base snapshot 已是 5-15KB，evidence + frame timing + storage signature 增量 5-15KB）
- `runChecks` 内部不持有 `result` 长期引用；调用者持有引用的内存由调用方负责
- `playtestRuntimeErrorBuffer` 仍为 50 容量，readiness 内只读 `latest 10`，不影响 buffer 大小
