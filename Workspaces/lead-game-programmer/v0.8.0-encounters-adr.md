## 1. 背景

### 1.1 版本目标摘要

制作人方向把 v0.8.0 定为「局内随机事件系统」——在 v0.7.0 战斗操控之上，加入**平行 ENCOUNTER_TYPES 注册表**，每关随机触发 1-3 个"非主线"遭遇事件（trader 漂流商船 / distress 救援请求 / ambush 伪装陷阱 / derelict 沉船秘窟 / signal 神秘信号），每个事件提供独立选择 + 后果反馈，让玩家在 OBJECTIVE_TYPES 主任务之外多一层"做选择"的乐趣。

整个版本是「新增并行系统」而非「现有系统重构」，OBJECTIVE_TYPES / `createObjective` / `notifyObjective` / `grantObjectiveReward` / v0.7.0 锁敌战斗 / v0.6.0 hostile-station 等全部不修改；玩家可"完全不参与"任何事件（不靠近 trader / 忽略 distress / 远离 ambush）继续游戏，行为等价 v0.7.0。

### 1.3 关键设计风险（高优先级）

1. **RNG 流隔离破坏 `__gameTest` 测试钩子**：现有 `findEscortSeed / getObjectiveType` 依赖 `seed ^ 0x9e3779b9 + level * 0x85ebca6b`。若 ENCOUNTER 与 OBJECTIVE 共用 RNG（哪怕在 RNG 之后追加调用），都会让"同 seed 同关 objective 类型"在 v0.7.0 和 v0.8.0 之间不一致，破坏所有现有种子穷举测试与回归脚本。**必须**为 ENCOUNTER 使用独立公式 `seed ^ 0xb5297a4d + level * 0xcc9e2d51`（与制作人方向 §4 一致），并且**完全不调用 objective 的 RNG**，包括 ENCOUNTER 抽池、spawn 位置、wreck/fragment 内部 RNG 等。
2. **`notifyObjective` 8 处调用追加 `notifyEncounters` 的副作用扩散**：现有 8 处 `notifyObjective` 调用都在关键事件分发点（NPC 死亡 / NPC 到达 / 击毁敌方 station / fragment dock / 采矿 / 击毁 guardian / 击毁敌人 / tick）。如果**修改**这些调用点（如替换为 `notifyAll` 统一派发），会让 v0.7.0 行为发生意外改变。**必须**严格追加策略：每处 `notifyObjective(event, payload)` 后**紧接一行** `notifyEncounters(event, payload)`，不删除、不重排、不合并、不抽象。
3. **`damageNpc` 派发分流陷阱**：现有 `damageNpc` 死亡时直接 `notifyObjective("npcDestroyed", npc)`，escort 任务用它判失败。v0.8.0 加入 ENCOUNTER NPC 后，事件 NPC 死亡如果同样调 `notifyObjective`，会**让 escort 任务 isFailed 误判**（escort.npcId 检查不通过则不影响，但 ENCOUNTER NPC 与 escort NPC 同时存在时仍要确认）。**必须**按 `npc.role` 分流：`role === "objective"` 走 `notifyObjective("npcDestroyed")`、`role === "encounter"` 走 `notifyEncounters("npcDestroyed")`；两者不重叠。`tickNpc` 同款处理 `npcArrived`。
4. **NPC kind 扩展破坏 `buildNpcHudAlerts` 假设**：现有 `buildNpcHudAlerts` 假定 `npc.kind === "friendly-cargo" && objective.type === "escort"`；v0.8.0 加入 `trader / distress-pilot / pirate-ambush` 后，如果不按 `role` 分流而是简单遍历 `state.npcs`，会让事件 NPC 错误触发 escort 失败提示。**必须**按 `npc.role === "objective"` 与 `npc.role === "encounter"` 分别走 `buildNpcHudAlerts` 与 `buildEncounterHudAlerts` 两条独立路径。
5. **`showHostileStationAlert` 多次调用 DOM 堆叠**：现有实现直接 `document.body.appendChild`，多次调用时 DOM 节点同时存在并叠加显示，玩家无法读到任何一条。v0.8.0 同关并发最多 3 个 encounter，每个有 spawn alert + 完成/失败 alert，叠加 v0.6.0 hostile-station alert 与 v0.4.0 escort alert，单关可能短时间内连续 5-8 次调用。**必须**改造为 priority 队列，同时只显示 1 个 DOM 节点，按 priority 抢占；priority 100 = hostile-station，priority 80 = ambush，priority 70 = distress，priority 50 = trader，priority 40 = derelict，priority 30 = signal（与制作人方向 §6 一致）。
6. **`makeWreckFragment` 多次调用与 `WRECK_FRAGMENT_MAX_COUNT` 上限**：现有 `seedSalvageFragments` 循环 spawn wreck fragment 时会检查 `countWreckFragments() >= WRECK_FRAGMENT_MAX_COUNT` 跳出；derelict 事件预期 spawn 4-6 个 wreck，**必须**确认这个上限不会让 derelict 中途丢 wreck；如有冲突，derelict 走独立 origin 且不计入 wreck 总数（或在数值方案与上限协调）。
7. **state.run.encounters 跨关清理与 SAVE_KEY**：v0.8.0 不修改 SAVE_KEY，encounter 字段全部在 `state.run` 而不进入 meta；`ensureRunRuntimeState` 必须在缺失时默认 `state.run.encounters = []`、`state.run.encounterCooldownThisGalaxy = new Set()`。**关键陷阱**：`Set` 不能直接 JSON 序列化（如果未来误把 state.run 写入 SAVE_KEY），但 v0.8.0 范围内 state.run 不进 SAVE_KEY，先保留 Set 性能优势；若数值方案后续需要持久化，改为 Array + Set helper。
8. **`updateEncounters(dt)` 独立函数与 update 顺序**：必须放在 `updateObjective(dt)` **之后**、`updateCamera(dt)` **之前**：
   - 之后：保证 objective 先 tick（关键决定 `state.run.objective.failed/complete`），encounter 后 tick 不影响 objective 判定。
   - 之前：保证 encounter 状态变化能在本帧 camera 更新前生效（如玩家锁定 encounter NPC 时锁定框位置同步）。
   - 如果误放到 updateNpcs / updateEnemies 之前，会让 encounter spawn 的新 NPC 在本帧无法 tick（无害但浪费一帧）；放到 updateEnemies 之后但 updateObjective 之前是次优选择，**保留**作为 fallback。
9. **`encounter.id` 唯一性**：encounter 反向引用 NPC / fragment（`npc.encounterId / fragment.encounterId`）要求 id 唯一；现有 npcIdSeed / fragmentIdSeed 是 module-level 闭包整数，模式可复用；v0.8.0 新增 `encounterIdSeed`，与现有 seed 独立，避免被 reset 跨干扰。
10. **`updateNpcs` / `damageNpc` 时序与 ENCOUNTER NPC 移除**：现有 `damageNpc` 死亡时 `state.npcs.filter` 移除 npc；事件 NPC 也走同款路径。**风险**：若 ENCOUNTER.isComplete 检测依赖"NPC 还在 state.npcs"，则 npc 移除瞬间 encounter 已无法找到引用。**必须**让 ENCOUNTER 在 `notifyEncounters("npcDestroyed", npc)` 收到事件时**先**更新自身计数（如 `pirateRemainingCount--`），再让 damageNpc 移除 NPC。OBJECTIVE_TYPES.escort 现有路径同款（npc.destroyed 标志 + filter），ENCOUNTER 复用即可。

## 2. 设计决定逐项（对应方向文档「关键技术疑问」9 项）

### 2.1 `ENCOUNTER_TYPES` 字段集（决定 1）

- **决定**：采用**专属 `ENCOUNTER_TYPE_DEFAULTS`**，与 `OBJECTIVE_TYPE_DEFAULTS` 部分重叠但语义独立。
- **字段集**：
  ```
  // 与 OBJECTIVE_TYPE_DEFAULTS 同形（沿用）
  tick: null,                         // (encounter, dt) -> void
  onMined: null,                      // (encounter, mined) -> void（一般不用）
  onEnemyKilled: null,                // (encounter, enemy) -> void
  onGuardianKilled: null,             // (encounter, enemy) -> void（一般不用）
  onHostileStationKilled: null,       // (encounter, enemy) -> void（一般不用）
  onFragmentDocked: null,             // (encounter, fragment) -> void
  onNpcArrived: null,                 // (encounter, npc) -> void
  onNpcDestroyed: null,               // (encounter, npc) -> void
  isComplete: null,                   // (encounter) -> boolean
  isFailed: null,                     // (encounter) -> boolean
  getDetail: () => "",                // (encounter, displayProgress) -> string
  getHint: () => "",                  // (encounter) -> string
  render: null,                       // (encounter) -> void（世界标记渲染）

  // v0.8.0 ENCOUNTER 专属字段（OBJECTIVE 无）
  cssClass: "alert-encounter",        // HUD alert 默认 CSS 类
  hudAlertClass: "alert-encounter",   // 中央闪烁 alert CSS 类
  priority: 50,                       // showHostileStationAlert priority（30-80）
  spawnDistanceMin: 400,              // px；与玩家最小距离
  spawnDistanceMax: 1000,             // px；与玩家最大距离
  cooldownTurns: 1,                   // 单星系内最多触发次数（trader=2，其他=1）
  defaultExpireMs: 30000,             // 默认过期时间（ms）

  // v0.8.0 不复用 OBJECTIVE_TYPE_DEFAULTS 的字段
  // rewardMultiplier: 不复用（事件走独立 applyEncounterReward，不计入 grantObjectiveReward）
  // researchReward: 不复用（同上）
  ```
- **理由**：
  - **沿用同形字段**让"主任务"和"事件"开发体验一致，玩法策划学习成本最低；ENCOUNTER NPC 死亡仍走 `onNpcDestroyed` 函数签名，与 escort 任务同款。
  - **新增 `cssClass / hudAlertClass / priority / spawnDistance* / cooldownTurns / defaultExpireMs`** 是 ENCOUNTER 专属语义：OBJECTIVE 只有一个 / 不限时 / 不需要 priority；ENCOUNTER 多个并发 / 限时 / 有 HUD 优先级。把这些字段放进字段集而非每个事件 make() 内部硬编码，便于数值方案统一调整。
  - **不复用 `rewardMultiplier / researchReward`**：避免 `notifyEncounters` 调用 `grantObjectiveReward` 触发 `meta.points / objectiveCompleteAt / awaitingObjectiveChoice` 等 OBJECTIVE 专属副作用；ENCOUNTER 走独立 `applyEncounterReward(encounter, rewardSpec)`，rewardSpec 在 encounter 自身的 onXxx 函数内构造。
- **与 OBJECTIVE_TYPE_DEFAULTS 的差异表**：

| 字段 | OBJECTIVE_TYPE_DEFAULTS | ENCOUNTER_TYPE_DEFAULTS | 说明 |
| --- | --- | --- | --- |
| tick / onMined / onEnemyKilled / onGuardianKilled / onHostileStationKilled / onFragmentDocked / onNpcArrived / onNpcDestroyed | 有 | 有 | 完全同形函数签名，仅第一参数从 objective 改为 encounter |
| isComplete / isFailed | 有 | 有 | 完全同形 |
| getDetail / getHint / render | 有 | 有 | 完全同形 |
| rewardMultiplier | 1.0 | （无） | OBJECTIVE 走 grantObjectiveReward；ENCOUNTER 走 applyEncounterReward |
| researchReward | 0 | （无） | 同上 |
| cssClass | （无） | "alert-encounter" | ENCOUNTER HUD 默认类，由 buildEncounterHudAlerts 读 |
| hudAlertClass | （无） | "alert-encounter" | showHostileStationAlert 调用时附加的 CSS 类 |
| priority | （无） | 50 | priority 队列的默认权重 |
| spawnDistanceMin / Max | （无） | 400 / 1000 | spawn 时的距离范围 |
| cooldownTurns | （无） | 1 | 单星系内最多触发次数 |
| defaultExpireMs | （无） | 30000 | 默认过期 ms |

- **回滚**：若节点 A 实测"ENCOUNTER_TYPE_DEFAULTS 与 OBJECTIVE_TYPE_DEFAULTS 高度重叠让代码冗余"，可在 v0.8.1 重构为"共享 BASE_TYPE_DEFAULTS + ENCOUNTER_EXTRA_FIELDS"两层；v0.8.0 范围内保持独立 default 表，避免抽象过度。

### 2.2 `state.run.encounters` 数据结构（决定 2）

- **决定**：独立数组 + 显式字段对象，**不复用 OBJECTIVE 字段结构**。
- **encounter 对象完整字段**：
  ```js
  {
    id: <integer>,                // 唯一 id，由 encounterIdSeed++ 生成
    type: "trader" | "distress" | "ambush" | "derelict" | "signal",
    state: "pending" | "active" | "resolving" | "complete" | "failed" | "expired",
    spawnAt: <state.time>,        // encounter 生成时刻
    triggerAt: <state.time + delay>,  // pending → active 切换时刻
    expireAt: <state.time + ttl>, // active → expired 切换时刻（ttl 由 defaultExpireMs 决定）
    spawnPos: { x, y },           // 主要 spawn 位置（NPC 起点 / fragment 中心）
    npcIds: [<integer>],          // 反向引用所属 NPC（trader / distress / ambush 用）
    fragmentIds: [<integer>],     // 反向引用所属 fragment（derelict / signal 用）
    enemyIds: [<id 或 reference>],// 反向引用 spawn 的 pirate（distress / ambush 用）；id 不稳定时直接 reference
    completed: false,             // 事件完成标志（玩家选择援助/拾取/交易完成等）
    failed: false,                // 事件失败标志（NPC 死亡 / 过期 / 玩家拒绝等）
    expired: false,               // 过期标志（与 failed 互斥；过期算"未参与"，不算失败）
    encounterData: {              // 类型专属字段（每种 ENCOUNTER make 时填）
      // trader: { wantedResource, wantedAmount, offeredResource, offeredAmount, traded: false }
      // distress: { pirateTargetCount, pirateKilledCount, npcDestroyed: false }
      // ambush: { triggered: false, pirateTargetCount, pirateKilledCount }
      // derelict: { fragmentTargetCount, fragmentDockedCount }
      // signal: { fragmentTargetCount: 1, fragmentDockedCount }
    }
  }
  ```
- **理由**：
  - **独立 type 字段**让 `notifyEncounters` 派发时通过 `ENCOUNTER_TYPES[encounter.type]` 查表（与 OBJECTIVE_TYPES 同款），代码一致。
  - **显式 state 字段**："pending"（延迟期）→ "active"（玩家可交互）→ "resolving"（玩家已选择，等结算）→ "complete/failed/expired"（结束态）的状态机比布尔标志更清晰；同时 HUD 可按 state 显示不同提示文字（"即将出现 / 进行中 / 结算中 / 已结束"）。
  - **npcIds / fragmentIds / enemyIds 反向引用**：保证 encounter 自己知道它生成了哪些实体；NPC / fragment 被销毁时（外部其他系统触发 dock 完成 / wreck 被拾光），encounter 通过反向引用更新自身计数；不依赖每帧遍历 state.npcs / state.fragments。
  - **encounterData 子对象**：每种 ENCOUNTER 类型专属字段不污染主结构；便于未来加事件类型时不改主 schema。
- **`state.run.encounters` 默认值**：`ensureRunRuntimeState` 中：
  ```js
  if (!Array.isArray(state.run.encounters)) state.run.encounters = [];
  if (!(state.run.encounterCooldownThisGalaxy instanceof Set)) {
    state.run.encounterCooldownThisGalaxy = new Set();
  }
  ```
- **`encounterIdSeed`** module-level 闭包整数，与 `npcIdSeed / fragmentIdSeed` 同款。
- **回滚**：若节点 A 实测 encounter.state 状态机过于复杂，可降级为布尔字段 `triggered / completed / failed`，但**不推荐**（HUD 显示语义会模糊）。

### 2.3 事件触发时机：`updateEncounters(dt)` 独立函数（决定 3）

- **决定**：新增 `updateEncounters(dt)` 独立函数，在 `update()` 末尾 `updateObjective(dt)` **之后** + `updateCamera(dt)` **之前**调用。
- **完整调用顺序**（v0.8.0 修改后）：
  ```
  ensureRunRuntimeState();
  state.toastTimer -= dt;
  state.time += dt;

  updatePowerAndFacilities(dt);
  updateMouseAim(dt);
  updateStationPhysics(dt);
  updateFragments(dt);
  updateNpcs(dt);                 // ← npc tick；可能触发 npcArrived 走 notifyObjective + notifyEncounters
  updateMiningAndResearch(dt);
  updateEnemies(dt);              // ← enemy tick；可能触发 enemyKilled / hostileStationKilled
  updateProjectiles(dt);
  updateRepair(dt);
  updateParticles(dt);
  updateObjective(dt);            // ← objective tick；notifyObjective("tick", dt)
  updateEncounters(dt);           // ← v0.8.0 新增；notifyEncounters("tick", dt) + state 推进
  updateCamera(dt);
  trackResearchGrowth(prevResearch);
  ```
- **`updateEncounters(dt)` 内部逻辑**：
  ```js
  function updateEncounters(dt) {
    if (!state.run.encounters || !state.run.encounters.length) return;
    for (const encounter of state.run.encounters) {
      if (encounter.state === "complete" || encounter.state === "failed" || encounter.state === "expired") continue;
      // pending → active 推进
      if (encounter.state === "pending" && state.time >= encounter.triggerAt) {
        activateEncounter(encounter);
      }
      // active 阶段 tick
      if (encounter.state === "active") {
        notifyEncounter(encounter, "tick", dt);
        // 过期判定
        if (encounter.expireAt && state.time >= encounter.expireAt && !ENCOUNTER_TYPES[encounter.type].isComplete?.(encounter)) {
          expireEncounter(encounter);
        }
      }
    }
    // 清理已结束 encounter（保留在数组用于 HUD 历史显示？或直接 filter？v0.8.0 范围保留）
  }
  ```
- **理由**：
  - **放在 updateObjective 之后**：objective 先 tick，避免 encounter tick 改变 enemy/fragment 状态后让 objective 误判。
  - **放在 updateCamera 之前**：camera 跟随 station，不依赖 encounter；但 HUD 渲染依赖 encounter state（如锁定框、NPC 显示），保证 HUD 帧间无 1-tick 延迟。
  - **独立函数**便于测试钩子调用（`__gameTest.advanceEncounters` 可单独 step）。
- **回滚**：若节点 A 实测 updateEncounters 在 updateObjective 之后导致 escort + encounter NPC 状态串扰，可调整为 updateNpcs 之后立即 updateEncounters；但**先**保持 updateObjective 后这一保守选择。

### 2.4 5 种事件 spawn 位置（决定 4）

- **决定**：每种事件有独立 `spawnDistanceMin / spawnDistanceMax` + 角度均匀分布 + 与现有实体的最小距离避让。
- **完整 spawn 距离表**：

| 事件类型 | spawnDistanceMin | spawnDistanceMax | 角度分布 | 与玩家 station 最小距离 | 与 hostile-station 最小距离 | 与 escort NPC 最小距离 | 与现有 enemy 最小距离 |
| --- | --- | --- | --- | --- | --- | --- | --- |
| trader | 400 | 600 | 全角 (0, 2π) 均匀 | 350 | 300 | 200 | 200 |
| distress | 500 | 800 | 全角均匀 | 450 | 400 | 200 | 200 |
| ambush | 600 | 900 | 全角均匀 | 550 | 500 | 250 | 250 |
| derelict | 700 | 1100 | 全角均匀 | 650 | 600 | 300 | 300 |
| signal | 800 | 1400 | 全角均匀 | 750 | 700 | 350 | 350 |

- **`pickEncounterSpawnPos(encounterType, rng)` 算法**：
  ```js
  function pickEncounterSpawnPos(encounterType, rng) {
    const def = ENCOUNTER_TYPES[encounterType];
    const stationPos = state.station.pos;
    for (let tries = 0; tries < 8; tries++) {
      const angle = rngFloat(rng, 0, Math.PI * 2);
      const radius = rngFloat(rng, def.spawnDistanceMin, def.spawnDistanceMax);
      const candidate = {
        x: stationPos.x + Math.cos(angle) * radius,
        y: stationPos.y + Math.sin(angle) * radius
      };
      if (!isInGalaxyBounds(candidate)) continue;
      if (dist(candidate, stationPos) < def.spawnDistanceMin) continue;
      let conflict = false;
      for (const enemy of state.enemies) {
        const minDist = enemy.kind === "hostile-station"
          ? ENCOUNTER_HOSTILE_STATION_AVOID[encounterType]
          : ENCOUNTER_ENEMY_AVOID[encounterType];
        if (dist(candidate, enemy) < minDist) { conflict = true; break; }
      }
      if (conflict) continue;
      for (const npc of state.npcs) {
        if (npc.role === "objective" && dist(candidate, npc.pos) < ENCOUNTER_OBJECTIVE_NPC_AVOID[encounterType]) {
          conflict = true; break;
        }
      }
      if (conflict) continue;
      // 与其他 encounter 实体的最小距离避让（避免 5 个事件挤在一起）
      for (const other of state.run.encounters) {
        if (other.state !== "active" && other.state !== "pending") continue;
        if (dist(candidate, other.spawnPos) < 200) { conflict = true; break; }
      }
      if (conflict) continue;
      return candidate;
    }
    return null;  // spawn 失败；调用方应跳过此 encounter
  }
  ```
- **理由**：
  - **距离递增**：trader 最近（玩家容易主动靠近交易）→ signal 最远（高风险高回报，玩家需主动飞过去）；与制作人方向 §"5 种事件 spawn 位置" 一致。
  - **角度均匀**：玩家不会"总在同一方向遇到事件"，避免可预测。
  - **8 次重试**：spawn 失败率应 < 5%（典型关卡 enemy < 10、其他 encounter < 3、escort npc < 1）；8 次重试足够。
  - **spawn 失败时跳过此 encounter**：不重新滚池（避免 RNG 流非确定），seedEncountersForLevel 抽 3 个时实际可能只 spawn 2 个，HUD 显示不变。
- **回滚**：若节点 B 实测某些事件 spawn 总失败（如 ambush 600-900px 在小星系中超出 bounds），可调小 max 距离；但**先**保持表中值。

### 2.6 `showHostileStationAlert` priority 队列设计（决定 6）

- **决定**：新增 `state.hudCenterAlertQueue: [{ text, ms, isSuccess, priority, cssClass, id }]`，同时只显示 1 个 DOM 节点；按 priority 抢占（高 priority 立即抢占低 priority 的 DOM，低 priority 排队等待）。
- **数据结构**：
  ```js

  if (typeof state.hudCenterAlertCurrent !== "object") state.hudCenterAlertCurrent = null;
  // hudCenterAlertCurrent: { id, el, expireAt, priority } | null
  ```
- **`showHostileStationAlert(message, durationMs, isSuccess, options)` 改造**：
  ```js
  function showHostileStationAlert(message, durationMs = 3000, isSuccess = false, options = {}) {
    const priority = options.priority ?? 100;  // 默认 100 兼容现有 hostile-station 调用
    const cssClass = options.cssClass ?? (isSuccess ? "hud-alert success" : "hud-alert");
    const entry = {
      id: hudAlertIdSeed++,
      text: message,
      ms: durationMs,
      isSuccess,
      priority,
      cssClass
    };
    state.hudCenterAlertQueue.push(entry);
    processHudCenterAlertQueue();
  }

  function processHudCenterAlertQueue() {
    if (!state.hudCenterAlertQueue.length) return;
    // 按 priority 降序排序
    state.hudCenterAlertQueue.sort((a, b) => b.priority - a.priority);
    const next = state.hudCenterAlertQueue[0];

    const current = state.hudCenterAlertCurrent;
    if (current && current.priority >= next.priority) {
      return;  // 当前正在显示更高/同优先级，等其结束
    }
    if (current && current.priority < next.priority) {
      // 抢占当前
      current.el?.remove();
      state.hudCenterAlertCurrent = null;
    }
    // 显示 next
    state.hudCenterAlertQueue.shift();
    const el = document.createElement("div");
    el.className = next.cssClass;
    el.textContent = next.text;
    document.body.appendChild(el);
    state.hudCenterAlertCurrent = { id: next.id, el, expireAt: state.time + next.ms / 1000, priority: next.priority };
    setTimeout(() => {
      el.style.transition = "opacity 0.5s";
      el.style.opacity = "0";
      setTimeout(() => {
        el.remove();
        if (state.hudCenterAlertCurrent?.id === next.id) state.hudCenterAlertCurrent = null;
        processHudCenterAlertQueue();  // 显示下一条
      }, 500);
    }, next.ms);
  }
  ```
- **priority 规范**：

| 来源 | priority | 说明 |
| --- | --- | --- |
| hostile-station 出现/击毁/警告 | 100 | 最高（v0.6.0 现有；与决战体验一致） |
| ambush 触发 | 80 | 高（伪装陷阱玩家必须立即关注） |
| distress 触发 / 失败 | 70 | 中高（救援请求，需玩家决策） |
| trader 出现 / 完成 | 50 | 中（友好事件，玩家可忽略） |
| derelict 发现 | 40 | 中低（拾荒事件，玩家可拖延） |
| signal 出现 | 30 | 低（最不紧急，玩家可后期拾） |

- **理由**：
  - **同时只显示 1 个**：避免 DOM 堆叠 + 玩家阅读困扰；priority 抢占让"重要事件"立即可见。
  - **同 priority 走 FIFO**：sort 是稳定排序（V8 实现），加上 push 顺序自然 FIFO。
  - **现有 hostile-station 调用兼容**：`showHostileStationAlert(message, durationMs, isSuccess)` 不传 options 时 priority 默认 100，行为等价 v0.6.0（除了"DOM 不再堆叠"这一改进）。
  - **不引入 setTimeout 串行**：setTimeout 不响应 priority 抢占，且嵌套深时难以调试；显式队列 + processQueue 状态机更清晰。
- **回滚**：若节点 A 实测队列复杂度过高，可降级为"DOM 最多 1 个但 push 时直接替换"（不排队）；但**不推荐**（玩家会错过低 priority 通知）。

### 2.9 事件并发上限（决定 9）

- **决定**：同关卡最多 3 个并发 encounter；按权重 + cooldown 抽取，超限按 weight 排序选 top 3。
- **`seedEncountersForLevel(level)` 滚池算法**：
  ```js
  const ENCOUNTER_MAX_CONCURRENT = 3;
  const ENCOUNTER_LEVEL_WEIGHTS = [
    // level 0：游戏起步，仅 1 个 trader（柔和介绍）
    { trader: 1.0 },
    // level 1：trader + distress
    { trader: 0.5, distress: 0.5 },
    // level 2-5：5 种全开，权重递增
    { trader: 0.3, distress: 0.25, derelict: 0.25, signal: 0.2 },
    { trader: 0.25, distress: 0.25, ambush: 0.2, derelict: 0.15, signal: 0.15 },
    { trader: 0.2, distress: 0.25, ambush: 0.25, derelict: 0.15, signal: 0.15 },
    { trader: 0.15, distress: 0.25, ambush: 0.3, derelict: 0.15, signal: 0.15 },
    // level 6：终末关无 encounter（保护 guardian 终局体验）
    {}
  ];

  function seedEncountersForLevel(level) {
    const lvl = levelIndex(level);
    if (lvl >= ENDGAME_LEVEL) {
      state.run.encounters = [];
      return;
    }
    const encounterSeed = (state.run.seed ^ 0xb5297a4d) + lvl * 0xcc9e2d51;
    const rng = mulberry32(encounterSeed);
    const weights = { ...ENCOUNTER_LEVEL_WEIGHTS[lvl] };
    const targetCount = pickEncounterCountForLevel(lvl, rng);  // 1-3 by level
    const picked = [];
    for (let i = 0; i < targetCount; i++) {
      if (Object.keys(weights).length === 0) break;
      const type = pickWeighted(rng, weights);
      if (!type) break;
      // cooldown 检查：单星系内此类型已触发过则跳过（trader 例外，可触发 2 次）
      const cooldownLimit = ENCOUNTER_TYPES[type].cooldownTurns ?? 1;
      const triggeredCount = picked.filter((e) => e.type === type).length;
      if (triggeredCount >= cooldownLimit) {
        delete weights[type];
        i--;  // 重抽
        continue;
      }
      // make encounter
      const encounter = makeEncounter(type, rng, lvl);
      if (encounter) {
        picked.push(encounter);
        state.run.encounterCooldownThisGalaxy.add(type);
      }
    }
    state.run.encounters = picked.slice(0, ENCOUNTER_MAX_CONCURRENT);
  }

  function makeEncounter(type, rng, level) {
    const def = ENCOUNTER_TYPES[type];
    const spawnPos = pickEncounterSpawnPos(type, rng);
    if (!spawnPos) return null;  // spawn 失败跳过
    const delayMs = pickEncounterDelay(type, rng, level);   // 60-180s，数值方案精化
    const ttlMs = def.defaultExpireMs;
    const encounter = {
      id: ++encounterIdSeed,
      type,
      state: "pending",
      spawnAt: state.time,
      triggerAt: state.time + delayMs / 1000,
      expireAt: state.time + delayMs / 1000 + ttlMs / 1000,
      spawnPos,
      npcIds: [], fragmentIds: [], enemyIds: [],
      completed: false, failed: false, expired: false,
      encounterData: def.make(rng, level, spawnPos)
    };
    return encounter;
  }
  ```
- **`pickEncounterCountForLevel(level, rng)`** 由数值方案精化（建议 level 0：1 / level 1-2：1-2 / level 3-5：2-3 / level 6：0）。
- **`pickEncounterDelay(type, rng, level)`** 由数值方案精化（建议 trader 60-90s / distress 90-120s / ambush 120-150s / derelict 150-180s / signal 180s）。
- **超限策略**：`picked.slice(0, ENCOUNTER_MAX_CONCURRENT)`；多抽的丢弃（不需要按 weight 重排，pickWeighted 已经按权重抽取）。
- **理由**：
  - **3 个并发上限**避免 HUD 信息过载（5 种 alert + objective + escort + hostile-station 总数 > 6 时阅读困难）。
  - **trader cooldown=2 其他=1**：trader 友好事件可重复（玩家爽快），其他事件单星系 1 次保持新鲜感。
  - **level 0/6 特殊化**：level 0 仅 trader 让新玩家平滑入门；level 6 无 encounter 保护 guardian 决战体验。
- **回滚**：若节点 B 实测 3 并发不够（玩家觉得"事件密度低"），可调到 4；但 v0.8.0 范围内保持 3。

### 4.1 `notifyEncounters(event, payload)` 派发

```js
function notifyEncounters(event, payload) {
  if (!state.run.encounters || !state.run.encounters.length) return;
  const handlerName = OBJECTIVE_EVENT_HANDLERS[event];  // 复用同款映射表
  if (!handlerName) return;
  // 复制数组防止派发过程中 splice/mutation
  const encounters = state.run.encounters.slice();
  for (const encounter of encounters) {
    if (encounter.state !== "active") continue;  // 仅 active 状态接收事件
    const def = ENCOUNTER_TYPES[encounter.type];
    if (!def) continue;
    const handler = def[handlerName];
    if (typeof handler !== "function") continue;
    handler(encounter, payload);
    // 检查 isComplete / isFailed，触发结算
    if (!encounter.completed && def.isComplete?.(encounter)) {
      encounter.completed = true;
      encounter.state = "complete";
      applyEncounterReward(encounter, def.completeReward?.(encounter) || null);
    } else if (!encounter.failed && def.isFailed?.(encounter)) {
      encounter.failed = true;
      encounter.state = "failed";
    }
  }
}

function notifyEncountersForNpc(npc, event) {
  // 仅派发给 npc 所属的 encounter（精准派发，避免广播）
  if (!npc.encounterId) {
    notifyEncounters(event, npc);  // 兜底广播
    return;
  }
  const encounter = state.run.encounters.find((e) => e.id === npc.encounterId);
  if (!encounter) return;
  const def = ENCOUNTER_TYPES[encounter.type];
  if (!def) return;
  const handlerName = OBJECTIVE_EVENT_HANDLERS[event];
  const handler = def[handlerName];
  if (typeof handler === "function") {
    handler(encounter, npc);
    if (!encounter.completed && def.isComplete?.(encounter)) {
      encounter.completed = true;
      encounter.state = "complete";
      applyEncounterReward(encounter, def.completeReward?.(encounter) || null);
    } else if (!encounter.failed && def.isFailed?.(encounter)) {
      encounter.failed = true;
      encounter.state = "failed";
    }
  }
}
```

### 4.2 `seedEncountersForLevel(level)` 滚池

```js
function seedEncountersForLevel(level) {
  const lvl = levelIndex(level);
  state.run.encounters = [];
  state.run.encounterCooldownThisGalaxy = new Set();
  if (lvl >= ENDGAME_LEVEL) return;  // 终末关无 encounter

  // 独立 RNG 流
  const encounterSeed = (state.run.seed ^ ENCOUNTER_RNG_XOR) + lvl * ENCOUNTER_RNG_MUL;
  const rng = mulberry32(encounterSeed);

  const weightsBase = ENCOUNTER_LEVEL_WEIGHTS[lvl] || {};
  if (Object.keys(weightsBase).length === 0) return;

  const targetCount = pickEncounterCountForLevel(lvl, rng);  // 1-3
  const weights = { ...weightsBase };
  const picked = [];

  for (let i = 0; i < targetCount && Object.keys(weights).length > 0; i++) {
    const type = pickWeighted(rng, weights);
    if (!type) break;

    const cooldownLimit = ENCOUNTER_TYPES[type].cooldownTurns ?? 1;
    const triggeredCount = picked.filter((e) => e.type === type).length;
    if (triggeredCount >= cooldownLimit) {
      delete weights[type];
      i--;
      continue;
    }

    const encounter = makeEncounter(type, rng, lvl);
    if (encounter) {
      picked.push(encounter);
      state.run.encounterCooldownThisGalaxy.add(type);
    }
  }

  state.run.encounters = picked.slice(0, ENCOUNTER_MAX_CONCURRENT);
}

function pickEncounterCountForLevel(level, rng) {
  // 数值方案精化；先用 ADR 默认
  if (level === 0) return 1;
  if (level <= 2) return 1 + Math.floor(rng() * 2);  // 1-2
  if (level <= 5) return 2 + Math.floor(rng() * 2);  // 2-3
  return 0;
}
```

```js
function activateEncounter(encounter) {
  encounter.state = "active";
  const def = ENCOUNTER_TYPES[encounter.type];

  // 根据类型 spawn 实体（NPC / pirate / fragment）
  switch (encounter.type) {
    case "trader":
      spawnTraderNpcForEncounter(encounter);
      break;
    case "distress":
      spawnDistressNpcAndPiratesForEncounter(encounter);
      break;
    case "ambush":

      break;
    case "derelict":
      spawnDerelictFragmentsForEncounter(encounter);
      break;
    case "signal":
      spawnSignalFragmentForEncounter(encounter);
      break;
  }

  // 中央 alert
  showHostileStationAlert(
    formatEncounterSpawnText(encounter),
    def.spawnAlertMs ?? 3000,
    false,
    { priority: def.priority, cssClass: def.hudAlertClass }
  );
}

function expireEncounter(encounter) {
  encounter.state = "expired";
  encounter.expired = true;
  // 清理实体：NPC 离场（不算死亡）/ fragment 保留至下关 nextLevel 清理
  for (const npcId of encounter.npcIds) {
    const npc = state.npcs.find((n) => n.id === npcId);
    if (npc && !npc.destroyed && !npc.arrived) {
      // 让 NPC "离场"：设置 npc.arrived = true 让它从 active 列表移除（视觉上飞出屏幕外）
      npc.arrived = true;
    }
  }
  // fragment 不清理（玩家可在剩余关卡时间内拾取）
}
```

### 4.6 `updateEncounters(dt)` 主循环

```js
function updateEncounters(dt) {
  if (!state.run.encounters || !state.run.encounters.length) return;

  for (const encounter of state.run.encounters) {
    if (encounter.state === "complete" || encounter.state === "failed" || encounter.state === "expired") continue;

    // pending → active 推进
    if (encounter.state === "pending" && state.time >= encounter.triggerAt) {
      activateEncounter(encounter);
    }

    // active 阶段 tick
    if (encounter.state === "active") {
      const def = ENCOUNTER_TYPES[encounter.type];
      // 调用类型自身 tick（如 trader 巡航逻辑、ambush 玩家距离触发）
      if (typeof def.tick === "function") {
        def.tick(encounter, dt);
        // tick 后可能立即 complete / failed
        if (!encounter.completed && def.isComplete?.(encounter)) {
          encounter.completed = true;
          encounter.state = "complete";
          applyEncounterReward(encounter, def.completeReward?.(encounter) || null);
          continue;
        }
        if (!encounter.failed && def.isFailed?.(encounter)) {
          encounter.failed = true;
          encounter.state = "failed";
          continue;
        }
      }
      // 过期判定
      if (encounter.expireAt && state.time >= encounter.expireAt) {
        expireEncounter(encounter);
      }
    }
  }
}
```

### 5.2 性能与缓存

- `updateEncounters(dt)` 每帧遍历 `state.run.encounters`（最多 3 个）；每个 encounter 内调一次 def.tick；总开销 < 0.05 ms。
- `notifyEncounters` 每次事件派发遍历 active encounter（最多 3 个）；事件频率与 OBJECTIVE 同款；总开销 < 0.05 ms。
- `buildEncounterHudAlerts` 每帧重算（最多 3 个）；与现有 `buildFragmentHudAlerts / buildNpcHudAlerts` 同款节流（HUD 整体走 `setHtmlIfChanged` 缓存）。
- `processHudCenterAlertQueue` 仅在 push / 当前 alert 结束时触发；非每帧。
- `pickEncounterSpawnPos` 仅在 activateEncounter 时调用 1 次；最多 8 次重试 × O(E + N + Enc) ≈ < 0.5 ms 一次性。

### 5.3 兼容现有 hostile-station 与 escort

- hostile-station 中央 alert 调用 `showHostileStationAlert(message, durationMs, isSuccess)` 默认 priority 100，行为等价 v0.6.0（唯一改进：DOM 不再堆叠）。
- escort NPC `role === "objective"`，走原 `damageNpc → notifyObjective("npcDestroyed")` 路径，v0.4.0 失败逻辑完全等价；`buildNpcHudAlerts` 仅看 `objective.type === "escort"`，不被事件 NPC 干扰。
- v0.7.0 priorityTarget 不变；玩家可右键锁定 encounter 派生的 enemy（pirate / pirate-ambush）。

## 6. 性能预算

| 项 | 估算 | 备注 |
| --- | --- | --- |
| `updateEncounters(dt)` 每帧 | O(E) E ≤ 3 | < 0.05 ms |
| `notifyEncounters(event, payload)` 单次 | O(E) | < 0.02 ms / 次 |
| `seedEncountersForLevel(level)` 一次性 | O(W × spawnRetries) W ≤ 5、retries ≤ 8 | < 1 ms（仅 nextLevel 时） |
| `buildEncounterHudAlerts()` 每帧 | O(E) + sort | < 0.05 ms |
| `processHudCenterAlertQueue` 单次 | O(Q log Q) Q ≤ 5 | < 0.05 ms / 次（非每帧） |
| `pickEncounterSpawnPos` 一次性 | O(retries × (E + N + Enc)) | < 0.5 ms / 次 |
| `applyEncounterReward` 单次 | O(F) F = fragmentSpawn.count ≤ 6 | < 0.5 ms / 次（仅 encounter 完成时） |
| **合计新增（3 个并发 encounter active）** | < 0.3 ms / 帧 | 远低于 v0.7.0 selectTarget + HUD 武器行总开销 |

- **关键决定**：`updateEncounters` 每帧执行；`seedEncountersForLevel / activateEncounter / applyEncounterReward / processHudCenterAlertQueue` 仅事件触发时执行。
- **HUD 武器行 / 任务行 / 资源条**性能与 v0.7.0 等价，不受 encounter 影响。

## 11. 待制作人 / PM 决策的开放项

1. **`pickEncounterCountForLevel` 抽数算法**：
   - 选项 A：纯权重 + cooldown 抽 max ENCOUNTER_MAX_CONCURRENT 个（本 ADR 默认）。
   - 选项 B：每 level 固定数量（如 level 3 固定 2 个 / level 4 固定 3 个）。
   - **本 ADR 推荐：选项 A**（保留 RNG 多样性，玩家不会"知道每关固定几个事件"）；数值方案可微调每 level 期望值。

2. **`encounter.state` 状态机粒度**：
   - 选项 A：5 状态（pending / active / resolving / complete / failed / expired）（本 ADR 默认）。
   - 选项 B：3 布尔字段（triggered / completed / failed）。
   - **本 ADR 推荐：选项 A**（HUD 显示语义更清晰）；若节点 A 实测复杂可降级。

3. **HUD priority 队列抢占策略**：
   - 选项 A：高 priority 立即移除当前 DOM（本 ADR 默认）。
   - 选项 B：高 priority 在当前 alert 结束后立即显示，不抢占。
   - **本 ADR 推荐：选项 A**（"陷阱！"等紧急 alert 必须立即可见）；若节点 B 实测视觉跳变过快可改为选项 B。

4. **trader 交易触发方式**：
   - 选项 A：玩家靠近 200px 自动消耗资源换取奖励（本 ADR 默认）。
   - 选项 B：玩家靠近后弹出 dialog "确认交易？"，玩家点击确认。
   - **本 ADR 推荐：选项 A**（与制作人方向"不引入额外教学层 / 不弹 dialog"一致）；若节点 B 玩家"误触发交易消耗资源"，可加 5s 冷却 + toast 提示。

5. **ambush 触发距离**：
   - 选项 A：玩家 station 中心 200px（本 ADR 默认）。
   - 选项 B：玩家 station 最外侧 cell + 100px。
   - **本 ADR 推荐：选项 A**（station 大小不定，固定距离最简单）；数值方案可微调。

6. **derelict / signal fragment 与现有 `WRECK_FRAGMENT_MAX_COUNT` 冲突**：
   - 选项 A：derelict / signal 不计入 WRECK_FRAGMENT_MAX_COUNT（独立计数）。
   - 选项 B：共享 WRECK_FRAGMENT_MAX_COUNT，超限时跳过 spawn。

7. **`fragment.encounterId` 反向引用**：
   - 选项 A：fragment 持有 encounterId（本 ADR 默认）；encounter dock 时按 fragment.encounterId 反查。
   - 选项 B：encounter 持有 fragmentIds 数组（已默认）+ fragment 不需要反向引用。
