## 1. 背景

### 1.1 版本目标摘要

制作人方向把 v0.6.0 定为「敌方空间站」中期 BOSS 决战：

- 在 level 3-5 玩家完成研发 build 后，由 `assault` 任务主动 spawn 一座 hostile-station，cells 化形成可战术拆解的多目标（先打武器 / 先打动力 / 直击核心）。
- 击毁后爆出 6-12 个 wreck fragment（origin = `"enemy-wreck"`），与 v0.3.0 拾荒 / 桥接 / 设施合并系统天然衔接，形成"决战 → 战利品 → 进入下个星系"的中期高潮。
- 完全静态悬停（不引入移动型 BOSS），多武器 cell 独立 reload + 独立 fire，击毁该武器 cell 即停火该武器（直接战术反馈）。
- 召唤 pirate minion，间隔 7-10s、上限 3-5（与 guardian 风格一致但更克制）。
- 复用 v0.3.0 `fragment.cells` 数据结构、v0.4.0 OBJECTIVE_TYPES 注册表、v0.5.0 `getCellStat` 抽象层；**不改 guardian / 不改 v0.5.0 研发 / 不改 SAVE_KEY**。

### 1.3 关键设计风险（高优先级）

1. **v0.5.0 `getCellStat` 全局加成漏隔离**：`getGlobalModFactor` 读 `state.station.hullMod / weaponMod` 时**只判断 facility 是否在 weapon mod 白名单**，**不判断 cell 是否属于玩家**。如果直接把 hostile-station 武器 cell 调 `getCellStat(cell, "damage")`，cell 会**被玩家 weaponMod ×1.05 增益**——这是严重越界。必须在 cell 上加 `cell.owner = "enemy"` 字段，并在 `getGlobalModFactor` 中判断 owner 跳过 globalMod；同时 `getTechLevelFactor` 也读 `state.station.techLevel`，敌方 cell 不应被玩家 techLevel 加成，必须同样隔离。**这是 v0.6.0 最高优先级风险点。**
2. **cells 化 hostile-station 与 fragment 命中冲突**：`updateProjectiles` 玩家弹分支当前是 `for enemy of state.enemies: dist(projectile, enemy) < enemy.r + projectile.r`；如果 hostile-station 用大 `r`（如 100）做包围球，玩家弹会直接命中"整个 hostile-station"扣血——但 hostile-station 应当 cell 粒度命中。**必须**在命中分支区分 kind：hostile-station 调用 `getHostileStationCellHit(projectile, enemy)`（仿 `getFragmentCellAtWorld`）；其他 kind 走原 `enemy.hp -= damage`。
3. **HP 池与击杀判定语义**：hostile-station 没有"整体 hp"概念（cells 是真实生命池），但所有现有 enemy 路径（`enemy.hp > 0` 过滤、HP bar 渲染、奖励计算 `enemy.reward`、`enemy.maxHp`）都依赖 `hp / maxHp` 标量。**必须**保留 `enemy.hp / enemy.maxHp` 作为聚合显示值：每次 `damageHostileStationCell` 后**重新聚合** `enemy.hp = sum(cell.hp)`、`enemy.maxHp = sum(cell.maxHp)`；这样 HP bar 渲染、enemy.hp > 0 过滤、`triggerGuardianDeathEffect`-style 死亡分支都自动 work。
4. **击杀判定竞态**：core cell 破坏判定与 `enemy.hp <= 0` 过滤之间存在竞态：如果玩家最后一击同时破坏 core 与多个其他 cell，必须保证只触发**一次**死亡分支（wreck 爆出、震屏、奖励、notifyObjective）。**必须**在 `damageHostileStationCell` 内一旦 core hp ≤ 0 立刻 `enemy.hp = 0`（同步），让 `updateEnemies` 末尾的 `filter` 走死亡分支；并加 `enemy._deathHandled` 守卫防重入。
5. **assault 任务与现有 `LEVEL_SPAWN_RATIO` 隔离**：方向明确"hostile-station 仅由 `OBJECTIVE_TYPES.assault.make` 主动 spawn，不污染随机波"。但 `spawnWave` 当前按 LEVEL_SPAWN_RATIO 随机生成 asteroid / pirate / station；**必须**在 spawnWave 中加防御：`if (kind === "hostile-station") return`（虽然 LEVEL_SPAWN_RATIO 不含 hostile-station，但为防未来误配置加保护）；同时 assault `make()` 直接调 `spawnHostileStationNearStation()`，不进 `spawnWave` 路径。
6. **wreck origin 三态**：v0.3.0 fragment.origin 只有 `"player" / "wreck"` 两态（player = 玩家脱落，wreck = salvage 预生成）。v0.6.0 新增 `"enemy-wreck"` 后，所有读 `fragment.origin` 的判断点都要审查：当前确认有 `countWreckFragments`（`origin === "wreck"`）/ `countPlayerFragments`（`origin !== "wreck"`）/ `renderFragments` 内 `isWreck = fragment.origin === "wreck"` / `renderFragmentOutline` 同款判断 / `salvage.onFragmentDocked` 内 `if (fragment.origin !== "wreck") return`。**决定**：`enemy-wreck` 在拾荒桥接路径上**视同 `wreck`**（玩家拖回去合并即得设施，与 salvage 任务无关；salvage 任务的 `onFragmentDocked` 已经过滤 `origin === "wreck"`，enemy-wreck 不会误算入 salvage 任务进度）。**风险点**：`countWreckFragments` 当前是 `origin === "wreck"`，**必须改为 `origin === "wreck" || origin === "enemy-wreck"`**，否则 `WRECK_FRAGMENT_MAX_COUNT` cap 在 hostile-station 爆出后失效；`renderFragmentOutline / renderFragments` 同款判断扩展。
7. **enemy-wreck wreck 数量 cap 与 salvage 冲突**：`seedSalvageFragments` 有 `if (countWreckFragments() >= WRECK_FRAGMENT_MAX_COUNT) break;` 保护。hostile-station 死亡时可能爆出 6-12 个 wreck，若同关已有 salvage 任务（可能性低，因 assault 与 salvage 任务互斥，但 v0.6.0 不强制互斥），会触发 cap。**决定**：hostile-station 死亡 wreck 爆出**不受 WRECK_FRAGMENT_MAX_COUNT 限制**（决战奖励语义优先）；但敌方爆出后总 wreck 数量**应该**临时高于 cap，下一关 `nextLevel` 内 `state.fragments = []` 时统一清理。
8. **HUD 警报与 guardian 警报互斥**：guardian 在 level 6 出现时有红色警报，hostile-station 在 level 3-5 出现时也是红色警报；两者**不会在同一 level 共存**（guardian 仅 level 6），但 HUD `state.run.escortLowHpAlerted` 类似的状态标志需要新增一个 `state.run.hostileStationAlerted` 防止 spawn 警报重复触发。

## 2. 设计决定逐项（对应方向文档「关键技术疑问」8 项）

### 2.1 hostile-station cells 数据结构

- **决定：完全复用 `fragment.cells` 模板**，hostile-station 在数据结构层等同 fragment，仅 `enemy.kind === "hostile-station"` 标签区分行为路径。
- **理由**：
  - v0.3.0 fragment 系统已经稳定运行 3 个版本（v0.3.0/v0.4.0/v0.5.0），cells Map + centerGrid + angle + edgeKeys + bounds 是当前项目内**唯一现成的多 cells 实体模板**；重写一份等于把同样的 bug 重新踩一遍。
  - cells 中的 cell 直接复用 `createCell` 工厂（含 tier=0/mod=null/upgradePath=null/baseMaxHp/baseMaxFrameHp/shield 初始化），无需新工厂。
  - `computeFragmentBounds / computeFragmentEdgeKeys / fragmentCellWorldPosition / damageFragmentCell` 4 个核心算子可**直接复用或最小适配**（伪代码见 §4）。
  - 未来若要做"hostile-station cells 脱落"（v0.7+ 高级特性），fragment 路径已经有 `removeFragmentCell + destroyFragmentIfEmpty` 现成参考。
- **cells 字段在 enemy 对象上的位置**：**`enemy.cells: Map<key, cell>` 平铺**（不嵌套 `enemy.body.cells`）。
  - 理由：与 fragment 等价的形态（fragment 也是平铺 `fragment.cells`），让两套渲染 / 命中算子可以直接共享；嵌套 `enemy.body` 会引入"什么是 body / 什么不是 body"的边界问题。
- **cells 中心点 / 旋转**：直接用 `enemy.x / enemy.y` 作为 cells 中心（与 fragment.pos 等价），`enemy.angle` 作为旋转（与 fragment.angle 等价），`enemy.centerGrid: { x: 0, y: 0 }`（默认 hostile-station 配置围绕 (0,0) 网格设计，不需要重心偏移；如配置文件不对称，centerGrid 可由 `computeFragmentBounds` 计算后取中点）。
- **`enemy.r` 计算**：基于 `computeFragmentBounds(cells)` 后取 `max(spanX, spanY) * 0.5 * CELL`；用于：
  - 雷达 / HUD 距离指示。
  - `dist(world, enemy) < enemy.r + offset` 类外层粗筛（实际命中走 cell 粒度）。
  - `spawnHostileStationNearStation` 时与 station / 其他 enemy 的距离冲突检查。
- **`enemy.hp / enemy.maxHp`**：**保留为聚合值**，每次 cell 受伤后 `enemy.hp = sum(cell.hp)`、`enemy.maxHp = sum(cell.maxHp)`（一次性算到底，不缓存）。用于：
  - HUD HP bar 渲染（`renderEnemies` line 4952 现有路径）。
  - `enemy.hp > 0` 过滤（`updateEnemies` line 4163 现有路径）。
  - 击杀判定**辅助**（主判定走 §2.2 core cell 路径）。
- **`enemy.owner` 字段**：**新增 `enemy.owner = "enemy"` 标签**，传递给 cells 创建路径（每个 cell 上 `cell.owner = "enemy"`），用于 §2.3 武器隔离。

### 2.2 击杀判定

- **决定：core cell HP ≤ 0 即触发死亡分支**（与方向 §"关键技术疑问 #2"建议一致，与玩家 station 死亡判定一致）。
- **理由**：
  - 与玩家 station `damageCell` 内 `if (cell.facility === "core" && cell.hp <= 0) { gameOver(); }` 完全对称；玩家熟悉这条规则，敌方应用同款规则有"反向直觉"。
  - 让"先打 core 速胜"成为可行战术（玩家研发武器 build 强、单次输出高时，可以选择强突 core；玩家武器 build 弱、需要持久战时，可以选择先拆武器 / shield 防御）。
  - 实现简洁：判定函数 `isHostileStationDestroyed(enemy)` 一行；与 §1.3 风险 #3 的"HP 聚合"路径配合（core 破即 sum < 整体，但聚合 hp 与击杀判定**正交**——hp 聚合用于显示，核心破坏用于死亡）。
- **死亡判定函数**：
  ```js
  function isHostileStationDestroyed(enemy) {
    if (!enemy.cells || !enemy.cells.size) return true;
    for (const cell of enemy.cells.values()) {
      if (cell.facility === "core" && cell.hp > 0) return false;
    }
    return true;  // 没有任何 core 或所有 core 都破坏
  }
  ```
- **边界**：如果 hostile-station 配置中**不含 core cell**（数值策划如果将 BOSS 设计为"无 core 纯结构"），则等价于 `cells 全部破坏` 判定（上面函数自然 work：找不到 core 即视为已死）。
- **死亡触发同步**：
  - `damageHostileStationCell` 内：每次伤害后调用 `isHostileStationDestroyed`，若 true，立刻 `enemy.hp = 0` + `enemy._deathHandled = false`（防重入守卫）。
  - `updateEnemies` 末尾 `filter` 已有 `enemy.hp > 0` 检查，自动走死亡分支（line 4163-4179）。
  - 死亡分支内调用 `triggerHostileStationDeath(enemy)`：含 wreck 爆出 + 相机震屏 + `notifyObjective("hostileStationKilled", enemy)` + 粒子。
- **回滚**：若节点 A 体验测试发现"直击 core 太容易"，可在 v0.6.1 引入"core 被 shield cells 保护"机制（shield cells 必须先破才能伤 core）；v0.6.0 不开。

### 2.3 武器 cell 行为（关键隔离）

- **决定：复用 player turret / missile update 逻辑（`getCellStat`），但通过 `cell.owner = "enemy"` 字段隔离玩家研发增益（选项 A）**。
- **理由**：
  - 复用现有 update 逻辑（`updateTurret` line 4241 / `fireMissiles` line 4291）的好处：少写一套 reload + fire + projectile 路径；hostile-station 武器 stat（damage / reload / range / projectileSpeed）可以走 `getCellStat`，让 v0.5.0 数据驱动的 baseStats / techLevel 公式自然适用。
  - 关键问题：`getCellStat` 现有实现读 `state.station.techLevel / hullMod / weaponMod` —— **敌方 cell 不应被玩家研发加成**。
  - **选项 A（cell.owner 显式字段 + getCellStat 内部判断）**：在 `createCell` 之后立刻 `cell.owner = "enemy"`；`getCellStat` 在 `getTechLevelFactor / getGlobalModFactor` 调用前判断 `cell.owner === "player"`（或默认无 owner 即玩家，向后兼容旧 cell），仅玩家走加成路径。
  - **选项 B（隐式：hostile-station cells 不进入玩家面板）**：依赖 `cell.tier=0 / mod=null / upgradePath=null` + UI 层不允许选中敌方 cell 升级。**问题**：getGlobalModFactor 仍会无条件应用 hullMod / weaponMod（不依赖 cell 字段），导致敌方武器 damage ×1.05、maxHp ×1.05；不安全。
  - **选项 A 优于 B**：显式 + 防御性 + 未来扩展（NPC / 友军 cell 可同款隔离）；唯一成本是 `getCellStat` 加 2 行 owner 检查，性能可忽略。
- **`getCellStat` 内部 owner 隔离实现**：
  ```js
  function getCellStat(cell, key) {
    // ... 现有 base / tier / mod 计算保持不变 ...

    // === v0.6.0 owner 隔离 ===
    const isPlayerOwned = !cell.owner || cell.owner === "player";

    const techFactor = isPlayerOwned
      ? getTechLevelFactor(cell.facility, key, state.station.techLevel || 0)
      : 1;
    const globalFactor = isPlayerOwned
      ? getGlobalModFactor(cell.facility, key, state.station)
      : 1;

    // ... 后续公式保持不变 ...
  }
  ```
- **hostile-station 武器 update 入口**：
  - **不直接复用 `updateTurret(cell, dt)`**（该函数硬编码 `nearestEnemy` + `hasLineOfSight` + `fireProjectile("player", ...)`，会让敌方武器误伤敌人 / 走 player projectile 路径）。
  - **新增 `updateHostileStationWeapon(enemy, cell, dt)`**：
    - 索敌：调用 `state.station.pos`（直接打玩家），不调 `nearestEnemy`。
    - 射程：`dist(cellWorld, state.station.pos) < getCellStat(cell, "range")`。
    - reload：`cell.reload -= dt; if (cell.reload > 0) return;` + `cell.reload = getCellStat(cell, "reload");`。
    - fire：`fireProjectile(cellWorld, dir, "enemy", getCellStat(cell, "damage"), getCellStat(cell, "projectileSpeed"));`（注意 owner = "enemy"，让 `updateProjectiles` 走玩家命中分支）。
  - 在 `updateEnemies` 内 hostile-station 分支遍历 `enemy.cells.values()`，对 `facility === "turret"` 调 `updateHostileStationWeapon`。
  - **missile cell**：v0.6.0 范围**不实现**（数值方案应给 hostile-station 配 turret + shield + armor + core + power 即可；missile 涉及 `state.resources` 检查，敌方没有资源池，会引入 `if (cell.owner === "enemy") skip resource check` 类硬编码；留 v0.7+）。
- **shield cell（敌方）**：方向方案有 shield，但 `updateShield(cell)`（line 4276）当前是"过滤敌方 projectile"——hostile-station shield 应当过滤**玩家** projectile。同款问题。
  - **新增 `updateHostileStationShield(enemy, cell)`**：遍历 `state.projectiles`，过滤 `projectile.owner === "player" || projectile.owner === "missile"`，距 `<getCellStat(cell, "range")` + 方向匹配，则 `projectile.dead = true`。
  - 与 player updateShield 是对称镜像，约 25 行。
- **风险**：`getCellStat` owner 判断改动会影响**所有现有 cell**（玩家 cell 默认 `cell.owner` 不存在 → `!cell.owner || cell.owner === "player"` 为 true → 走原路径），不破坏 v0.5.0。但建议节点 A 用 5 个 seed 跑完 turret / missile / shield / mining / processor 行为对比，确保 owner 改动零回潮。
- **回滚**：若 owner 判断引入回归，可在 `getCellStat` 内默认 owner 处理改为"显式判断 enemy 才隔离"：`const isEnemyOwned = cell.owner === "enemy"; if (!isEnemyOwned) { ... 原路径 ... } else { techFactor = 1; globalFactor = 1; }`；语义等价但更保守。

### 2.4 HP 与平衡

- **决定**：HP 池由数值方案精确化，本 ADR 给出区间约束：
  - 总 HP 池（cells maxHp 总和）：**level 3 → ~1500、level 4 → ~2200、level 5 → ~3000**（介于 pirate 400-450 和 guardian 1200 之间，体感"BOSS 级"但不超 guardian）。
  - 单 cell HP：**150-300**（core 250-300、turret 150-200、shield 200-250、armor 200-250、power 150-180）。
  - 武器 damage：**12-15**（与 pirate 同档但更准；level 3-5 区间 LEVEL_STATS.pirate.projectile 是 `[6,7,8,10,12,13]`）。
  - 武器 reload：**1.6s**（与 enemy 通用 reload 一致）。
  - 武器 range：**480-550**（介于 pirate 380 和 guardian 600 之间）。
- **是否随 techLevel / level 缩放**：
  - **基础 HP 按 level 缩放**：与 `ENEMY_LEVEL_STATS` 模式一致，在 `getEnemyData("hostile-station", level)` 内按 level 索引返回不同 hp 总和；具体每 cell 的 hp 在配置中按比例缩放。
  - **cells HP 不受玩家 techLevel 影响**：由 §2.3 owner 隔离自动实现（hostile-station cells `owner="enemy"`，`getCellStat` 不应用 techFactor）；保证玩家 techLevel 自动升级（每 35+22n research 触发一次 `improveStationHp(1.08)`）**只增益玩家自身 cell**。
  - **不实现"敌方 techLevel"**：v0.6.0 范围内 hostile-station 不进化（与制作人不做 v0.5.1 候选项一致）。
- **配置形式**：
  ```js
  const HOSTILE_STATION_LEVEL_HP = {
    3: 1500,
    4: 2200,
    5: 3000
  };

  const HOSTILE_STATION_LAYOUT = {
    // 网格坐标 + facility + hp 比例（数值方案 §X.X 精化）
    cells: [
      { x: 0, y: 0, facility: "core",    hpRatio: 0.20 },
      { x: 1, y: 0, facility: "turret",  hpRatio: 0.12 },
      { x: -1, y: 0, facility: "turret", hpRatio: 0.12 },
      { x: 0, y: 1, facility: "power",   hpRatio: 0.10 },
      { x: 0, y: -1, facility: "shield", hpRatio: 0.15 },
      { x: 1, y: 1, facility: "armor",   hpRatio: 0.15 },
      { x: -1, y: -1, facility: "armor", hpRatio: 0.15 }
      // 总 hpRatio = 1.0（cells maxHp 由 levelHP × hpRatio 计算）
    ]
  };
  ```
  - 数值策划在 numerics 文档中可调整 cells 数量（5-8 个）、布局、hpRatio 与 facility 类型。
- **风险**：hostile-station HP 池过低（玩家武器秒杀）或过高（30s+ 拖战）；节点 B 由 PM 用 3 个不同 build seed 实测击毁时间 15-30s 为合理区间。

### 2.5 AI 静态程度

- **决定**：
  - **复用 `enemyUsesStationAi` 标签**（添加 hostile-station 到 `kind === "station" || kind === "guardian" || kind === "hostile-station"`）；自动获 470 距离悬停模板。
  - 通过 `getEnemyData("hostile-station", level)` 返回**极小** `accel: 4 / drag: 0.5`，让位置漂移 < 几像素但视觉静止。
  - 缓慢自转：在 `updateEnemies` 内 hostile-station 分支独立加 `enemy.angle += enemy.angularVel * dt`，其中 `enemy.angularVel = rand(0.1, 0.3)`（rad/s，与 fragment 同款语义）。
- **理由**：
  - 复用 `enemyUsesStationAi` 比单写 AI 函数少 30 行代码；悬停 470 距离的逻辑（line 4108-4111）已经稳定。
  - accel = 4 / drag = 0.5 测算（玩家在 470 内 hostile-station 不主动靠近、玩家在 470 外 hostile-station 微推几像素后立刻被 drag 吃掉）：每帧加速度 4 × 0.0167 = 0.067 px/s²；drag 衰减 1 - min(0.5 × 0.0167, 0.3) = 1 - 0.008 ≈ 0.992，**实际峰值速度 ≈ 8 px/s，1 秒内移动 ≈ 8 px**，玩家视觉感知静止。
  - 缓慢自转（angularVel 0.1-0.3 rad/s ≈ 6°-17°/s）：12 秒一圈到 60 秒一圈，视觉活力但不会改变武器朝向（hostile-station 武器索敌玩家是动态计算，不依赖 enemy.angle）。
- **`enemy.angularVel` 字段**：新增 enemy 字段（与现有 `enemy.spin` 区分；spin 是 asteroid/pirate 用，固定每帧加 `enemy.spin * dt`；angularVel 是 hostile-station 用，语义同 fragment.angularVel）；在 `updateEnemies` 内 if hostile-station 分支独立处理，避免覆盖 spin 路径。
- **风险**：accel=4 / drag=0.5 在玩家拉远 470+ 距离时 hostile-station 可能位置漂移过大，建议节点 A 实测；如漂移 > 50 px，accel 进一步降到 2，drag 提升到 0.7。
- **回滚**：若 `enemyUsesStationAi` 扩展引入回归（如 station kind 行为变化），改为独立判断 `if (enemy.kind === "hostile-station") { ... 单独 AI ... }`，与 station kind 解耦；多 ~20 行但更安全。

### 2.6 召唤手下

- **决定**：
  - 召唤间隔：**spawnInterval 7-10s**（数值方案精化，区间内为合理范围）。
  - 召唤上限：**spawnCap 3-5**（与 guardian 一致风格但更克制）。
  - minion 类型：**pirate**（沿用现有 pirate AI）。
  - minion 行为：**直接索敌玩家**（沿用现有 pirate AI，**不**绑定 hostile-station 周围旋转）。
  - minion 标签：`hostileStationMinion: true`（仿 `guardianMinion: true`）便于平衡 / 数据采集 / 后续统计。
- **理由**：
  - minion 直接索敌玩家比"绕站旋转"省一套 AI（绕站旋转需要新 AI 状态机：维持距离 + 切线速度 + 索敌切换；与 v0.6.0"不引入复杂 AI"硬约束冲突）。
  - 制作人方向建议"跟随 hostile-station 周围"，但代码实现成本高（需要新 AI 状态）；本 ADR 优先**直接索敌玩家**作为 v0.6.0 选择；如节点 B 体验测试发现"minion 立刻冲玩家显得突兀"，可在 v0.6.1 引入"出生 3s 内绕站旋转 + 之后冲玩家"折衷方案，v0.6.0 不做。
  - `hostileStationMinion` 标签便于：数值平衡（如未来 hostile-station 死亡后清场 minion）、击杀统计（assault 任务可选附加目标：击毁 hostile-station + 清掉 minion）。
- **实现**：
  - hostile-station 在 `updateEnemies` 内的 spawn 分支独立处理（**不**走 guardian / station kind 的 spawn 分支）：
    ```js
    if (enemy.kind === "hostile-station" && enemy.spawnInterval > 0) {
      enemy.spawn -= dt;
      if (enemy.spawn <= 0) {
        enemy.spawn = enemy.spawnInterval;
        const minions = state.enemies.filter((m) => m.hostileStationMinion && m.hp > 0).length;
        if (minions < enemy.spawnCap) {
          spawnEnemy("pirate", enemy.x + rand(-100, 100), enemy.y + rand(-100, 100), {
            level: state.run.level,
            hostileStationMinion: true
          });
        }
      }
    }
    ```
  - `spawnEnemy` 需要接受 `options.hostileStationMinion` 字段并写入 enemy（仿现有 `guardianMinion: Boolean(options.guardianMinion)`）。
- **风险**：spawnCap 4 + interval 8s 在 60 秒战斗中可能召唤多达 8-10 个 pirate（首次 3 个 + 后续 5-7 次补充），玩家压力过大；节点 B 实测玩家负担，若过载，spawnCap 调到 3 + interval 10s。
- **回滚**：若直接索敌玩家在某些 build 下"过于不公"，可在 v0.6.1 引入"出生 2-3s 内绕站旋转"过渡 AI。

### 2.7 wreck 爆出

- **决定**：
  - 数量：**6-12 个 wreck fragment**（数值方案精化，按 hostile-station level 缩放）。
  - 设施分布：**turret 50% / shield 25% / armor 15% / repair 10%**（数值方案精化；与玩家 salvage 用 `WRECK_FACILITY_WEIGHTS` 的 turret 60% / shield 20% / armor 15% / repair 5% 略有差异——敌方爆出**偏战斗设施**，玩家拾荒后可强化 build）。
  - 新增独立常量：**`ENEMY_WRECK_FACILITY_WEIGHTS`**（与 `WRECK_FACILITY_WEIGHTS` 并列，**不**修改后者）。
  - 新增 fragment.origin 第 3 态：**`"enemy-wreck"`**。
  - 爆出位置：从 hostile-station 中心半径 **80-150 像素**内随机分布（更近 → 玩家可以集中拾荒；不过散 → 避免与 boss spawn 位置冲突）。
- **理由**：
  - 独立 `ENEMY_WRECK_FACILITY_WEIGHTS` 让"敌方决战奖励"有特色（偏战斗 = 玩家 build 在击毁 hostile-station 后**直接获得战斗设施加强**，形成"打 BOSS → 拾荒 → 下一关变强"的爽快闭环）；玩家正常 salvage 任务则保留通用 facility 分布。
  - `origin = "enemy-wreck"` 与 `"wreck"` 区分但**视同 wreck** 走桥接 / 拾荒路径（修改点见 §1.3 风险 #6 的 `countWreckFragments / renderFragmentOutline / renderFragments` 扩展）。
  - 80-150 像素散布：hostile-station r ≈ 80（cells span 3-4 格 × 32 + 周边 8-16 padding），爆出位置在 80-150 半径内，保证 wreck 不与 hostile-station "重叠生成"（避免视觉混乱），同时也不太远（150 内玩家移动 1-2 秒内可到）。
- **实施位置**：
  - 在 `updateEnemies` 内 enemy 死亡分支（line 4163-4179）加分支：`if (enemy.kind === "hostile-station") { triggerHostileStationDeath(enemy); }`。
  - `triggerHostileStationDeath(enemy)` 内：
    1. 调用 `spawnEnemyWrecks(enemy, level)`：按 `ENEMY_WRECK_LEVEL_COUNT[level]` 数量调 `makeEnemyWreckFragment(pos, rng)`，pos 从 enemy.x/y + 80-150 半径随机；fragment.origin = "enemy-wreck"。
    2. 调用 `triggerCameraShake(72, 1.3)`（比 guardian 64/1.1 略强，强调中期 BOSS 决战感）。
    3. 调用 `notifyObjective("hostileStationKilled", enemy)`。
    4. spawn 30-50 个粒子（红色 / 黄色混合，仿 guardian 死亡效果但更大）。
- **wreck cell HP 状态**：默认 `hp = maxHp * rngFloat(0.65, 0.9)`（与 `createWreckCell` line 2870 一致）；这是"残骸"语义。
- **风险**：
  - `WRECK_FRAGMENT_MAX_COUNT`（v0.3.0 cap，约 8）若被 hostile-station 一次性爆出 12 个 wreck 超过，影响 salvage 任务；§1.3 风险 #7 处理。
  - wreck 与 hostile-station 残骸视觉重叠（hostile-station enemy 已死亡被 filter 移除，但 wreck 立刻 spawn 在原位置）：节点 B 测试是否出现"wreck 重合"问题；可在 spawn 时按"环形分布"（每个 wreck 占 360°/N 角度 + 半径 100-150 抖动）改善散布。
- **回滚**：若 enemy-wreck 引入 v0.3.0 salvage / wreck 路径回归，临时把 `origin = "wreck"` 即可（敌方爆出会被 salvage 任务误算入进度，但是 v0.6.1 修复成本低）。

### 2.8 生成节奏与冷却

- **决定**：
  - assault 任务在 `OBJECTIVE_LEVEL_WEIGHTS` 中扩展：
    - level 3：**`assault: 0.25`**（占该 level 总权重 25%）。
    - level 4：**`assault: 0.35`**（35%）。
    - level 5：**`assault: 0.30`**（30%）。
    - level 0/1/2/6 **不出**（与 guardian level 6 互斥；level 0-2 玩家尚无 build 不适合 BOSS）。
  - 单星系上限：**1 次**（state.run 新增 `hostileStationSpawnedThisGalaxy` 标记，nextLevel / jumpToNextGalaxy 时重置为 false）。
- **理由**：
  - 概率分布：level 4 最高（玩家 build 接近成熟，决战体验最佳）；level 3 较低（玩家 build 起步）；level 5 较低（不与即将到来的 guardian level 6 抢戏）。
  - 单星系 1 次：避免 assault 任务在同一 galaxy 内被刷出多次（任务系统当前每 level 选一次 objective，不会重复刷新；但 `nextLevel` 进入新 galaxy 时会重新 createObjective，逻辑上需要冷却标志防御）。
- **`OBJECTIVE_TYPES.assault.make` 函数**：
  ```js
  assault: {
    ...OBJECTIVE_TYPE_DEFAULTS,
    rewardMultiplier: 1.5,
    researchReward: 9,
    make(rng, level, galaxy) {
      const objective = {
        type: "assault",
        text: "摧毁敌方空间站",
        target: 1,
        progress: 0,
        spawnPending: true
      };
      // 延迟 spawn：让玩家有 2-3s 反应时间（HUD 警报 → 雷达定位）
      objective._spawnDelay = 2.5;
      return objective;
    },
    tick(objective, dt) {
      if (objective.spawnPending) {
        objective._spawnDelay -= dt;
        if (objective._spawnDelay <= 0) {
          spawnHostileStationNearStation(rng);
          objective.spawnPending = false;
          state.run.hostileStationSpawnedThisGalaxy = true;
        }
      }
    },
    onEnemyKilled(objective, enemy) {
      // 不在这里完成；走 hostileStationKilled 事件以精确匹配 enemy.kind
    },
    onHostileStationKilled(objective, enemy) {
      objective.progress = objective.target;
    },
    getDetail(objective, displayProgress) {
      if (objective.spawnPending) return "扫描中…";
      if (displayProgress.current >= displayProgress.target) return "已摧毁";
      const target = state.enemies.find((e) => e.kind === "hostile-station" && e.hp > 0);
      if (!target) return "扫描中…";
      const hpPct = Math.floor(target.hp / target.maxHp * 100);
      return `敌方空间站 HP ${hpPct}%`;
    },
    getHint() {
      return "敌方空间站已经出现：先打武器 cell 让它停火，或者直击核心快速终结。";
    },
    render(objective) {
      if (isObjectiveComplete()) return;
      const target = state.enemies.find((e) => e.kind === "hostile-station" && e.hp > 0);
      if (!target) return;
      renderer.ring(target, target.r + 28, 4, [1, 0.2, 0.15, 0.5], 52);
      renderer.ring(target, target.r + 56, 2, [1, 0.2, 0.15, 0.2], 52);
    }
  }
  ```
  - **关键**：assault 不直接在 `make()` 内 spawn hostile-station（避免 nextLevel 后玩家还没看到星系就被 BOSS 出现）；通过 `tick` + spawnPending 延迟 2.5s 后再 spawn，玩家有 HUD 警报 + 雷达定位时间。
- **`spawnHostileStationNearStation(rng)` 仿 `spawnGuardianNearStation`**：距离 1000-1300 像素（与 guardian 1100-1300 接近），随机角度，确保不与 station / 已有 enemy 重叠。
- **`OBJECTIVE_EVENT_HANDLERS` 扩展**：新增 `hostileStationKilled: "onHostileStationKilled"`（第 8 个事件入口）。
- **单星系冷却**：
  - `state.run.hostileStationSpawnedThisGalaxy: boolean` 新字段（默认 false）。
  - `assault.make` 内（或 `tick` 内）先判断：if `state.run.hostileStationSpawnedThisGalaxy` 已为 true，则跳过 spawn（理论上同 level 不会重复 createObjective，但防御性）。
  - `nextLevel()`（line 4606）内加 `state.run.hostileStationSpawnedThisGalaxy = false;`。
  - `resetRun()` 内同款重置。
- **`pickWeighted` 不需要改**（已基于 `OBJECTIVE_TYPES[type]` 过滤，assault 注册后自动可选）。
- **风险**：
  - level 3-5 weight 总和需要重新平衡：当前 level 3 是 `mine 0.3 / explore 0.3 / battle 0.2 / salvage 0.2`（总 1.0）；加 assault 0.25 后总 1.25，相当于 assault 占 0.25/1.25 = 20% 概率，其他任务相应稀释。**数值方案**应给出具体配置（如把 mine 调到 0.25 等，使总和回到 1.0），本 ADR 仅给区间。
  - 单星系冷却字段如果未在 `resetRun` 重置，跨 run 会污染；T4 必须显式重置。
- **回滚**：若 assault 概率过高（玩家每关都遇到），数值方案下调 weight；若实现复杂阻塞，可临时把 assault 仅注册 level 4 一关（最重点位置）。

## 3. 数据结构草案

### 3.1 hostile-station enemy schema

```js
// spawnHostileStationNearStation(rng) 后产生的 enemy 对象
const enemy = {
  // === 标准 enemy 字段 ===
  kind: "hostile-station",
  x: 0, y: 0,               // 世界坐标，cells 中心
  vx: 0, vy: 0,             // 极小漂移（accel=4/drag=0.5）
  angle: 0,                 // 当前自转角度
  angularVel: 0.18,         // rad/s，缓慢自转（仿 fragment）
  reload: 0,                // hostile-station 自身不持有武器；reload 由每个武器 cell 独立持有
  spawn: 8,                 // 召唤 minion 倒计时
  spawnInterval: 8,         // 召唤间隔
  spawnCap: 4,              // minion 上限
  spawnLevel: 3,            // minion 等级（= hostile-station 出现的 run.level）

  // hp 聚合（用于 HUD 显示和 enemy.hp > 0 过滤）
  hp: 2200,                 // sum(cell.hp)
  maxHp: 2200,              // sum(cell.maxHp)
  r: 86,                    // 由 computeFragmentBounds 后的 max(spanX, spanY) * 0.5 * CELL

  // === v0.6.0 新增字段 ===
  cells: new Map(),         // Map<key, cell>，cell 由 createCell + cell.owner="enemy" 包装
  centerGrid: { x: 0, y: 0 }, // cells 重心（用于 fragmentCellWorldPosition 复用）
  edgeKeys: new Set(),      // 边缘 cell 集合（命中粗筛用）
  bounds: { minX: -1, maxX: 1, minY: -1, maxY: 1 }, // 缓存
  owner: "enemy",           // 标签，传递给 cells 用于 getCellStat 隔离
  level: 3,                 // run.level 快照

  // 现有字段（spawnEnemy 写入）
  collisionDamage: 12,      // 玩家撞击 hostile-station 时受到的伤害
  reward: 80,               // 击毁后 metal 奖励（hostile-station 在死亡分支会**改为不直接给 metal**，由 wreck + 任务 reward 替代；或保留小额 reward）
  projectileDamage: 0,      // hostile-station 自身不射，武器 cell 各自处理
  reloadTime: 99,           // 同上，禁用
  spin: 0,                  // 不用 spin（用 angularVel）
  range: 0,                 // 同上，禁用
  accel: 4, drag: 0.5,      // 近静态

  // 死亡守卫
  _deathHandled: false      // 防 wreck 爆出重复触发
};
```

### 3.2 hostile-station cell schema（与 fragment cell 等价）

```js
// 由 createHostileStationCell(x, y, facility, hpRatio, totalHp) 工厂创建
const cell = {
  // === 标准 cell 字段（createCell 工厂保留所有字段）===
```

```js
function tryDamageHostileStationCell(projectile) {
  for (const enemy of state.enemies) {
    if (enemy.kind !== "hostile-station") continue;
    if (enemy.hp <= 0) continue;
    // 粗筛（包围球）
    if (dist(projectile, enemy) > enemy.r + projectile.r + CELL * 0.6) continue;
    // 精筛（cell 粒度）
    const hit = getHostileStationCellAtWorld(enemy, projectile);
    if (hit) {
      damageHostileStationCell(enemy, hit, projectile.damage);
      return true;
    }
  }
  return false;
}

function getHostileStationCellAtWorld(enemy, world) {
  // 复用 fragment 命中模型（fragmentCellWorldPosition）
  const transform = { pos: { x: enemy.x, y: enemy.y }, angle: enemy.angle, origin: enemy.centerGrid };
  for (const cell of enemy.cells.values()) {
    const cellPos = cellWorldPositionByTransform(cell, transform);
    if (dist(world, cellPos) < FRAGMENT_PROJECTILE_HIT_RADIUS) return cell;
  }
  return null;
}

function damageHostileStationCell(enemy, cell, damage) {
  if (enemy._deathHandled) return;
  // 仿 damageFragmentCell：core / frame cell 走 frameHp，其他走 hp
  if (cell.facility !== "frame" && cell.facility !== "core") {
    cell.hp -= damage;
    if (cell.hp <= 0) {
      const hitPos = cellWorldPositionByTransform(
        cell,
        { pos: { x: enemy.x, y: enemy.y }, angle: enemy.angle, origin: enemy.centerGrid }
      );
      enemy.cells.delete(key(cell.x, cell.y));
      enemy.edgeKeys = computeFragmentEdgeKeys(enemy.cells);
      enemy.bounds = computeFragmentBounds(enemy.cells);
      spawnParticle(hitPos, [1, 0.42, 0.32, 0.9]);
    }
  } else {
    cell.hp -= damage;
    cell.frameHp -= damage;
    if (cell.facility === "core" && cell.hp <= 0) {
      // 核心破坏即触发死亡
      enemy.hp = 0;
      return;  // 让 updateEnemies 末尾 filter 走死亡分支
    }
    if (cell.frameHp <= 0 && cell.facility !== "core") {
      enemy.cells.delete(key(cell.x, cell.y));
      enemy.edgeKeys = computeFragmentEdgeKeys(enemy.cells);
      enemy.bounds = computeFragmentBounds(enemy.cells);
    }
  }
  // 重新聚合 hp / maxHp（用于 HUD 显示 + filter 判定）
  const agg = aggregateHostileStationHp(enemy.cells);
  enemy.hp = agg.hp;
  enemy.maxHp = agg.maxHp;

  if (enemy.cells.size === 0) {
    enemy.hp = 0;
```

```js
function pickEnemyWreckFacility(rng) {
  // 与 pickWreckFacility 同款实现，但读 ENEMY_WRECK_FACILITY_WEIGHTS
  let total = 0;
  for (const [, w] of ENEMY_WRECK_FACILITY_WEIGHTS) total += w;
  let roll = rng() * total;
  for (const [facility, weight] of ENEMY_WRECK_FACILITY_WEIGHTS) {
    roll -= weight;
    if (roll <= 0) return facility;
  }
  return ENEMY_WRECK_FACILITY_WEIGHTS[0][0];
}

function spawnEnemyWrecks(enemy, rng) {
  const level = enemy.level || state.run.level;
  const count = HOSTILE_STATION_WRECK_COUNT_BY_LEVEL[level] || 6;
  const minR = HOSTILE_STATION_WRECK_SPAWN_RADIUS_MIN;
  const maxR = HOSTILE_STATION_WRECK_SPAWN_RADIUS_MAX;
  const placed = [];
  for (let i = 0; i < count; i++) {
    let pos = null;
    for (let tries = 0; tries < 6; tries++) {
      const ang = rngFloat(rng, 0, Math.PI * 2);
      const rad = rngFloat(rng, minR, maxR);
      const candidate = {
        x: enemy.x + Math.cos(ang) * rad,
        y: enemy.y + Math.sin(ang) * rad
      };
      // 与已 placed 的 wreck 保持最小间距
      let ok = true;
      for (const existing of placed) {
        if (dist(candidate, existing) < WRECK_MIN_SEPARATION * 0.7) { ok = false; break; }
      }
      if (ok) { pos = candidate; break; }
    }
    if (!pos) continue;
    placed.push(pos);
    state.fragments.push(makeEnemyWreckFragment(pos, rng));
  }
}
```

  - 整体 HP bar 由 `renderEnemies` 末尾 `clamp(enemy.hp / Math.max(1, enemy.maxHp || enemy.hp))` 路径自动渲染（聚合 hp 字段已经写好）。
  - 危险圈：由 `assault.render` 内 `renderer.ring` 双环渲染（仿 guardian 红圈）。

```js
// renderEnemies 内 hostile-station 分支：
} else if (enemy.kind === "hostile-station") {
  const transform = {
    pos: { x: enemy.x, y: enemy.y },
    angle: enemy.angle,
    origin: enemy.centerGrid
  };
  for (const cell of enemy.cells.values()) {
    renderCellAt(cell, transform, {
      alpha: 1,
      desaturation: 0.15,
      drawHpBar: false,
      drawEffects: true,
      forceInactive: false
    });
  }
  // 整体红色危险光晕（hostile-station 标识，可选）
  const pulse = 0.5 + 0.5 * Math.sin(state.time * 2.0);
  renderer.ring(enemy, enemy.r + 14, 3, [1, 0.25, 0.2, 0.18 + pulse * 0.12], 48);
}
```

## 6. 性能预算

| 项 | 估算 | 备注 |
| --- | --- | --- |
| spawnHostileStationNearStation 单次 | O(N cells) ≈ 7 次 createCell + computeFragmentBounds + computeFragmentEdgeKeys | 一次性 < 1 ms |
| damageHostileStationCell 单次 | O(N cells) for aggregate + O(N) for edgeKeys | < 0.1 ms（hostile-station cells 数 5-8） |
| getHostileStationCellAtWorld 单次 | O(N cells) dist 计算 | < 0.05 ms |
| updateHostileStationWeapon × 每个武器 cell 每帧 | O(1) 索敌 + reload 检查 + 可能 fireProjectile | < 0.01 ms / cell |
| updateHostileStationShield × 每个 shield cell 每帧 | O(P 玩家弹数 × 1) | < 0.05 ms（典型玩家弹 < 20 个） |
| hostile-station 渲染 | O(N cells) × renderCellAt | < 0.5 ms（与 fragment 等价） |
| spawnEnemyWrecks 死亡一次性 | O(W) ≈ 6-12 次 makeEnemyWreckFragment | 一次性 < 5 ms（瞬时峰值，不持续）|
| getCellStat owner 判断额外开销 | 每次调用增 1 个 && 比较 | 可忽略（< 0.001 ms） |
| assault.tick spawnPending | 仅 spawn 前 2.5s 内每帧扣 dt | < 0.001 ms / 帧 |
| assault.render | 每帧 2 个 ring 渲染 | < 0.1 ms / 帧 |
| **合计新增（hostile-station 存在时）** | < 1.5 ms / 帧（峰值死亡瞬时 < 5 ms） | 远低于 v0.5.0 getCellStat 总开销（5400 次/秒）|

- **关键决定**：不引入 hostile-station cells 命中缓存（每帧重新 dist 计算；cell 数量 ≤ 8，性能足够）；如节点 B 实测发现 > 1.5 ms / 帧，可在 `getHostileStationCellAtWorld` 内加 AABB 粗筛（先用 enemy.bounds 排除显然不命中的 projectile）。
- **wreck spawn 瞬时峰值**：6-12 个 fragment 一次性创建，对当前 fragment 系统（v0.3.0 测试过 8 个 wreck 同时存在）已经验证可承受；hostile-station 12 个略超 cap，但只是瞬时（下一关 nextLevel 清空）。

    - 击毁单个武器 cell 后该武器停火（cell 从 enemy.cells.delete 后不再 update）。
    - shield cell 拦截玩家弹道。
  - 通过后才能启动 T4。
- **节点 C（T4 + T5 + T6 完成后）**：
  - PM 全量回归：
    - 完整 assault 任务流程：level 3-5 创建 objective → assault 出现 → spawn 延迟 2.5s → 玩家决战 → 击毁 → 奖励发放（metal × 1.5、+9 research）→ HUD 显示"已摧毁"。
    - level 3-5 跑 30 seed 验证 assault 出现概率（25%/35%/30% ± 10%）。
    - 单星系冷却：手动连续 spawn 测试，第二次不重复 spawn。
    - 跨关 nextLevel：hostileStationSpawnedThisGalaxy 重置。
    - 跨 run resetRun：同款重置。
    - v0.1.x / v0.2.0 / v0.2.1 / v0.3.0 / v0.4.0 / v0.5.0 全部回归点（PM 看板 §QA 回归清单）。
    - 现有 7 种任务（mine / explore / battle / survive / guardian / salvage / escort）行为不变。
    - 完整 build 路线：跑 level 0-6 完整 run，level 4 遇 assault 决战，wreck 拾荒后强化 build，level 5 再战，level 6 guardian。
    - HUD 警报 + 击毁庆祝 + 危险圈渲染。
    - 专项审核（`expert-agent-planner-gpt55`，与执行不同基底）介入。

## 11. 待制作人 / PM 决策的开放项

1. **hostile-station 是否给击毁后 metal 直接奖励**：
   - 选项 A：reward = 0，所有奖励通过 wreck 拾荒 + assault 任务 reward（metal × 1.5 + 9 research）。
   - 选项 B：reward = 30-50 metal（与 guardian 80 接近）+ wreck + 任务奖励，让玩家"双重收益"。
   - **本 ADR 推荐：选项 A**（避免 metal 通胀；让玩家感受"拾荒 = 主要奖励"，与 v0.3.0 拾荒系统强协同）；如 PM 节点 C 体验测试发现"击毁后即时反馈不足"，再由 v0.6.1 引入 reward 30 metal。
2. **assault 任务失败条件**：
   - 选项 A：不设失败条件（与 battle / survive 一致；玩家可一直在该 galaxy 战斗到击毁为止）。
   - 选项 B：设超时（如 SALVAGE_TIMEOUT_SEC 480s），超时后 assault 失败 + 不阻止跃迁。
   - **本 ADR 推荐：选项 A**（assault 不设失败：BOSS 决战是 mandatory 进度，玩家应该被鼓励完成；如玩家长时间无法击毁，本身就是 build 不足的信号）；如 PM 节点 B 发现"玩家被卡住无法跃迁"问题，可在 v0.6.1 引入超时但不阻止跃迁。
3. **minion 是否在 hostile-station 死亡后清场**：
   - 选项 A：不清场（minion 继续追击玩家直到被击毁）。
   - 选项 B：清场（hostile-station 死亡后所有 hostileStationMinion 在 1-2s 内自爆 / 撤退）。
   - **本 ADR 推荐：选项 A**（更自然；minion 是 pirate kind，玩家可正常清理获得 metal 奖励）；如 PM 节点 B 发现"清完 BOSS 还要追 minion 累赘"，再考虑选项 B。
4. **enemy-wreck cell 是否带 cell.owner 字段**：
   - 选项 A：enemy-wreck cell 由 `createWreckCell` 创建，**不**带 owner 字段（默认玩家）；桥接后立即享受 techLevel / hullMod。
   - 选项 B：enemy-wreck cell 创建时设 `owner = "enemy"`；桥接后玩家手动"接管"（清除 owner 字段）；增加交互复杂度。
   - **本 ADR 推荐：选项 A**（与"决战奖励 = 强化设施"语义一致；桥接即获得加成是玩家预期）；如未来引入"敌方 cells 有 buff/debuff"机制再考虑选项 B。
5. **HOSTILE_STATION_LAYOUT 是否多套配置**：
   - 选项 A：单一 LAYOUT（5-8 cells），所有 level 共用；按 level 缩放总 HP 即可。
   - 选项 B：多套 LAYOUT（如 small/medium/large），按 level 切换不同 cells 结构。
   - **本 ADR 推荐：选项 A**（v0.6.0 范围控制；多 LAYOUT 引入配置管理 + 多套美术 / 平衡问题）；v0.7+ 可引入选项 B。
6. **hostile-station spawn 距离是否随 level 缩放**：
   - 选项 A：固定 1000-1300（与 guardian 接近）。
   - 选项 B：level 3 距离 1200-1500（玩家更早察觉）、level 5 距离 900-1100（更紧逼）。
   - **本 ADR 推荐：选项 A**（统一节奏，玩家可学习经验）；如 PM 节点 C 体验发现"level 3 BOSS 太突然"，再考虑选项 B。
7. **hostile-station 是否在 endgame（level 6）出现**：
   - 选项 A：不出现（与 guardian 互斥；guardian 是终局 BOSS）。
   - 选项 B：可与 guardian 共存（同 galaxy 同时挑战两个 BOSS）。
   - **本 ADR 推荐：选项 A**（与方向"hostile-station 在 level 3-5"硬约束一致）；node C 验证 level 6 仅 guardian。
8. **dev 调试开关**：
   - 选项 A：不引入（与 v0.5.0 一致）。
   - 选项 B：临时 `window.DEBUG_SPAWN_HOSTILE_STATION = () => spawnHostileStationNearStation(mulberry32(Date.now()))` 调试入口；T6 范围内移除。
   - **本 ADR 推荐：选项 B**（T1 必备，便于节点 A 验证 spawn / 渲染 / 命中；T6 必须移除）。
