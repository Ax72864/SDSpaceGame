## 3. D1 / D2：SAVE_KEY、schemaVersion 与迁移

### 3.1 SAVE_KEY 决策

**保留** `SAVE_KEY = "star-ring-station-meta-v1"`。

理由：

1. 新建 key 会留下旧 key 幽灵数据，需要额外清理逻辑，反而引入新风险。
2. 旧 key 中 `points` 是玩家积累的真实价值，保留 key 意味着读到旧值就是玩家原值，不需要二次复制。
3. PM 看板 QA 要求"localStorage 中无幽灵 key 造成的读写混乱"——保留单 key 是最直接的实现。

key 字符串"-v1"是历史命名残留，**不代表 schemaVersion**。schemaVersion 由 JSON payload 内的 `schemaVersion` 字段决定，与 key 字符串解耦。这条原则写入 ADR 是为了避免未来误以为"新版本要 -v2 key"。

### 3.2 schemaVersion 定义

| 值 | 形态 | 触发条件 |
| --- | --- | --- |
| 缺省 / 0 / 1 | 旧 schema（`points / mining / hull / weapons / unlocks`） | v0.1 - v0.9 留下的存档 |
| 2 | 新 schema（见 3.3） | v0.10.0 起 |

未来扩展只需把当前 `schemaVersion` 加 1 并在 `migrateMetaSave` 中追加 case，不再变更 key。

### 3.3 新 schema 形状

```javascript
// 注：字段语义说明，不在游戏运行时写注释
state.meta = {
  schemaVersion: 2,
  points: 0,                       // 玩家未花费的局外点数（保留旧值）
  talents: {                       // 已购买节点的当前 rank，键为 META_TALENT_TREE.id
    miningYield: 0,
    hullIntegrity: 0,
    weaponCalibration: 0,
    weaponFrame: 0,
    efficientCore: 0
    // 数值方案补充的新节点也以 id 为键
  },
  unlockedProtocols: ["balanced"], // 已解锁的协议 id 列表；balanced 永远在内
  selectedProtocol: "balanced"     // 当前选定，下一局开局时应用
};
```

字段不变量：

- `schemaVersion` 永远是 number，写盘时必填。
- `points` 始终 >= 0；不直接对外暴露写入，统一走 `grantMetaPoints / spendMetaPoints` helper。
- `talents` 是 plain object，键集 = `META_TALENT_TREE` 所有 `id`，rank 范围 `[0, maxRank]`。
- `unlockedProtocols` 是字符串数组，恒包含 `"balanced"`，无重复；每次 talents 变化后由 `recomputeUnlockedProtocols` 同步重算。
- `selectedProtocol` 必须 ∈ `unlockedProtocols`；否则在 `migrate / ensure` 阶段回退到 `"balanced"`。

### 3.4 旧字段映射规则

旧 → 新映射表（id 命名与制作人方向一致）：

| 旧字段 | 新 talent id | 映射规则 | 备注 |
| --- | --- | --- | --- |
| `mining`（number, 旧无上限） | `miningYield` | `rank = clamp(mining, 0, maxRank)` | 超过 maxRank 部分不退点（保护"已获得能力"） |
| `hull`（number, 旧无上限） | `hullIntegrity` | `rank = clamp(hull, 0, maxRank)` | 同上 |
| `weapons`（number, 旧无上限） | `weaponCalibration` | `rank = clamp(weapons, 0, maxRank)` | 同上 |
| `unlocks.weaponFrame: true` | `weaponFrame` | `rank = 1`（maxRank=1） | bool → 数值 |
| `unlocks.efficientCore: true` | `efficientCore` | `rank = 1`（maxRank=1） | bool → 数值 |
| `points` | `points` | 原值保留 | 不动 |

**clamp 行为说明**：

- 数值方案目前建议 maxRank = 3-5。若玩家旧 `mining = 10`，迁移后 rank = maxRank（例如 5）。
- 多出的 5 级**不退点**（不写回 `points`）。理由：(a) 旧字段每级成本恒为 1 点，迁移退点对未参与超买的玩家不公平；(b) 数值方案可吸收多余 rank 为"已经获得的能力上限"，无成本计算争议。
- 若数值方案最终将 maxRank 设为高于现有旧字段的最大可观测值（例如线上玩家观测最高 rank=6），可改为不 clamp 直接保留，由数值方案决定。这条策略在数值方案定稿后回到 ADR 更新。

**双倍生效防护（关键）**：

迁移成功后，从 `state.meta` 中**显式 `delete`** 旧字段 `mining / hull / weapons / unlocks`。即使迁移后 saveMeta 再次写盘，旧字段也不会再写入 localStorage。

迁移期间，如果某个步骤抛错导致部分字段已 delete 但 talents 未写入，`migrateMetaSave` 在 try 内构造完整新对象再原子赋值给 `state.meta`，避免半成品状态。

### 3.5 迁移流程伪代码

```javascript
function loadMeta() {
  let raw = "";
  try { raw = localStorage.getItem(SAVE_KEY) || ""; } catch { raw = ""; }
  const parsed = safeParseMeta(raw);
  return ensureMetaState(migrateMetaSave(parsed));
}

function safeParseMeta(raw) {
  if (!raw) return {};
  try { return JSON.parse(raw); }
  catch (e) {
    try { sessionStorage.setItem("star-ring-station-meta-corrupt-raw", raw); } catch {}
    console.warn("[meta] save corrupt, fallback to empty", e);
    return {};
  }
}

function migrateMetaSave(parsed) {
  const v = parsed && typeof parsed === "object" ? (parsed.schemaVersion | 0) : 0;
  if (v >= 2) return parsed;
  return migrateV0ToV2(parsed);
}

function migrateV0ToV2(old) {
  const talents = makeDefaultTalents();
  if (Number.isFinite(old.mining))  talents.miningYield        = clampRank("miningYield",        old.mining);
  if (Number.isFinite(old.hull))    talents.hullIntegrity      = clampRank("hullIntegrity",      old.hull);
  if (Number.isFinite(old.weapons)) talents.weaponCalibration  = clampRank("weaponCalibration",  old.weapons);
  if (old.unlocks?.weaponFrame)     talents.weaponFrame        = 1;
  if (old.unlocks?.efficientCore)   talents.efficientCore      = 1;
  return {
    schemaVersion: 2,
    points: Math.max(0, Number.isFinite(old.points) ? old.points : 0),
    talents,
    unlockedProtocols: ["balanced"],
    selectedProtocol: "balanced"
  };
}

function ensureMetaState(meta) {
  meta.schemaVersion = 2;
  meta.points = Math.max(0, Number.isFinite(meta.points) ? meta.points : 0);
  meta.talents = ensureTalentsDefaults(meta.talents);
  meta.unlockedProtocols = ensureUnlockedProtocols(meta.unlockedProtocols);
  meta.selectedProtocol = meta.unlockedProtocols.includes(meta.selectedProtocol)
    ? meta.selectedProtocol
    : "balanced";
  recomputeUnlockedProtocols(meta);
  return meta;
}
```

`ensureTalentsDefaults`：以 `META_TALENT_TREE` 当前所有 id 为模板补齐，未知键删除（防节点删除后残留）。

`recomputeUnlockedProtocols`：扫描 `META_TALENT_TREE` 中带 `effects: [{ key: "unlockProtocol", protocol: "...", atRank: n }]` 的节点，按当前 ranks 计算应解锁集合并合并 `["balanced", ...]`，写入 `meta.unlockedProtocols`。

`ensureMetaState` 在三个时机调用：

1. `loadMeta` 末尾。
2. `purchaseMetaTalent` 写盘前（保险）。
3. `selectStartProtocol` 写盘前。

## 4. D3：META_TALENT_TREE 与 START_PROTOCOLS 数据结构

### 4.1 META_TALENT_TREE 节点形态

```javascript
const META_TALENT_TREE = [
  {
    id: "miningYield",
    name: "采集校准",
    desc: "提升所有采集设施的产出。",
    category: "resource",        // "resource" | "defense" | "weapon" | "exploration"
    tier: 1,                     // 1-4
    cost: [1, 1, 1],             // 每级成本数组，长度 = maxRank
    maxRank: 3,                  // 数值方案定
    prereq: [],                  // 数组：{ nodeId, rank } 全部满足
    effects: [
      { key: "miningYield", op: "addPercent", perRank: 0.08 }
    ],
    uiHint: { color: "resource", icon: "ore", recommendTag: "" }
  }
  // ... 其余节点由数值方案补齐
];
```

字段不变量：

| 字段 | 类型 | 约束 |
| --- | --- | --- |
| `id` | string | 全局唯一；**发布后稳定不可改**（直接进存档） |
| `name` | string | 中文短名，UI 显示 |
| `desc` | string | 节点说明 |
| `category` | enum | 四选一，决定 UI 分组与颜色 |
| `tier` | int | 1-4，控制 UI 排序与前置节奏 |
| `cost` | int[] | 长度 = maxRank；`cost[rank-1]` 为升到该 rank 的成本 |
| `maxRank` | int | >= 1 |
| `prereq` | `{nodeId, rank}[]` | 全部满足才能购买第一级 |
| `effects` | EffectDecl[] | 见 4.3 |
| `uiHint` | object | UI 提示，结构非强约束，由 UI 方案补充 |

**注册表查找**：定义后立即构建 `META_TALENT_INDEX = new Map(META_TALENT_TREE.map(n => [n.id, n]))`，供 helper O(1) 查找。

**节点删改禁忌**：

- v0.10.0 发布后，`id` 永远不删；如某节点废弃，将其 `maxRank` 设 0 并在 UI 隐藏，但保留 id 让旧存档迁移不报错。
- 新增节点直接 append，`ensureTalentsDefaults` 会补 rank=0。

### 4.2 START_PROTOCOLS 数据结构

```javascript

    unlock: null,                          // null = 默认解锁

  },
  {
    id: "miningStart",

    desc: "初始金属与采集设施折扣。",
    unlock: { nodeId: "miningYield", rank: 2 },
    effects: [
      { key: "startingMetal", op: "add", value: 30 },
      { key: "miningFacilityDiscount", op: "mul", value: 0.85 }
    ]
  },
  {
    id: "defenseStart",
    name: "防御起步",
    desc: "初始核心与框架耐久增益。",
    unlock: { nodeId: "hullIntegrity", rank: 2 },
    effects: [
      { key: "startingHullBonus", op: "addPercent", value: 0.10 }
    ]
  },
  {
    id: "weaponStart",
    name: "武器起步",
    desc: "初始武器框架折扣。",
    unlock: { nodeId: "weaponCalibration", rank: 2 },

```

字段不变量：

| 字段 | 类型 | 约束 |
| --- | --- | --- |
| `id` | string | 全局唯一；进存档 |
| `name / desc` | string | UI 显示 |
| `unlock` | `{nodeId, rank}` 或 null | null 表示默认解锁 |
| `effects` | EffectDecl[] | 与 talent effects 同语义（4.3） |

`balanced` 永远存在且 `unlock = null`；helper 内对该 id 做硬保底。

### 4.3 EffectDecl 与 effect 聚合

声明格式（D5）：

```javascript
{ key: "miningYield", op: "addPercent", perRank: 0.08 }
{ key: "startingMetal", op: "add", value: 30 }
{ key: "weaponFacilityDiscount", op: "mul", value: 0.85 }
{ key: "unlockProtocol", protocol: "miningStart", atRank: 1 }   // 特殊：派生 unlockedProtocols
```

`op` 取值与聚合规则（在 `getMetaEffect(effectKey, ctx)` 内统一处理）：

| op | 含义 | 聚合方式 | 默认初值 |
| --- | --- | --- | --- |
| `addPercent` | 加法百分比 | `total += rank * perRank`（talent）或 `total += value`（protocol） | 0 → 返回 `1 + total` |
| `add` | 绝对加法 | `total += rank * perRank` 或 `total += value` | 0 → 返回 `total` |
| `mul` | 乘法倍率（折扣） | `total *= value`（按 effect 出现次数累乘） | 1 → 返回 `total` |
| `flag` | 是否解锁 | 任意来源出现即返回 true | false |
| `unlockProtocol` | 派生解锁 | 不进 `getMetaEffect`，由 `recomputeUnlockedProtocols` 单独消费 | — |

调用方式：

```javascript
// 局内系统问效果，不需要关心来源是 talent 还是 protocol
const miningMul = getMetaEffect("miningYield");                 // 返回 1.24（例）
const startMetal = getMetaEffect("startingMetal");              // 返回 30 或 0
const weaponDiscount = getMetaEffect("weaponFacilityDiscount"); // 返回 0.85 或 1
const hasWeaponFrame = hasMetaTalent("weaponFrame");            // bool
```

`getMetaEffect` 聚合时同时扫描：

1. `META_TALENT_TREE` 中所有 rank > 0 的节点的 effects（按 rank 倍数）。
2. `START_PROTOCOLS` 中 `selectedProtocol` 的 effects（按 value 直读）。

数值方案补充节点时，只需在节点 effects 数组里追加 decl，无需改 helper 实现。如果数值方案要新 op，必须在 ADR 增补一条记录并扩展 helper。

## 5. D4：集中 helper 接口

所有 helper 集中在 `Game/game.js` 内一段连续区块（约在 `loadMeta` 后），不分文件。命名以 `Meta` 前缀，避免和 v0.5 研发系统的 `Research`、v0.9 `Galaxy` 混淆。

| Helper | 签名 | 职责 | 调用方 |
| --- | --- | --- | --- |
| `getMetaTalentRank(id)` | `(string) => int` | 读 rank；未知 id 返回 0 | UI、效果、测试 |
| `hasMetaTalent(id)` | `(string) => bool` | rank >= 1 | 局内 unlock 判断 |
| `getMetaEffect(effectKey, ctx?)` | `(string, object?) => number\|bool` | 聚合 talent + protocol 的某 effectKey | 采集、耐久、武器、科研、初始资源 |
| `getMetaPurchaseState(id)` | `(string) => { canBuy, reason, nextCost, currentRank, maxRank }` | UI 渲染节点状态 | UI |
| `purchaseMetaTalent(id)` | `(string) => { ok, reason?, rankAfter? }` | 扣点 + rank+1 + saveMeta + recompute | UI 购买按钮 |
| `getSelectedStartProtocol()` | `() => string` | 当前选定协议 id | 开局应用 |
| `selectStartProtocol(id)` | `(string) => { ok, reason? }` | 校验 unlocked + saveMeta | UI 协议选择 |
| `applyMetaStartProtocol(stateRef)` | `(state) => void` | 在 reset / 新局初始化时把协议效果写进初始 station / resources | `resetRun` / `startNewRun` |
| `migrateMetaSave(raw)` | `(object) => object` | 旧 → 新 schema 迁移；纯函数 | `loadMeta`、测试 |
| `grantMetaPoints(amount, source)` | `(int, string) => void` | 加点 + saveMeta，可选记录来源 | `grantObjectiveReward` / `endRunSettlement` |
| `getMetaSettlementHints()` | `() => { recommended: [], shortfalls: [], advice: [] }` | 给结算 UI 提供推荐节点 / 差多少点 / 下一局建议 | T3 结算 |
| `recomputeUnlockedProtocols(meta?)` | `(meta?) => string[]` | 扫描 talents 派生 unlockedProtocols | 内部 |
| `__metaResetForTest()` | `() => void` | 清空 meta（仅测试） | `__gameTest.meta.resetSave` |

**调用边界硬规则**：

1. 任何文件、任何函数都**不得**再读 `state.meta.mining / .hull / .weapons / .unlocks`。这五个字段在迁移后不存在；如代码不慎读取，访问 undefined → NaN → 易爆。本规则作为 PR 自检和审核 checklist。
2. 任何文件、任何函数**不得**直接写 `state.meta.points`。写入必须经 `grantMetaPoints / purchaseMetaTalent`。
3. UI、采集、武器、耐久、科研、初始资源 → 全部走 `getMetaEffect / hasMetaTalent`，不允许把 `META_TALENT_TREE` 的节点 id 直接散落在局内系统硬编码（节点 id 字符串只能出现在 helper 和 UI 渲染层）。
4. `state.meta` 在序列化前后必须满足 §3.3 不变量；写盘走 `saveMeta()` 不变。

## 6. D6：开局协议应用时机

### 6.1 状态机

```text
┌─────────────────┐     resetRun(seed) /        ┌────────────────────┐
│ MetaPanel 选择   │ ─── startNewRun() ──────► │ applyStartProtocol │
│ selectedProtocol│                              │ (本次开局前一次性) │
└─────────────────┘                              └────────────────────┘
                                                        │
                                                        ▼
                                                ┌────────────────────┐
                                                │ generateGalaxy /    │
                                                │ buildInitialStation │
                                                └────────────────────┘
                                                        │
                                                        ▼
                                                  局内运行（不可改）
```

### 6.2 时机定义

`applyMetaStartProtocol(stateRef)` 的调用点：

1. `__gameTest.resetRun(seed, level, galaxyType)` 入口：现有第 8693 行 `ensureRunRuntimeState()` 之后、`createObjective()` 之前。
2. 玩家点击"开始新一局"：现有 `window.gameActions.startNewRun() → location.reload()` 会触发整个页面重载，`loadMeta` + 首次 `state` 初始化时调用一次 `applyMetaStartProtocol(state)`。
3. 不挂在 `tickFrame` / 任何持续运行循环里。

**禁止时机**：

- 局中任何时刻不允许 `selectStartProtocol` 改变**当前**局状态；只影响下一次 `resetRun / 新局 reload`。
- UI 允许玩家在局中查看 / 切换 `selectedProtocol`，但 toast 明确："已保存为下一局开局协议；当前局不变更"。
- 结算面板 / `endRunSettlement` 中不调用 `applyMetaStartProtocol`，避免双倍应用。

### 6.3 协议影响范围（白名单）

`applyMetaStartProtocol(stateRef)` 仅允许写入以下字段：

| 协议字段 | 写入位置 | 上限 |
| --- | --- | --- |
| `startingMetal / startingOre / startingGas / startingPlasma` | `state.resources.<key> += effect` | 单项不得超过 v0.1 初始值的 50% |
| `startingHullBonus` | 初始 station core/frame hp 乘加因子 | <= 0.15 |
| `miningFacilityDiscount / weaponFacilityDiscount` | 写入 `state.run.startupFacilityDiscounts[facility]`，由 `createBuildUi` 与 `getBuildCost` 读取 | <= 0.85（即最多 15% 折扣） |
| `earlyHint` | 写入 `state.run.startupHints[]`，由 HUD 在前 90 秒按顺序播放 | 字符串数组，无数值影响 |

**协议不得修改**：

- `state.run.galaxyMap`（v0.9 边界）。
- `state.run.currentGalaxyType`（v0.9 边界）。
- `state.run.encounters` / `state.run.encounterCooldownThisGalaxy`（v0.8 边界）。
- `state.run.objective`（v0.4 边界）。
- `state.run.completedObjectives / totalObjectiveRewardBase / settlementBonusGranted`（结算边界）。
- `state.resources.research`（v0.5 局内研发独立性）。
- 任何 ENEMY / SPAWN / ASSAULT / GUARDIAN 常量。

上限超出时 `applyMetaStartProtocol` 直接 clamp 并在 console.warn 提示数值方案校准。clamp 行为优先保护体验稳定。

## 7. D7 / D8：UI 状态机与重置策略

### 7.1 UI 状态机（购买）

```text
       点击节点
           │
           ▼
   getMetaPurchaseState(id)
           │
   ┌───────┼────────────┐
   ▼       ▼            ▼
```

**关键约束**：

- 双击 / 高频点击：`purchaseMetaTalent` 内部不做时间窗节流，但每次点击都重新读 `getMetaPurchaseState`；状态一旦变为 `not_enough_points` 立即拒绝。不会出现重复扣点。
- saveMeta 失败（浏览器阻止）→ 现有 `showToast("浏览器阻止了存档写入...")` 已存在；T2 在此基础上额外回滚 ranks 与 points（撤销内存写入），避免存档与内存不一致。

### 7.3 重置策略（D8）

v0.10.0 **不实现重置**。

- UI 在 Meta 面板底部明显位置显示提示文本：`当前版本暂不支持局外天赋重置；后续平衡期可能提供一次性重置入口。`（具体文案由 UI 方案）
- helper 不暴露 `resetMetaSave` 给 `window.gameActions`；仅保留 `__gameTest.meta.resetSave` 供 PM 测试。
- 若数值方案在 #3 中明确"必须配套重置"，回到 ADR 增补 D8a 决策（含确认弹窗、测试钩子、点数处理策略）。

理由：

1. 减少经济争议（点错节点的玩家更愿意接受"等下个版本"，比"误点重置"风险低）。
2. 节省 #6 T2 UI 复杂度、#9 PM 回归成本（节省一整套重置回归矩阵）。
3. helper 接口保留扩展点，下个版本接入成本 < 1 天。

## 8. D11：迁移失败保护与回滚

### 8.1 保护点

`safeParseMeta` 在 JSON.parse 抛错时：

1. 把原始字符串写到 `sessionStorage["star-ring-station-meta-corrupt-raw"]`（仅本会话，可被 PM 取证）。
2. console.warn 打印 raw 截断信息（前 200 字符）便于复现。
3. 返回 `{}`，触发 `migrateV0ToV2({})`，玩家从默认 schema 起步（points = 0）。

**这是降级路径，不是常态路径**：JSON 解析失败的概率极低，且无法从错误 raw 中恢复点数；ADR 选择保护"游戏可启动"而不是保护"点数不丢"。同时把 raw 保留到 sessionStorage，理论上 PM 拿到本会话可手工拷出 raw 提交 issue，制作人决策是否赠点。

### 8.2 迁移异常

`migrateMetaSave` 内部如果某字段值非预期（例如 `points: "abc"`）：

- Number.isFinite 检查 → 不是 finite 直接当 0 处理，不抛错。
- `unlocks.weaponFrame` 非 bool/truthy → 当 false 处理。
- 整个迁移过程**不允许抛错**；try/catch 包一层 outer 兜底，异常时返回 `migrateV0ToV2({})` 等价默认值。

### 8.3 回滚机制（版本级）

v0.10.0 发布前如果出现严重问题（点数批量丢失等）：

- **代码回滚**：git revert v0.10.0 相关 commits，回到 v0.9 HEAD。
- **存档兼容性**：v0.10.0 写入的新 schema `{ schemaVersion: 2, talents, ... }` 在 v0.9 老代码下 `loadMeta` 走 `{...base, ...parsed}`，旧代码只读 `points`，新字段被忽略但保留。**老代码不会因为新 schema 字段而报错**——这是保留 SAVE_KEY 的额外收益。
- **点数继承**：v0.10.0 写盘的 `points` 字段格式与 v0.9 一致，回滚后玩家点数无损。

如果回滚发生在玩家已经购买大量 v0.10.0 新节点之后，回滚至 v0.9 时这些 rank 会被 v0.9 代码忽略（仍在 talents 字段下，但 v0.9 不读），玩家感受是"v0.10 的成长进度看不见了，但点数还在"。这是可接受的回滚体验。

## 9. 模块边界（不污染 v0.9 / v0.5）

### 9.1 v0.9 galaxyMap 边界

- `state.run.galaxyMap / currentGalaxyType / galaxyChoicesShown / pendingChoices` → Meta 系统**只读**，不写。

### 9.2 v0.5 research 边界

- 局内 `state.resources.research` → Meta 系统**只读**（计算结算时可读取本局获得量），不写。

- UI 文案规则（写入 UI 方案验收）：
  - v0.5 局内研发面板 → "研发"、"科研点"
  - v0.10 局外天赋 → "局外天赋"、"局外点数"
  - 严禁文案出现"局外研发"或"局内天赋树"等混淆词。

### 9.3 v0.1-v0.8 主路径边界

- v0.6 hostile station / v0.7 锁敌齐射 / v0.8 encounters 的状态机不被 Meta 触碰。
- `applyMetaStartProtocol` 只写 §6.3 白名单字段；任何企图写入 ENEMY_TYPES / SPAWN_TABLES / ENCOUNTER_LEVEL_WEIGHTS 等常量的实现必须被审核驳回。

## 14. 风险与未决项

| 风险 | 影响 | 缓解 | 触发回到 ADR 的条件 |
| --- | --- | --- | --- |
| 数值方案 maxRank 高于旧字段最大值，clamp 策略需调整 | 旧玩家迁移后 rank 不饱和 | 数值方案定稿后回到 §3.4 决策"是否 clamp" | #3 数值方案产出后 |
| 数值方案要求探索型节点写星系候选权重 | 突破 v0.9 边界 | 必须先增补 D12 决策 | 数值方案中出现 galaxy effectKey |
| UI 方案要求协议在结算面板"立即应用本局" | 突破协议时机白名单 | 必须先增补 D6a 决策 | UI 方案出现"协议立即生效"字样 |
| `getMetaEffect` 性能：每帧多次调用聚合 | 单局 60fps × 多个调用点 | 数据结构小（节点数 < 20），聚合是 O(节点数)；如成瓶颈引入 per-effectKey memo（在 talents 变化时失效） | profile 显示该函数 > 0.5ms/帧 |
| `applyMetaStartProtocol` 累积写入 / 重入 | 协议被多次应用导致初始资源叠加 | 在 `state.run` 加 `metaProtocolAppliedThisRun: bool`，第二次调用直接 return | 出现重入路径（例如 endgame 重生流程） |
| 旧字段映射后玩家"已购数量"减少（clamp 后看起来变少） | 玩家可能感受成长进度被砍 | UI 方案首屏明确告知"v0.10.0 整理了旧成长进度" | UI 方案需采纳 |

未决项需要在数值方案 / UI 方案产出后回到 ADR 更新：

1. 节点 maxRank 与 cost 数组（数值方案）。
2. 各 effectKey 的具体 perRank / value（数值方案）。
3. 探索型协议是否进首版（制作人决定 + 数值方案）。
4. UI 方案对"差多少点" / "下一局建议"的文案模板（决定 `getMetaSettlementHints` 输出 shape）。
