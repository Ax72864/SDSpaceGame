## 问题摘要

v0.20.0 已建立采矿覆盖诊断能力，但仍未取得 `harvestingCount > 0`、目标资源 `body.amount` 下降或可归因收益。`suggestedExpansionPath` 在 helper 中可入圈，但真实铺设受 `frame` 金属成本与起始资源约束阻断；原 `mining=-1,1` 路径输入稳定性差，HUD 覆盖与相机 / 工站移动造成证据漂移。

本文档目的：在不破坏当前规则与存档稳定的前提下，给出至少 4 个候选方案，覆盖“仅引导 / 调布局 / 调范围 / 调成本 / 调移动”五个产品方向，并推荐一个适合 1.0 正式交付候选的优先方案。

## 方案 A：仅强化引导与诊断反馈（零规则改动）

### 玩家收益
- 玩家在首屏即看到"先扩建框架到彩色外环再造采矿站"或"先驾驶工站靠近资源再建采矿站"的明确路径建议，不再只看到"还差 X"。
- 失败采矿尝试时，HUD 给出 `nearestResource` 距离、`gap`、`suggestedExpansionPath` 的可读化版本，降低首局学习成本。
- 减少误把"采矿站建好就会出资源"误判为 BUG 的概率。

### 规则 / 数值影响
- 不改 `MINING_RANGE_OFFSET`、不改 `mining/frame/power/thruster.cost`、不改资源生成、不改存档 / Meta。
- 仅在现有 `resourceGuide` 与 HUD 提示层做增强：把 helper 已经产出的 `bestCandidate`、`suggestedExpansionPath`、`failure` 转译为玩家文案，并在采矿站红色范围环外加"扩建提示"灰色虚线圈。
- 数值层面只新增"提示触发阈值"暂定值：当 `gap > 40` 显示"过远，建议靠近"；当 `gap <= 40 && frameMetalNeeded > currentMetal` 显示"需要 N 金属再扩建"。

### 风险
- 不解决"金属不够支付扩建"的根本问题。若玩家完全没采矿收益，可能依旧卡死；但 v0.20.0 已有 storage / runtime 稳定证据，至少不会卡崩。
- 文案过多会污染 HUD 可读性；需要 UI 美术评审显示位置与字号。
- 仍依赖玩家主动响应提示；若玩家忽视提示，路径仍可能不闭环。

### 验收证据
- 首局 P0 真实鼠标路径：玩家在 90 秒内能根据提示成功建出一座入圈采矿站，并取得 `harvestingCount > 0`。
- 小屏触控完成同样路径。
- 浏览器 JSON 包含 `resourceGuide.action`、`suggestedExpansionPath` 提示触发次数、玩家选择路径。
- 复测覆盖 `mining=-1,1` 原路径，确认提示出现且不阻塞。

### 回滚方式
- 仅涉及提示文案与 HUD 渲染层，回滚为恢复 `resourceGuide` v0.20.0 版本即可，不会牵动数据。

### 是否高风险
- 否。属于引导 / UI 增强类改动，按常规任务流即可。

## 方案 B：调整首局资源布局 / 起始 seed，保证近距资源可达

### 玩家收益
- 玩家空档进入首局，核心附近 1-2 圈范围内必有至少一颗矿物天体在采矿范围内（即 `dist(core, body) <= body.r + MINING_RANGE_OFFSET + frameStep * 1`）。
- 玩家只需扩 1-2 节框架即可让首个采矿站入圈，金属预算 130 足够支付。
- 让 1.0 首局的"建造 → 采矿收益 → 再建造"循环在真实输入下稳定建立。

### 规则 / 数值影响
- 改资源生成布局：保留现有 `world.bodies` 类型与数量，但约束首局 seed：
  - 至少一颗 `ore` 或 `metal` 矿物天体满足 `dist(core, body) - body.r <= MINING_RANGE_OFFSET + 16`（即一格 frame 即可入圈，frame 步长按现工程值，暂定 16-24 像素）。
  - 该天体 `amount >= 80`，保证不会刚开局就被采空（暂定值，需实测调优）。
- 保留现有星系类型权重（`rich/nebula/storm/balance/endgame`），仅在首局 seed 前置一道"近距可达资源"硬约束；进入后续阶段恢复原随机布局。
- 起始资源不动；frame 成本不动；`MINING_RANGE_OFFSET` 不动。
- 配置字段建议：
  - `worldgen.firstSession.guaranteedNearMiner = true`（暂定）。
  - `worldgen.firstSession.nearMinerExtraRange = 16`（暂定，单位与 frame 步长一致，需实测）。
  - `worldgen.firstSession.nearMinerMinAmount = 80`（暂定）。

### 风险
- 触及资源布局，属于高风险项。需要专项审核：是否破坏后续阶段难度曲线、是否让"富矿 / 风暴"星系特色失效、空档 vs 旧档生成是否一致。
- 若 seed 约束过强，首局可能丧失探索感；需控制只在第一个星系阶段生效。
- 该约束与 `miningStart` 天赋叠加可能让首局过于轻松，需要在专项审核中确认无叠加爆炸。
- 若玩家刷新或更换 seed，资源位置变化要保持承诺；需说明 seed 持久化策略。

### 验收证据
- 真实人工鼠标 + 小屏触控：连续 5 次新档，全部能在不消耗 `miningStart` 天赋的前提下，于 90 秒内取得 `harvestingCount > 0`。
- 浏览器 JSON 记录：首局核心到最近矿体距离、`MINING_RANGE_OFFSET`、玩家成功建造的 frame 数量、采矿站位置、首次 `harvestingCount > 0` 时间。
- 旧档 / 空档兼容：旧档不触发新 seed 约束，新档触发。
- 后续阶段星系不强制此约束，保留探索差异。

### 回滚方式
- 把 `worldgen.firstSession.guaranteedNearMiner` 设为 `false`，回到原随机布局；不污染存档。

### 是否高风险
- 是。资源布局属项目高风险项，必须走专项流程。

## 方案 C：调整 `mining range` 或 `resource ring` 半径

### 玩家收益
- 通过把 `MINING_RANGE_OFFSET` 从 `130` 提到（暂定）`150-170`，或把首局矿物天体半径 `body.r` 整体放大 10-15%，让首局采矿站不依赖远距扩建即可入圈。
- 玩家学习成本最低：建好采矿站就有收益，且 v0.20.0 已建立的诊断 / 提示文案继续兼容。

### 规则 / 数值影响
- 只动 1-2 个常量：
  - 暂定 `MINING_RANGE_OFFSET = 150`（+15%）或 `160`（+23%），需实测确定最低有效值。
  - 或保留 `MINING_RANGE_OFFSET = 130`，将首局矿物天体的 `r` 乘 1.12-1.20（暂定）。
- 不改成本、不改起始资源、不改布局、不改 schema。
- 后续阶段是否同步放大需要专项评审：默认建议保持后续阶段原值，仅放大首局；但这会让数值变成"分段"，可能引入与天赋、研究、采集校准的耦合 bug。

### 风险
- 触及采矿范围 / 公式，属于项目高风险项。
- 范围放大会同时让"采空速度变快"，影响首局资源耐久；需配套调整 `body.amount` 或采矿速率，否则前期玩家会很快采空近距资源，产生新的卡点。
- 若整体放大 range，可能破坏后续战斗 / 防御 / 工站避让距离设计；需要主程评估视觉环渲染与碰撞判定是否需要同步调整。
- 与 v0.20.0 helper 输出耦合：`suggestedExpansionPath`、`gap` 等字段会自动适配新值，但旧证据 JSON 与新值不可直接对比。

### 验收证据
- 连续 5 次新档真实输入：90 秒内取得 `harvestingCount > 0`，且首局首颗矿体 `amount` 不在 3 分钟内归零。
- 后续阶段（第二、三星系阶段）的采矿站行为未变（若选择"仅首局放大"方案）。
- 自动检查：`MINING_RANGE_OFFSET` 引用点全部用同一常量；视觉范围环、碰撞、覆盖判定都基于该常量，回归测试一次性通过。

### 回滚方式
- 把常量恢复为 `130`，立即回到 v0.20 行为；不污染存档。

### 是否高风险
- 是。采矿范围 / 公式是直接命中的高风险项。

## 方案 D：调整早期 `metal` / `frame` 成本或起始储备

### 玩家收益
- 起始金属从 `130` 提到（暂定）`160-200`，或把 `frame.cost.metal` 从 `8` 降到 `6`（暂定），玩家在首局不靠采矿收益也能扩建 4-6 节 frame，足以把采矿站推进入圈。
- 让 v0.20.0 helper 的 `suggestedExpansionPath` 在真实成本下可支付，从"理论可入圈"变成"实际可入圈"。

### 规则 / 数值影响
- 三种子选项，需要专项评审选其一：
  - D1：仅提升起始 `metal` 储备到 `170-200`（暂定 180），不动单位成本。最小数值改动，最稳。
  - D2：`frame.cost.metal: 8 → 6`，单位扩建更便宜；影响整局节奏，需要全程平衡评估。
  - D3：在首局阶段对 `frame` 成本叠加一个 0.75 的折扣系数，仅首星系阶段生效；其余阶段恢复原成本。
- 不改 `MINING_RANGE_OFFSET`、不改资源布局、不改 `mining.cost`、不改 schema。
- 建议优先 D1：调整面最小，回滚最容易，正式交付稳定性最高。

### 风险
- 触及数值与建造合法性，属于项目高风险项。
- D2 / D3 改全程成本曲线，会影响 `power`、`thruster`、`mining` 等所有 `metal` 消耗设施的相对价值；若不同步评审整套经济，可能导致中后期通胀。
- D1 看似最稳，但若与方案 B / C 叠加会过度补偿；不能与其它高风险方案同时实施。
- 与 `miningStart` 天赋叠加会让首局过于轻松，需在专项审核中确认上限。

### 验收证据
- 连续 5 次新档真实输入：玩家在不依赖 `miningStart` 的前提下能于 120 秒内取得 `harvestingCount > 0`，且 frame 扩建过程中不出现"金属为 0 卡住"的失败路径。
- 浏览器 JSON 记录：每次建造的 metal 余额、frame 总数、首次 `harvestingCount > 0` 时间。
- 旧档读取后是否补差额：建议旧档保持原值，新档使用新起始资源；专项审核明确兼容策略。

### 回滚方式
- D1：把 `1076` 行的 `metal: 180` 恢复为 `130`，一行回滚。
- D2 / D3：恢复 `frame.cost.metal: 8`，并移除阶段折扣字段。

### 是否高风险
- 是。数值改动 + 建造合法性，属项目高风险项。

## 方案 E（可选）：突出移动 / 工站推进到资源外环

### 玩家收益
- 不改资源、范围、成本；改为强化"驾驶工站靠近资源后重建采矿站"路径：
  - HUD 在采矿失败时明确提示"推进器尚未建造，建议先建推进器并移动工站"或"工站可移动至 P 点附近重建采矿站"。
  - 在 helper 已有 `suggestedExpansionPath` 之外，新增 `suggestedMovePath`：以工站重心移动到候选采矿点附近的目标方向 / 距离。
- 让玩家在原规则下感受到"移动是另一条有效采矿路径"，而不是被迫扩建框架。

### 规则 / 数值影响
- 不动 `MINING_RANGE_OFFSET`、`mining.cost`、`frame.cost`、起始资源、布局、schema。
- 涉及 helper 增量：新增 `suggestedMovePath` 字段，描述工站需要移动的方向与距离，但不写入 state。
- 涉及 UI：在采矿失败提示中加入"建造推进器 → 移动工站 → 重建采矿站"步骤。
- 涉及数值：可选轻调整 `thruster.cost.metal: 26 → 22`（暂定，需平衡评估），或保留原值仅做引导。

### 风险
- 推进器路径目前真实输入稳定性未知；v0.20.0 证据中相机 / 工站移动会让证据漂移。在没有打通真实输入稳定的前提下，把玩家引导到移动路径，可能制造新的卡点。
- 工站移动后采矿站重建涉及拆除 / 重新放置，多步操作小屏触控失败率高；需 UI 美术与体验设计配套优化。
- 如果调整 `thruster.cost`，也属于高风险项；建议仅做引导不动成本。

### 验收证据
- 真实人工鼠标完成"建推进器 → 移动 → 拆采矿站 → 重建采矿站 → 取得 `harvestingCount > 0`"完整链路。
- 小屏触控同样路径可完成。
- 浏览器 JSON 记录：移动路径、工站位姿变化、重建采矿站位置、首次 `harvestingCount > 0` 时间。
- 输入失败案例（HUD 误触、相机滑动）有复现步骤与缺陷单。

### 回滚方式
- 仅引导版本：移除 `suggestedMovePath` 与对应 HUD 提示。
- 若动 `thruster.cost`：恢复 `metal: 26` 一行回滚。

### 是否高风险
- 是。移动能力 / 建造合法性（拆除重建）属高风险项；不动成本时风险等级中。

## 推荐 1.0 优先方案

### 推荐：方案 A 为主，方案 B 为备选

**主推：方案 A（仅强化引导与诊断反馈，零规则改动）**

理由：
- **正式交付稳定性最高**：不触采矿范围 / 公式、资源布局、移动能力、建造合法性、数值、存档 / Meta；不需要走完整项目高风险流程的"实施 + 专项审核 + 回归"长链路，能在 1.0 release candidate 时间窗内最快落地。
- **最小改动**：只动 `resourceGuide` 文案、HUD 提示渲染、helper 输出转译；与 v0.20.0 helper 输出无冲突，已有 `bestCandidate / suggestedExpansionPath / failure` 字段直接复用。
- **回滚成本最低**：单文件文案与渲染层改动，故障时秒级回滚；不污染存档。
- **不破坏 v0.19/v0.20 既得成果**：维修链不回潮、诊断能力不丢、存档结构不变。
- **可加性强**：若 A 落地后真实输入证据仍未取得 `harvestingCount > 0`，可叠加方案 D 的子选项 D1（仅提升起始 metal）作为补救——D1 是数值类高风险中改动面最小的一个子选项，组合后仍可控。

主推方案 A 时禁止同时实施 B/C/D/E 的规则与数值改动。

**备选：方案 B（调整首局 seed，保证近距可达资源）**

仅当方案 A 在真实输入复测中连续 3 次新档无法在 120 秒内取得 `harvestingCount > 0` 时，进入备选 B：触发首局 `worldgen.firstSession.guaranteedNearMiner` 硬约束。备选 B 是高风险项，必须走完整项目高风险流程：制作人确认方向 → PM 拆分 → 专项实施 → 专项审核 → PM 汇总 → 制作人最终决定。

### 不推荐作为 1.0 主路径

- 方案 C（改 `MINING_RANGE_OFFSET` 或 `body.r`）：单点常量牵动碰撞、渲染、覆盖、采空速度、后续阶段难度，回归面大，正式交付窗口内不建议。
- 方案 D2/D3（改 `frame.cost` 或全程折扣）：影响整局经济曲线，需要完整经济回归。
- 方案 E（移动 / 推进路径）：依赖真实输入稳定性，目前未达成；引导若误导玩家进入更复杂操作，反而恶化首局体验。
