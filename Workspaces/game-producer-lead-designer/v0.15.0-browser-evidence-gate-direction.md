# v0.15.0-browser-evidence-gate 制作人方向

## 结论

v0.15.0 的版本名为 `v0.15.0-browser-evidence-gate`。

核心目标：把 v0.14.0 已建立的 release readiness checks、隐藏人工 checklist 和小屏/HUD 密度打磨真正跑进真实浏览器，形成可复查证据；再根据证据决定继续补证、进入小修 polish，或进入发布准备交接。

制作人判断：本版本不是新增玩法版本，也不是正式发布版本。v0.14.0-release-candidate-playtest 已完成开发方向上的候选验收能力，但 Chrome headless `--dump-dom` 未形成有效 browser 证据，真实浏览器人工 P0 路径、DevTools 运行态 JSON、真实触控、真实帧耗时、三视口截图/遮挡证据仍缺失。因此 v0.15.0 应把工作收束到“证据闸门”，先证明当前纵向切片在真实浏览器中可运行、可观察、可复查，再讨论发布准备。

## 玩家与团队收益

- 团队不再只靠静态检查、Node/VM 或 headless 判断发布风险，而是能用真实浏览器证据验证首局体验。
- 能确认空存档首局、建造、采矿、资源不足、断电、敌人接近、武器/护盾/维修、战斗后复盘、星系跳转和 Meta 回到新局在浏览器中可用。
- 能把小屏、HUD 密度、关键 DOM、运行时错误、帧耗时和测试钩子输出记录为可复查材料，减少“感觉能发布”的误判。
- 若证据不足，团队能明确继续补证；若发现阻塞缺陷，能按原路径修复并复测；若证据通过，才进入发布准备交接。

## 核心目标

v0.15.0 要回答一个问题：v0.14.0 的发布候选验收能力，是否已经在真实浏览器和人工路径中形成足够可信的证据。

必须覆盖以下判断：

- 真实浏览器取证是否可行：在不安装依赖、不改系统设置的前提下，能否通过现有 Edge/Chrome/DevTools/CDP/远程调试或人工 DevTools 取得证据。
- 运行态 JSON 是否可复查：`runReleaseCandidateChecks({ sampleFrames: 30 })` 能否在真实浏览器中产出结构化摘要。
- P0 人工路径是否闭环：玩家从空存档进入首局，能否完成建造、采矿、战斗、复盘、星系跳转和 Meta 回流到新局。
- 三视口是否有证据：`760x640`、`420x740`、`390x844` 或等价浏览器 viewport 下，关键 HUD、面板和操作入口是否可读、可滚动、可点击。
- 缺陷处理是否克制：只修复证据路径中发现的 S0/S1/S2 明确缺陷；手感 polish 由制作人判断是否纳入 v0.15.0 或转后续版本。

## 必做范围建议

### 1. 浏览器运行态取证方案

目标是在现有本机环境内找到可复现的真实浏览器取证方式，不以安装工具链作为前提。

允许尝试：

- 使用现有 Edge 或 Chrome 打开本地游戏。
- 使用浏览器 DevTools 控制台人工调用检查函数。
- 使用现有浏览器远程调试能力或 CDP 方式读取控制台、运行态 JSON 和关键 DOM。
- 若已有项目脚本能启动本地服务或静态页面，可以复用，但不得新增依赖。

必须记录：

- 使用的浏览器、版本或可识别环境。
- 启动命令、页面入口、视口设置方式和 DevTools 操作步骤。
- 成功取得了哪些证据，失败在哪一步。
- 若远程调试、CDP 或 headless 不可用，记录具体失败原因和是否影响验收结论。

验收标准：

- 至少形成一份可复查的浏览器取证步骤。
- 如果无法取得真实浏览器证据，v0.15.0 只能给出“继续补证”结论，不能进入发布准备。
- 不得通过安装 Playwright、Puppeteer、jsdom、浏览器驱动或修改系统设置来绕过限制。

### 2. 真实浏览器运行态 JSON 证据

目标是在真实浏览器中调用 `runReleaseCandidateChecks({ sampleFrames: 30 })`，得到能支撑发布闸门判断的 JSON 摘要。

JSON 证据必须关注：

- `storageSignature`：检查前后存储签名，用于证明取证没有写存档或污染进度。
- `runtimeStateSignature`：当前运行态摘要，用于复查游戏状态是否符合取证起点。
- `frameTiming`：至少 30 帧粗采样结果，记录平均值、最大值或长帧等关键信息。
- `runtimeErrors`：运行时错误、未捕获异常、关键 console error/warning 摘要。
- `keyDom`：HUD、建造面板、战斗提示、复盘、星系图、Meta 入口等关键 DOM 存在性和可见性。
- `hookStatus`：发布候选检查钩子、人工 checklist 钩子和只读状态钩子的可用性。

验收标准：

- 至少获得一次真实浏览器运行态 JSON，并归档摘要。
- JSON 能说明采样环境、视口、调用方式、通过项、失败项和未覆盖项。
- 如果字段缺失或函数不可调用，必须记录为证据路径缺陷；是否修复取决于其是否阻塞本版本闸门。

### 3. P0 人工路径最小体验

目标是让 PM、制作人或执行成员按同一条最小路径在真实浏览器中体验，并记录人工结论。该路径可以人工记录，不要求自动化。

必须覆盖的 P0 路径：

- 空存档首局：确认入口、初始 HUD、建造入口、资源和电力信息可读。
- 建造/采矿：完成最小建造和采矿链，看到资源变化。
- 资源不足/断电：触发或观察资源不足、电力不足时，系统给出可理解反馈。
- 敌人接近：敌人出现或靠近时，玩家能看到威胁提示并理解风险。
- 武器/护盾/维修：确认基础攻击、防御和维修状态可读，失败原因可解释。
- 战斗后复盘：一次交火后能看到损伤、问题或下一步设计建议。
- 星系跳转：从局内进入星系图，理解节点选择或跳转入口。
- Meta 回到新局：从局外或 Meta 路径回到新局，关键状态不混乱、不误导。

验收标准：

- 至少形成一次 P0 人工路径记录，包含浏览器环境、视口、起始条件、操作步骤、实际结果和结论。
- 每个路径节点标记为通过、失败、未覆盖、阻塞或非阻塞风险。
- 没有 P0 人工路径记录时，制作人结论只能是“继续补证”。

### 4. 三视口证据

目标是确认 v0.14.0 的小屏/HUD 密度打磨在真实浏览器 viewport 中没有阻塞首局路径。

建议视口：

- `760x640`：小桌面或窄窗口，重点看 HUD 密度、建造面板、复盘和星系图。
- `420x740`：窄屏移动近似视口，重点看遮挡、滚动、按钮可点性和信息优先级。
- `390x844`：更窄移动近似视口，重点看关键入口是否被挤出、面板是否可关闭。

必须记录：

- 视口设置方式，是否为真实浏览器 viewport。
- HUD、建造面板、战斗提示、复盘、星系图、Meta 入口是否可见。
- 是否出现遮挡、不可滚动、按钮不可点击、文本严重溢出或核心信息被挤掉。
- 若无法真实调整视口，明确标记为未覆盖，不能用默认窗口结果替代三视口结论。

验收标准：

- 至少三种目标视口或等价 viewport 的证据记录进入 PM 工作区。
- 若某视口存在阻塞首局路径的问题，修复后必须按同一视口复测。
- 真实触控若未覆盖，应明确作为残留风险，不得宣称移动端已完整通过。

### 5. PM 工作区证据包

目标是把 v0.15.0 的证据集中归档到 PM 工作区，供审核、复盘和发布准备判断使用。本版本不把证据包放入 `Releases/`。

证据包建议包含：

- 浏览器运行态 JSON 摘要。
- `runReleaseCandidateChecks({ sampleFrames: 30 })` 调用步骤和结果。
- P0 人工路径记录。
- 三视口记录，包含遮挡、滚动、可点性和未覆盖项。
- 运行环境、命令、浏览器入口、失败原因。
- 阻塞缺陷、非阻塞风险、已修复项和复测结果。

验收标准：

- PM 能基于证据包判断当前状态属于继续补证、小修 polish，还是发布准备交接。
- 证据包进入 PM 工作区，不进入 `Releases/`。
- 证据必须区分真实浏览器、headless、Node/VM、人工体验和未覆盖项。

### 6. 必要小修边界

目标是在证据路径中发现明确缺陷时允许小修，但禁止把 v0.15.0 扩张为新玩法或大改版。

允许修复：

- S0：导致游戏无法启动、核心路径无法进入、检查函数完全不可用的阻塞问题。
- S1：导致 P0 路径无法完成、关键 UI 不可点击、关键状态错误或运行时异常的问题。
- S2：导致证据不可信、关键 DOM 缺失、小屏核心信息被遮挡、只读检查字段明显错误的问题。

需要制作人判断后再纳入：

- 手感 polish。
- 文案微调。
- 视觉密度优化。
- 非阻塞小屏舒适度优化。

验收标准：

- 所有修复都必须来自证据路径中的明确现象。
- 修复后必须按原浏览器、原路径或原视口复测。
- 非阻塞 polish 若影响范围变大，应转入后续版本。

## 不做范围

v0.15.0 禁止变成新玩法、核心规则或发布归档版本。以下内容不进入本版本：

- 不新增玩法系统。
- 不新增敌人。
- 不新增建筑。
- 不新增 Meta 内容。
- 不修改核心数值。
- 不修改伤害、敌人 AI、RNG、资源公式、维修或护盾判定。
- 不修改 `SAVE_KEY`。
- 不修改 meta schema。
- 不安装 Playwright、Puppeteer、jsdom、浏览器驱动或其他依赖。
- 不修改系统设置、浏览器全局设置、权限或驱动。
- 不发布、不打 tag、不归档 `Releases/`，除非最终另行批准。
- 不以取证为由写入存档、伪造战斗结果、伪造 Meta 进度或改变玩家状态。
- 不改动与 v0.15.0 证据闸门无关的代码、文档、注释、资源或工程结构。

## 风险级别

本版本风险级别为中低风险。

风险偏低的原因：主要工作是浏览器取证、人工记录、证据归档和必要小修，不主动改变核心玩法、数值、存档和发布产物。

中风险来源：

- 真实浏览器取证可能受本机浏览器、远程调试端口、DevTools 操作方式影响，导致证据不可重复。
- `runReleaseCandidateChecks` 字段可能缺失或误报，影响发布闸门判断。
- 三视口问题若需要 UI 修复，可能触及 HUD 和面板布局。
- 人工路径记录如果不够结构化，容易把“体验过”误读为“证据充分”。

风险升级条件：

- 需要安装依赖、浏览器驱动或新增测试框架。
- 需要修改系统配置、浏览器全局配置、权限或服务。
- 需要修改发布产物、tag、`Releases/` 或正式发布流程。
- 需要修改核心规则、数值、伤害、AI、RNG、`SAVE_KEY` 或 meta schema。

只要触发上述任一条件，必须停下确认并升级流程；涉及本机环境危险操作时，必须先向用户说明风险并等待确认。

## 制作人接受标准

v0.15.0 的接受标准是证据闸门，不是功能完成数量。

最低接受条件：

- 至少获得一次真实浏览器运行态 JSON。
- 至少获得一次 P0 人工路径记录。
- 证据包能说明浏览器环境、操作步骤、视口、结果、失败项和未覆盖项。
- 若存在阻塞缺陷，已完成修复并按原路径复测。

结论规则：

- 如果没有真实浏览器运行态 JSON，结论只能是“继续补证”。
- 如果没有 P0 人工路径记录，结论只能是“继续补证”。
- 如果有 S0/S1 阻塞缺陷，必须修复并复测，不能进入发布准备交接。
- 如果只有非阻塞 S2 或 polish 项，由制作人判断纳入 v0.15.0 小修，或转入后续版本。
- 如果真实浏览器 JSON、P0 人工路径和三视口证据均可复查，且无阻塞缺陷，才允许进入发布准备交接。

## 角色流程

v0.15.0 按以下流程推进：

1. 制作人给出方向：确认版本目标是浏览器证据闸门，不新增玩法、不发布。
2. PM 建立看板：拆分浏览器取证、人工路径、三视口证据、证据包和必要小修任务。
3. 主程给出浏览器取证技术方案：优先使用现有 Edge/Chrome/DevTools/CDP/远程调试或人工 DevTools，不安装依赖。
4. PM 给出人工证据方案：明确 P0 路径、记录格式、通过/失败/未覆盖/阻塞口径。
5. 执行取证和必要小修：先取证，只有发现 S0/S1/S2 明确缺陷时才修复。
6. 专项审核：主程审核只读检查、证据可信度和修复边界；PM 审核记录完整性。
7. PM 汇总证据：把 JSON 摘要、操作步骤、三视口记录、人工路径和风险结论放入 PM 工作区。
8. 制作人决定：继续补证、进入小修 polish，或进入发布准备交接。

## 版本边界声明

v0.15.0-browser-evidence-gate 的交付物是“证据与判断”，不是“新内容”。本版本成功的标志，是团队能用真实浏览器证据回答当前纵向切片是否可以进入发布准备；如果证据不足，诚实给出继续补证，就是正确结论。
