# v0.4.0 任务多样化 制作人方向

版本名：`v0.4.0-mission-diversity`
基线：HEAD `739cc49`（v0.3.0 框架韧性发布后）
风险级别：**项目高风险**（涉及核心玩法和关键数据结构）

## 本轮目标

让 rogue 局内的每个关卡有真正不同的玩法体验，不再被"清剿敌人 + 采矿"单一循环主导。引入与 v0.3.0 fragment 系统强协同的拾荒任务，以及需要保护友方 NPC 的护送任务，并把现有任务系统重构为可扩展的注册表 + 事件分发框架，为后续版本继续扩展任务类型铺路。

## 体验目标

按重要程度排序：

1. **可探索**：玩家进入每个星系时都不知道任务类型，需要看 HUD / 引导才知道本关要做什么；同一关多次重启不一定是同一任务（任务选择参考 seed）。
2. **可参与**：拾荒任务给玩家"在星系里搜寻老旧残骸 → 靠近桥接回来"的明确动作；护送任务给玩家"保护一艘缓慢移动的友方货船"的明确动作。
3. **可重玩**：完成一种新任务后，下一局可能遇到完全不同任务，给玩家"再开一局看看下一关是啥"的动机。

## 必做范围

### 框架重构（基础工程）

- 建立 `OBJECTIVE_TYPES` 注册表（位于 `createObjective` 上方），每个类型声明 `make / tick / onEvent.../ getDetail / getHint / render`。
- 建立 `notifyObjective(event, payload)` 事件分发函数，由现有进度事件触点（`updateMining` 采集、enemy reaper 击杀、guardian 击杀）统一调用，不再各自直接修改 `objective.progress`。
- `updateObjective(dt)`、HUD `progressDetail`、`pickNextForObjective`、`buildGuideText` 全部改为查注册表（不再硬编码 switch）。
- 任务选择改为 seed 化：`createObjective` 用 `state.run.seed` + level 派生 RNG 选任务（保留每个 level 的"基础类型"作为兜底权重，避免破坏 v0.3.0 节奏曲线）。
- 现有 5 种任务（mine / explore / battle / survive / guardian）按新注册表重构，行为完全等价。

### 拾荒任务（salvage，与 v0.3.0 fragment 强协同）

- 进入星系时在世界中预生成 N 个"古老残骸 fragment"（origin = "wreck"），漂浮在星系内固定位置，速度低（明显比玩家脱落的 fragment 慢）。
- 玩家在限定数量内桥接 X 段残骸（X 可由数值方案决定，建议 2-4 段）即完成。
- 残骸内的设施使用现有设施集合（不引入新设施），但可以是"已升级"或"高 HP"版本，让玩家"赚到了"。
- 复用 v0.3.0 桥接系统：拾荒 fragment 与玩家脱落 fragment 行为一致，桥接判定、HUD、维修无人机都自然兼容。
- 拾荒任务不与玩家脱落 fragment 混淆：HUD 上"古老残骸"用区别色或特殊标记区分；任务计数只统计 `origin = "wreck"` 的桥接。

### 护送任务（escort，新增友方 NPC）

- 进入星系时生成 1 艘友方 NPC 货船（friendly vessel），从星系一端缓慢移动到跃迁点（与跃迁出口反向起步，自然接近终点）。
- NPC 有自己的 HP、移动速度、外观（与敌人 / 玩家明显区分）；可被敌方 projectile / 撞击 / asteroid 伤害。
- 玩家任务：保护 NPC 到达跃迁点不死。NPC 抵达跃迁点 → 任务完成；NPC 死亡 → 任务失败（本关无法跃迁 / 直接进入失败结算，或重置任务进度由数值方案决定）。
- NPC 在世界中可见、有 HUD 提示（方向 + 距离 + HP）；玩家炮塔不会自动锁定 NPC（敌我识别）。
- 护送是高紧张度任务，敌人 spawn 节奏可微调（数值方案决定）。

## 不做范围

- 不做并行多任务（保持单 `state.run.objective` 结构，避免 HUD 重构 + 跃迁门控重构）。
- 不做防御 / 救援 / 拾遗 / 探险 / 交付等其它新任务类型（留 v0.4.1 评估）。
- 不做敌方 NPC AI 多样化（敌人保持现有 pirate / asteroid / station / guardian）。
- 不做设施 / 武器 / 科技升级 / 局外天赋扩展。
- 不做联机 / 异步对战。
- 不修改 `SAVE_KEY = "star-ring-station-meta-v1"`、局外存档结构、局外点数算法。
- 不做 v0.3.1 转入评估池的 9 条非阻塞建议（HUD 四档分档、§7.1 60s 减压、文案 pass、anchor 语义等）；这些是独立的体验润色版本。
- 不修改 v0.3.0 fragment 物理（FRAGMENT_* 常量）、桥接判定（BRIDGE_DISTANCE / BRIDGE_ANGLE）、弹道命中（FRAGMENT_PROJECTILE_HIT_RADIUS）。
- 不修改 v0.2.1 三引擎手感 / 推进器松键不续推 / `updateMouseAim` 转向 / 双模式 WASD。
- 不修改 level 0 首局 60 秒减压（spawnWave ≤ 1）+ 初始资源 130/60/35/8/0 + 旧 meta 兼容。

## 兼容要求

- v0.1.x / v0.2.0 / v0.2.1 / v0.3.0 全部回归点必须通过。
- 现有 5 种任务（mine / explore / battle / survive / guardian）在新注册表下行为完全等价（PM 回归必须复测）。
- 终局守护者关（level 6）仍按 guardian 任务执行，不被 seed 化打乱。
- 旧存档 meta 注入后正常读取，包括无 unlocks、空 `{}`、仅 `{points:5}` 形态。

## 已有代码概览（基于探索调研）

- 任务对象：`state.run.objective` 字段 `{type, text, target, progress[, beacon]}`，定义于 `Game/game.js` 中 run 状态块；当前 5 种 type 硬编码。
- 任务生成：`createObjective()` 用 `choices[state.run.level % choices.length]`（确定性，不 seed 化）。
- 任务进度：散落在 `updateMining`、enemy 死亡清算、guardian 死亡清算、`updateObjective(dt)` 共 4 处。
- 完成态：`state.run.objectiveCompleteAt` + `awaitingObjectiveChoice`；跃迁门控 `isObjectiveComplete()`。
- 奖励：`grantObjectiveReward()`，发 `2 + level` 点数，写 `state.meta.points` 并 `saveMeta()`。
- 世界对象：`state.world.bodies` 只有 `star / planet / asteroid`，没有"信号源 / 残骸 / 友方"实体。
- NPC 抽象：`state.enemies` 全部默认敌对，`enemy.kind = pirate / asteroid / station / guardian`，没有友方 / 中立概念。
- Fragment：`state.fragments` 写入口只有 `detachComponents`（玩家脱落），不支持预生成；`nextLevel` 强制清空。
- HUD：`#objective` 由 `setHtmlIfChanged` 整段 innerHTML 重绘，扩展类型需补 detail / hint 分支。

## 关键技术疑问（给主程）

请主程在 ADR 中明确回答以下 8 项决定：

1. **`OBJECTIVE_TYPES` 注册表结构**：每个类型应有哪些必选 / 可选方法？是否需要支持异步事件（如"NPC 状态变化"延迟通知）？
2. **`notifyObjective` 事件分发**：事件命名约定？是否支持多类型同时监听同一事件（如"敌人击杀"同时触发 battle 和"护送途中击退追兵"加分）？
3. **任务选择 seed 化**：如何在保留 v0.3.0 节奏曲线（level 0-2 减压 / 3-4 上升 / 5-6 紧张）的前提下，让任务选择参考 seed？是否每个 level 有"任务类型权重表"？
4. **拾荒 fragment 预生成**：在 `applyGalaxy → createObjective` 之间还是 `createObjective` 内部生成？fragment 上加 `origin` 字段是否影响 v0.3.0 序列化 / HUD / 维修无人机判定？
5. **拾荒残骸位置**：星系内随机散布、距离 station 起点 X-Y 之间，避免太靠近也太远？是否需要避开 planet / asteroid 碰撞？
6. **友方 NPC 抽象**：新建 `state.npcs` 数组，还是扩展 `enemy.kind` 加 `"friendly"`？哪种风险小？
7. **NPC 移动 AI**：直线移动还是 A* 避障？速度多少（建议慢于玩家全速，让玩家能跟上）？
8. **NPC 死亡处理**：任务失败 → 直接 gameOver，还是任务失败 → 留在本关但无法跃迁（玩家自由游玩或重启）？

请数值策划在数值方案中明确以下参数：

- 拾荒任务每级目标 X（建议 level 0-2 X=2，level 3-5 X=3，避免新玩家压力）。
- 拾荒残骸数量 N（建议 X+1 到 X+2，给玩家"找最好的"选择空间）。
- 拾荒残骸速度（建议 v0.3.0 玩家脱落 fragment 速度的 30%-50%）。
- 拾荒残骸内设施类型 + HP 范围（建议复用 turret / shield / armor / repair，HP 在 60%-90% 区间）。
- 护送 NPC HP / 速度 / 跃迁点距离 / 完成时间窗。
- 护送任务期间敌人 spawn 倍率（建议保持或略降，避免 NPC 立刻死）。
- 任务选择权重表（每个 level 各任务类型权重）。
- 拾荒任务奖励倍率（建议比 mine / explore 高，反映"主动搜寻 + 桥接成本"）。
- 护送任务奖励倍率（建议比 battle 高，反映"全程紧张"）。

## 高风险流程要求

按团队规则，本任务属于核心玩法 + 关键数据结构改动，必须走完整高风险流程：

1. 制作人方向（本文档）已 commit + push 后才能进入下一步。
2. PM 拆分 v0.4.0 任务看板，包含 ADR / 数值 / 实现 / 专项审核 / PM 回归 / 制作人最终决定 6 个核心任务节点。
3. 主程 ADR 必须先 commit + push 才能进入实现。
4. 数值方案与 ADR 可并行启动，但必须在实现前 commit。
5. 实现可考虑拆分 T1 框架重构 / T2 拾荒 / T3 护送，每个子任务后做最小验证。
6. 实现完成后进入专项审核（与执行不同基底 agent），通过后 PM 完整回归。
7. 通过后制作人最终决定接受 + 是否触发发布。
