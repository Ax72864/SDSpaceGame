## 体验目标（必须命中）

让"框架脱落 → 玩家重新接入"成为玩家在一局内**可感知、可操作、可回收**的循环：

1. **可感知**：站体某段链路被打断时，脱落部分明显从站体上"飞出来"成为漂浮残骸；HUD 与音画反馈让玩家立刻意识到"丢了一块"。
2. **可操作**：玩家可以操控站体靠近残骸，把它视作世界中的目标移动 / 围绕；不强制要求精确对齐到 grid。
3. **可回收**：玩家在合适位置扩展 frame 时，能够把附近的残骸整段重新合并回站体，并恢复其上设施功能（包括之前的耐久 / 状态）。

体验风格基线：
- 仍然是 v0.2.0 节奏紧凑的太空站建造，不引入复杂物理模拟；残骸物理简化到"惯性漂移 + 阻尼"层级即可。
- 不强制玩家立刻回收残骸，允许玩家选择放弃；但应让"试图回收"的玩家有可读路径和明确反馈。

## 必做范围（v0.3.0 主版本范围）

### 1. 脱落残骸作为独立浮动实体

- 现状：`checkConnectivity` 已经把无法连接 core 的 cell 标 `detached=true` 并塞个 `drift` 向量，但仍存储在 `state.station.cells` Map 中并跟随站体世界位移。
- 必做：把整段脱落 cells 抽离成独立 fragment（建议 `state.fragments[]`），具备自己的世界坐标、速度、角度、角速度、cells 子集；station 主 Map 内不再保留这些 cell。
- 残骸物理：惯性漂移 + 轻度阻尼，无主动 AI；可被敌人 / 玩家投射物物理影响（最小：被弹道误伤减 HP；不必强制）。

### 2. 残骸视觉与 HUD 反馈

- 残骸用区别于站体的视觉表现：例如外环虚线、灰阶、半透明、闪烁警示。
- HUD 新增"脱落"提示：最近残骸方向 / 距离 / 设施数量；可复用现有 `statusAlerts` 或新增最小一段。
- 残骸被敌人摧毁或漂离世界边界后应明确消失，并给一次性提示。

### 3. 玩家靠近 + 重新接入

- 当玩家空间站某个候选 grid 位的世界坐标距任意 fragment cell 的世界坐标 < 阈值（建议 ~`CELL * 1.2`）、且 fragment cell 朝向与 station grid 朝向夹角 < 阈值（允许偏差，做投影修正）时，扩展 frame 可以"桥接"。
- 桥接成功：把整段 fragment 合并回 station Map；fragment cells 的位置按相对偏移转回 station grid；保留其 facility、hp、frameHp、enabled、active 等运行时状态；触发"重新接入 N 个结构"成功反馈。
- 桥接失败：与现有 `setBuildError` 风格一致提示，例如"未在脱落残骸附近，无法直接桥接"。

### 4. 平衡与可读性兜底

- 脱落不应是惩罚机制叠加炸弹：当 fragment 数量过多（例如 > 6）或玩家短时间内丢失过多 cell 时，给一次性平衡提示并允许玩家通过维修无人机 / 重连恢复部分压力。
- 维修无人机应对 detached cell 提供有限支持：例如允许维修"刚脱落的 fragment 上的 cell"（具体策略由数值策划在执行阶段决定）。

## 不做范围（明确推迟）

- 不做新设施（如新武器 / 新维修类型 / 新建造资源）。
- 不做完整物理碰撞重做（残骸不与小行星 / 行星实体碰撞，仅与站体桥接判定相关）。
- 不做局外天赋树扩展（即使脱落与韧性相关）。
- 不做联机 / 异步对战。
- 不做新关卡任务类型（护送 / 撤离）。
- 不做存档结构变更：`SAVE_KEY="star-ring-station-meta-v1"` 不动；新增运行态字段一律放 `state.run` / 新建 `state.fragments`。
- 不做发布归档，除非制作人在版本验收后另行决定。

## 待 `/lead-game-programmer` 在 ADR 中评估的关键技术决定

1. **fragment 数据结构与生命周期**：是否用独立 `state.fragments[]`；每个 fragment 是否保留 `Map<"x,y", Cell>` 子集 + 世界变换；如何处理 fragment 内 cell 的相对 grid 坐标。
2. **station ↔ fragment 坐标系切换**：脱落瞬间如何把 cells 从 station grid 转世界坐标；重连瞬间如何把 fragment 子集投影回 station grid 而不引起 grid 冲突（重叠位置应被拒绝或重定向）。
3. **物理简化策略**：fragment 是否参与与 station 的碰撞反弹；是否被敌人弹道伤害；是否能被推走以避免桥接干扰。
4. **桥接判定算法**：候选距离阈值、角度对齐阈值、是否允许多 fragment 同时桥接、是否允许桥接到 fragment 内部任意 cell 还是仅边缘 cell。
5. **重连后状态恢复**：fragment cells 的 facility / hp / frameHp / enabled / active / priority 是否完整保留；reload / fire 等运行时累积是否归零。
6. **性能预算**：fragments 上限（建议 ≤ 8）、单 fragment cells 上限、每帧 BFS / 距离判定 budget。
7. **渲染管线**：fragment 是否复用 `renderStation` 转换、是否新增 `renderFragments`、是否支持多 fragment 旋转。
8. **回归风险**：检查改动是否影响 `damageCell` / `checkConnectivity` / `buildAt` / `updateRepair` 的现有路径。
