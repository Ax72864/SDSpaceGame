# v0.12.0-station-design-feedback 制作人方向

## 结论

v0.12.0 的版本名为 `v0.12.0-station-design-feedback`。

本版本不继续叠加新系统，而是把现有建造、资源、移动、战斗、研发、星系图和 Meta 成长从“能造东西、能跑流程”推进到“玩家能理解自己的空间站设计是否合理”。玩家应该在建造和检查空间站时，看懂设施角色、资源/电力短板、采矿范围、武器射线、推进与防御缺口，并据此做出设计决策。

v0.11.0-playtest-readiness 已接受为“开发完成 / 自动与 headless 浏览器回归通过”，但真实人工发布验收仍未完成。v0.12.0 可以继续推进制作，不把人工发布验收卡成本版本启动条件；但发布判断仍应在后续由 PM 和制作人补齐真实体验证据。

制作人方向：优先增强诊断、解释和可读性，不新增建筑类型，不改底层规则，不自动替玩家修复空间站。若玩家能在战斗、采矿或跃迁前主动发现问题并调整设计，本版本目标即成立。

## 玩家收益

- 减少瞎点建造：建造面板能说明设施用途、成本、电力压力和当前是否负担得起。
- 知道为什么设施不工作：选中设施或准备建造时，能看到资源不足、未连接核心、断电、范围不达标等原因。
- 主动修正空间站：玩家能在采矿、战斗和跃迁前发现采矿范围、武器火力、护盾/装甲、维修能力和推进能力短板。
- Meta 与协议收益更可观察：局外 Meta 和开局协议带来的成本、电力或初始能力变化，在建造和健康摘要中更容易被看见。
- 设计取舍更明确：设施布局、资源供给、推进器/武器遮挡、维修、护盾和装甲不再只是背景数值，而是能被玩家读取和比较的设计信号。

## 核心目标

让玩家在建造和检查空间站时能理解以下问题：

- 设施角色：这类设施是采矿、供电、推进、武器、防御、维修还是支撑设施。
- 资源/电力短板：当前为什么造不起、为什么断电、建造后会不会拉低电力余量。
- 采矿范围：采矿站是否覆盖资源外环，离最近资源是否太远。
- 武器射线：武器、导弹等攻击设施的作用方向、边界或可用状态是否清楚。
- 推进/防御缺口：推进能力、护盾/装甲、维修覆盖是否足以支撑下一次战斗或跃迁。

本版本的体验目标不是让系统更复杂，而是让玩家从“我能放建筑”升级为“我知道为什么这样放更好”。

## 必做范围

### 1. 建造面板可读性

目标是让建造入口从“按钮列表”变成“可比较的设计选择”。不要求大改 UI 框架，但必须让玩家能快速判断每个设施的角色、代价和当前可行性。

建议方向：

- 设施按角色或标签展示，例如采矿、供电、推进、武器、防御、维修、支撑。
- 卡片显示建造成本、电力产出/消耗、核心用途和当前能否负担。
- 卡片显示已建数量，必要时给出简短推荐理由，例如“当前无采矿能力”“电力余量偏低”“已有武器但缺防护”。
- 当前资源不足或电力压力明显时，使用低噪声状态提示，不把每张卡都刷成警告。
- 若 Meta 或协议改变成本、初始资源或建造能力，卡片上应能观察到变化后的结果。

验收标准：

- 空存档玩家打开建造面板后，能区分采矿、供电、推进、武器、防御和维修设施的主要作用。
- 资源不足时，玩家能在尝试建造前看到至少一处明确原因。
- 已建数量或推荐理由能帮助玩家判断下一步优先补什么，而不是只看按钮是否可点。

### 2. 放置/选择反馈

目标是让玩家在准备建造或选中设施时，能看到和当前位置、连接、供电、范围、战斗能力相关的诊断。

必须覆盖的反馈：

- 资源不足：显示缺哪类资源或当前不可负担。
- 未连接核心：提示该位置或设施无法接入空间站核心。
- 电力压力：提示建造后可能导致断电，或当前设施因缺电无法工作。
- 采矿范围：采矿站距最近资源外环是否过远，是否能有效覆盖资源。
- 武器/导弹边界：提示射线、方向、作用范围或可攻击状态，避免玩家误以为有武器就一定能打到。
- 护盾边界：提示护盾保护对象或覆盖逻辑，至少说明当前防护能力是否存在。
- 维修覆盖：提示维修站覆盖逻辑，玩家能判断受损设施是否可能被维修。

验收标准：

- 准备建造采矿站时，玩家能判断该位置是否接近资源外环。
- 选中武器、护盾或维修相关设施时，玩家能看到其作用边界或状态说明。
- 断电、未连接核心和资源不足不会只表现为“按钮没反应”，必须有可读原因。

### 3. 空间站设计健康摘要

目标是在右侧状态或更多状态中，用短项总结空间站当前设计状态。该摘要只做诊断和建议，不自动修复、不替玩家下决策。

建议短项控制在 4-6 个：

- 电力余量：电力是否充足，是否有大量设施断电。
- 采矿能力：是否有采矿站，是否有效覆盖资源。
- 推进能力：推进设施是否存在，跃迁或移动前是否明显不足。
- 武器火力：是否有武器，是否缺电或作用边界受限。
- 护盾/装甲：是否具备基础防御，是否有明显防御空窗。
- 维修能力：是否有维修设施，覆盖是否能解释给玩家。

验收标准：

- 玩家无需逐个点设施，也能从摘要看出空间站最关键的 1-2 个短板。
- 摘要文案是诊断和建议，不出现“自动优化”“一键修复”等超出本版本范围的承诺。
- 战斗前或跃迁前，摘要能帮助玩家判断是否需要补电、补武器、补防御或补采矿。

### 4. 低噪声问题提示

目标是只在关键问题出现时打断或提醒玩家，避免把诊断系统做成持续刷屏的红字列表。

建议触发条件：

- 有采矿站，但全部采不到资源。
- 有武器或导弹设施，但因缺电或关键状态无法发挥作用。
- 敌人来袭或即将战斗，但空间站没有武器/护盾等基本防御能力。
- 大量设施断电，且影响采矿、武器、防御或推进等核心能力。
- 玩家准备跃迁或进入高风险节点时，空间站健康摘要存在明显短板。

验收标准：

- 普通轻微低效不频繁打断玩家。
- 关键风险出现时，提示能落到具体问题，例如“采矿站未覆盖资源”“武器断电”“敌袭前无防御”。
- 提示不会替代建造面板和健康摘要，只作为关键时刻的短提醒。

### 5. 测试/验收钩子

目标是给 PM、主程和制作人提供只读验证信息，用于自动检查、headless 浏览器回归和后续人工验收记录。

建议只读输出：

- build palette：建造面板设施列表、角色标签、成本、电力、可负担状态和已建数量。
- design health：空间站健康摘要，包括电力余量、采矿能力、推进能力、武器火力、防御和维修状态。
- selected cell diagnostics：当前选中格或准备建造位置的连接、资源、电力、范围和作用边界诊断。
- resource reachability：采矿站与资源外环的可达或覆盖状态。
- power margin：当前电力产出、消耗、余量和断电影响摘要。

约束：

- 只读输出，不写存档。
- 不修改 `SAVE_KEY` 或 meta schema。
- 不改变普通玩家玩法规则。
- 默认不污染普通 UI，可通过测试钩子、debug snapshot 或现有测试入口读取。

验收标准：

- PM 能用工具读取建造面板、设计健康、选中格诊断、资源可达性和电力余量。
- 读取测试钩子不会改变存档、资源、敌人、任务进度或随机结果。
- 输出字段能支撑本版本验收路径，而不是只返回笼统的“ok”。

## 不做范围

本版本禁止扩展为新系统版本。以下内容不进入 v0.12.0：

- 不新增建筑类型。
- 不修改 `SAVE_KEY`。
- 不修改 meta schema。
- 不重写建造网格。
- 不修改敌人、任务或 RNG。
- 不改核心战斗判定。
- 不做全新教程系统。
- 不做发布归档、发布 tag 或版本发布。
- 不把 v0.11.0 真实人工发布验收作为本轮开发阻塞项。
- 不以诊断需求为由重构无关 UI、资源公式或存档结构。

## 风险级别

本版本风险级别为中风险。

中风险来源主要是 UI 信息架构、诊断准确性、测试钩子字段和体验解释是否一致。若实现只涉及展示、只读诊断、文案和非侵入式测试输出，可以按中风险推进。

若实现过程中触及以下内容，风险升级为高风险，并需要制作人重新确认、PM 调整看板、主程专项审核：

- 设施数值。
- 资源生产或消耗公式。
- 电力结算公式。
- 战斗命中、射线、导弹、护盾、装甲或维修判定。
- 存档结构、`SAVE_KEY` 或 meta schema。
- 敌人、任务、星系图推进或 RNG。

## 验收路径

v0.12.0 必须围绕“玩家能否理解空间站设计”验收，而不是只检查 UI 是否显示。

必测路径：

- 空存档建造采矿链：玩家打开建造面板，理解采矿、供电和基础支撑设施，并能完成最小采矿链。
- 资源不足建造失败：资源不够时，建造面板和放置反馈能说明缺口，不只是失败。
- 断电设施：制造电力压力后，玩家能从设施反馈或健康摘要看出哪些关键能力受影响。
- 采矿站远离外环：采矿站离资源过远时，系统能提示覆盖不足或采不到资源。
- 武器状态：武器/导弹设施能说明作用边界、可用状态或缺电问题。
- 护盾状态：护盾能力能被玩家识别，至少知道当前是否具备基础防护。
- 维修状态：维修站覆盖逻辑可读，玩家能判断哪些受损设施可能被维修。
- 战斗前健康摘要：敌人来袭或战斗前，摘要能指出武器、防御、电力或维修短板。
- Meta/协议加成下成本变化显示：已有 Meta 或开局协议影响成本、资源或能力时，建造面板能展示变化后的结果。

制作人接受标准：

- 玩家能说出当前空间站最缺什么，以及为什么。
- 至少三个关键设施角色能被建造面板或选中反馈解释清楚。
- 至少一个采矿范围问题、一个电力问题、一个战斗/防御问题能被诊断出来。
- 测试钩子能只读输出本版本验收所需信息。
- 没有新增系统范围膨胀，没有修改存档结构或核心玩法判定。

若以上任一核心项缺失，制作人结论应为“继续打磨，不进入发布候选”。

## 角色流程

建议执行流程如下：

1. 制作人方向：确认 v0.12 聚焦空间站设计反馈，不新增建筑和新玩法系统。
2. PM 看板：拆分任务、记录允许修改范围、验收标准、风险升级条件和回归路径。
3. UI 信息架构：整理建造面板标签、卡片字段、状态摘要和低噪声提示优先级。
4. 主程技术边界：确认只读诊断数据来源、测试钩子字段、不会改存档和核心判定。
5. 数值/文案表：列出设施角色、用途说明、成本/电力展示规则和短诊断文案。
6. 分批实现：先建造面板可读性，再放置/选择反馈，再健康摘要和测试钩子，最后做低噪声提示。
7. 专项审核：主程审核诊断是否越界到规则修改，UI 审核可读性，数值策划审核文案是否误导。
8. PM 回归：覆盖空存档、资源不足、断电、采矿范围、武器/护盾/维修、战斗前摘要和 Meta/协议加成显示。
9. 制作人决定：根据 PM 记录和体验证据，决定 v0.12 是否可进入发布候选或继续打磨。
