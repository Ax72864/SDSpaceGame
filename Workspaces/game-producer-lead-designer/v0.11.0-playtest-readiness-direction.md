# v0.11.0-playtest-readiness 制作人方向

## 结论

v0.11.0 的版本名为 `v0.11.0-playtest-readiness`。

本版本不继续叠加新系统，而是把 v0.9 星系图与 v0.10 局外成长从“功能可用”推进到“可真实体验、可验收、可准备发布”。当前最关键的问题不是系统数量不足，而是玩家能否在第一局理解目标、完成一次局内到局外的闭环，并在失败或通关后明确知道下一局要做什么。

制作人方向：优先打通首局理解、面板互斥、小屏可读性和浏览器验收证据。若这些体验证据不足，v0.10 不应直接发布；若 v0.11 完成并通过人工体验验收，再由制作人决定发布 v0.11 或继续打磨。

## 玩家收益

- 首局路径更清楚：玩家能理解当前目标、靠近资源、建造采矿、完成任务、跃迁、结算和购买 Meta 的关系。
- 信息焦点更稳定：Meta 面板、星系图、结算和任务选择不会互相抢视线，也不会形成多个大面板叠加。
- 失败和通关后有下一步：玩家能从结算反馈中知道获得了什么、为什么失败或通关、下一局该选择协议还是购买 Meta。
- 小屏体验更可靠：窄屏下右侧面板、候选卡、抽屉和关键按钮可读、可滚动、不遮挡核心 HUD。
- 发布判断更有证据：制作人和 PM 能用浏览器级证据判断版本是否可发布，而不是只依赖工具回归通过。

## 必做范围

### 1. 首局体验引导重构

目标是让玩家在空存档第一局能顺着游戏意图走完最小闭环。现有目标、提示和下一步文案需要更聚焦，不追求长教程。

必须覆盖的引导节点：

- 当前要做什么：优先让玩家理解“先建立采矿能力”。
- 为什么靠近资源：提示玩家靠近资源点后再建造或采集更有效。
- 如何完成任务：任务目标文案要能指向局内行动，而不是只显示抽象条件。
- 何时跃迁：完成关键目标后，明确告诉玩家可以进入星系图或继续推进。
- 结算后看什么：失败或通关结算要明确展示本局结果、获得内容和下一步入口。
- 为什么购买 Meta：Meta 购买应被解释为“让下一局更强或更稳定”的局外成长，而不是孤立商店。

验收标准：

- 空存档玩家在不读外部说明的情况下，能完成至少一次“采矿目标 -> 跃迁/结算 -> 购买 Meta 或选择下一局协议”的路径。
- 每个关键节点都有一个清晰主提示，不出现多个同级提示争抢注意力。
- 提示文案不引入新系统承诺，不暗示尚未实现的玩法。

### 2. 多面板互斥与 Escape 行为

目标是稳定大面板的打开、关闭和优先级，避免玩家在首局闭环中被叠层 UI 打断。

必须纳入互斥管理的大面板：

- `metaPanel`
- `galaxyMapPanel`
- `runSettlementPanel`
- `objectiveChoicePanel`

建议方向：

- 同一时间最多展示一个主流程大面板。
- 打开更高优先级流程面板时，应稳定关闭或隐藏其他冲突面板。
- Escape 行为应按“当前最上层面板 -> 次级弹层 -> 返回游戏”的顺序关闭。
- 结算面板打开期间，不应被星系图或 Meta 抽屉意外盖住关键结果。

验收标准：

- 连续打开 Meta、星系图、结算、任务选择时，不出现两个以上大面板同时遮挡主视野。
- Escape 能关闭当前最需要关闭的面板，且不会误触发新局、购买、选择协议等不可逆动作。
- 面板关闭后 HUD 状态、按钮可用性和焦点状态保持一致。

### 3. 小屏与右侧面板可读性打磨

目标是让窄屏视口下仍能完成首局路径，不因按钮遮挡、滚动缺失或信息密度过高阻断体验。

重点对象：

- Meta 抽屉
- 失败/通关结算面板
- 星图候选卡
- 右侧目标/状态面板
- 关键 HUD 和底部操作区

建议方向：

- 窄屏下大面板内容必须可滚动，主要按钮不得落到不可点击区域。
- 右侧信息不应永久遮挡局内关键 HUD。
- 星图候选卡在小屏下应优先保留“当前选择、奖励/风险、确认按钮”三类信息。
- Meta 抽屉应优先显示可购买项、当前货币和返回路径，弱化次要说明。

验收标准：

- 在小屏视口下，玩家能完成任务选择、结算确认、Meta 购买和新局开始。
- 所有主按钮可见或可通过明确滚动触达。
- 重要 HUD 不被右侧面板长期遮挡，玩家仍能判断资源、生命/状态和当前目标。

### 4. 浏览器验收辅助

目标是让 PM 和制作人能记录浏览器证据，判断“功能回归通过”之外的真实可验收状态。

建议新增轻量 playtest/debug snapshot 或 `__gameTest` 汇总钩子。若实现成本可控，应输出以下信息：

- 当前主面板状态：哪些大面板打开、当前焦点面板、是否存在冲突叠加。
- `localStorage` 中 Meta 相关状态摘要：不泄露无关存储，不修改 `SAVE_KEY`。
- console/runtime error 计数：如浏览器环境可行，记录错误数量和最近摘要。
- 关键 DOM 存在性：HUD、Meta 面板、星系图、结算面板、任务选择面板、关键按钮是否存在。
- 完整路径自检结果：空存档首局、任务完成、跃迁、结算、购买 Meta、选择协议、新局生效等检查点能否被工具读取。

约束：

- 该能力必须隐藏在测试钩子、query 参数或 debug 开关下。
- 不影响普通玩家体验，不在默认 UI 暴露调试信息。
- 不新增依赖，不安装系统工具。
- 不改变存档 key，不为了测试便利重写核心状态。

验收标准：

- PM 能在浏览器环境读取一份当前状态快照，并用于回归记录。
- 快照能辅助定位面板叠加、Meta 状态异常、关键 DOM 缺失和运行时错误。
- 默认普通游玩路径不显示调试内容。

### 5. 发布前体验记录

目标是补齐 v0.10 未覆盖的真实体验证据，避免“工具回归通过但玩家体验不可发布”。

PM 记录至少区分：

- 自动检查：脚本或工具能确认的状态。
- Headless 检查：浏览器无头环境覆盖到的流程。
- 有头浏览器检查：真实浏览器中是否能看清、点到、关闭和继续。
- 人工体验覆盖项：首局理解、失败后下一步、通关后下一步、小屏可读性。
- 人工未覆盖项：明确列出未验证路径，供制作人判断是否阻塞发布。

验收标准：

- PM 能给出“可发布 / 继续打磨 / 阻塞”的判断依据。
- 制作人能根据记录判断 v0.11 是否达到发布标准。
- 主观体验问题必须落到具体路径、面板或文案，不接受只写“感觉还行”。

## 不做范围

本版本禁止把范围扩成新内容版本。以下内容不进入 v0.11.0：

- 不新增核心系统。
- 不新增 Meta 节点。
- 不修改 `SAVE_KEY`。
- 不新增 enemy、objective 或 encounter。
- 不做联机。
- 不做发布归档或 tag，除非本轮最后由制作人另行批准。
- 不安装新依赖或系统工具。
- 不重做底层架构。
- 不以调试便利为由改变普通玩家流程。

## 风险级别

本版本风险级别为中风险偏体验。

主要风险来自首局理解、UI 互斥、小屏布局和验收证据是否足够。若实现过程中只调整文案、UI 状态管理、调试快照和验收记录，按中风险处理。

若改动触及以下内容，风险升级为高风险，并需要主程专项审核：

- 存档结构或 `SAVE_KEY`。
- 核心 run 状态机。
- Meta 生效逻辑。
- 星系图推进规则。
- 任务完成、失败或通关判定。

## 验收路径

v0.11.0 必须围绕完整玩家路径验收，而不是只看单个按钮是否可点击。

必测路径：

- 空存档首局：玩家从进入游戏开始，能理解第一目标并完成一次局内推进。
- 旧存档迁移后首局：已有 v0.9/v0.10 数据时，不出现阻断性 UI 或 Meta 状态异常。
- 任务完成跃迁：完成任务后能进入星系图或看到明确下一步。
- 失败结算：失败原因、获得内容和下一步入口清楚。
- 通关结算：通关反馈、奖励和下一步入口清楚。
- 购买 Meta：能看懂货币、购买项、生效意义和购买结果。
- 选择协议：能理解协议选择对下一局的影响，并成功进入新局。
- 新局生效：已购买 Meta 或已选择协议在下一局有可观察反馈。
- 小屏视口：窄屏下能完成上述关键路径，按钮不被遮挡。

## 角色流程

建议执行流程如下：

1. 制作人方向：确认 v0.11 聚焦 playtest readiness，不新增系统。
2. PM 看板：拆分任务、记录允许修改范围、验收标准、风险和阻塞项。
3. UI/主程轻量方案：先确认面板互斥、小屏布局和测试钩子的最小实现方式。
4. 分批实现：按首局引导、面板互斥、小屏打磨、浏览器验收辅助、体验记录分批推进。
5. 专项审核：主程审核状态机/存档风险，UI 审核可读性和遮挡风险。
6. PM 浏览器回归：记录自动检查、headless、有头浏览器和人工体验覆盖。
7. 制作人决定：根据证据决定发布 v0.11，或继续打磨到下一个候选版本。

## 制作人接受标准

v0.11.0 可以进入发布候选的条件：

- 首局玩家能完成从局内目标到局外成长的闭环。
- 四个主流程大面板不会形成阻断性叠加。
- 小屏下关键路径可完成。
- PM 能提供浏览器级验收记录，而不只是工具回归结果。
- 没有新增系统范围膨胀，没有破坏旧存档和核心状态机。

若以上任一项缺失，制作人结论应为“继续打磨，不发布”。
