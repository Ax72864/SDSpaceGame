# v0.5.0 研发与升级 制作人方向

版本名：`v0.5.0-research-and-upgrade`
基线：HEAD `4a2ac6e`（v0.4.0 任务多样化发布后）
风险级别：**项目高风险**（涉及核心玩法、关键数据结构和大量散布参数迁移）

## 本轮目标

让玩家在每局 15-30 分钟内有"研发路线 build"决策，使每局有渐进感和路线分歧；玩家完成任务后可以主动选择升级 / 改造 / 激活全局加成，让 13 种设施有真正的差异化深度，同时不破坏 v0.1.x - v0.4.x 已验收的 techLevel 自动难度曲线。

## 体验目标

按重要程度排序：

1. **可感知 build 决策**：玩家完成任务后获得 research 点数，可以在面板上看到"升级炮塔"/"改造导弹井"/"激活船体强化"等可选项；选择后立刻可感知效果（弹道更快 / 伤害更高 / HP 增加）。
2. **可演化的 build 路线**：每局结束时，同一玩家的两次 run 因为研发选择不同，build 表现可能差异较大；这是 rogue-like 的"再开一局看看不同路线"动机。
3. **可平衡的难度曲线**：v0.1.x-v0.4.x 的 `techLevel` 自动升级机制（敌人难度上升）保留不变；玩家研发是额外加成路线，不替代曲线，不让早期玩家暴打中后期敌人，也不让中后期敌人暴打研发不足的玩家。

## 必做范围

### 工程基础（先做这些再做面板）

- **`getCellStat(cell, key)` 集中读取层**：把散布在 5+ 个 update 函数体内的硬编码 damage / reload / range / repairRate / produceRate 等参数集中到 `getCellStat` 查询接口；现有 update 函数改为 `getCellStat(cell, "damage")` 等查询；这一步**仅做查询层抽象，不改变任何数值**（v0.1.x-v0.4.x 行为完全等价）。
- **`TYPES` 字典扩展**：每个设施 type 添加 `baseStats` 字段（含 damage / reload / range / repairRate / produceRate 等），将硬编码值填入；同时添加 `upgrades` 数组（v0.5.0 范围内每个设施 ≤ 3 个升级路径）和 `modifications` 数组（每个设施 ≤ 2 个 mutually exclusive 改造）。
- **`cell.tier`（升级层级）+ `cell.mod`（改造选项）字段**：cell 上记录当前升级 tier 和已选改造；`getCellStat` 在查询时应用 tier 倍率 + 改造效果；v0.1.x-v0.4.x 既有 cell 默认 tier=0 / mod=null（向后兼容）。

### 研发点积累（任务系统挂钩）

- **`researchReward` 字段加入 `OBJECTIVE_TYPES`**：每个任务类型注册研发点奖励（数值方案给具体数值，建议基础 3-8 点）；任务完成时 `state.resources.research += floor(researchReward * (1 + level * 0.1))`。
- **保留现有 research 资源体系**：当前 research 由 research 设施产出 + 部分任务消耗；v0.5.0 不破坏这些路径，仅新增"任务完成 → research 奖励"+"玩家选研发 → research 消耗"。

### 研发面板 UI

- **选中 cell 弹出"升级此设施"按钮**（在现有 `selectedCellPanel` 上扩展）：显示 cell 当前 tier / 当前 mod / 可用升级路径（含 research 成本 + 效果说明）/ 可用改造选项（含 research 成本 + 效果说明）；点击"升级"按钮消耗 research 提升 tier；点击"改造"按钮消耗 research 应用 mod。
- **全局研发面板**（新增独立 DOM 区域）：显示全局 build 加成（`hullMod` / `weaponMod` 等死代码激活点）的可激活节点，玩家消耗较高 research 解锁全局加成；与"升级单个 cell"不同，全局加成对全 station 所有同类设施生效。
- **HUD research 资源条更新**：当前 HUD 已显示 research 数字，v0.5.0 在数字旁加"+ 增长率 / 秒"提示（research 设施产出 + 任务奖励近 30 秒滑动平均）。

### 设施升级清单（每个设施 ≤ 3 个升级路径）

- **核心战斗设施**（turret / missile / shield）：
  - turret：升级 1 +20% damage / 升级 2 +20% reload speed / 升级 3 +20% range。
  - missile：升级 1 +25% damage / 升级 2 +20% reload speed / 升级 3 +30% missile speed。
  - shield：升级 1 +25% maxShield / 升级 2 +30% regen rate / 升级 3 +50% range。
- **资源 / 维护设施**（power / mining / harvester / repair / research）：
  - power：+25% produceRate（最多 +75% 共 3 tier）。
  - mining：+20% mineRate（最多 +60% 共 3 tier）。
  - harvester：+20% produceRate（最多 +60% 共 3 tier）。
  - repair：+25% repairRate（最多 +75% 共 3 tier）。
  - research：+25% produceRate（最多 +75% 共 3 tier）。
- **结构设施**（armor / frame / core）：
  - armor：+30% maxHp（最多 +90% 共 3 tier）。
  - frame：+20% maxFrameHp（最多 +60% 共 3 tier）。
  - core：不可升级（保持核心简单）。
- **其它设施**（engine 等）：
  - engine：+15% thrust（最多 +45% 共 3 tier）。

具体倍率与 research 成本由数值策划定，本方向给区间建议。

### 设施改造清单（每个设施 ≤ 2 个 mutually exclusive 改造）

- turret：A "高速点射"（reload -40% / damage -30%）vs B "重型炮"（reload +50% / damage +80%）。
- missile：A "导弹蜂群"（projectile count ×2 / damage -40%）vs B "穿甲弹"（damage +60% / reload -20%）。
- shield：A "全向场"（range +100% / regen -50%）vs B "强化场"（maxShield +80% / range -20%）。
- engine：A "加速器"（thrust +30% / mass +20%）vs B "稳定器"（angularDamping +50% / thrust -10%）。
- 其他设施改造选项由数值策划补齐，本方向给 4 个核心样例。

### `hullMod / weaponMod` 死代码激活

- 当前 `state.station.hullMod` / `state.station.weaponMod` 是死代码（已定义但未实际应用）。
- v0.5.0 将其作为研发面板的"全局加成节点"，玩家用 research 解锁后生效：
  - `hullMod`（已激活）：所有 station cell maxHp ×1.05。
  - `weaponMod`（已激活）：所有 turret + missile damage ×1.05。
- 后续 v0.5.x 可以扩展更多全局节点（如 `engineMod` / `shieldMod`）。

## 关键技术疑问（给主程）

请主程在 ADR 中明确回答以下 8 项决定：

1. **`getCellStat(cell, key)` 接口形式**：纯函数查询，还是 cell 上挂载 `cell.getStat(key)`？默认值如何 fallback（TYPES.baseStats → cell 上覆盖 → tier 倍率 → mod 效果）？
2. **`cell.tier` / `cell.mod` 序列化**：旧存档 / 跨关 nextLevel / resetRun 时如何处理？建议每关重置 cell.tier=0 / cell.mod=null（与 fragments / npcs 清空一致），让研发是"本局有效"。
3. **research 成本曲线**：每个 tier 的 research 成本是线性、指数还是平方？建议指数（tier 1 = 10, tier 2 = 20, tier 3 = 40）让玩家有取舍。
4. **改造唯一性**：选择改造 A 后能否反悔切到 B？建议**不可逆**（防止玩家反复横跳），如要回退需消耗 1 个"科研重置点"（v0.5.0 不引入，留 v0.5.1）。
5. **全局加成 `hullMod / weaponMod`**：是 binary（解锁 / 未解锁）还是 stackable（叠加多次）？建议 binary（简化 v0.5.0 范围）。
6. **`techLevel` vs 玩家研发的叠加方式**：是乘法（`(1 + tier * 0.2) * 1.08^techLevel`）还是加法（`tier * 0.2 + 0.08 * techLevel`）？建议**乘法**（与既有 HP 倍乘逻辑一致），但要给 PM 关注数值爆炸的回归点。
7. **研发面板 UI 位置**：在 `selectedCellPanel` 内追加按钮，还是独立全局面板？建议两者兼有——cell 升级走 `selectedCellPanel`，全局加成走新增独立面板（"研发树"按钮打开）。
8. **现有 `weaponFrame / efficientCore` 解锁**：是否要把它们也迁入研发面板（成为研发树节点）？建议**保留现状不动**（v0.5.0 不动现有局外解锁路径，仅引入"局内研发"）。

## 数值边界建议（给数值策划）

- **任务 researchReward**：mine = 3、explore = 4、battle = 5、survive = 6、salvage = 5、escort = 7、guardian = 12。
- **tier 倍率（升级）**：每 tier 加成 15%-25%（按设施类型，建议范围 0.15 - 0.30）。
- **tier 成本**：线性或指数（建议 tier 1 = 10、tier 2 = 20、tier 3 = 40）。
- **改造成本**：固定 30 research（一次性选择）。
- **全局加成成本**：50 research（binary 解锁）。
- **techLevel 与玩家 tier 叠加方式**：乘法（按方向 §1 / §6 建议）。
- **数值上限**：单 cell stat 不超过 `2 × baseStats[key]`（避免爆炸）。

## 高风险流程要求

按团队规则，本任务属于核心玩法 + 关键数据结构改动 + 大量散布参数迁移，**必须走完整高风险流程**：

1. 制作人方向（本文档）commit + push 后才能进入下一步。
2. PM 拆分 v0.5.0 任务看板（7 个核心任务节点，参考 v0.4.0 看板风格）。
3. 主程 ADR 必须先 commit + push 才能进入实现。
4. 数值方案与 ADR 可并行启动，但必须在实现前 commit。
5. 实现建议拆分 T1 工程基础（getCellStat + cell.tier/mod + TYPES 扩展）/ T2 研发点积累 / T3 升级面板 UI / T4 改造与全局加成 / T5 死代码清理 / T6 完整调参与文案；每个子任务后做最小验证。
6. 实现完成后进入专项审核（与执行不同基底 agent），通过后 PM 完整回归。
7. 通过后制作人最终决定接受 + 是否触发发布。
