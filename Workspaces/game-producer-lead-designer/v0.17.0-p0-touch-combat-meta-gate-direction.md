# v0.17.0-p0-touch-combat-meta-gate 制作人方向

## 结论

v0.17.0 的版本名为 `v0.17.0-p0-touch-combat-meta-gate`。

核心目标：补齐发布闸门最后几类真实体验证据，并只修复证据路径中明确阻塞或高价值低风险 polish；不新增玩法。

制作人判断：v0.15.0 已取得真实浏览器 JSON，`ok=true`、runtime errors 0、storage signature 稳定、frameTiming rAF 正常，说明浏览器运行态基础证据已经成立。但 PM 证据包明确不进入发布准备，因为完整真人 P0 尚未闭环，真实触控未覆盖，武器/护盾/维修实际反馈、战斗后复盘、Meta 点击回流仍缺少完整证据，小屏叠层还有 polish 风险。因此 v0.17.0 应尊重最新用户指令，直接把目标收束为最后一轮 P0、触控、交火、复盘和 Meta 回流证据闸门；即使 v0.16.0 未形成完整交付，也不回头扩张中间版本范围。

本版本成功的标志不是新增内容，而是团队能用真实玩家路径和 PM 证据包决定：继续补证、小修 polish，还是进入发布准备交接。

## 玩家与团队收益

- 确认真实玩家是否能从首局自然走过建造、采矿、资源不足/断电、交火、复盘、星系跳转和 Meta 回流。
- 确认触控/小屏是否可继续，尤其是叠层、滚动、按钮 hit、结算入口和复盘入口是否阻塞核心路径。
- 让武器开火、护盾/受击或防御缺口、维修状态和战后 `combatReview` 不再停留在 hook 或推断层面，而是形成真实交互证据。
- 让 Meta 点击回流和 storage 影响被实际记录，避免只证明 UI 存在却没有证明玩家能回到新局。
- 为是否进入发布准备提供最终依据：证据充分则交接发布准备；证据不足则明确补证或小修，不凭感觉推进。

## 核心目标

v0.17.0 要回答一个问题：当前 SSDSpaceGame 纵向切片是否已经具备进入发布准备的最后一组真实体验证据。

必须覆盖以下判断：

- 真人 P0 主路径是否闭环：尽量不用 test hook 替代真实点击；若必须使用辅助，必须标注具体辅助、使用原因和它不能证明的体验部分。
- 自然交火与复盘是否成立：至少形成武器开火、护盾/受击或防御缺口、维修状态、战后 `combatReview` 的真实证据。
- Meta 点击回流是否成立：真实点击结算、Meta、开始新局或等价 UI 回流，并记录 storage 影响。
- 触控/小屏是否可继续：真实触控优先；若无真机，必须诚实记录不可用原因，以及 DevTools/鼠标替代的边界。
- polish 是否克制：只处理 S0/S1/S2 明确阻塞，或制作人判定高价值低风险的小屏、按钮 hit、复盘入口、触控滚动问题。
- 发布闸门决定是否有证据支持：PM 证据包必须能支撑通过、继续补证、小修 polish 或进入发布准备交接。

## 必做范围建议

### 1. 完整真人 P0 路径

目标是让真实玩家或 PM/制作人在真实浏览器中尽量用自然点击完成首局主要路径，而不是用 test hook 拼出通过状态。

必须覆盖的路径节点：

- 空存档或隔离存档进入首局，确认入口、初始 HUD、建造入口和基础状态可读。
- 自然完成最小建造与采矿链，看到资源变化。
- 触发或观察资源不足、断电或电力压力，确认反馈可理解。
- 遇到敌人或威胁接近，确认玩家能理解风险来源。
- 进入自然交火，观察武器、防御、维修和战斗后反馈。
- 从局内进入星系跳转或等价推进入口。
- 从结算/Meta 路径回到新局或等价可继续状态。

记录要求：

- 优先真实点击、真实浏览器、真实 UI 路径。
- 若必须用辅助、debug、hook、控制台或人为预置状态，逐项标注原因和影响。
- 每个节点标记为通过、失败、未覆盖、辅助覆盖或阻塞。

验收标准：

- 至少主要路径闭环，且能说明哪些步骤是自然点击完成。
- 不得把 hook 结果直接等同于真实玩家体验通过。
- 若主要路径无法闭环，制作人结论只能是继续补证或小修 polish。

### 2. 自然交火与复盘

目标是补齐 v0.15.0 证据包中最影响发布判断的战斗真实反馈缺口。

必须形成证据：

- 武器开火：玩家能看到或理解武器正在攻击，或明确知道为什么未开火。
- 护盾/受击或防御缺口：玩家能看到空间站受到威胁、防御生效或防御不足的反馈。
- 维修状态：玩家能理解维修是否启动、是否缺资源、是否仍有损伤。
- 战后 `combatReview`：一次交火后能看到复盘入口、内容或等价总结。

记录要求：

- 证据优先来自自然交互、截图、人工步骤、PM 记录和只读状态摘要的组合。
- 允许使用只读 hook 辅助确认状态，但不能用它替代玩家是否看懂反馈的结论。
- 若敌人、战斗节奏或等待时间导致自然交火难以触发，只记录为证据路径问题，不改 AI、伤害、RNG 或核心数值。

验收标准：

- 至少有一次真实浏览器交火记录能串起开火、防御/受击、维修、复盘。
- 若缺少任一关键反馈，必须判断是阻塞发布、非阻塞 polish，还是继续补证。

### 3. Meta 点击回流

目标是确认玩家能从战斗后或局内推进路径真实点击到 Meta/结算，并回到新局或等价继续状态。

必须记录：

- 结算、Meta 或局外入口是否可见、可理解、可点击。
- 点击后的状态变化、文案、按钮和下一步入口。
- 开始新局或回到局内的真实 UI 路径。
- 点击前后 storage signature、关键 storage 字段或等价摘要，说明是否写入、是否符合预期。

验收标准：

- 至少形成一次真实点击回流记录。
- storage 影响必须被说明：未写、按预期写入、无法判断或异常写入。
- 若只能用 hook 或控制台模拟回流，不能视为 Meta UI 回流通过。

### 4. 触控/小屏

目标是确认小屏和触控不会阻塞 P0 主路径，尤其是叠层、按钮 hit、滚动和关闭入口。

优先级：

- 优先真实触控设备或真实触控浏览器环境。
- 若无真机，记录不可用原因，并使用 DevTools 移动视口、鼠标或触控模拟作为替代证据。
- 替代证据必须说明边界：它能证明布局、可见性和鼠标点击，不能完全证明真实手指触控、软键盘或系统浏览器行为。

重点检查：

- 小屏 HUD 与叠层是否遮挡建造、战斗、复盘、星系图或 Meta 入口。
- 结算按钮、复盘入口、Meta 入口、开始新局按钮是否有足够 hit 区域。
- 面板滚动、关闭、返回和继续按钮是否可操作。
- 触控滚动是否被遮罩、全屏容器或事件处理阻塞。

验收标准：

- 触控或替代边界说明足够诚实，不能把 DevTools/鼠标结果包装成真机通过。
- 若发现阻塞级叠层、按钮 hit 或滚动问题，允许纳入本版本必要 polish，并按原路径复测。

### 5. 必要 polish 修复

目标是允许证据路径中的小修，但禁止把 v0.17.0 扩张为新功能或系统改造。

允许修复：

- S0：游戏无法启动、首局无法进入、核心路径完全不可用。
- S1：P0 主路径无法完成，关键 UI 不可点击，复盘/Meta/新局回流无法进入。
- S2：证据不可信，小屏核心信息被遮挡，按钮 hit 明显过小，触控滚动阻塞，结算或复盘入口不可点。
- 制作人判定高价值低风险的 polish：小屏叠层、结算按钮 hit、复盘入口不可点、触控滚动阻塞等。

修复约束：

- 每个修复必须来自证据路径中的明确现象。
- 修复后必须按原浏览器、原视口、原步骤或原触控替代条件复测。
- 若修复需要改核心规则、数值、AI、RNG、存档或 meta schema，立即升级风险，不纳入普通 polish。

验收标准：

- 无 S0/S1 阻塞遗留。
- S2 或 polish 项必须有纳入理由、复测结果和残留风险说明。

### 6. PM 证据包与发布闸门决定

目标是把 v0.17.0 的最终证据集中到 PM 工作区，支撑制作人做发布闸门判断。本方向文档不创建 PM 看板，也不进入 `Releases/`。

PM 证据包建议包含：

- 完整真人 P0 路径记录，区分自然点击、辅助覆盖和未覆盖。
- 自然交火与复盘证据，包含武器、护盾/受击或防御缺口、维修、`combatReview`。
- Meta 点击回流记录，包含 storage 影响。
- 触控/小屏证据，说明真机、DevTools、鼠标替代或未覆盖原因。
- 必要 polish 的问题、修复、复测和未覆盖风险。
- 发布闸门建议：通过、继续补证、小修 polish、进入发布准备交接。

验收标准：

- PM 能基于证据包给出明确建议，而不是只列出零散截图或自动检查结果。
- 制作人能基于证据包决定是否进入发布准备。
- 证据类型必须区分真实玩家、真实浏览器、真实触控、DevTools/鼠标替代、hook 辅助和未覆盖。

## 不做范围

v0.17.0 禁止扩张为新玩法、核心规则改动或发布版本。以下内容不进入本版本：

- 不新增玩法。
- 不新增敌人。
- 不新增建筑。
- 不新增 Meta 节点。
- 不修改核心数值。
- 不修改伤害、敌人 AI、RNG、维修、护盾或战斗结算规则。
- 不修改 `SAVE_KEY`。
- 不修改 meta schema。
- 不安装依赖、浏览器驱动或测试框架。
- 不修改系统设置、浏览器全局设置、权限、服务或本机环境。
- 不发布、不打 tag、不归档 `Releases/`，除非最终另行批准。
- 不为取证伪造战斗结果、伪造 Meta 进度、伪造 storage 状态或改变玩家状态。
- 不改动与 v0.17.0 证据闸门无关的代码、文档、注释、资源或工作区文件。

## 风险级别

本版本风险级别为中风险。

风险来源：

- 真实 P0、交火、复盘、Meta 回流和触控证据需要走实际交互，可能暴露隐藏阻塞。
- 小屏 UI polish 可能触碰 HUD、叠层、按钮 hit、滚动和面板布局。
- 证据类型容易被误读，需要 PM 明确区分真实点击、hook 辅助、DevTools/鼠标替代和未覆盖。

风险升级条件：

- 需要修改核心规则、核心数值、伤害、敌人 AI、RNG、维修、护盾或战斗结算。
- 需要修改存档结构、`SAVE_KEY`、meta schema 或局外成长数据。
- 需要安装依赖、浏览器驱动、测试框架，或修改系统/浏览器全局环境。
- 需要发布、打 tag、归档 `Releases/` 或调整发布流程。

只要触发上述任一条件，必须升级为高风险或等待用户确认；涉及本机环境危险操作时，必须先说明风险并等待用户确认。

## 制作人接受标准

v0.17.0 的接受标准是“最后证据闸门”，不是功能数量。

最低接受条件：

- 完整真人 P0 至少主要路径闭环，并明确哪些步骤自然点击完成、哪些步骤使用辅助。
- 自然交火/复盘有证据，至少覆盖武器开火、护盾/受击或防御缺口、维修状态、战后 `combatReview`。
- Meta UI 回流有证据，包含真实点击结算/Meta/开始新局或等价回流，以及 storage 影响说明。
- 触控或替代边界说明足够诚实；若无真机，必须说明不可用原因和 DevTools/鼠标替代边界。
- 无 S0/S1 阻塞；S2 和 polish 项有纳入理由、复测结果或明确残留风险。
- 发布建议有证据支持，能在通过、继续补证、小修 polish、进入发布准备交接之间做出明确判断。

结论规则：

- 如果真人 P0 主路径未闭环，结论只能是继续补证或小修 polish。
- 如果自然交火、复盘或 Meta UI 回流缺少真实证据，不能进入发布准备交接。
- 如果触控无真机且替代证据发现阻塞级问题，必须先修复并复测。
- 如果只剩非阻塞小屏舒适度或轻 polish，由制作人判断是本版本收口还是转入后续版本。
- 如果主要路径、交火复盘、Meta 回流、触控/小屏边界和 PM 证据包均可复查，且无 S0/S1 阻塞，才允许进入发布准备交接。

## 角色流程

v0.17.0 按以下流程推进：

1. 制作人方向：确认版本聚焦 `v0.17.0-p0-touch-combat-meta-gate`，不新增玩法、不发布。
2. PM 看板：拆分 P0 路径、交火复盘、Meta 回流、触控/小屏、必要 polish、证据包和闸门决定。
3. PM 证据方案：定义记录模板，明确真实玩家、真实浏览器、真实触控、DevTools/鼠标替代、hook 辅助和未覆盖的口径。
4. UI 小屏 polish 方案：只针对证据路径中的叠层、按钮 hit、复盘入口、结算入口和触控滚动风险提出最小修复方案。
5. 主程边界复核：确认不会触碰核心规则、存档、数值、AI、RNG、`SAVE_KEY`、meta schema 或系统环境。
6. 执行补证/修复：先补证，只有证据路径发现 S0/S1/S2 或高价值低风险 polish 时才修复。
7. 专项审核：UI 审核小屏和触控 polish，主程审核规则/存档边界，PM 审核证据完整性。
8. PM 证据包：汇总 P0、交火复盘、Meta 回流、触控/小屏、修复复测和残留风险。
9. 制作人决定：通过、继续补证、小修 polish，或进入发布准备交接。

## 版本边界声明

v0.17.0-p0-touch-combat-meta-gate 的交付物是“真实体验证据、必要小修和发布闸门判断”。如果证据不足，诚实给出继续补证就是正确结论；如果证据充分且无 S0/S1 阻塞，才进入发布准备交接。
