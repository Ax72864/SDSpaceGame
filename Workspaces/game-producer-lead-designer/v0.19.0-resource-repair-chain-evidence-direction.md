## 核心目标

本阶段只补齐 v0.18.0 残留的三个证据缺口：

1. `repair` 建成条件：确认维修站无法建成时，是资源规划不足、提示不可理解，还是路径存在阻塞。
2. 采矿覆盖 / 收益证据：证明矿站覆盖资源外环后，能产生可归因的资源收益。
3. 资源不足提示可理解性：玩家应能看懂为什么维修站建不起、缺什么、差多少，以及下一步如何补资源。

## 玩家收益

玩家收益不是新增内容，而是让首局建设链从“能点中建筑”推进到“能理解资源缺口并规划解决”。玩家需要知道维修站为什么建不起、如何获得或规划缺口资源，并能看到矿站覆盖资源外环后产生可归因收益。

这是后续解锁自然交火、战后复盘和 Meta 回流补证的前置条件。若资源、维修、采矿链条没有闭环，战斗和局外成长会建立在玩家尚未理解的基础建设问题上。

## 必做范围

- PM 建立 `v0.19.0-resource-repair-chain-evidence` 看板，写清任务、负责人、允许修改范围、验收标准、证据点和发布边界。
- 制定资源 / 维修 / 采矿证据方案，明确 `repair` 建成或失败的复现路径、资源缺口记录方式、采矿覆盖资源外环的验证方式。
- 补可见浏览器证据，保留 runtime errors、console errors、unhandled rejections、关键资源变化、目标格 / 覆盖范围 / 收益归因的记录。
- 做必要最小提示或诊断修复，只服务于让资源不足原因和采矿收益可被追踪，不扩大到新系统。
- 回到 v0.18.0 的原路径复测，验证 `repair` 与采矿证据是否关闭原失败点。
- 安排专项审核，重点检查是否越界修改规则、数值、存档、战斗、AI、RNG 或 Meta。
- PM 汇总证据与风险，给出是否关闭本阶段闸门的建议。
- 制作人基于 PM 汇总给出接受、继续追或阻塞决定。

## 不做范围

- 不新增建筑、敌人、任务、星系、Meta 节点或局外成长内容。
- 不修改资源成本、采矿公式、维修规则、建造合法性、设施数值或电力规则。
- 不修改战斗、AI、RNG、`SAVE_KEY`、存档结构或 meta schema。
- 不发布、不打 tag、不创建或归档 `Releases/`。
- 不把 CDP、hook 或状态构造证据包装成人类体验通过。

## 风险判断

本阶段默认风险为中低风险：目标是补证、提示理解和诊断能力，不改变游戏规则。

若执行中需要修改资源成本、采矿公式、维修规则、建造合法性、存档结构、`SAVE_KEY` 或 meta schema，必须立刻升级为高风险流程，由制作人确认方向、PM 重新拆分，专项审核后再决定是否继续。

## 接受标准

- `repair` 路径必须满足其一：证明维修站可按正常路径建成；或明确证明资源不足提示可理解，且 metal / plasma 等缺口可追踪到来源和下一步补足路径。
- `mining` 路径必须满足其一：证明矿站覆盖资源外环并产生可归因收益；或明确记录覆盖 / 距离 / 资源类型 / 运行条件中的阻塞点。
- 可见浏览器证据中 runtime errors 为 0，console errors 和 unhandled rejections 不得出现未解释的新错误。
- 原 v0.18.0 失败路径必须复测，不能只用新构造路径证明成功。
- CDP、hook、诊断 helper 或状态构造只能作为浏览器级 / 辅助证据，不得写成人类鼠标或触控体验通过。
- 专项审核需确认未越界修改资源成本、采矿公式、维修规则、建造合法性、战斗、AI、RNG、`SAVE_KEY`、存档结构或 meta schema。

## 下一阶段条件

只有在资源、维修和采矿闭环后，才允许进入自然交火、战后复盘和 Meta 回流补证。

若本阶段只能证明资源不足提示可理解，但仍无法完成 `repair` 建成或采矿收益归因，下一阶段仍应继续追资源 / 维修 / 采矿，不解锁交火和 Meta。

## 制作人结论

批准进入 `v0.19.0-resource-repair-chain-evidence` 内部阶段。目标不是做更多内容，而是把玩家首局建设中最容易卡住的资源、维修和采矿因果链补清楚。

本阶段完成后，制作人只根据证据闭环程度决定是否解锁自然交火、战后复盘和 Meta 回流；不因单个浏览器自动化成功就提前放开发布或后续系统。
