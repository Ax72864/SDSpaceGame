# v0.10.0 局外持久天赋树 制作人方向

版本名：`v0.10.0-meta-progression`
基线：HEAD `ec82018`（v0.9.0 局间星系图系统代码层已实现，#9 浏览器 / 主观回归仍进行中）
风险级别：**项目高风险**（涉及 `SAVE_KEY`、局外存档迁移、持久化天赋树、开局状态、Meta UI 面板与长期平衡）

## 本轮目标

把当前局外点数从“几个加数值按钮”升级为**可规划的持久天赋树 + 开局策略选择**。玩家完成一局后，不论失败还是通关，都应该能理解自己获得了什么成长、下一局可以追什么节点、下一局要用什么开局路线。

v0.10.0 不追求堆新战斗内容，而是把 v0.1-v0.9 已经形成的局内建造、任务、研发、战斗和星系路径，接到一个清晰的跨局成长闭环上：

1. 本局游玩获得局外点数。
2. 结算面板告诉玩家“你赚了多少、能买什么、下一局建议做什么”。
3. 玩家在局外天赋树中规划资源型 / 防御型 / 武器型 / 探索型路线。
4. 已解锁天赋影响下一局的开局协议、初始资源、初始设施或蓝图折扣。
5. 玩家带着明确意图进入下一局，验证自己选择的路线。

## 与原始设计的对应

本版本落实 `Docs/game desc.txt` §8 rogue like 要素中的“局外持久化天赋树”：

- **升级**：提高资源获取、结构耐久、武器输出、科研收益等数值。
- **改造**：用轻量机制变化影响开局或设施策略，例如特定开局协议、蓝图折扣、初始框架倾向。
- **解锁**：解锁 `weaponFrame`、`efficientCore` 等现有简化 meta 已经暗示的能力。

明确不做 §9 联机、§10 异步对战，也不在 v0.10.0 中扩展完整新建筑大类。v0.10.0 的价值不是“更多系统”，而是让局外成长从临时按钮变成可读、可规划、可回归的长期框架。

## 体验目标

按重要程度排序：

1. **失败也有去处**：玩家核心被摧毁或未能通关时，结算仍显示获得点数、可购买节点和下一局建议，减少“白玩一局”的挫败。
2. **路线选择明确**：玩家能在资源型 / 防御型 / 武器型 / 探索型之间做方向选择，不只是无脑点最高数值。
3. **开局策略可感知**：至少 3 个开局协议会改变下一局前 3 分钟体验，例如更快采矿、更稳防御、更早武器成型。
4. **局内 / 局外边界清楚**：v0.5.0 研发系统仍是“本局内临时成长”，v0.10.0 Meta 天赋是“跨局持久成长”，UI 文案和代码接入都不能混淆。
5. **不污染 v0.9 星系图**：v0.9 galaxyMap 是局间路径选择；v0.10 Meta 是局外长期成长。两者可以互相提示，但不互相改写核心状态机。

## 当前问题判断

当前代码已有简化 meta：

- `SAVE_KEY = "star-ring-station-meta-v1"`
- `state.meta = { points, mining, hull, weapons, unlocks: { weaponFrame, efficientCore } }`
- 右侧“更多状态”中已有局外天赋按钮。

制作人判断：这套实现证明局外成长方向成立，但体验还停留在“几个加数值按钮”。它缺少前置关系、节点层级、路线规划、开局协议、结算引导和存档迁移策略，因此不像原始设计中的“局外持久化天赋树”。

v0.10.0 的核心不是重做全部存档，而是在保留玩家已有点数和旧字段价值的前提下，把这套简化 meta 升级为正式结构。

## 核心机制

### 1. `META_TALENT_TREE` 注册表

新增集中注册表，所有局外天赋节点必须从这里定义，不允许散落在 UI 或结算逻辑里硬编码。

节点字段建议：

| 字段 | 说明 | 要求 |
| --- | --- | --- |
| `id` | 稳定节点 ID | 存档中使用，发布后避免改名 |
| `name` | UI 显示名 | 中文短名，可一眼看懂 |
| `desc` | 节点说明 | 说明每级收益或解锁效果 |
| `category` | 路线分类 | `resource` / `defense` / `weapon` / `exploration` |
| `tier` | 层级 | 1-4，控制前置节奏 |
| `cost` | 每级成本 | 支持数字或数组，便于多级递增 |
| `maxRank` | 最大等级 | 升级类可多级，解锁类通常 1 级 |
| `prereq` | 前置条件 | 节点 ID + rank 条件 |
| `effects` | 效果声明 | 只描述数据，不直接写逻辑 |
| `uiHint` | UI 提示 | 图标、颜色、推荐标签、路线说明 |

示例方向：

| 路线 | 早期节点 | 中期节点 | 解锁 / 改造节点 |
| --- | --- | --- | --- |
| 资源型 | 采矿效率提升 | 金属加工折扣 | 采矿起步协议 |
| 防御型 | 框架 / 核心耐久 | 初始维修余量 | 防御起步协议 |
| 武器型 | 炮塔输出 | 弹药 / 能量效率 | `weaponFrame` |
| 探索型 | 科研收益 | 星系选择信息提示 | `efficientCore` 或蓝图折扣 |

### 2. 存档 schema 兼容与迁移

制作人拍板：**绝不能清空玩家点数，也不能让旧字段价值消失**。

主程 ADR 必须在实现前明确以下选择：

- 保留现有 `SAVE_KEY = "star-ring-station-meta-v1"` 并做原地兼容，或建立清晰迁移策略。
- 旧字段 `mining / hull / weapons / unlocks.weaponFrame / unlocks.efficientCore` 必须映射到新 `talents` ranks。
- `points` 必须原样保留；如旧字段映射后出现成本差额，优先保护玩家已获得能力，不做扣点争议。
- 新 schema 建议引入：

```javascript
state.meta = {
  points: 0,
  talents: {
    miningYield: 0,
    hullIntegrity: 0,
    weaponCalibration: 0,
    weaponFrame: 0,
    efficientCore: 0
  },
  selectedStartProtocol: "balanced",
  migrationVersion: 2
};
```

兼容底线：

- 旧存档缺 `talents` 时自动迁移。
- 旧存档字段仍存在时不报错。
- 迁移只增加 / 映射字段，不删除玩家点数。
- 测试钩子必须能注入旧存档并验证迁移后 ranks。

### 3. Meta UI 重做

把右侧“更多状态”的简单局外按钮升级为树 / 卡片面板。UI 不要求复杂大地图，但必须让玩家看懂路线和前置关系。

面板必须显示：

- 当前可用局外点数。
- 每个节点的名称、路线、tier、当前等级 / 最大等级。
- 节点效果，包括当前效果和下一级效果。
- 前置条件，未满足时说明差什么。
- 可购买 / 点数不足 / 前置不足 / 已满级状态。
- 已选开局协议及其来源。
- 购买后的结算提示或即时反馈。

重置策略拍板：

- v0.10.0 可以先不做自由重置，避免点数经济争议。
- 如果不做重置，UI 必须明确“当前版本暂不支持重置，后续平衡期可能提供一次性重置”。
- 如果主程认为实现重置成本低，必须加测试钩子和确认弹窗，且默认只重置 ranks，不丢失总点数。

### 4. 开局协议 / 路线选择

至少做 3 个可选“开局协议”，由已解锁天赋决定。开局协议的收益要小而可感知，避免第一版直接数值膨胀。

建议首批协议：

| 协议 | 解锁来源 | 开局影响 | 玩家意图 |
| --- | --- | --- | --- |
| 采矿起步 | 资源型 tier 2 | 少量初始金属 / 采矿设施折扣 / 早期采集提示增强（三选一或组合小幅） | 快速滚资源 |
| 防御起步 | 防御型 tier 2 | 少量核心 / 框架耐久加成或初始维修余量 | 更稳撑过前期 |
| 武器起步 | 武器型 tier 2 | 初始炮塔折扣 / 初始武器框架 / 弹药效率小幅提升 | 更早主动战斗 |

可选第 4 条探索型协议留作加分项：影响科研收益、星系候选信息提示或初始扫描奖励。若工期紧，探索型可以先表现为 talent 路线，不强制首版协议。

协议约束：

- 同一局只能选择一个开局协议。
- 默认协议 `balanced` 不需要解锁，保证新玩家不被卡住。
- 协议必须在开局前 / 新局开始时选择，不在局中临时切换。
- 协议影响范围限制在初始资源、初始设施、初始蓝图折扣、早期提示或少量倍率，不改写核心关卡结构。

### 5. 效果集中接入

所有 meta 效果必须通过 helper 查询，禁止在采矿、耐久、武器、科研、UI、结算中散落硬编码。

建议 helper：

- `getMetaTalentRank(id)`
- `hasMetaTalent(id)`
- `getMetaEffect(effectKey, context?)`
- `getSelectedStartProtocol()`
- `applyMetaStartProtocol(state)`
- `getMetaPurchaseState(nodeId)`
- `purchaseMetaTalent(nodeId)`
- `migrateMetaSave(rawMeta)`

首批接入效果：

- 采集效率。
- 结构 / 核心 / 框架耐久。
- 武器输出或发射效率。
- 科研收益。
- 初始资源。
- 初始框架或初始设施折扣。
- `weaponFrame` 解锁。
- `efficientCore` 解锁。

关键原则：局内系统只问“当前 meta 给我的效果是多少”，不要知道天赋树怎么画、点数怎么花、旧字段怎么迁移。

### 6. 结算反馈

结算面板必须从“本局结束”升级为“下一局目标入口”。

必须显示：

- 本局获得局外点数。
- 当前总点数。
- 已满足购买条件的推荐节点。
- 如果点数不足，显示“还差多少点可以购买某个路线节点”。
- 下一局建议，例如“你已接近采矿起步协议，下一局建议优先完成资源任务”。
- 失败与通关都走同一套反馈，只是文案语气不同。

验收标准：玩家结束一局后，不打开外部文档，也能知道“我现在能买什么 / 还差什么 / 下一局试什么”。

### 7. 测试钩子

v0.10.0 必须预留测试入口，避免持久化系统只能靠手点验证。

必做钩子：

- 重置 meta。
- 注入旧存档。
- 执行迁移并读取迁移结果。
- 设置 / 增加局外点数。
- 购买指定节点。
- 查询节点 rank 和效果值。
- 选择开局协议。
- 模拟结算奖励。
- 导出当前 meta 简要快照。

测试钩子只服务开发和回归，不作为玩家 UI 暴露。

## 必做范围

### T1 注册表与数据结构

- `META_TALENT_TREE` 注册表。
- 节点字段：`id / name / desc / category / tier / cost / maxRank / prereq / effects / uiHint`。
- 新 meta schema：`points / talents / selectedStartProtocol / migrationVersion`。
- `ensureMetaState` 或同等入口，保证新旧存档都能得到完整默认值。

- 旧字段 `mining / hull / weapons / unlocks` 映射到新 `talents` ranks。

- 显示已选开局协议。
- 购买交互有确认 / 反馈。

- 至少 3 个协议：采矿起步、防御起步、武器起步。

### T6 结算反馈

- 显示获得点数。
- 显示可购买节点提示。
- 显示下一局建议。

### T7 测试与回归钩子

- 重置 meta。
- 注入旧存档。
- 购买节点。
- 查询效果。
- 模拟结算。

## 兼容要求

- 旧存档必须兼容。
- `42.md` 不动。
- v0.1-v0.9 回归点不回潮。
- v0.5.0 研发系统不被局外天赋污染：局内研发仍在本局内生效，结算后不保存。
- v0.9.0 galaxyMap 不被局外天赋污染：星系候选、路径状态、跃迁状态机仍按 v0.9 规则运行。
- 旧的 `mining / hull / weapons / unlocks` 字段即使仍在存档中，也不能导致双倍收益。
- 新玩家空存档必须能正常开局，默认协议可用。
- 玩家已有 `points` 必须保留。

## 主程 ADR 必答问题

1. `SAVE_KEY` 是否保持不变？如果变更，旧 key 如何迁移、何时写回？
2. 旧字段映射到哪些新 talent ID？多级旧数值如何对应新 rank？
3. `effects` 采用何种声明格式？数值叠加是加法、乘法还是按 effectKey 决定？
4. helper 的命名、职责和调用边界是什么？
5. 开局协议何时应用？是在 reset run、新局创建、还是首帧初始化？
6. UI 购买是即时写存档，还是先预览后确认？
7. 是否提供重置？如果不做，如何在 UI 中解释？
8. 如何避免旧字段和新 `talents` 同时存在导致重复生效？
9. 测试钩子挂在 `__gameTest` 还是单独 meta debug 对象？
10. 迁移失败或存档损坏时，如何保护玩家点数并给出降级路径？

## 数值方向建议

第一版追求“可感知但不失控”：

- 单个数值型节点每级收益建议 5%-12%，最大 3-5 级。
- tier 1 节点成本低，让玩家 1-2 局内能完成第一次购买。
- tier 2 解锁开局协议，给玩家明确短期目标。
- 解锁类节点通常 1 级，成本高于同 tier 数值节点。
- 初始资源 / 折扣类收益只影响开局节奏，不应让玩家跳过核心采集和建造循环。
- 武器输出收益必须低于“新增武器设施”的边际体验，避免玩家只点武器树不建造。
- 防御收益保护容错，不应让撞击和敌人完全失去威胁。
- 探索 / 科研收益要和 v0.5 研发系统保持边界，优先影响科研获取量或信息提示，不直接赠送大量局内研究完成项。

## 风险预警

1. **存档迁移风险**：`SAVE_KEY` 和旧字段已经存在，任何误判都可能清空点数或丢失已购能力。
2. **Meta 数值膨胀**：跨局成长一旦过强，会压扁 v0.1-v0.9 的首局难度、研发节奏和星系风险。
3. **UI 拥挤**：天赋树、协议、结算提示都放进现有右侧 UI 容易过载，需要卡片化和折叠策略。
4. **旧按钮兼容**：旧“局外天赋按钮”要么被新面板替换，要么明确转接，不能同时出现两套购买入口。
5. **点数重置争议**：如果没有重置，玩家可能担心点错；如果有重置，经济平衡和测试成本上升。
6. **局内 / 局外 research 混淆**：v0.5 研发是本局系统，v0.10 Meta 是跨局系统，文案和代码 helper 必须分清。
7. **开局协议污染关卡结构**：协议只能轻量影响开局，不应绕过 v0.9 galaxyMap 或新建关卡分支。

## 项目高风险流程要求

本版本涉及核心持久化、存档迁移、跨局成长和开局状态，必须走项目高风险流程：

1. 制作人方向（本文档）commit + push。
2. PM 建立 v0.10.0 看板，拆分任务、验收标准和回归清单。
3. 主程 ADR 先行，明确 schema、迁移、helper、UI 状态机和测试钩子。
4. 数值策划给出首批节点表、成本曲线、路线矩阵和协议收益上限。
5. UI 设计给出 Meta 面板布局、节点状态、协议选择和结算提示。
6. 分批实现：注册表 / 迁移 / UI / 协议 / 效果接入 / 结算 / 测试钩子。
7. 专项审核检查需求满足、存档安全、helper 边界、无越界修改。
8. PM 回归覆盖旧存档、空存档、失败结算、通关结算、购买节点、开局协议、v0.5 研发、v0.9 galaxyMap。
9. 制作人最终接受，并判断是否发布 v0.10.0。

## 产品接受标准

v0.10.0 完成时，制作人按以下标准接受：

- 空存档玩家完成首局后，能看到局外点数、推荐节点和下一局建议。
- 旧存档玩家进入游戏后，点数不丢失，旧 `mining / hull / weapons / unlocks` 能映射到新天赋等级。
- 玩家至少能选择资源型 / 防御型 / 武器型三条路线中的一种，并理解其短期目标。
- 至少 3 个开局协议可被解锁、选择，并在下一局前 3 分钟产生可感知差异。
- 采矿、耐久、武器、科研、初始资源、解锁项等效果均通过集中 helper 接入。
- 失败和通关结算都能把玩家导向“下一局我要做什么”。
- v0.5 研发系统、v0.9 galaxyMap 和 v0.1-v0.9 主要回归点不被破坏。

## 制作人结论

v0.10.0 的方向正式定为 `v0.10.0-meta-progression`。

本版本的核心玩家收益是：每一局之后都有可理解、可规划、可带入下一局的成长目标。只要这条闭环成立，即使首版天赋树节点数量克制，也比继续增加敌人、建筑或事件更能提升长期可玩性。

本版本不追求发布承诺。是否归档发布，取决于旧存档迁移是否安全、Meta UI 是否可读、开局协议是否真的改变首局策略，以及 PM 回归是否证明 v0.5 / v0.9 等关键系统没有回潮。
