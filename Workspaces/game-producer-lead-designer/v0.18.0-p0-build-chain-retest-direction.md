## 核心目标

用正确真实操作复测 P0 建造链：

1. 先选择 `frame` 并扩出新的空框架。
2. 再点击已有空 `frame`，将该格替换为 `power` / `mining` / `turret` / `repair`。
3. 每一步记录资源变化、目标 `gridKey`、目标格 `facility`、toast / statusAlerts 和 runtime errors。

本阶段要证明的问题不是“辅助脚本能不能构造成功状态”，而是玩家按当前规则能否理解并完成首局最小建设动作。

## 玩家收益

玩家收益是确认首局核心动作是否成立：玩家能从核心出发扩出框架，在框架上建设施，并形成采矿、武装和维修能力。

这条链路是后续交火复盘和 Meta 回流的前置条件。若玩家不能自然完成“扩框架 -> 在框架上建设施 -> 形成能力”，后续的敌人压力、战后复盘、局外成长都会变成建立在不可靠入口上的补丁。

## 必做范围

- PM 建立本阶段看板，明确负责人、允许修改范围、验收标准、风险和状态。
- 执行正确操作复测证据：先扩 `frame`，再点击已有空 `frame` 替换为 `power` / `mining` / `turret` / `repair`。
- 证据必须逐步记录资源、`gridKey`、`facility`、toast / statusAlerts、runtime errors。
- 若正确真实操作失败，只允许做最小新手提示、点击反馈或取证断言修复。
- 修复后必须按原路径复测，证明原失败路径关闭或仍失败。
- 需要专项审核确认需求满足、未越界修改、未把辅助输入包装成人类体验通过。
- PM 汇总证据、阻塞、残留风险和是否满足接受标准。
- 制作人基于 PM 汇总作出阶段决定。

## 不做范围

- 不新增玩法、建筑、敌人或 Meta 节点。
- 不修改建造合法性规则。
- 不修改资源成本或设施数值。
- 不修改战斗、AI、RNG、`SAVE_KEY` 或 meta schema。
- 不修改存档结构或局外成长结构。
- 不启动发布准备，不打 tag，不归档 `Releases/`。

## 风险边界

本阶段为中低风险，前提是复测、取证和最小提示反馈不触碰核心规则。

如果任务中需要修改建造合法性、存档、核心数值、战斗判定、AI、RNG、`SAVE_KEY` 或 meta schema，必须立即升级为高风险流程：制作人重新确认方向，PM 重新拆分，专项成员审核，PM 汇总后再由制作人决定。

## 接受标准

- 至少证明正确操作下能建成 `power` / `mining` / `turret` / `repair` 中的关键链路，并能解释玩家如何完成该链路。
- 或者明确真实失败点，给出最小修复、原路径复测结果和仍不通过的原因。
- runtime errors 必须为 0。
- 证据中必须包含资源变化、`gridKey`、目标格 `facility`、toast / statusAlerts。
- 不把 hook、CDP 辅助构造、直接改状态或断言注入包装成人类体验通过。
- 若使用 CDP 或脚本辅助取证，必须明确其边界，并记录它不能替代真实鼠标、真实触控或人类理解。

## 下一阶段条件

只有建造链闭环后，才能继续自然交火、武器 / 护盾 / 维修反馈、战后复盘和 Meta 回流补证。

若建造链未闭环，下一阶段仍应继续围绕首局建设理解和点击反馈收束，而不是提前扩展战斗、Meta 或发布流程。

## 制作人决定

批准进入 `v0.18.0-p0-build-chain-retest` 内部阶段。目标只有一个：用正确真实操作关闭 P0 建造链前置缺口。

本阶段不追新内容，不扩系统，不做发布。完成标准是首局玩家建设链路被证明成立，或失败被诚实定位并以最小修复复测。
