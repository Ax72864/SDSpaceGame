# v0.14.0-release-candidate-playtest 制作人方向

## 结论

v0.14.0 的版本名为 `v0.14.0-release-candidate-playtest`。

核心目标：把 SSDSpaceGame 从“功能开发完成”推进到“发布候选可验收”。本版本不继续叠新玩法，而是补齐浏览器级证据、只读验收钩子和人工体验路径，让团队能够保守判断当前纵向切片是否可以进入发布准备。

v0.12.0 已完成空间站设计反馈，v0.13.0 已完成战斗威胁可读性；两者都已具备开发完成基础，但仍缺少真实浏览器人工交火、小屏触控、真实帧耗时、完整运行态只读签名等证据，因此尚不应宣称发布就绪。v0.14.0 的价值在于把这些缺口收束为可执行、可记录、可复查的发布候选验收闭环。

制作人方向：本版本只补发布候选验收能力和必要可读性修正。若玩家能从空存档跑完建造、采矿、战斗、维修、复盘、星系跳转和 Meta 回到新局，且 PM 能在浏览器中记录控制台错误、runtime errors、关键 DOM、视口/小屏、帧耗时粗采样和 playtest hooks 结果，才允许进入发布准备讨论。

## 玩家收益

- 减少首局阻塞：空存档玩家能按一条明确路径完成最小建造、采矿、战斗和复盘，不被隐藏错误或不可读状态卡住。
- 确认真实浏览器可读性：HUD、建造、战斗、复盘、星系图和 Meta 不只在 headless 或静态检查中成立，也要在真实浏览器视口中看得懂。
- 降低发布误判：控制台错误、runtime errors、关键 DOM、视口、小屏和帧耗时都有记录，避免凭感觉宣布“能发布”。
- 保留纵向切片价值：若验收通过，玩家可获得一个更稳定、更可交付的太空站建造纵向切片。
- 让后续版本更有依据：若验收不通过，团队能知道是首局路径、浏览器兼容、小屏可读性、性能粗采样还是测试钩子缺口阻塞发布。

## 核心目标

v0.14.0 要回答一个问题：当前游戏是否已经具备进入发布准备的证据，而不是只证明功能列表已完成。

必须覆盖以下判断：

- 玩家路径：从空存档开始，能否完成首局建造、采矿、资源不足处理、断电理解、敌人接近、武器/护盾/维修、战斗后复盘、星系跳转、Meta 回到新局。
- 浏览器证据：真实浏览器或可说明边界的 browser/headless 流程，能否记录错误、DOM、视口、关键 hooks 和帧耗时粗采样。
- 只读安全：发布候选检查只能读取状态，不写存档，不改变 RNG、战斗、资源、Meta 或星系推进。
- 小屏可读：关键容器在小屏或窄窗口下不出现阻塞性溢出、遮挡或无法读取的核心信息。
- 人工结论：没有真实人工体验记录时，制作人不得给出“发布就绪”结论。

## 必做范围建议

### 1. 发布候选验收路径定义

目标是把“从头玩一局”的体验拆成 PM、制作人和实现成员都能复现的最小路径。路径应围绕现有系统，不新增玩法、不改规则。

必须定义的路径节点：

- 空存档首局：清空或隔离存档后进入游戏，确认默认入口、初始 HUD、建造入口和基础状态可读。
- 最小采矿链：玩家能找到采矿相关设施，完成能工作的基础采矿/资源链，并看到资源变化。
- 资源不足：尝试资源不足的建造或操作时，系统给出可读原因，不只是按钮无反应。
- 断电：制造或观察电力不足时，玩家能知道哪些关键能力受影响。
- 敌人接近：敌人出现或靠近时，玩家能看到威胁方向、数量、距离或关键提示。
- 武器/护盾/维修：玩家能判断基础攻击、防御和维修是否在工作，失败原因是否可读。
- 战斗后复盘：一次交火后，玩家能看到损伤、主要问题或下一步设计建议。
- 星系跳转：完成或离开当前节点后，玩家能进入星系图并理解下一步选择。
- Meta 回到新局：局后或 Meta 入口能被找到，回到新局后关键状态不混乱、不误导。

验收标准：

- PM 看板或检查记录中能逐项标记上述路径为通过、失败、未覆盖或阻塞。
- 任一路径失败时，记录失败步骤、实际现象、浏览器环境和是否阻塞发布候选。
- 该路径不要求完成 15-30 分钟完整局，但必须覆盖当前纵向切片的核心循环闭环。

### 2. 浏览器取证与只读验收钩子

目标是新增或整理发布候选可读取的 readiness snapshot / `runReleaseCandidateChecks`，用于浏览器级或 headless 回归取证。该能力服务验收，不服务普通玩家 UI。

建议输出：

- `runtimeErrors`：运行时错误、未捕获异常、关键 console error/warning 摘要。
- `keyDom`：HUD、建造面板、战斗提示、复盘、星系图、Meta 入口等关键 DOM 存在性与可见性摘要。
- `viewport`：当前视口尺寸、设备像素比、是否命中小屏/窄屏检查条件。
- `combatHooks`：威胁摘要、武器有效性、护盾、维修、战斗后复盘关键只读状态。
- `buildHooks`：建造面板、成本、资源不足、电力压力、放置/选择诊断等只读状态。
- `designHooks`：空间站健康摘要、采矿能力、供电、推进、防御、维修和设计短板。
- `metaHooks`：Meta 入口、局外点数或节点可见性、新局返回状态的只读摘要。
- `galaxyHooks`：星系图入口、节点选择、跳转状态和回到局内的只读摘要。
- `frameTiming`：粗粒度帧耗时采样，例如采样数量、平均值、最大值或长帧计数；不把它包装成完整性能分析。
- `storageSignature`：当前运行态存储签名或只读摘要，用于确认检查前后未写存档。

约束：

- 只读不写存档，不修改 `SAVE_KEY`，不改 meta schema。
- 不改变资源、敌人、任务、RNG、战斗、维修、星系跳转或 Meta 结算。
- 检查函数可以读取现有测试钩子、debug snapshot 或浏览器状态，但不得为了通过检查伪造游戏状态。
- 若真实浏览器自动化不可用，必须明确记录“真实浏览器自动化未覆盖”，不能用 headless 结果替代人工体验。

验收标准：

- PM 能调用 release readiness snapshot 或 `runReleaseCandidateChecks` 并得到结构化结果。
- 调用前后 storage signature 一致，证明只读检查没有写存档。
- 输出字段能支撑发布候选判断，而不是只返回笼统的 `ok`。

### 3. 小屏与窗口可读性检查

目标是在不大改 UI 的前提下，确认关键容器在小屏、窄窗口和触控假设下不会阻塞首局路径。

建议检查对象：

- HUD 资源、电力、威胁和空间站健康摘要。
- 建造面板、设施卡片、成本/不可负担原因和放置反馈。
- 战斗提示、武器/护盾/维修状态和战斗后复盘。
- 星系图节点、跳转按钮、Meta 入口和新局返回入口。
- 任何覆盖主操作区的弹层、面板或提示。

允许修正：

- 关键容器的折行、最大高度、滚动、溢出、遮挡和验收标记。
- 不改变信息架构前提下的文案长度压缩。
- 为 PM 检查增加只读可见性标记或 DOM 标记。

不允许扩展：

- 不重做 UI 框架。
- 不新增完整移动端交互系统。
- 不因为小屏检查重排核心玩法、数值或流程。

验收标准：

- 小屏或窄窗口下，玩家仍能看到资源、电力、建造入口、关键失败原因、敌人/战斗提示和继续流程入口。
- 关键面板不遮住无法关闭的核心操作，不出现阻塞发布候选路径的文本溢出。
- 若触控体验未做真实设备验证，记录为未覆盖风险，不得宣称移动端已通过。

### 4. 发布候选人工检查面板或文档化 checklist

目标是让人类测试者能按统一路径体验并记录结论。承载形式可以是游戏内隐藏测试 hook、轻量人工检查面板，也可以先由 PM 文档化 checklist 承载。

必须包含：

- 起始条件：空存档、浏览器、视口、是否使用真实浏览器、是否使用 headless。
- 操作路径：从首局建造到 Meta 回到新局的步骤。
- 观察项：玩家是否看懂 HUD、建造、资源不足、断电、敌人接近、武器/护盾/维修、复盘、星系图和 Meta。
- 证据项：console/runtime errors、关键 DOM、viewport、frame timing 粗采样、hooks 输出、storage signature。
- 结论项：通过、失败、未覆盖、阻塞发布、非阻塞待优化。

验收标准：

- PM 和制作人可以用同一 checklist 得到可比较的验收记录。
- checklist 能明确区分“自动通过”“浏览器/headless 通过”“人工体验通过”和“未覆盖”。
- 没有人工体验记录时，checklist 结论只能是“发布候选证据不足”。

### 5. PM browser/headless 回归脚本与记录

目标是在不安装依赖、不做系统配置的前提下，优先复用现有 Edge/Chrome、Node/VM 或项目已有测试方式，形成可重复的发布候选回归记录。

建议策略：

- 先盘点现有脚本、测试入口和浏览器可用方式。
- 能使用现有 Edge/Chrome 时，优先记录真实浏览器 console/runtime/DOM/viewport/hook 结果。
- 若只能使用 headless 或 Node/VM，明确它覆盖了什么、没覆盖什么。
- 帧耗时只做粗采样，用于发现明显长帧或异常，不作为性能优化专项。
- 记录结果进入 PM 看板或 PM 工作区，由 PM 判断是否阻塞发布候选。

验收标准：

- 至少有一份自动检查记录，包含 runtime errors、关键 DOM、hooks 输出和 storage signature。
- browser/headless 记录明确环境、命令或操作方式、通过项、失败项和未覆盖项。
- 若真实浏览器无法自动化，必须在验收记录中明确说明，并安排人工浏览器体验补证。

## 不做范围

本版本禁止扩展为新玩法或发布版本。以下内容不进入 v0.14.0：

- 不新增玩法系统。
- 不新增敌人、建筑、Meta 节点或星系图内容。
- 不修改数值平衡、伤害、敌人 AI、RNG、资源公式、维修或护盾判定。
- 不修改 `SAVE_KEY`。
- 不修改 meta schema。
- 不重写 UI 框架或做完整移动端改版。
- 不安装依赖，不做系统配置，不引入新测试框架。
- 不发布、不打 tag、不归档 `Releases/`，除非制作人最终批准后另走发布流程。
- 不以取证需求为由写入存档、伪造通关、伪造战斗结果或改变玩家进度。
- 不改动与发布候选验收无关的文案、注释、资源或工程结构。

## 风险级别

本版本风险级别为中风险偏低。

风险偏低的原因：目标主要是验收路径、浏览器取证、只读 hook、checklist 和小屏可读性补洞，不直接改变核心玩法规则。若实现保持只读、展示和记录层面，可以按中风险偏低推进。

中风险来源：

- release readiness snapshot 字段可能误报或漏报，影响发布判断。
- 小屏可读性修正可能触及已有 HUD 布局，需要 UI 和 PM 审核。
- browser/headless 结果容易被误读为人工体验通过，需要 PM 明确区分证据类型。
- storage signature 若设计不清，可能无法证明检查前后未写存档。

若实现过程中触及以下内容，风险升级为高风险，并需要制作人重新确认、PM 调整看板、主程专项审核：

- 核心规则、战斗结算、敌人 AI、RNG、资源公式、维修/护盾/脱落判定。
- 存档结构、`SAVE_KEY`、meta schema 或局外成长数据。
- 发布产物、tag、归档目录或正式发布流程。
- 新依赖安装、系统配置、浏览器驱动安装或其他本机环境危险操作。

涉及本机环境危险操作时，必须先向用户说明风险并等待确认。

## 验收路径

v0.14.0 必须把自动检查、browser/headless、人工体验和未覆盖风险分开记录。不能把其中任一类证据混同为完整发布结论。

### 自动检查

必须记录：

- release readiness snapshot 或 `runReleaseCandidateChecks` 是否可调用。
- `runtimeErrors`、`keyDom`、`viewport`、combat/build/design/meta/galaxy hooks、`frameTiming`、`storageSignature` 是否输出。
- 调用前后 storage signature 是否一致。
- 自动检查发现的错误、缺失 DOM、缺失 hook 或异常长帧。

接受标准：

- 自动检查能证明“可读取、结构完整、未写存档”。
- 自动检查不能单独证明“发布就绪”。

### Browser / Headless

必须记录：

- 使用的浏览器或运行方式，例如真实 Edge/Chrome、headless、Node/VM。
- 视口尺寸、小屏或窄屏参数。
- console/runtime errors、关键 DOM、hooks 输出和帧耗时粗采样。
- 哪些路径由真实浏览器覆盖，哪些仅由 headless 或 VM 覆盖。

接受标准：

- 真实浏览器覆盖越多，发布候选可信度越高。
- 若真实浏览器无法自动化，必须明确未覆盖，并由人工体验补证。

### 人工体验

必须记录：

- 从空存档到新局返回的实际操作路径。
- 玩家是否能看懂 HUD、建造、采矿、资源不足、断电、敌人接近、武器/护盾/维修、战斗复盘、星系跳转和 Meta。
- 小屏或窄窗口下是否存在阻塞阅读或操作的问题。
- 制作人或 PM 的体验结论：通过、继续打磨、阻塞发布候选。

接受标准：

- 没有真实人工体验，不得宣称发布就绪。
- 人工体验如果无法完成核心路径，制作人结论应为“继续打磨，不进入发布准备”。

### 未覆盖风险

必须单列：

- 未覆盖的浏览器、视口、小屏、触控、真实交火、帧耗时或存档签名验证。
- 仅 headless 覆盖但未人工验证的路径。
- 可能影响发布判断但本版本不处理的技术或体验风险。

接受标准：

- 未覆盖风险可以不全部阻塞，但必须被 PM 和制作人看见。
- 若未覆盖项包含真实人工体验、完整运行态只读签名或核心路径闭环，则不得进入发布准备。

## 制作人接受标准

v0.14.0 只有在以下条件同时成立时，才可进入发布准备讨论：

- 空存档首局、最小采矿链、资源不足、断电、敌人接近、武器/护盾/维修、战斗后复盘、星系跳转、Meta 回到新局都有记录。
- release readiness snapshot / `runReleaseCandidateChecks` 能输出 runtime errors、关键 DOM、viewport、combat/build/design/meta/galaxy hooks、frame timing 粗采样和 storage signature。
- 只读检查前后不写存档，不改变核心规则、RNG、战斗、资源、星系或 Meta。
- 小屏/窄窗口没有阻塞核心路径的 HUD、建造、战斗、复盘、星系图或 Meta 可读性问题。
- PM 明确区分自动检查、browser/headless、人工体验和未覆盖风险。
- 至少有一次真实浏览器人工体验记录；若没有，结论只能是“发布候选证据不足”。

若以上任一核心项缺失，制作人结论应为“继续补证或打磨，不进入发布准备”。

## 角色流程

建议执行流程如下：

1. 制作人方向：确认 v0.14 聚焦发布候选可验收，不新增玩法、不改核心规则、不启动发布流程。
2. PM 看板：拆分任务，记录允许修改范围、禁止范围、验收标准、证据类型和未覆盖风险。
3. 主程取证方案：设计 release readiness snapshot / `runReleaseCandidateChecks` 的只读字段、storage signature 和调用边界。
4. UI / PM checklist 方案：确认小屏关键容器检查点，以及人工路径 checklist 的承载方式。
5. 实现：按只读取证能力、关键可读性补洞、checklist/记录方式、browser/headless 回归顺序推进。
6. 专项审核：主程审核是否越界改规则/存档，UI 审核小屏可读性，PM 审核 checklist 是否能承载验收结论。
7. PM browser 回归：记录自动检查、browser/headless、人工体验和未覆盖风险，不能把 headless 结果冒充真实体验。
8. 制作人决定：根据 PM 证据和人工体验结论，决定是否进入发布准备、继续补证，或回退到具体体验问题修复。
