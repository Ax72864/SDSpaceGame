# v0.8.0 局内随机事件系统 制作人方向

版本名：`v0.8.0-encounters`
基线：HEAD `ad8799c`（v0.7.0 战斗操控发布后）
风险级别：**项目中高风险**（核心系统扩展 + 多 NPC 类型 + HUD 聚合 + RNG 流并行 + 与 OBJECTIVE_TYPES 平行架构）

## 本轮目标

在 v0.7.0 战斗操控之上，加入**局内随机事件系统**——玩家每关都会遭遇 1-3 个"非主线"随机事件（漂流商船 / 救援请求 / 伪装陷阱 / 沉船秘窟 / 神秘信号），每个事件提供独立选择和后果，让玩家在主任务之外有"决策深度 + 单局记忆点"；完整复用 v0.5.0 研发 / v0.6.0 hostile-station / v0.7.0 锁敌战斗等已有系统。

## 体验目标

按重要程度排序：

1. **单局记忆点**：玩家每关都会遇到 1-3 个随机遭遇事件，每个有独立选择和后果；玩家能记住"我那局遇到了什么"，形成碎片化的局内叙事。
2. **决策深度**：每个事件提供 2-3 个选项（援助 / 拒绝 / 抢劫 / 交易 / 忽略 / 伏击），选择影响奖励 / 风险 / 后续遭遇；让玩家在战斗和拾荒之外多一层"做选择"的乐趣。
3. **与已有系统协同**：部分事件触发战斗，玩家用 v0.7.0 锁敌 + F 键齐射处理；部分事件涉及友方 NPC，复用 v0.4.0 escort 系统；部分事件涉及沉船群，复用 v0.3.0 / v0.6.0 wreck 拾荒。事件不抢主任务（OBJECTIVE_TYPES）的注意力，作为"事件叠加在主任务之上"的并行层。

## 核心机制

### 1. 平行 ENCOUNTER_TYPES 注册表（不塞入 OBJECTIVE_TYPES）

**关键架构决策**：v0.8.0 必须采用**平行 ENCOUNTER_TYPES** 而非扩展 OBJECTIVE_TYPES。

理由：
- `OBJECTIVE_TYPES` 是「单局唯一目标 + 决定能否跃迁 + 决定结算奖励」的**强语义槽位**。
- ENCOUNTER 是「关卡内随机插入、可拒绝/可错过、不影响主线」的**弱语义层**。
- 塞进同一注册表会让 `createObjective()` 变成「事件 vs 主任务」二选一，破坏 `OBJECTIVE_LEVEL_WEIGHTS` 可读性，并污染 `awaitingObjectiveChoice` 等 UI 状态。

实施清单：
- 新建 `ENCOUNTER_TYPES = { trader, distress, ambush, derelict, signal }`，字段沿用 `make / tick / onXxx / isComplete / isFailed / getDetail / getHint / render` 同形结构。
- 新建 `state.run.encounters: []`（复数，允许并发）。
- 新建 `state.run.encounterCooldownThisGalaxy: Set<string>`（按类型单星系冷却）。
- 新建 `notifyEncounters(event, payload)` 平行于 `notifyObjective`，遍历 `state.run.encounters` 派发；在现有 8 处 `notifyObjective` 调用点之后**追加一行 `notifyEncounters` 调用**（不修改既有事件源）。
- 新建 `applyEncounterReward(encounter)` 独立函数，复用 `state.resources.*` 但**不**走 `grantObjectiveReward`（避免触发 `objectiveCompleteAt`/结算逻辑 / `meta.points` 双倍）。
- 新建 `seedEncountersForLevel(level, rng)` 在 `nextLevel` 末尾调用：按权重抽 1-3 个事件，延迟 60-180s 触发，并清空 cooldown set。

### 2. 5 种核心事件类型

| 编号 | 类型 | 简述 | 玩家选项 | 战斗 | 主要奖励 |
|---|---|---|---|---|---|
| E1 | **trader 漂流商船** | 友好商船路过，30s 内可交易 | 交易（金属 → 等离子）/ 忽略 | 否 | 等离子 / 罕见科研点 |
| E2 | **distress 救援请求** | 友方 NPC 被海盗围困 | 援助（清海盗）/ 忽略 | 是 | 大量金属 + 科研 + 跨关声望 |
| E3 | **ambush 伪装陷阱** | 看似 trader / distress，靠近后变海盗 | 靠近触发战斗 / 远离脱险 | 是 | 战斗后大量金属 + 海盗 wreck |
| E4 | **derelict 沉船秘窟** | 大型 wreck 群（4-6 wreck），不限时 | 拾荒（自由 dock） | 否 | 等离子 cell 高权重 wreck |
| E5 | **signal 神秘信号** | 远处单个高价值 fragment | 飞过去拾取 / 忽略 | 否 | 单个 origin="signal" 大 fragment（含罕见 facility cell） |

每关按权重抽 1-3 个事件，60-180s 间隔触发，每种事件单星系内最多 1 次（cooldown set）。

### 3. NPC 系统扩展

当前 `npc.kind` 仅有 `"friendly-cargo"`（escort 任务专用）。v0.8.0 必须扩展：

- 新增 kind：`"trader"` / `"distress-pilot"` / `"pirate-ambush"`（伪装时显示为 distress，靠近后切换 kind）。
- 新增字段：`npc.role: "encounter" | "objective"`（区分事件 NPC 与任务 NPC）。
- 新增字段：`npc.encounterId: string | null`（事件 NPC 反向引用所属 encounter）。
- `tickNpc(npc, dt)` 按 kind 分支：trader 巡航 / distress 跟随玩家 / pirate-ambush 触发条件改为攻击玩家。
- `buildNpcHudAlerts()` 按 `npc.role` 分组聚合，事件 NPC 不污染 escort 任务的 alert 通道。
- `damageNpc()` 在 `npc.role === "encounter"` 时调用 `notifyEncounters("npcDestroyed", npc)`，不调 `notifyObjective`。

### 4. 独立 RNG 流

ENCOUNTER 系统**必须使用独立 RNG 种子**：`state.run.seed ^ 0xb5297a4d + level * 0xcc9e2d51`。

理由：与 OBJECTIVE 共用种子会破坏 `__gameTest.findEscortSeed/getObjectiveType` 等测试钩子。

### 5. HUD 集成

- 新建 `buildEncounterHudAlerts()` 函数，在 `updateHud()` 的 `allAlerts` 数组中按 priority 插入到 hostile-station 之后、npc 之前。
- 中央闪烁告警复用 `showHostileStationAlert(text, ms, isSuccess)`，但**新增 priority 队列**：assault 优先级 > 战斗事件 > 友好事件，避免短时间多事件触发时 DOM 堆叠。
- 新增 CSS 类：`.alert-encounter-trader / .alert-encounter-distress / .alert-encounter-ambush / .alert-encounter-derelict / .alert-encounter-signal`。
- HUD 显示事件状态：`贸易商队 ×1 · 距离 450 · 30 秒后离开`。
- 玩家选择 UI 通过 HUD 提示文字 + 已有交互（如靠近 NPC 自动触发 / 锁敌 + 攻击）。

### 6. 跨关清理

`nextLevel()` 末尾追加：
- 清空 `state.run.encounters = []`
- 清空 `state.run.encounterCooldownThisGalaxy.clear()`
- 调用 `seedEncountersForLevel(state.run.level, encounterRng)` 重新滚事件池

跨关持续型事件**不在 v0.8.0 范围**（避免引入 run 持久化的存档迁移负担）。

### 7. fragment 新 origin

引入 `fragment.origin` 新类型：
- `"derelict"`：derelict 事件的 wreck 群（使用 WRECK_FACILITY_WEIGHTS 但权重偏向稀有 cell 如 plasma / hangar）
- `"signal"`：signal 事件的单个高价值 fragment（origin 仅 1 个 fragment，权重偏 plasma / fusion 等稀有 cell）

`buildFragmentHudAlerts()` 增加对这两种 origin 的分色 + 文字（沿用 v0.6.0 enemy-wreck 的扩展模式）。

## 体验目标

**目标 1：单局记忆点**——玩家每局都会经历 1-3 个事件，每个事件有视觉化触发 + 玩家选择 + 后果反馈；玩家能在结算时回忆"这局遇到了什么"。
**目标 2：决策深度**——5 种事件覆盖"友好交易 / 战斗援助 / 真伪判断 / 自由拾荒 / 高风险高回报"5 种决策模式；每种事件的选项后果明显（不是"一律捡起来就好"）。
**目标 3：与已有系统协同**——distress / ambush 触发战斗自动启用 v0.7.0 锁敌；derelict / signal 复用 v0.3.0 fragment dock；trader 复用 v0.4.0 NPC 系统；事件不抢 OBJECTIVE_TYPES 的注意力。

## 必做范围

### 工程基础

- `ENCOUNTER_TYPES` 注册表 + 5 种事件实现（trader / distress / ambush / derelict / signal）

- `nextLevel()` 集成清理 + 重新滚池

### NPC 系统扩展

- 扩展 `npc.kind ∈ {"friendly-cargo", "trader", "distress-pilot", "pirate-ambush"}`

- `damageNpc()` 按 role 选择 `notifyObjective` 或 `notifyEncounters` 派发
- `buildNpcHudAlerts()` 按 role 分组聚合

### HUD 与视觉反馈

- `buildEncounterHudAlerts()` 聚合 5 种事件状态
- `showHostileStationAlert` 加 priority 队列防 DOM 堆叠（不破坏现有 assault 行为）
- 新增 CSS：`.alert-encounter-*` 5 种语义类

### 战斗与拾荒接入

## 不做范围

- **不破坏 v0.4.0 现有 OBJECTIVE_TYPES**：事件是**完全独立**的平行系统，OBJECTIVE_TYPES 注册表 / `createObjective` / `notifyObjective` / `grantObjectiveReward` 等不修改。
- **不破坏 v0.5.0 研发系统 / v0.6.0 hostile-station / v0.7.0 锁敌**：事件触发战斗时复用现有 enemy 路径，玩家 turret / shield / missile / 锁敌行为不变。
- **不修改 SAVE_KEY**：`state.run` 仍不存盘，事件状态刷新即重置（接受这个边界，避免引入存档迁移负担）。
- **不引入新 enemy kind**：ambush 使用现有 pirate；distress 围困者也用 pirate。
- **不引入新建造 cell 类型**。
- **不做联机 / 异步对战**（game desc §9 / §10）。
- **不做局外天赋树重构**（game desc §8）。
- **不做新探索 / 地图系统**（game desc §7）。
- **不做跨关持续事件**（如"贸易商队 N 关后归来"），全部事件单关结束清空。
- **不做炮塔朝向 / 平面护盾方向性**（game desc §5 的 v0.7.0 第 4/5 步留 v0.8.x 或 v0.9）。

## 关键技术疑问（给主程）

请主程在 ADR 中明确回答以下 9 项决定：

1. **`ENCOUNTER_TYPES` 字段集**：与 OBJECTIVE_TYPE_DEFAULTS 完全相同还是子集？是否引入新字段（如 `priority` / `cooldownGalaxies`）？**建议**：复用 `make / tick / onXxx / isComplete / isFailed / getDetail / getHint / render` 加新字段 `cssClass / priority / hudAlertClass`，不复用 `rewardMultiplier / researchReward`（事件走独立 `applyEncounterReward`）。
2. **`state.run.encounters` 数据结构**：每个 encounter 是 `{ id, type, state, spawnAt, expireAt, ... }` 还是直接复用 OBJECTIVE 的字段结构？**建议**：独立结构，明确 type 字段，避免与 objective 混淆。
3. **事件触发时机**：在 `update()` 主循环里 tick 还是单独 `updateEncounters(dt)`？**建议**：独立 `updateEncounters(dt)` 在 `update()` 末尾调用，便于隔离 + 测试。
4. **5 种事件 spawn 位置**：trader / distress / ambush / derelict / signal 距离玩家多远？**建议**：trader 400-600px / distress 500-800px / ambush 600-900px / derelict 700-1100px / signal 800-1400px。
5. **NPC role 字段位置**：`npc.role` 还是 `npc.encounterContext: { encounterId, kind }`？**建议**：扁平字段 `npc.role` + `npc.encounterId`，与现有 `npc.kind / hp / target` 同级，无嵌套。
6. **`showHostileStationAlert` priority 队列设计**：是新增 `state.hudAlertQueue` 还是用 `setTimeout` 串行？**建议**：简单队列 `state.hudCenterAlertQueue: [{ text, ms, isSuccess, priority }]`，按 priority 排序，同时只显示 1 个。
7. **跨关清理边界**：encounters 完全清空 + 重新滚池还是保留未完成事件？**建议**：完全清空（与 fragments / npcs / enemies 行为一致），v0.8.0 不做跨关持续。
8. **fragment origin 扩展**：`buildFragmentHudAlerts` 怎么显示 `derelict / signal`？**建议**：derelict 用紫色 `alert-fragment-derelict` / signal 用金色 `alert-fragment-signal`，文字"沉船残骸 / 神秘信号"。
9. **事件并发上限**：同关卡最多并存几个事件？**建议**：max 3 个并发，避免 HUD 信息过载（5 种事件类型 → 每关随机选 1-3 个）。

## 数值边界建议（给数值策划）

- **每关事件数量**：level 0-5 各 1-3 个；level 6 终末关 0 个（保护 guardian 终局体验）。
- **每事件触发延迟**：spawn 后 60-180s（按事件复杂度递增；trader 60s / distress 90s / ambush 120s / derelict 150s / signal 180s）。
- **事件过期时间**：trader 30s 内交易 / distress 45s 内援助 / ambush 触发后即时 / derelict 关内永久 / signal 关内永久。
- **trader 交易比例**：50 金属 → 1 等离子 / 100 金属 → 3 科研。
- **distress 援助奖励**：清掉 4-6 个海盗后 +120 金属 + 18 科研 + 5 等离子。
- **ambush 战斗奖励**：清掉 6-8 个海盗后 +200 金属 + 25 科研 + 3 海盗 wreck（origin="enemy-wreck"，使用 v0.6.0 ENEMY_WRECK_FACILITY_WEIGHTS）。
- **derelict wreck 数量**：4-6 个（按 level 递增）/ 半径 80-120px / 权重偏 plasma / hangar / fusion。
- **signal fragment**：1 个 / 距离 800-1400px / cell 数量 6-9 / 权重偏 plasma / fusion / energy。
- **事件冷却**：每种事件单星系内最多 1 次（trader 例外，可 2 次）。
- **HUD priority**：hostile-station 100 > ambush 80 > distress 70 > trader 50 > derelict 40 > signal 30。
- **alert 持续**：trader 4s / distress 5s / ambush 3s（红字）/ derelict 4s / signal 3s。

## 项目高风险流程要求

按团队规则，本任务属于**核心系统扩展 + 5 种新事件 + NPC 多 kind + HUD 聚合层重构 + RNG 流并行**，**走项目中高风险流程**（与 v0.6.0 / v0.7.0 一致）：

1. 制作人方向（本文档）commit + push 后才能进入下一步。
2. PM 拆分 v0.8.0 任务看板（7 个核心任务节点）。
