# v0.13.0-combat-threat-readability 制作人方向

## 结论

v0.13.0 的版本名为 `v0.13.0-combat-threat-readability`。

核心目标：让玩家在战斗前、战斗中、战斗后看懂威胁来源、防御状态、武器是否有效、维修是否跟上，从“知道空间站短板”推进到“能对战斗做反应”。

v0.12.0-station-design-feedback 已接受为“开发完成 / 自动与 headless 浏览器回归通过”，但尚未发布，真实人工验收仍未完成。v0.13.0 可以继续推进制作，但发布候选判断必须补齐真实浏览器或发布包体验证据。

制作人方向：本版本不做战斗数值重平衡，不扩展敌人或武器系统，只把现有敌袭、武器、护盾、维修和受损反馈解释清楚。若玩家能在敌人接近时判断危险来自哪里，在交火中看懂哪些防御/武器正在工作，在战后知道该修正什么设计，本版本目标即成立。

## 玩家收益

- 敌袭不再像突然损坏：玩家能提前看到来袭数量、方向、距离和主要威胁类型。
- 知道被什么打：战斗中能分辨小行星、海盗、空间站、守护者或敌站带来的压力。
- 知道哪里危险：最近受损模块、防御缺口和脱落风险能被明确提示。
- 知道武器是否有效：炮塔、导弹和齐射状态能解释“正在攻击 / 射程不足 / LOS 阻挡 / 缺电 / 装填中”。
- 知道护盾和维修是否跟上：护盾存在、缺电、维修无人机状态和维修站是否有目标不再只靠猜。
- 战斗后能修正设计：玩家能根据主要伤害来源、损坏最重区域和防御短板决定补武器、补电、补护盾、补维修或调整布局。
- 导弹/锁敌/齐射的价值更清晰：玩家能看到锁敌和齐射是否形成有效输出，而不是只看到按钮冷却。

## 核心目标

v0.13.0 要把 v0.12.0 的空间站健康摘要向战斗实时决策延伸，重点回答以下问题：

- 战斗前：敌人从哪里来，距离多远，数量和类型是否足以构成威胁。
- 战斗中：当前主要威胁是谁，武器是否打得到，护盾是否挡得住，维修是否在处理损伤。
- 战斗后：这次主要被什么伤害，哪里损坏最重，空间站设计暴露了什么短板。

体验目标不是让战斗更难或更复杂，而是让玩家从“我看见设施有状态”升级为“我能对战斗变化做出反应”。

## 必做范围

### 1. 威胁预警与敌情摘要

目标是让敌袭从“突然进入受损状态”变成玩家可提前理解的战斗事件。提示必须低噪声，避免持续刷屏。

建议方向：

- 在 HUD 或状态摘要中显示敌人来袭数量、方向、最近威胁和距离。
- 敌种类型至少能区分小行星、海盗、空间站、守护者和敌站。
- 最近威胁应优先展示对空间站最有决策价值的信息，例如“最近敌站 420m”“海盗从右上接近”。
- 安全态不显示夸张警告；只有敌人接近、进入射程、突破防御或造成损伤时提高提示优先级。
- 信息表现可以是短状态项、轻量 HUD 行或现有摘要扩展，不需要新弹窗系统。

验收标准：

- 无敌安全态时，玩家能明确知道当前没有战斗压力。
- 敌人接近时，玩家能读到数量、方向、最近威胁类型和距离。
- 多个敌人存在时，摘要优先解释最近或最危险威胁，而不是输出无意义列表。

### 2. 武器有效性反馈

目标是解释现有炮塔、导弹、锁敌和齐射是否正在产生有效输出。不改伤害公式，不改射程、命中、AI 或敌人生成。

必须覆盖的反馈：

- 炮塔当前目标、是否在射程内、是否被 LOS 阻挡、是否缺电。
- 导弹当前目标、装填状态、齐射准备状态和发射结果。
- 锁敌目标是否有效，若无效应能解释失效原因，例如目标死亡、超距或没有可攻击目标。
- 护盾的当前防御状态可以与武器摘要并列显示，但不能暗示护盾会改变攻击逻辑。
- 选中面板和状态摘要都可以承载信息：选中面板解释单个设施，摘要解释整体火力状态。

验收标准：

- 选中炮塔时，玩家能判断它是否有目标、是否被 LOS 阻挡、是否因缺电停火。
- 武器缺电时，提示落到具体原因，不只表现为“没有开火”。
- 导弹齐射准备和发射后状态可读，玩家能知道齐射按钮是否值得按。

### 3. 受损与维修反馈

目标是让受损、脱落风险、护盾阻挡和维修行动可读，只做反馈解释，不改维修算法。

必须覆盖的反馈：

- 最近受损模块：显示近期被打到或被碰撞影响的关键模块。
- 脱落风险：当模块接近脱落或核心链路风险明显时给出短提示。
- 护盾存在/缺电：玩家能知道当前是否有护盾能力，护盾是否因缺电失效。
- 维修无人机状态：正在前往、正在维修、空闲或无法维修。
- 维修站是否有目标：有损伤但维修站无目标时，应说明是无覆盖、无可修对象、缺电还是其他现有规则导致。

验收标准：

- 战斗中出现损伤后，玩家能看到最近受损模块或区域。
- 有维修设施时，玩家能判断维修是否正在跟上损伤。
- 无维修目标或维修停摆时，玩家能看到可读原因，不需要理解底层算法。

### 4. 战斗后复盘摘要

目标是在一次战斗结束后，用短文案把本次战斗暴露的问题讲清楚。该摘要可以使用运行时临时统计，不进存档。

建议输出：

- 主要伤害来源：例如小行星碰撞、海盗火力、敌站远程输出。
- 损坏最重区域：例如核心附近、左翼武器区、外层采矿区。
- 防御短板：例如无护盾、护盾缺电、维修跟不上、炮塔被遮挡、导弹未形成齐射。
- 下一步建议：例如补电、补护盾、增加维修覆盖、调整武器外露位置、保留导弹齐射窗口。

验收标准：

- 一次战斗结束后，玩家能看到 2-4 条简短复盘信息。
- 复盘基于本次运行时观察，不写入存档，不影响 meta 成长或任务结果。
- 复盘文案是建议，不替玩家自动修复空间站。

### 5. 只读测试钩子

目标是给 PM、主程和制作人提供自动检查和 headless 回归可读取的只读信息。

建议只读输出：

- `threatSummary`：敌人数量、类型、方向、最近威胁和距离。
- `weaponEffectiveness`：炮塔/导弹/锁敌/齐射的目标、射程、LOS、缺电、装填和可发射状态。
- `damageRecent`：近期受损模块、损伤来源、脱落风险和关键区域受损摘要。
- `repairStatus`：维修无人机、维修站目标、缺电、空闲和无法维修原因。
- `combatReview`：战斗后主要伤害来源、损坏最重区域、防御短板和下一步建议。

约束：

- 只读输出，不写存档。
- 不修改 `SAVE_KEY` 或 meta schema。
- 不改变敌人 AI、RNG、生成、伤害、命中、护盾或维修判定。
- 默认不污染普通玩家 UI，可通过现有测试入口、debug snapshot 或回归钩子读取。

验收标准：

- PM 能读取 `threatSummary`、`weaponEffectiveness`、`damageRecent`、`repairStatus`、`combatReview`。
- 读取测试钩子不会改变存档、资源、敌人、任务进度、战斗结果或随机序列。
- 输出字段能支撑本版本验收路径，而不是只返回笼统的 `ok`。

## 不做范围

本版本禁止扩展为战斗系统重做。以下内容不进入 v0.13.0：

- 不新增敌人种类。
- 不修改伤害、HP、射程、命中、攻击频率或 AI。
- 不修改敌人生成、星系图事件或 RNG。
- 不修改护盾、维修、脱落、核心死亡等判定逻辑。
- 不修改 `SAVE_KEY`。
- 不修改 meta schema。
- 不新增新武器或新弹药资源。
- 不做战斗数值重平衡。
- 不做发布归档、发布 tag 或版本发布。
- 不安装依赖，不引入新测试框架。
- 不以可读性需求为由重构无关 UI、存档、敌人系统或资源公式。

## 风险级别

本版本风险级别为中风险。

中风险来源主要是战斗状态解释、HUD 信息优先级、只读测试字段和复盘统计准确性。若实现只涉及展示、只读诊断、运行时临时统计和文案，可以按中风险推进。

若实现过程中触及以下内容，风险升级为高风险，并需要制作人重新确认、PM 调整看板、主程专项审核：

- 伤害、命中、射程、冷却、HP 或敌人 AI。
- 敌人生成、敌种行为、星系图战斗事件或 RNG。
- 护盾阻挡、维修目标选择、脱落、核心死亡等判定。
- 存档结构、`SAVE_KEY` 或 meta schema。
- 导弹齐射、锁敌、炮塔攻击等现有战斗规则的实际结算。

## 验收路径

v0.13.0 必须围绕“玩家能否对战斗做反应”验收，而不是只检查状态字段是否存在。

必测路径：

- 无敌安全态：没有敌人或敌人无威胁时，HUD/摘要不制造假警报。
- 敌人接近：出现敌人后，玩家能看到数量、方向、类型、最近威胁和距离。
- 武器缺电：炮塔或导弹因缺电无法工作时，选中面板或摘要能解释原因。
- 炮塔 LOS 阻挡：炮塔被空间站结构遮挡时，玩家能看懂不是“武器坏了”，而是被挡住。
- 导弹齐射准备/发射：玩家能判断导弹是否就绪、齐射是否可用、发射后是否进入装填。
- 护盾存在/缺电：有护盾时能看到防御状态，护盾缺电时能看到失效原因。
- 维修有/无目标：维修无人机或维修站能解释正在修、无目标、缺电或无法维修。
- 战斗后损伤回顾：战斗结束后能看到主要伤害来源、损坏最重区域、防御短板和下一步建议。
- v0.7 不回潮：锁敌、齐射、LOS 和武器反馈不退回到不可读或误导状态。
- v0.12 不回潮：建造诊断、空间站健康摘要、星系图和 Meta 成长可读性不被破坏。

制作人接受标准：

- 玩家能说出当前战斗威胁来自哪里，以及最近威胁是什么。
- 玩家能判断至少一个武器无效原因，例如缺电、射程不足、LOS 阻挡或装填中。
- 玩家能判断护盾/维修是否正在发挥作用，以及失败原因是否可读。
- 战斗后复盘能给出至少一个可执行设计修正建议。
- 测试钩子只读输出本版本验收所需信息，不改变存档或战斗结果。

若以上任一核心项缺失，制作人结论应为“继续打磨，不进入发布候选”。

## 角色流程

建议执行流程如下：

1. 制作人方向：确认 v0.13 聚焦战斗威胁可读性，不改战斗数值和核心判定。
2. PM 看板：拆分任务、记录允许修改范围、验收标准、风险升级条件和回归路径。
3. UI 信息架构：整理威胁摘要、武器状态、受损维修状态和战斗后复盘的信息优先级。
4. 主程技术边界：确认只读数据来源、运行时临时统计、测试钩子字段和不改存档/判定的边界。
5. 文案/阈值表：列出敌情、武器无效原因、护盾/维修状态、损伤风险和复盘建议文案。
6. 分批实现：先威胁摘要，再武器有效性，再受损/维修反馈，再战斗后复盘，最后接入只读测试钩子。
7. 专项审核：主程审核是否越界到战斗规则修改，UI 审核可读性和噪声，数值策划审核阈值/文案是否误导。
8. PM 回归：覆盖无敌安全态、敌人接近、武器缺电、LOS 阻挡、导弹齐射、护盾、维修、战斗复盘和 v0.7/v0.12 回潮检查。
9. 制作人决定：根据 PM 记录和体验证据，决定 v0.13 是否可进入发布候选或继续打磨。
