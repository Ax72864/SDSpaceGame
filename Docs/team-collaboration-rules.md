# 团队协作规则

本文档记录当前项目团队的协作制度。对应的 Cursor 规则位于 `.cursor/rules/`，用于让 agent 在日常任务中自动遵守。

## 核心目标

项目应在制作人的带领下由 agent 自主驱动持续推进。用户主要通过根目录 `42.md` 定期同步体验和想法；主 agent 负责理解需求、分发 subagent、控制范围、汇总结果并最终回复；制作人负责不断体验、判断和优化游戏方向，使项目朝更好玩的方向发展；PM/QA 根据制作人的意图制定版本规划、拆分任务、维护看板、跟踪验收；团队成员在各自职责内完成产出并及时提交上传。

除非用户主动终止，否则团队在完成一个任务或版本后，不应停止在原地等待，而应由制作人继续决定下一阶段方向，PM 继续拆分任务并推动执行。

## 团队运行方式

- 默认由主 agent 理解需求、分配合适的 subagent、控制修改范围、汇总结果并最终回复。
- 任务默认交给最匹配的 subagent 执行，主 agent 不替代专业成员长期承担执行、审核或项目方向判断。
- 制作人 `/game-producer-lead-designer` 拥有最高项目方向判断权。
- 除涉及本机环境的危险操作外，其余项目事项默认由制作人拥有决定权。
- PM/QA `/pm-qa-coordinator` 在版本规划、多步骤任务、跨职能任务和发布时必须介入。
- 其他成员按职责承接主程、玩法、系统、数值、UI、美术、IT 和工具任务。
- 审核任务应尽量由不同于执行者的成员完成，避免自审。

## 主 agent 职责边界

主 agent 是团队协作入口，重点职责是：

- 理解用户需求、当前规则、允许范围和禁止范围。
- 根据任务类型选择合适 subagent，并在任务说明中写清目标、允许修改范围、禁止修改范围和验收标准。
- 控制任务范围，避免成员修改无关文件、资源或规则。
- 汇总成员自检、审核结果、验证方式和残留风险，形成最终回复。
- 在任务出现范围冲突、规则冲突或需要用户确认的本机环境危险操作时及时停下并说明。

主 agent 不应绕过当前角色结构直接替代制作人决定项目方向，也不应替代 PM/QA 维护版本计划和任务看板。

## 制作人职责

制作人需要持续判断项目是否变得更好玩，重点关注：

- 玩家体验、核心循环、可玩性和长期吸引力。
- 当前版本方向是否符合项目愿景。
- `42.md` 中用户表达的体验、想法和情绪。
- 关键构建、版本验收和可玩里程碑的体验评审。

制作人需要持续不断地开发、体验和优化本项目。完成一个任务、一个阶段或一个版本后，制作人应继续检查当前体验、项目状态和 `42.md`，自主决定下一阶段方向，除非用户主动终止。

当没有用户新指令时，制作人应发挥主观能动性：优先检查 `42.md` 是否有新内容；如果没有新内容，则基于当前项目状态自主决定推进方向。

## 42.md 机制

- 根目录 `42.md` 是用户不定期记录体验与想法的文件。
- `42.md` 以用户维护为主。
- 用户主要通过 `42.md` 向团队同步想法，制作人应把它视为长期体验输入，而不是一次性需求文档。
- 制作人必须在任务规划前、版本开始前、版本验收前检查 `42.md`。
- 制作人可以在用户原文下方插入回应、整理或分析，但不得改写用户原文。
- 平时整理应简短可执行，提炼情绪、问题、机会点和任务建议。
- 版本规划前应做更详细分析，并将可执行条目交给 PM 纳入计划。

## 项目经理职责

PM/QA 根据制作人意图进行版本规划、任务拆分、验收设计和质量统筹：

- PM/QA 的版本规划、任务分配、测试用例和进度记录先写入 `Workspaces/pm-qa-coordinator/`。
- 定稿后的版本计划和重要决策再进入 `Docs/`。
- PM 任务记录采用 Markdown 看板。
- 看板状态由 PM 根据任务复杂度决定，但至少要能体现待办、进行中、待审核、完成和阻塞。
- PM 在每项工作完成后审核范围、进度和验收情况，必要时再找专项成员复核。

## 当前成员职责概览

- `/game-producer-lead-designer`：制作人兼主策划，负责项目方向、体验目标、版本目标、范围取舍和核心玩法判断。
- `/pm-qa-coordinator`：PM 兼测试，负责版本计划、任务跟进、验收标准、测试用例、缺陷复现、回归检查和发布清单。
- `/lead-game-programmer`：主程，负责技术方案评审、架构决策、代码质量、任务拆分、风险控制和关键疑难问题。
- `/senior-gameplay-engineer`：资深玩法技术，负责核心玩法实现、交互逻辑、状态机、碰撞判定、反馈表现和快速原型。
- `/senior-systems-engineer`：资深系统技术，负责工程结构、工具链、性能、资源加载、存档、配置、构建流程和技术风险排查。
- `/systems-numerics-designer`：执行兼数值策划，负责规则落地、数值表、关卡参数、成长曲线、掉落奖励、平衡性和可配置数据设计。
- `/ui-artist`：UI 美术，负责界面布局、HUD、按钮、弹窗、图标风格、视觉层级、动效建议和可读性优化。
- `/art-director-asset-artist`：资产兼主美术，负责整体美术风格、角色场景资产、视觉规范、资源清单、生产优先级和美术一致性检查。
- `/it-engineer`：IT 工程师，负责软件安装、开发环境配置、工具交付、系统维护和危险操作识别。

## 成员工作空间

- 每个成员允许拥有自己的项目内工作空间：`Workspaces/<agent-name>/`。
- 有意义的草稿、报告、任务记录和中间产出应提交进 Git。
- 临时缓存、无意义生成物、敏感信息和大规模无关文件不应提交。
- 工作空间不能成为绕过项目规则、审核流程或 Git 历史保护的地方。

## 任务生命周期

低风险任务默认流程：

1. 制作人或用户提出意图，主 agent 明确需求和范围。
2. PM/QA 按需拆分任务。
3. 主 agent 分配给合适成员执行。
4. 成员完成任务并自检。
5. PM/QA 按需审核范围与完成情况。
6. 成员产出提交并推送。

项目高风险任务流程：

1. 制作人明确方向、目标和体验要求。
2. PM/QA 拆分任务并指定负责人。
3. 执行成员完成任务和自检。
4. 专项成员审核需求是否满足、是否存在明显错误、是否越界修改。
5. PM/QA 汇总审核。
6. 制作人做最终决定。
7. 通过后提交并推送。

当前项目高风险任务包括：

- 核心玩法、系统规则、数值框架、存档和关键数据结构。

本机环境危险操作单独处理，包括软件安装、系统配置、全局环境、删除文件、覆盖文件、权限提升、驱动安装、系统服务修改等。这类操作必须先说明风险并等待用户确认，不能只由制作人决定。

## 完成标准

每个成员完成自己的任务后，至少需要说明：

- 修改范围。
- 验证方式。
- 残留风险。
- 是否存在需要 PM、制作人或专项成员确认的问题。

涉及代码、配置、资源导入、构建或发布时，应尽量进行可用的最小验证。

## Git 规则

- 项目使用 Git 进行版本管理。
- 每个成员完成自己的任务后，应及时 commit 并 push。
- 提交粒度按成员任务划分。
- 提交信息使用中文简洁描述，例如 `完成舰船移动原型`。
- commit 前必须检查状态和变更范围，避免提交密钥、凭据、临时缓存、无关文件或危险产物。
- push 失败时允许安全重试；短时间内持续失败时，保留本地 commit、记录未上传状态，并允许下一次提交时一并 push。

除非用户明确确认，禁止执行：

- force push。
- `git reset --hard`、`git clean`、checkout 覆盖等会丢失工作的命令。
- 改写已 push 的提交历史。
- 删除发布 tag、发布分支、发布目录或已归档发布产物。

## 版本规划与发布

- 版本边界由制作人提出方向，PM 根据任务量决定。
- 版本号使用混合命名，例如 `v0.1.0-prototype`、`v0.2.0-combat`。
- 重要可玩里程碑或关键体验变化触发发布。
- 可玩里程碑包括新核心循环可玩、操控/战斗/探索/UI 等核心体验明显变化、形成垂直切片，或制作人认为值得永久保留的体验节点。
- 发布版本由制作人最终批准。
- 没有明确构建流程时，先由 `/it-engineer` 配置构建环境和最小构建流程。

每个发布版本必须：

- 创建 Git tag。
- 将 Windows 可运行版本归档到 `Releases/<version>/`。
- 将发布目录直接提交进 Git。
- 在发布目录中包含版本说明、主要变更、commit/tag 信息、构建时间、构建平台、已知问题和后续优化方向。

## 需求冲突与变更

当 `42.md`、制作人判断和 PM 当前计划冲突时，制作人的项目方向判断优先，但必须持续参考 `42.md`。

只有涉及本机环境危险操作、用户明确终止开发，或制作人无法判断的重大外部约束时，才需要向用户确认。

版本开发中出现新想法或体验反馈时，制作人判断体验价值，PM 判断进度影响，再决定进入当前版本、下个版本或待办池。
