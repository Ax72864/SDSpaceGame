# 团队成员名单

本文档汇总当前项目中的 Cursor subagent 团队成员，供主 agent 在拆分任务、分配工作、轻量严格模式执行和结果审核时参考。

成员文件位于 `.cursor/agents/`，调用时使用对应的 `/agent-name`。

## 相关规则

- 完整协作制度见 `Docs/team-collaboration-rules.md`。
- 项目级 Cursor 规则位于 `.cursor/rules/`，默认始终生效。
- 团队默认由主 agent 理解需求、分配给合适 subagent 执行、控制范围、汇总结果并最终回复。
- 本项目由 agent 自主驱动持续开发；除非用户主动终止，否则完成一个任务或版本后，制作人应继续决定下一阶段方向。
- 用户主要通过根目录 `42.md` 同步体验和想法，制作人必须持续参考；制作人可以整理和回应，但不得改写用户原文。
- PM/QA 在 `Workspaces/pm-qa-coordinator/` 中维护 Markdown 看板，记录任务负责人、目标、允许修改范围、验收标准、当前状态和风险。
- 每个成员允许使用 `Workspaces/<agent-name>/` 作为个人工作空间；有意义的草稿、报告和任务记录应提交进 Git，临时缓存不提交。
- 每个成员完成自己的任务后，应自检并说明修改范围、验证方式和残留风险，再按 Git 规则提交并推送。

## 分配原则

- 制作方向、核心玩法、体验目标、版本范围和任务取舍：优先交给 `/game-producer-lead-designer`。
- 进度、排期、任务拆分、验收标准、测试用例、缺陷复现和发布清单：优先交给 `/pm-qa-coordinator`。
- 技术方案评审、架构决策、代码质量、任务拆分和关键疑难问题：优先交给 `/lead-game-programmer`。
- 核心玩法实现、交互逻辑、状态机、碰撞判定、反馈表现和快速原型：优先交给 `/senior-gameplay-engineer`。
- 工程结构、工具链、性能、资源加载、存档、配置和构建流程：优先交给 `/senior-systems-engineer`。
- 规则落地、数值表、关卡参数、成长曲线、掉落奖励和平衡性：优先交给 `/systems-numerics-designer`。
- UI 布局、HUD、按钮、弹窗、图标风格、视觉层级和可读性优化：优先交给 `/ui-artist`。
- 整体美术风格、角色场景资产、视觉规范、资源清单和美术一致性检查：优先交给 `/art-director-asset-artist`。
- 软件安装、环境配置、工具交付、系统维护和危险操作识别：优先交给 `/it-engineer`。
- 审核任务应优先选择更资深、职责更匹配且不同于执行者的成员。
- 版本规划、多步骤任务、跨职能任务和发布任务必须让 `/pm-qa-coordinator` 介入。
- 除涉及本机环境的危险操作外，其余项目事项默认由 `/game-producer-lead-designer` 拥有最终决定权。
- 核心玩法、系统规则、数值框架、存档和关键数据结构属于项目高风险任务，应走专项审核、PM 汇总审核和制作人最终决定。
- 环境配置、权限提升、删除覆盖类操作属于本机环境危险操作，必须先说明风险并获得用户确认。

## Git 与发布约束

- 当前仓库使用 Git。成员完成任务并通过必要自检后，应及时 commit 并 push；本次任务如用户明确要求不提交，则只保留工作区修改。
- commit 前必须检查状态和变更范围，避免提交密钥、凭据、临时缓存、无关文件或危险产物。
- 提交信息使用中文简洁描述，例如 `完成舰船移动原型`。
- 禁止无确认执行 force push、`git reset --hard`、`git clean`、checkout 覆盖、改写已 push 历史、删除发布 tag/分支/目录或已归档发布产物。
- 发布版本由制作人最终批准，版本边界由制作人提出方向、PM/QA 根据任务量决定。
- 发布版本必须创建 Git tag，并将 Windows 可运行发布产物归档到 `Releases/<version>/` 后直接提交进 Git。

## 常见分工示例

- 继续推进项目或完成阶段后没有新指令：`/game-producer-lead-designer` 先检查 `42.md` 并决定下一阶段方向，`/pm-qa-coordinator` 再拆分任务并维护看板。
- 评估“核心循环是否好玩”：`/game-producer-lead-designer` 主评，必要时 `/systems-numerics-designer` 补规则和可调参数。
- 拆分一个版本计划：`/pm-qa-coordinator` 主拆，必要时让各职能成员补充工作量、风险和验收方式。
- 设计成长曲线、奖励掉落或战斗参数：`/systems-numerics-designer` 主导，制作人确认体验目标。
- 快速验证新玩法：`/senior-gameplay-engineer` 执行，`/game-producer-lead-designer` 或 `/systems-numerics-designer` 审核玩法目标和规则边界。
- 实现正式玩法或修复交互问题：`/senior-gameplay-engineer` 执行，复杂技术由 `/lead-game-programmer` 评审或攻关。
- 改进资源加载、存档、配置、构建或性能问题：`/senior-systems-engineer` 执行，影响公共架构时由 `/lead-game-programmer` 把关。
- 制作 UI mockup、HUD 或图标方案：`/ui-artist` 执行，`/art-director-asset-artist` 审核风格一致性。
- 生成角色、道具、场景资产方案或美术规范：`/art-director-asset-artist` 执行，并与制作人确认题材和体验识别。
- 安装 Unity、Blender、SDK 或外部工具：`/it-engineer` 先检查环境和风险，再在确认边界内执行安装配置。
